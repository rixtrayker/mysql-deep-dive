## 15.7 · Binlog / redo loss & PITR gaps ★

**Mental model.** The recovery *mechanisms themselves* can fail — and then you *can't* recover. **PITR** (M13/13.3) needs the **binlog continuously retained and durable** (`sync_binlog=1`); if the binlog has a **gap** (not retained long enough — expired before the disaster window; or not durable — `sync_binlog=0` lost recent events on a crash) or is **corrupt**, you *can't* replay it → the changes in that gap are **unrecoverable** (you can only restore to before the gap, losing everything since). Similarly, a corrupt **redo log** breaks crash recovery (→ force_recovery, 15.6). The catastrophe: you *think* you can rewind to any moment (PITR), but a binlog gap means *that specific window is gone forever*. This is *the* PITR gap — the recovery you counted on, absent when you need it.

**How it happens (root cause).** **(1) Retention too short:** `binlog_expire_logs_seconds` purges binlogs after N days; if the disaster's recovery window (last backup → disaster) is *longer* than retention, the needed binlogs are *already deleted* → a gap. **(2) Not durable:** `sync_binlog=0` (or N) means recent binlog events sit in the OS cache; a crash loses them → a gap at the crash point (the transactions committed but their binlog events are gone → can't PITR them, and replicas/CDC miss them too). **(3) Corruption:** a torn/corrupt binlog file (15.5) breaks replay. **(4) No base backup in range:** if the last *full backup* is older than the retained binlog, there's no base to replay *onto*. Any of these = a window you can't reconstruct.

**Why it's catastrophic.** PITR is the platform's *fine-grained* recovery (rewind to just before any disaster, M13/13.3) — a gap *defeats* it, forcing a coarse restore (to before the gap, *losing* all the good data since) *or* an unrecoverable loss (the gap's changes are simply gone). For money, this means a bad `DELETE` you *can't* surgically undo (you lose the whole window, not just the bad change), or confirmed transfers in the gap *permanently lost*. The catastrophe is *discovering the gap during recovery* — the recovery tool you relied on has a hole exactly where you need it.

**The fix (recovery).** If there's a gap, you're limited: **restore to before the gap** (lose the gap's changes — coarse) + **reconcile** (M12/12.14 — find what's missing vs external records) + **re-drive** the lost transactions idempotently (M12/12.9 — re-process from the external source-of-truth, if available). If the binlog is *corrupt but partially readable*, salvage what you can (`mysqlbinlog` up to the corruption). There's no clean recovery for a true gap — which is *why* prevention is everything.

**Prevention.** **`sync_binlog=1`** (durable binlog — no crash gap, the money setting, M09/14.9) + **binlog retention ≥ the recovery window** (`binlog_expire_logs_seconds` long enough to cover last-backup → now, comfortably) + **frequent base backups** (so the binlog-replay distance is short, M13/13.3) + **archive binlogs off-host** (so a host loss doesn't lose them) + **tested PITR drills** (M13/13.5 — *verify* you can actually replay the binlog to a target; a drill *finds the gap* before a real disaster). The tested restore drill (M13/13.5) is the key — it *proves* PITR works (or reveals the gap).

**Generics / MySQL reality.** Universally, point-in-time recovery depends on a *complete, durable, retained* change log — a gap anywhere defeats it (the same for Postgres WAL archiving). In MySQL: `sync_binlog=1` (durability) + retention + off-host archive + GTID (precise replay, M10/10.7) + tested drills. The deep link: the binlog's *durability* (M09) and *retention* (M13) directly gate *recoverability*. The ★ SVG (Pass C) draws the PITR gap (missing/corrupt binlog → unrecoverable window).

**💰 Money verdict.** **Money UNRECOVERABLE in the gap** (the changes are gone; coarse restore loses the window) — a *recovery* catastrophe (the tool you counted on has a hole). Prevention: `sync_binlog=1` + retention + tested PITR drills. Detection: a tested drill *reveals* the gap before disaster; reconciliation finds what's lost. The lesson: **a PITR gap is the recovery you didn't have — make the binlog durable + retained, and *test* the replay.**

## 15.8 · Recovering a dropped table / database

**Mental model.** A `DROP TABLE ledger_entry` or `DROP DATABASE payments` — a fat-finger, a bad migration, a wrong-environment deploy — instantly removes the table/database. Recovery is a *specific case* of PITR (15.7/M13/13.3): **restore a backup + replay the binlog up to *just before* the `DROP`** (excluding the `DROP` statement) → the table is back as of one second before. GTIDs (M10/10.7) make the "just before" precise. The catch: it requires the *prerequisites* (a base backup + retained durable binlog, 15.7) — without them, a `DROP` is unrecoverable. It's the most *common* "I lost data" incident, and the most *cleanly recoverable* (if you have backups + binlog).

**How it happens (root cause).** Human/deploy error: a `DROP` run against the wrong environment (prod instead of staging), a migration script with a `DROP` that shouldn't have run, a `DROP DATABASE` meant for a test DB. Or a bad `TRUNCATE`. The `DROP` is *committed* and *replicated* (it's a real DDL) — so it's gone from the primary *and* all replicas (failover doesn't help — the `DROP` replicated everywhere). Only *backup + PITR* recovers it.

**Why it's catastrophic.** Losing the ledger table is total (every transfer's record gone) — and because the `DROP` *replicated* (M10), the replicas are *also* dropped (so HA/failover is useless here — the standard "node loss" recovery doesn't apply to a *logical* error that propagated). It's catastrophic *unless* you have the PITR prerequisites; with them, it's cleanly recoverable.

**The fix (recovery).** **PITR** (M13/13.3): **(1)** restore the most recent base backup (to before the `DROP`); **(2)** replay the binlog *up to just before* the `DROP` statement (`--stop-position`/`--stop-datetime` at the `DROP`, or `--exclude-gtids` for the `DROP`'s GTID, M10/10.7) — recovering the table as of one second before; **(3)** reconcile (M12/12.14 — verify the recovered ledger). If the `DROP` is *very* recent and the table was large, *extracting from a physical backup* (the table's `.ibd` file, if using file-per-table) is an alternative. With backups + binlog, this is a clean recovery.

**Prevention.** The PITR prerequisites (15.7 — base backup + durable retained binlog) + **operational guards**: restrict `DROP`/`TRUNCATE` privileges (least privilege, M13/13.14 — most accounts can't `DROP`), require review for destructive migrations, use distinct prod/staging credentials (so a wrong-environment `DROP` fails on privilege), and `--i-am-a-dummy`/safe-updates modes. Prevention is *both* recoverability (backups) *and* not-running-the-DROP (privilege + process).

**Generics / MySQL reality.** Recovering a dropped object via backup + change-log-replay-excluding-the-drop is universal (the same in any database with PITR). In MySQL: PITR (`mysqlbinlog` + `--stop-position`/GTID exclusion) + least-privilege on DDL. The Pass C Mermaid draws restore + binlog-replay-excluding-the-DROP.

**💰 Money verdict.** **Money RECOVERABLE** (cleanly, via PITR to just before the `DROP`) — *if* you have the prerequisites (backup + durable retained binlog, 15.7); **UNRECOVERABLE** without them. Detection: obvious (the table is gone). The lesson: **a dropped table is cleanly PITR-recoverable with backups + binlog — which is why the PITR prerequisites (15.7) and DDL least-privilege (M13/13.14) matter.**

## 15.9 · App-level loss the DB faithfully persists (lost updates, RMW races) ★

**Mental model.** The subtlest class: **the database does *exactly* what it's told — correctly, durably — but the *application logic* loses money**. The classic is the **lost update** (read-modify-write race): two concurrent transfers both *read* balance=$100, each computes "new = 100 − 80 = 20", each *writes* 20 → one overwrites the other → the account shows $20 after *two* $80 debits (should be -$60, i.e., the second rejected). The DB persisted both writes *faithfully*; the *app* lost an update by reading-then-writing without protection. Also in this class: the **dual-write** problem (M12/12.10 — DB updated but the event lost) and general RMW races. The catastrophe: **the DB is correct, the application is wrong** — so it's invisible to DB-level integrity checks; only *application-level* correctness (locking, atomicity, idempotency) and *reconciliation* catch it.

**How it happens (root cause).** **Lost update (RMW race):** the app does `SELECT balance` → (compute in app) → `UPDATE balance = computed` as *separate* statements, with *no* protection — so two concurrent transactions interleave: both read the *same* old balance, both compute from it, both write — the second write *overwrites* the first (the first update is *lost*). The DB did nothing wrong (it serialized the writes); the *app* lost the update by not making read-then-write atomic or locked. **Dual-write (M12/12.10):** the app writes the DB, then a separate write (queue/cache) fails after the DB commit → inconsistency. The root cause is *application logic that isn't concurrency-safe or atomic* — the DB faithfully persists what the (flawed) app tells it.

**Why it's catastrophic.** It's *silent to the database* — no error, no corruption, the DB is internally consistent (balance = some value it was told to store). The *wrongness* is *logical* (the balance is wrong relative to the transfers that should have applied) — invisible to `CHECK TABLE`, page checksums, crash recovery (all DB-level). For money, a lost update *is* lost/duplicated money (a debit that didn't take, a balance that's wrong) — and it's caught *only* by application correctness (preventing the race) or **reconciliation** (M12/12.14 — balance ≠ Σ entries reveals it). It's the catastrophe that *the database can't catch for you*.

**The fix (recovery).** **Detect:** reconciliation (M12/12.14 — re-derive balance from the immutable ledger entries; a lost update shows as balance ≠ Σ entries). **Recover:** correct the balance from the authoritative ledger (the immutable entries are the truth, M01/1.17 — re-derive the correct balance, with an audit trail), and re-apply any lost transfer (idempotently, M12/12.9). The immutable ledger (append-only entries) is what *enables* recovery — the *derived* balance can be wrong, but the *entries* are the source of truth.

**Prevention.** Make the app concurrency-safe and atomic: **atomic conditional `UPDATE`** (`UPDATE account SET balance_minor = balance_minor - :amt WHERE account_id=:a AND balance_minor >= :amt` — no read-then-write gap, the DB does the read+check+write atomically, M07/7.16) OR **`SELECT … FOR UPDATE`** (lock the row, serialize the transfers, M08) OR **optimistic concurrency** (a version column — write only if the version matches). Plus **idempotency** (M12/12.9 — no double-apply), **outbox** (M12/12.11 — no dual-write), and **reconciliation** (M12/12.14 — the backstop). The immutable ledger (M01/1.17) makes the truth re-derivable.

**Generics / MySQL reality.** The lost-update/RMW race is *the* classic concurrency bug, universal to all databases — prevented by atomic operations, locking, or optimistic concurrency (the same everywhere). In MySQL: atomic conditional `UPDATE`, `FOR UPDATE` (M08), version columns; the immutable-ledger pattern (M01/1.17) for re-derivable truth; reconciliation (M12/12.14) for detection. The deep lesson: **the DB faithfully persists what the app tells it — application correctness (atomicity, concurrency-safety, idempotency) is the app's job, and reconciliation is the backstop.** The ★ SVG (Pass C) draws the lost-update race.

**💰 Money verdict.** **Money LOST/DUPLICATED by the app** (a lost update = wrong balance; the DB is correct, the app is wrong) — the *silent application* catastrophe, invisible to DB integrity checks. Prevention: atomic/locked/optimistic writes + idempotency + outbox. Detection/recovery: **reconciliation against the immutable ledger** (M12/12.14, M01/1.17). The lesson: **the DB can't save you from app-level races — use atomic operations, and reconcile against the immutable source of truth.**

## 15.10 · The backup that won't restore ★

**Mental model.** The catastrophe M13/13.5 *warned* of, *realized*: you go to restore — corruption, a bad deploy, hardware loss — and the backup **doesn't work**: it's *corrupt* (silent bit-rot, a backup-process bug), *incomplete* (a table silently excluded), *un-decryptable* (the encryption key was lost/rotated, M13/13.14), *missing the binlog* (no PITR, 15.7), or *un-restorable* (version mismatch). The backup "succeeded" every night for a year — but "backup succeeded" *never meant* "restore works." Now the data is gone *and* the recovery is gone — *the* operational money catastrophe. The *only* prevention is **tested restores** (M13/13.5 — automated drills that *prove* recoverability); here we see it *unprevented*.

**How it happens (root cause).** The backup *job* reports success based on "did the backup *run*" — *not* "can it *restore*." So *silent* failures accumulate undetected: the backup file slowly bit-rots (15.5); a config change silently excludes a table; the encryption key is rotated and the old one discarded (so old backups can't decrypt, M13/13.14); the binlog isn't backed up alongside (so no PITR, 15.7); a MySQL upgrade makes old physical backups incompatible. *None* of these fail the *backup* — they fail the *restore*, discovered only when you *try to restore* (during the disaster). The root cause is **never testing the restore** — trusting "backup succeeded."

**Why it's catastrophic.** It's the *worst-timed* failure: you discover it *mid-disaster* (when the data is *already* lost and you're counting on the backup), and now *both* are gone. The whole point of backups is to be the *last line of defense* — an unrestorable backup means *there is no defense*. For money, an unrestorable ledger backup after a disaster = *permanent* loss of the money. And it's *avoidable* — the failure was *latent for a year*, catchable by a single restore drill.

**The fix (recovery).** *Mid-disaster* with a dead backup: try *other* backups (an older one? a replica? a logical dump alongside the physical?), salvage what you can (`force_recovery` on the corrupt source, 15.6; partial restore), and **reconcile against external records** (M12/12.14 — reconstruct as much as possible from the payment processor's records). It's a *damage-limitation* scramble — there's no clean recovery, which is the point. The *real* fix is *having tested* (so you never get here).

**Prevention.** **Tested restores** (M13/13.5 — *the* prevention): **automated, frequent restore drills** that restore the backup to a scratch instance, apply PITR, and **verify** (checksums + **reconciliation** — the recovered ledger is correct). This catches *every* silent failure (corruption → restore fails; incompleteness → reconciliation mismatch; key loss → decryption fails; missing binlog → PITR fails) *before* the disaster. Plus: **multiple backup types** (physical + logical, 13.2 — redundancy), **off-site copies**, **key management** (M13/13.14 — don't lose the encryption key), and **monitoring backup *and* restore-drill success**. "You don't have backups, you have restores" (M13/13.5).

**Generics / MySQL reality.** Universal, hard-won ops truth: an untested backup is not a backup; only a *tested restore* proves recoverability (SRE gospel, every system). In MySQL: tested restore drills (XtraBackup `--prepare`/`--copy-back`, logical reload, `mysqlbinlog` PITR) + reconciliation + key management (M13/13.5/13.14). This is M13/13.5's catastrophe *realized* — the cautionary tale. The ★ SVG (Pass C) draws the untested-backup catastrophe (the failure realized).

**💰 Money verdict.** **Money PERMANENTLY LOST** (the data is gone *and* the backup can't restore it) — the *operational* money catastrophe, and *entirely preventable* by tested restores. Prevention: automated restore drills + reconciliation (M13/13.5). The lesson: **"backup succeeded" ≠ "restore works" — test the restore, or discover during the disaster that you have neither.** This is *the* M13/13.5 cautionary tale.

## 15.11 · Disk-full / out-of-space mid-write

**Mental model.** The disk fills up — from **binlog growth** (un-purged, especially if a replica is down and holding them), **undo bloat** (a long transaction, 15.13), **data growth**, or **a runaway query's temp files** — and when MySQL tries to write (a transfer, a binlog event, a sort) and *can't*, the write **fails**, and the server may **crash, hang, or stop replication** → an outage (and possibly corruption if a write is half-done). It's a *loud* (it errors) but *rare-ish* failure that's *entirely preventable* by the disk-space early-warning signal (M13/13.11). The catastrophe is letting a *predictable, gradual* condition (the disk filling) become an *outage* by not watching it.

**How it happens (root cause).** Disk usage grows over time (data, binlogs, undo, temp) and someone isn't watching. Common triggers: a **down replica** (the primary retains binlogs until the replica catches up — if the replica is down for days, binlogs pile up, M10); **undo/HLL bloat** (a forgotten long transaction prevents undo purge, 15.13/M08); a **runaway query** writing huge temp files (a bad `ORDER BY`/`GROUP BY` without an index, M06); or just **organic growth** past capacity. When the disk hits 100%, the *next write fails* → MySQL errors (writes rejected), may crash, or replication stops (can't write the relay log). A half-completed write at the failure point risks corruption (mitigated by the doublewrite buffer, M09).

**Why it's catastrophic.** A full disk *stops writes* — for a payments platform, *transfers stop* (an outage). If MySQL *crashes* on the full disk, you get crash recovery (M09) — fine *if* durability config is right (15.2), but the disk is *still full* on restart (a crash loop). If *replication* stops, HA/replicas fall behind (M10). And it's *predictable* — the disk didn't fill suddenly; it crept up, *un-watched*. The catastrophe is the *failure to watch* a gradual, fatal condition.

**The fix (recovery).** **Free space *immediately*:** purge old binlogs (`PURGE BINARY LOGS` — *after* ensuring replicas/CDC consumed them, M10/M12), drop temp files, add disk (cloud: expand the volume). Then **restart** if crashed (crash recovery, M09 — durable if config is right). **Fix the root cause:** restart the down replica (so binlogs purge), kill the long transaction (15.13 — so undo purges), fix the runaway query (M06). Reconcile (M12/12.14) if any write was lost in the failure.

**Prevention.** **The disk-space early-warning signal** (M13/13.11 — alert at 80%, well before full → add space / purge proactively) + **binlog retention limits + monitoring replica health** (so binlogs don't pile up from a down replica, M10) + **HLL/long-transaction monitoring** (15.13/M13 — so undo doesn't bloat) + **query limits** (so a runaway doesn't fill temp). The disk-space alert (M13/13.11) is the single most important prevention — a *predictable* failure should *never* be a surprise outage.

**Generics / MySQL reality.** Disk-full is a universal operational failure (every stateful system) — prevented by capacity monitoring with early warning. In MySQL: binlog purging (M10), undo/HLL management (M08/15.13), temp-file limits, and the disk-space early-warning signal (M13/13.11). The Pass C Mermaid draws disk-full → write failure → recover + prevention.

**💰 Money verdict.** **Money movement STOPS (outage), possible corruption** at the failure point — a *predictable, preventable* operational catastrophe. Prevention: the disk-space early-warning signal (M13/13.11). Detection: the alert (if watched) or the outage (if not). The lesson: **a full disk is a predictable failure — watch disk space (M13/13.11) and it's never a surprise; ignore it and it's an outage.**
