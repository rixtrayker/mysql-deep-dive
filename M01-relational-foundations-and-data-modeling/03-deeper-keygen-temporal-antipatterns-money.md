## 1.15 · Surrogate-key generation strategies

**Mental model.** Once you've decided to use a surrogate key (1.4), you still must choose **how to mint it** — and the choice is not cosmetic. The fault line is **monotonic vs random**: monotonic IDs (auto-increment, time-ordered) give you tight index locality but a single hot insertion point and guessable IDs; random IDs (UUIDv4) spread inserts and resist guessing but shred your index's physical order. Most modern answers (ULID, UUIDv7, Snowflake) are attempts to get *time-ordered-but-distributed* — the best of both.

**How it actually works.** The main families:
- **Auto-increment (BIGINT)** — the database hands out 1, 2, 3, …. Compact (8 bytes), perfectly monotonic, ideal clustered-index locality (every insert appends to the right edge). Downsides: a single sequence is a coordination point (hard across shards), inserts hammer one "hot" page (M09 — but mitigated by InnoDB's append behavior), and IDs are guessable/enumerable (a security and business-intelligence leak — competitors can count your transactions).
- **UUIDv4 (random 128-bit)** — globally unique with no coordination, unguessable. But **random order is poison for a clustered index**: each insert lands at a random point in the key space, causing page splits, fragmentation, and a working set that won't stay in the buffer pool (M05/M09). Stored as a 36-char string it's catastrophic; as 16 bytes binary it's merely bad.
- **ULID / UUIDv7** — 128-bit but **time-prefixed**: the high bits are a timestamp, the low bits random. This restores *roughly* monotonic insert order (good locality) while keeping global uniqueness and no central coordinator. UUIDv7 is the standardized version of this idea; ULID is the popular pre-standard one. This is the current default recommendation when you need distributed/unguessable keys.
- **Snowflake-style IDs** — 64-bit composed of (timestamp | machine/shard id | per-ms sequence). Time-ordered, compact (fits in BIGINT), coordination-free across nodes, and encodes the origin shard. Used by Twitter/Discord/etc. for distributed monotonic-ish IDs (revisited for distributed ID generation in M11).

**Why it exists / what it solves.** The generation strategy resolves a three-way tension between **uniqueness without coordination**, **index/storage locality**, and **non-guessability** — plus the distributed need to mint IDs on many nodes without a shared sequence. You can't maximize all of them with one scheme, so the strategies are different points on that surface.

**Tradeoffs & alternatives.** Auto-increment: smallest and fastest locally, worst for sharding and security. UUIDv4: best distribution/unguessability, worst locality. ULID/UUIDv7/Snowflake: the pragmatic middle — near-monotonic locality + distributed minting, at the cost of larger keys than BIGINT (128 vs 64 bits, except Snowflake) and slightly leaky timing information (the embedded timestamp reveals creation time). A frequent hybrid: BIGINT auto-increment as the internal PK for locality, plus a separate unguessable public ID (ULID/UUID) exposed in APIs.

**Generics / first-principles.** "How you allocate identifiers determines your data's physical clustering and your system's coordination needs." This is a distributed-systems decision wearing a modeling costume — the same reasoning governs partition keys, sequence servers, and content-addressing. Time-ordered IDs are a recurring trick to keep "globally unique" and "locally sequential" simultaneously true.

**MySQL-specific reality.** This matters *more* in MySQL than most engines because of the **clustered PK** (1.3, M05): the PK dictates physical row order, so a random PK (UUIDv4) directly causes B-tree page splits, fragmentation, and buffer-pool thrash — a well-documented MySQL performance killer. Concrete MySQL guidance: prefer **BIGINT auto-increment** for internal PKs when a single sequence is acceptable; if you need distributed/unguessable keys, use **time-ordered IDs (UUIDv7/ULID) stored as BINARY(16)**, never as CHAR(36) (the string form triples storage and wrecks comparisons). MySQL's `auto_increment` also has its own locking nuances (the auto-inc lock modes) covered in M08. The physical consequences of this choice are explored in depth in M05 (index design) and M09 (page splits/fragmentation).

## 1.16 · Temporal & bitemporal modeling

**Mental model.** Most schemas store only "what is true *now*," overwriting the past. Real systems — especially financial ones — need to answer "what did we believe was true *as of* some date?" That needs **time as a modeled dimension**. And there are actually *two* independent time axes: **valid time** (when a fact was true in the real world) and **transaction time** (when our database recorded/knew it). Tracking both is **bitemporal** modeling, and it's the difference between "we can't reconstruct what we knew last Tuesday" and "we can audit exactly that."

**How it actually works.**
- **Valid time** — the period during which a fact holds in reality. An FX rate "USD/EUR = 1.10" was valid *from Monday to Wednesday*. Modeled with `valid_from` / `valid_to` columns (a row per version, not an overwrite).
- **Transaction time** — the period during which the database *recorded* that fact as current. We *entered* that rate on Monday but *corrected* it on Friday after finding an error. Modeled with `recorded_from` / `recorded_to` (or an append-only audit log).
- **Bitemporal** = both pairs together. A single real-world fact can have multiple bitemporal rows: the originally-recorded (wrong) value and the later-recorded correction, each with its own valid-time and transaction-time ranges. Queries then ask two-dimensional questions: "what did we *think on Friday* the rate *was on Tuesday*?"

The mechanics are append-only versioned rows: you never UPDATE-in-place a temporal fact; you close the old version's time range and insert a new version.

**Why it exists / what it solves.** Finance, insurance, and compliance constantly need *as-of* and *as-believed* reconstructions: restating a report with the data known at the time, auditing a correction, computing interest on the balance *as it was valid then* even if a backdated correction arrived later. Overwriting destroys the ability to answer these and is often *illegal* for regulated records. Bitemporal modeling makes "what was true" and "what we knew" independently reconstructable.

**Tradeoffs & alternatives.** Bitemporal modeling is powerful and *expensive in complexity*: every read must filter on time ranges, every write must close-and-insert, uniqueness constraints become "unique *within an overlapping time range*" (hard to express), and queries get genuinely tricky. Simpler alternatives, in increasing power: (1) overwrite + a separate audit log (cheap, but reconstructing point-in-time state is manual); (2) **uni-temporal** (valid-time *or* transaction-time only) — covers many needs at half the complexity; (3) full bitemporal only where the regulation/business truly demands it. Don't make everything bitemporal — apply it to the facts that need restatement (rates, prices, balances, entitlements).

**Generics / first-principles.** "There is more than one kind of time, and overwriting conflates them." The distinction between *when something was true* and *when we learned it* is fundamental to auditing, debugging, and event-sourced systems everywhere — it's the same insight behind event timestamps vs ingestion timestamps in data pipelines, and "effective date" vs "knowledge date" in accounting.

**MySQL-specific reality.** MySQL has **no built-in bitemporal support** (unlike SQL:2011 system-versioned tables that MariaDB and some other engines offer natively). So in MySQL you model temporality **by hand**: explicit `valid_from`/`valid_to` (+ `recorded_from`/`recorded_to` for bitemporal) columns, append-on-change instead of update, and application or trigger logic to maintain the ranges. Practical MySQL notes: use DATETIME(6) for sub-second precision and be deliberate about time zones (M03); "no two overlapping valid periods for the same key" usually can't be a simple UNIQUE constraint and needs application enforcement or careful generated-column tricks; and the append-only growth interacts with partitioning/archival (M11) and with the event-sourcing/audit patterns in M16.

## 1.17 · Modeling history: append-only vs mutable state

**Mental model.** Two fundamentally different ways to represent something that changes over time, like a wallet balance. **Mutable state:** store the *current* value and overwrite it on every change (`account.balance = 150`). **Append-only:** store the *immutable stream of events* that caused the changes (`+100`, `+80`, `-30`) and derive the current value by **folding** over them. The first answers "what is it now?" instantly but forgets how it got there; the second remembers everything but must compute the present. This choice *is* your audit story.

**How it actually works.**
- **Mutable / current-state:** a column you UPDATE in place. Reads are O(1), there's exactly one row to look at, but each update destroys the prior value — history is gone unless separately logged. Concurrency requires care (two writers updating a balance race — lost-update problem, M07/M08).
- **Append-only / event log:** an immutable table where every change is a new row (a `ledger_entry` with a signed `amount`), never updated or deleted. Current state = `SUM(amount)` over the entries (a fold/reduction). You get a perfect, ordered, auditable history for free; you pay by computing aggregates (mitigated by snapshots/summary balances — 1.14).
- **The standard hybrid:** keep the append-only log as the **system of record** *and* maintain a denormalized current `balance` as a **derived cache** that you update in the same transaction as the entry. Reads hit the fast cached balance; integrity and audit live in the immutable log; the log can always rebuild the cache (and reconciliation, M16, checks they agree).

**Why it exists / what it solves.** Append-only modeling solves auditability, debuggability, and *correctness under correction*: you can always answer "how did we get this balance?", replay to reconstruct state at any point, and never silently lose a fact. For money this is close to mandatory — the immutable ledger is the legal and operational truth. Mutable state solves "I need the current value cheaply and don't need its history."

**Tradeoffs & alternatives.** Append-only: bulletproof audit + replay, but the table grows forever (archival/partitioning, M11), and naive current-state reads get slower as history accumulates (needs snapshots). Mutable: simple and fast, but lossy and harder to audit, and the in-place update is where lost-update and concurrency bugs concentrate. **Event sourcing** (M16) is the maximalist form of append-only (the event log is the *only* source of truth, all state is a projection); CRUD-with-audit-log is a lighter middle ground. The general rule for fintech: **the money record is append-only and immutable; everything derived from it is a rebuildable cache.**

**Generics / first-principles.** "Store the events, derive the state — or store the state and lose the events." This is the deepest recurring choice in data modeling, reappearing as event sourcing vs CRUD, Git's commit log vs a mutable working tree, write-ahead logs vs in-place updates (which is *literally* how InnoDB's redo log works — M09), and Kafka's log-as-source-of-truth. Folding an immutable log into current state is a universal pattern.

**MySQL-specific reality.** MySQL implements this at multiple levels, which is a nice teaching mirror: *you* choose append-only vs mutable for your domain tables, but **InnoDB itself is internally append-then-apply** — the redo log is an append-only WAL and MVCC keeps old row versions in undo logs (M08/M09), i.e., the engine logs events and derives current pages. For your schema: an append-only ledger is just a table you only ever INSERT into (enforce with permissions/triggers and never writing UPDATE/DELETE against it), plus a derived `balance` updated transactionally alongside each insert. The binlog (M10/M13) is *another* append-only event stream MySQL already maintains, and CDC off it (M12/M16) is how the event log reaches the rest of the system. The throughline you'll see again and again: **logs are primary, state is derived.**

## 1.18 · Data-modeling anti-patterns ★

**Mental model.** A catalog of schema shapes that **look flexible or convenient but rot over time** — they trade a little short-term ease for permanent pain in querying, integrity, and performance. Recognizing them by name is half the battle, because each has a known, better replacement.

**How it actually works.** The repeat offenders:
- **EAV (Entity-Attribute-Value).** Instead of columns, a generic `(entity_id, attribute_name, value)` table to allow "any field." Sounds infinitely flexible; in reality it destroys typing (everything's a string), makes constraints and FKs nearly impossible, turns one logical row into many physical rows, and forces monstrous self-joins or pivots to reconstruct an entity — wrecking query plans (M06). **Fix:** real columns; if you genuinely need dynamic attributes, a JSON column (M03) with discipline, or a proper extension table per attribute set.
- **The god-table / "wide table of everything."** One table with 80+ mostly-nullable columns trying to cover several entities or many optional concepts (a `transactions` table holding card fields, bank fields, crypto fields, dispute fields, all nullable). **Fix:** split by entity/subtype; use proper subtype modeling (separate tables per type, or a shared table + type-specific tables). Reduces NULLs (1.7), shrinks the hot row, clarifies meaning.
- **Comma-separated values in a column.** Storing `"123,456,789"` as a list of account IDs in one field. Violates 1NF (M02), makes joins/filters impossible (`WHERE account_id = 456` can't use an index — leading-wildcard `LIKE`), and has no referential integrity. **Fix:** a junction table (1.12).
- **Polymorphic / generic foreign keys.** A `(target_type, target_id)` pair pointing at "whatever table `target_type` names." The database **cannot enforce a real FK** (it doesn't know which table), so referential integrity is gone and you get dangling references silently. **Fix:** separate nullable FK columns per target type (each a real FK), or an exclusive-arc pattern, or a supertype table the others reference.
- **Misusing NULL as a flag / overloading columns** — a column meaning different things based on another column; using NULL to mean several different "specials." **Fix:** explicit status columns, separate columns, CHECK constraints.

**Why it exists / what it solves (to recognize the temptation).** Each anti-pattern is a *premature optimization for flexibility or fewer tables*: EAV and polymorphic FKs promise "add anything without migrations"; god-tables promise "one place for related stuff"; CSV columns promise "no extra join." The trap is that they move complexity from the schema (where the engine helps you) into application code and ad-hoc queries (where it festers).

**Tradeoffs & alternatives.** There are *narrow* legitimate uses — EAV-ish patterns for truly open-ended user-defined fields (better served by JSON today), a wide table when columns genuinely belong to one entity and are co-accessed. The skill is distinguishing "I have a real need for dynamic structure" (rare, handle with JSON + care) from "I'm avoiding the work of modeling properly" (common, costs you later). The replacements above are almost always worth the extra tables/joins.

**Generics / first-principles.** "Flexibility you get by abandoning structure is flexibility the database can no longer help you keep correct." Every one of these anti-patterns works by hiding structure from the engine — so typing, constraints, FKs, and the optimizer can't assist. The agnostic lesson: prefer schemas the system can *reason about*; reach for generic/dynamic shapes only when the requirement is genuinely dynamic, and even then contain the blast radius.

**MySQL-specific reality.** MySQL makes some of these *especially* costly: EAV/polymorphic designs defeat the optimizer (M06) and can't use FKs (no integrity); CSV-in-a-column can't use indexes (M05), forcing full scans; god-tables of nullable columns bloat the clustered-index row and waste buffer pool (M09). MySQL's **JSON type + generated columns** (M03) is the sanctioned modern escape hatch for *genuinely* dynamic attributes — you store flexible data in JSON but can index specific paths via generated columns, getting some structure back. The fintech stakes are high: a polymorphic FK on a `transactions` table means the database can't guarantee every transaction points at a real account — exactly the dangling-reference, money-can-go-missing failure the *money-never-lies* thread warns against (1.5).

## 1.19 · Fintech capstone — modeling a money system ★

**Mental model.** Almost every payments/wallet/banking system, however it starts, **converges on the same core shapes**: *parties* (who), *accounts* (containers of value), *transactions* (a transfer event), and an **immutable ledger of entries** (the per-account debits/credits that *are* the money record). If you internalize this canonical model, most fintech data design becomes "specialize this skeleton," and it sets up the entire M16 capstone.

**How it actually works (the canonical skeleton).**
- **`party` / `customer`** — the legal/real-world actor (a person or business). Surrogate PK (1.4/1.15); KYC/PII often split into a 1:1 child table (1.11) for isolation.
- **`account`** — a container of value owned by parties. Surrogate PK; `currency` (a domain — 1.8); status (active/closed, never hard-deleted — 1.6); related to parties **1:N or M:N** (joint accounts → junction `account_holder`, 1.12).
- **`transaction`** — one business-level money movement event (a transfer, a payment). Carries metadata (when, type, idempotency key — M16), and is the **strong entity** that owns its entries.
- **`ledger_entry`** — the heart. A **weak entity** (1.13) belonging to a transaction, **append-only and immutable** (1.17), recording a signed amount against one account. **Double-entry** (developed fully in M16): every transaction produces entries that **sum to zero across accounts** — every debit has an equal credit — so money is conserved by construction. The invariant `SUM(amount) over a transaction = 0` is the structural guarantee that money is neither created nor destroyed.
- **Derived `balance`** — each account's current balance is the fold over its entries (1.17), kept as a transactionally-maintained cache and continuously reconcilable against the log (M16).

This single ER model exercises nearly every M01 concept: keys (1.3), natural-vs-surrogate (1.4), FKs + RESTRICT-not-CASCADE (1.5/1.6), constraints/domains (1.8), cardinality + junction for joint accounts (1.11/1.12), weak entity for entries (1.13), design-for-queries for balance/statement access (1.14), surrogate-key generation for distributed IDs (1.15), temporal needs for FX/rates (1.16), and append-only history (1.17) — while avoiding the anti-patterns (1.18).

**Why it exists / what it solves.** This shape exists because **money demands conservation, auditability, and integrity simultaneously**, and double-entry bookkeeping (600 years old, predating computers) is the proven structure that delivers all three. Encoding it relationally — immutable entries, balanced transactions, derived balances — gives you a system where the books always balance, every movement is attributable, and the full history is reconstructable. It's the data-model embodiment of the *money-never-lies* thread.

**Tradeoffs & alternatives.** The canonical ledger is more tables and more discipline than a naive `account.balance` column you just UPDATE — and that naive shape is the single most common (and most dangerous) fintech modeling mistake, because it loses history, races on concurrent updates (M07/M08), and can silently lose or invent money. The "alternatives" are mostly *simplifications you'll regret*: a balance-only model, a single-entry log (no double-entry, so no conservation guarantee), or hard-deletable transactions (no audit). The trade you *do* make deliberately is read cost (derive balances) vs a maintained cache, and normalization purity vs query-shaped summaries (1.14) — both resolved by "log is source of truth, derived structures are rebuildable."

**Generics / first-principles.** "Model money as a conserved quantity moved by balanced, immutable events." Conservation laws + immutable event logs + derived state is a pattern far beyond finance (inventory, energy accounting, any system where a quantity must be neither lost nor duplicated). Double-entry is just the most battle-tested instance. The agnostic takeaway: when correctness means "nothing is created or destroyed," make the conservation invariant *structural*, not something application code remembers to check.

**MySQL-specific reality.** On MySQL this skeleton leans on: **InnoDB transactions** to post all of a transaction's entries + update the balance atomically (M07); the **clustered PK** chosen for the dominant access pattern — often `ledger_entry (account_id, created_at, id)` so an account's history clusters for fast statements (1.14/M05); **FKs with RESTRICT** and never-delete policies for audit safety (1.6); **DECIMAL (or integer minor units), never FLOAT**, for amounts (the canonical M03 rule — floating point silently loses cents); **compact monotonic or time-ordered keys** (1.15) to keep the ever-growing ledger's indexes healthy (M09); and **partitioning/archival** as the ledger grows unbounded (M11). Hot-account contention (many entries racing on one popular account's balance row) is a real MySQL concurrency problem addressed in M08/M16. This model is the seed that M16 grows into a full payments-platform design.
