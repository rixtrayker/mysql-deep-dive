## 10.1 · Why replicate? read scaling, HA, durability ★

**Mental model.** Up to now there was *one* server holding all the data — a single point of failure and a single machine's capacity ceiling. **Replication copies the data to other servers**, buying three things: **read scaling** (send read queries to replicas, so the source isn't the bottleneck for reads), **high availability** (if the source dies, promote a replica — no total outage), and **durability / geo-redundancy** (a second physical copy, possibly in another datacenter, surviving the loss of one node or one region). The unavoidable cost — and the defining theme of the whole module — is that **the copies are not instantly identical**: a replica is always slightly behind, so reads can be stale and a crash can lose what hadn't propagated yet. Replication is the first step from "one box" to "a distributed system."

**How it actually works.** A **source** (primary) server records every committed change to its **binary log** (M09 — the same durable log used for crash recovery's binlog-redo consistency, 9.11, and PITR). A **replica** connects to the source, **streams the binlog**, and **replays** those changes to keep an up-to-date copy (10.2). This gives you N copies of the data. You then *use* the copies: route read-only queries (reports, dashboards, reconciliation) to replicas to **scale reads** (the source handles writes; replicas handle the read fan-out); keep a replica ready to be **promoted** to source on failure (**HA/failover**, 10.10); and place a replica in another datacenter for **geo-redundancy / DR** (survive a regional outage). Because the binlog is replayed *asynchronously* by default (10.4), replicas lag the source slightly (10.5) — the source doesn't wait for them — so the copies are *eventually* consistent, not instantly.

**Why it exists / what it solves.** Replication solves the three fundamental limits of a single server: **read capacity** (one machine can only serve so many reads — replicas multiply read throughput), **availability** (one machine is a single point of failure — replicas provide failover targets), and **durability/locality** (one machine in one location — replicas provide a second copy and geographic distribution). For a payments system, all three matter: heavy reporting/reconciliation (M02/2.17) would overload the transactional primary if not offloaded to replicas; a node failure must not be a total outage (HA); and a committed payment should survive even total loss of the source's node (a replica has it). It's the foundational mechanism for scaling and surviving beyond one machine.

**Tradeoffs & alternatives.** Replication's central tradeoff is **scale/availability/durability vs consistency** — the copies diverge (lag, 10.5), so you trade instant consistency for the benefits. Read scaling trades freshness (replica reads can be stale, 10.6); HA trades a possible data-loss window (a promoted replica may have missed the source's last transactions, 10.10); semi-sync durability trades commit latency (10.4). The alternatives don't avoid the tradeoff, they pick a different point: a *single bigger server* (vertical scaling — simpler, consistent, but a capacity ceiling and a single point of failure); *synchronous/consensus replication* (group replication, 10.9 — stronger consistency, more latency/coordination); *sharding* (M11 — scales *writes*, which replication doesn't). Replication scales *reads* and provides *HA/durability*; it does *not* scale writes (every replica still applies every write) — that's sharding's job (M11), and real systems combine them.

**Generics / first-principles.** "Copy the data to multiple nodes for read capacity, fault tolerance, and locality — accepting that the copies are only eventually identical." This is the universal **replication** principle in distributed systems: every replicated datastore (Postgres streaming replication, MongoDB replica sets, Kafka replication, Raft/Paxos-backed stores) copies data to followers for the same three reasons (scale reads, survive failures, geo-distribute), and all face the same core tension (the copies diverge until propagation completes). The transferable insight: **replication buys availability/scale/durability at the cost of consistency between copies** — the copies are a *cache-like* second source of truth that lags, and every use of a replica must account for that lag. This is the entry point to the whole consistency-vs-availability tradeoff (CAP/PACELC, M12).

**MySQL-specific reality.** MySQL replication ships the **binary log** (M09) from a source to replicas (10.2). It's **asynchronous by default** (the source doesn't wait for replicas — fast, but replicas lag, 10.5), with **semi-synchronous** (wait for a replica to *receive* the binlog, 10.4) and **group replication** (consensus-based, 10.9) as stronger options. Modern MySQL uses **GTIDs** (10.7) for robust transaction tracking. The terminology modernized from "master/slave" to **source/replica** (and `CHANGE MASTER` → `CHANGE REPLICATION SOURCE`). For our domain: a payments primary handles transfers (writes); read-replicas serve reporting and reconciliation (M02/2.17 — offloading the primary); a semi-sync replica provides durable redundancy; a standby replica is the failover target (10.10); a cross-region replica provides DR. The whole module elaborates *how* this works and *where it can silently lose or stale money* (10.5/10.10/10.11/10.12). Replication is the foundation of every scaled, highly-available MySQL deployment — and of the fintech platform's topology (10.16, M16).

## 10.2 · The mechanism: binlog → relay log → apply ★

**Mental model.** All MySQL replication is **log shipping**: the source writes every change to its **binary log** (M09), a replica **streams** that binlog over the network and writes it to a local **relay log**, and an **applier thread** on the replica **replays** the relay log to reproduce the changes — so the replica's data ends up matching the source's. It's "ship the log of changes, replay it elsewhere" — the *same* idea as crash recovery (replay the log to reconstruct state, M09/9.14) and event sourcing (replay events to rebuild state, M01/1.17), now across machines. Understanding this three-step pipeline (binlog → relay log → apply) is understanding all of replication's behavior, lag, and failures.

**How it actually works.** The pipeline, with the threads involved (10.8):
1. **Source writes the binlog:** every committed transaction is recorded in the source's **binary log** (M09 — durably, via group commit + the binlog-redo 2PC, 9.11). The binlog is the authoritative stream of changes.
2. **Replica's IO (receiver) thread streams it:** the replica connects to the source and continuously **pulls the binlog** as it's written, writing the received events to a local **relay log** (a local copy of the binlog stream). This is just network transfer + local logging — fast, and it's how far the replica has *received* (vs *applied*).
3. **Replica's SQL (applier) thread(s) replay it:** the applier reads the **relay log** and **executes** the changes against the replica's data, bringing it up to date. With **parallel replication** (10.8), multiple applier workers replay independent transactions concurrently.
So there are *two* positions on a replica: how much it has **received** (relay log written) and how much it has **applied** (executed) — and the gap between the source's latest and the replica's *applied* position is **replication lag** (10.5). The replica tracks its position via **GTIDs** (10.7) or binlog file+position. (In **semi-sync**, 10.4, the source waits for the replica's *receipt* acknowledgment before the commit returns — a synchronization point in step 2.)

**Why it exists / what it solves.** Log shipping is the *mechanism* that makes replication possible and efficient: instead of copying entire tables or diffing data, you ship the *compact stream of changes* (the binlog) and replay it — so a replica stays in sync by applying the same operations the source did, in the same order. The relay log decouples *receiving* from *applying* (the replica can receive faster than it applies, buffering in the relay log — important for catching up and for not losing received-but-not-applied data on an applier crash). This pipeline is what every replication feature (formats 10.3, sync modes 10.4, GTIDs 10.7, parallel apply 10.8, failover 10.10) operates on. It's the foundational machinery.

**Tradeoffs & alternatives.** The binlog→relay→apply pipeline trades **simplicity/efficiency for the received-vs-applied gap** (lag, 10.5 — the replica can fall behind on *applying* even if it *receives* promptly). The relay log adds local storage and a step but buys the decoupling (receive-ahead-of-apply). Alternatives to logical log shipping: **physical replication** (ship the redo/page changes — what Postgres streaming replication does; tighter, but ties replicas to the exact storage format) — MySQL uses *logical* binlog shipping (more flexible: replicas can differ in version, indexes, even engine; enables filtering and CDC, 10.14). The format choice (statement/row/mixed, 10.3) is a sub-tradeoff within logical shipping. The pipeline's structure (separate IO and SQL threads, the relay log) is what enables both robustness (received data survives an applier restart) and the parallel-apply optimization (10.8).

**Generics / first-principles.** "Replicate state by shipping the ordered log of changes and replaying it on followers — the log is the source of truth, and replaying it reconstructs the state anywhere." This is the universal **state-machine replication via log shipping**: a deterministic replay of an ordered operation log produces an identical copy of the state (the foundation of Raft/Paxos replicated logs, Kafka's log replication, database replication generally, and event sourcing, M01/1.17). The transferable insight: **if you have an ordered, durable log of all changes, you can reconstruct the state on any node by replaying it — replication is just shipping that log and replaying it remotely** (the same "the log is primary, state is derived" theme as M01/1.17, M08/8.2, M09/9.14). The received-vs-applied distinction (a follower's two positions) is universal to any log-replication system (Kafka consumer lag, Raft commit-vs-apply index).

**MySQL-specific reality.** MySQL replication: the source's **binary log** → the replica's **IO thread** streams it to the **relay log** → the **SQL/applier thread(s)** replay it. `SHOW REPLICA STATUS` exposes the two positions (received: `Relay_Master_Log_File`/`Read_Master_Log_Pos`; applied: `Relay_Log_File`/`Exec_Master_Log_Pos`; `Seconds_Behind_Source`/`Master` ≈ lag). With **GTIDs** (10.7), positions are tracked as transaction sets (`Retrieved_Gtid_Set` vs `Executed_Gtid_Set`). The logical binlog stream is *why* MySQL replicas can differ (version, indexes, even be filtered) and *why* the same stream feeds CDC and PITR (10.14). For our domain: a committed transfer is written to the source's binlog, streamed to each replica's relay log, and applied — so reporting replicas reflect it (after lag, 10.5), the standby has it for failover (10.10), and the cross-region replica has it for DR. This pipeline is the spine of the module — every later concept is about *this* stream (its format, its sync point, its lag, its failover, its silent failures). The ★ SVG (Pass C) draws binlog → relay log → apply.

## 10.3 · Binary log formats: statement, row, mixed

**Mental model.** The binlog can record a change in three ways, and the choice has real correctness and efficiency consequences. **Statement-based (SBR):** log the *SQL statement* itself (`UPDATE account SET balance = balance - 100 WHERE id = 42`) — compact, but *dangerous* because some statements are **non-deterministic** (they produce different results when replayed: `NOW()`, `UUID()`, `RAND()`, certain `LIMIT`-without-`ORDER BY`), so the replica can diverge from the source. **Row-based (RBR):** log the *actual row changes* (the before/after image of each affected row) — exact and replay-safe (no determinism issues), but larger (a statement touching a million rows logs a million row changes). **Mixed:** use statement-based normally but automatically switch to row-based for non-deterministic statements — a compromise. **Row-based is the safe modern default** for correctness.

**How it actually works.**
- **Statement-based (SBR):** the binlog stores the literal SQL. The replica *re-executes* it. Compact (one statement, however many rows it affects), human-readable, but **unsafe for non-deterministic statements**: `INSERT … VALUES (UUID())` generates a *different* UUID on the replica → divergence; `UPDATE … LIMIT 10` without `ORDER BY` may affect *different* rows; `NOW()`/`RAND()`/`CONNECTION_ID()` differ. Also sensitive to differences in execution (triggers, stored functions, auto-inc with interleaved mode, M08/8.9).
- **Row-based (RBR):** the binlog stores the *resulting row images* (for an UPDATE: the before and after row values; for INSERT: the new row; for DELETE: the deleted row's key). The replica *applies the row changes directly* (no re-execution) → **deterministic and safe** (the source already computed the result; the replica just copies it). Larger binlog (logs every changed row), and less human-readable. The default and recommended format.
- **Mixed:** SBR by default, auto-switching to RBR when the optimizer detects a non-deterministic/unsafe statement — gets SBR's compactness where safe and RBR's safety where needed.

**Why it exists / what it solves.** The formats exist to navigate the **compactness vs correctness** tradeoff. SBR was the original (compact, low binlog volume), but its non-determinism hazard causes *silent replica divergence* — the replica's data drifts from the source's, which for money is catastrophic (the ledger forks). RBR solves correctness (the replica copies the exact result the source computed, so it *can't* diverge from non-determinism) at the cost of binlog size. The format choice is fundamentally about *guaranteeing replicas stay identical to the source* — and for any system where divergence is unacceptable (money), RBR's safety is non-negotiable.

**Tradeoffs & alternatives.** **SBR:** smallest binlog (compact statements), but **unsafe** (non-determinism → divergence) — disqualifying for correctness-critical data. **RBR:** safe (exact row images, no divergence), but **larger binlog** (logs every row changed — a bulk update of millions of rows produces a huge binlog) and less readable. **Mixed:** a compromise, but adds complexity (you must understand when it switches) and still has SBR's pitfalls for the statement-based cases. The modern consensus: **use RBR** (the default in modern MySQL) for correctness — the binlog-size cost is acceptable, and tools (binlog_row_image=minimal to log only changed columns) mitigate it. RBR is also *required* for some features (Group Replication, certain CDC setups, 10.14). The tradeoff is essentially settled in favor of RBR for any serious deployment.

**Generics / first-principles.** "Replicate by shipping the *result* (deterministic, safe) rather than the *operation* (compact but re-execution can diverge) when correctness demands it." This is the universal **operation-replication vs state-replication** choice: shipping operations (SBR — "do this command") is compact but requires deterministic re-execution (any non-determinism diverges); shipping results (RBR — "the data is now this") is larger but guaranteed identical. The transferable insight: **command replication needs determinism (same command + same state → same result); when you can't guarantee determinism, replicate the resulting state instead** — the same issue in any state-machine replication (commands must be deterministic for log replay to converge — Raft/Paxos require deterministic state machines for exactly this reason), in CRDTs, and in event sourcing (events should record *what happened*, not *what to do*, to avoid replay divergence — M01/1.17). Non-determinism is the enemy of operation-based replication.

**MySQL-specific reality.** `binlog_format = ROW` (the modern default and recommendation) / `STATEMENT` / `MIXED`. RBR is required for **Group Replication** (10.9) and is the safe choice for everything (no non-determinism divergence). `binlog_row_image = minimal` reduces RBR size (logs only changed columns + the key, not the full before/after row). The classic SBR bug — `INSERT … VALUES (NOW(), UUID())` producing different values on the replica → silent divergence — is exactly why RBR is the default. For our domain: the ledger and accounts replicate with **ROW format** — so a transfer's exact row changes (the precise balance values, the exact entry rows) are copied to replicas *identically*, with *no* chance of non-deterministic divergence (a statement-based `balance = balance - 100` is actually deterministic, but `idempotency_key = UUID()` or timestamp defaults would not be — RBR sidesteps the whole class). RBR's correctness guarantee (replicas can't drift from the source) is essential for money: a forked ledger between source and replica would be a money-never-lies catastrophe. Use ROW format for fintech, full stop.

## 10.4 · Async, semi-sync & sync replication ★

**Mental model.** The single most important replication *decision*: **how long does the source wait for the replica before the commit returns?** This is the **consistency/durability vs latency dial**, with three settings. **Asynchronous (async):** the source **doesn't wait** at all — it commits locally and returns immediately; the binlog ships to replicas in the background. Fast, but a source crash can **lose** transactions the replicas hadn't received yet. **Semi-synchronous (semi-sync):** the source **waits for at least one replica to *receive* (acknowledge) the binlog** before the commit returns — so a committed transaction is durable on *at least one other node* (it survives total loss of the source). Slower (a network round-trip per commit), but **replica-confirmed durability**. **Synchronous / consensus (group replication):** the source waits for replicas to *agree* (and often *apply*) — strongest consistency, most latency/coordination. Choosing the wait point *is* choosing your durability/consistency vs latency tradeoff.

**How it actually works.**
- **Async (default):** the source commits (durable locally via the redo log, M09), writes the binlog, returns to the client — *then* replicas stream and apply the binlog whenever they can (lag, 10.5). The source never waits for replicas. **Failure exposure:** if the source crashes after returning "committed" but before a replica received the transaction, and that replica is promoted (10.10), the transaction is **lost** (the promote-before-apply data-loss window). Async = lowest latency, highest throughput, but a real loss window on failover.
- **Semi-sync:** on commit, the source writes the binlog and **waits for a replica to acknowledge it *received* the binlog event** (wrote it to its relay log) before returning "committed" to the client. So *at least one other node has the transaction* when the client is told it committed → it survives total source-node loss (a promoted replica has it). Cost: one network round-trip added to *every commit* (latency), and it can **silently degrade to async** on timeout (10.12 — a critical gotcha). Note: semi-sync guarantees *receipt*, not *apply* (the replica has it in its relay log, will apply it) — "after-sync" variants tighten when the ack happens relative to the source's own commit.
- **Sync / Group Replication (10.9):** a group of nodes uses a **consensus protocol** (Paxos-based) — a transaction must be *certified/agreed* by a majority before commit. Strongest consistency (no divergence, automatic conflict handling), but the most coordination/latency, and write throughput limited by the slowest majority.

**Why it exists / what it solves.** The three modes exist because **durability/consistency and latency are in direct tension**, and different systems need different points. Async maximizes throughput/latency for systems that can tolerate a small loss window on rare failover (or that have other durability, like backups). Semi-sync provides **replica-confirmed durability** — the guarantee that a committed transaction survives even total loss of the source's node (not just a process crash, M09 — an *entire node/disk* loss) — essential for money (a confirmed payment must not vanish if the primary's datacenter burns down). Group replication provides strong consistency for systems that need it (no divergence, automatic failover). The choice resolves "how much am I willing to slow down every commit to guarantee it's durable beyond one node / consistent across nodes?"

**Tradeoffs & alternatives.** The dial: **async** (fastest, but loss window on failover — 10.10) → **semi-sync** (durable beyond the source node, but +1 round-trip per commit latency, and the silent-degrade-to-async hazard, 10.12) → **group replication** (strongly consistent, but most latency/coordination, throughput capped by majority agreement). It directly extends M09's durability posture: M09 made a commit durable *on the source's disk*; semi-sync makes it durable *on another node too* (surviving node loss, which M09's local fsync can't). For money: **semi-sync is the typical fintech choice** — it gives replica-confirmed durability (a committed transfer survives node loss) at an acceptable latency cost, *if* you guard against the silent degrade-to-async (10.12). Group replication is chosen when strong consistency + automatic failover are worth the coordination cost. Pure async is for loss-tolerant systems or where the loss window is acceptably mitigated. There's no free lunch — every mode trades latency for a durability/consistency guarantee.

**Generics / first-principles.** "How many replicas must acknowledge before you call a write 'done' determines its durability/consistency vs its latency — the universal replication-acknowledgment dial." This is *the* core distributed-replication tradeoff, present everywhere: Kafka's `acks` (0/1/all), quorum systems' W (write quorum), Raft/Paxos majority-commit, Postgres `synchronous_commit` levels, Cassandra/Dynamo tunable consistency. The transferable insight: **a write's durability/consistency is defined by *how many nodes confirmed it before you acknowledged the client* — wait for more nodes (stronger guarantee, more latency) or fewer (weaker guarantee, less latency)**, and you tune this per the cost of loss. It's the same "what does 'acknowledged' mean" question as M09's fsync (9.7 — durable to *which layer*), now lifted to *how many nodes*. This dial is the heart of CAP/PACELC (M12): in the latency-vs-consistency choice (the PACELC "else"), this *is* the knob.

**MySQL-specific reality.** MySQL: **async** (default — `source_log_pos` based, no wait), **semi-synchronous** (the `rpl_semi_sync_source`/`rpl_semi_sync_replica` plugins — source waits for a replica's receipt ack, with `rpl_semi_sync_source_timeout` controlling how long before it **silently falls back to async**, 10.12 — the critical gotcha), and **Group Replication** (the consensus-based, multi-primary-capable cluster underlying InnoDB Cluster, 10.9/10.13). Semi-sync's `rpl_semi_sync_source_wait_for_replica_count` sets how many replicas must ack; "after-sync" (`rpl_semi_sync_source_wait_point=AFTER_SYNC`) acks before the source's own commit is externalized (tighter durability). For our domain: a payments platform typically runs **semi-sync** (so a committed transfer is durable on ≥1 replica → survives total loss of the primary node, extending M09's posture, 9.16) — *with monitoring/alerting on the semi-sync status* so it doesn't silently degrade to async (10.12) and leave you *thinking* you have replica durability when you don't. Group replication (InnoDB Cluster) is the alternative for strong consistency + automated failover. This sync-mode choice is the most consequential replication decision for money durability (10.15/10.16). The ★ SVG (Pass C) draws the sync spectrum (wait point → guarantee → latency).
