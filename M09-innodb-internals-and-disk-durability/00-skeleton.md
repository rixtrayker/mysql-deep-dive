# M09 · InnoDB Internals & Disk Durability ★ — Pass A (Skeleton)

> **Pass A goal:** lock the ordered concept list, a one-line mental model per concept, and the planned diagram + bespoke-SVG inventory + worked example per concept. No prose yet — that's Pass B.
>
> **Module mental model:** *Everything above this module assumed two magic guarantees: that committed data is* durable *(M07's D) and that old row versions exist for* MVCC *(M08's snapshots). M09 is where the magic becomes mechanism — the actual machinery InnoDB uses to be fast* and *crash-safe at the same time. The core trick is **write-ahead logging**: don't write changed data pages to their final home on every commit (too slow — random I/O); instead, write the change* sequentially *to a **redo log** and **fsync** that, then update the in-memory page lazily and flush it to disk later. The **buffer pool** caches pages in RAM (most reads/writes never touch disk); the **redo log** makes commits durable without random writes; the **undo log** holds old versions (for rollback* and *MVCC); and crash recovery replays the redo log to reconstruct anything not yet flushed. But durability is only as good as the **fsync actually reaching stable storage** — and that's where it gets dangerous: OS caches, lying disk caches, torn page writes, and mis-set flush settings are how "committed" data silently vanishes. This module is the engine room — and the place where you learn exactly what survives a crash and what doesn't.*
>
> **Threads carried in this module:**
> - **Durability** — *the* thread: this is the durability module. What is *actually* written to disk, in what order, and what survives a crash (vs. what silently doesn't). The disk-sync chain, fsync semantics, lying disks, and the flush-tradeoff matrix are the heart of it.
> - **Money-never-lies** — a lost committed payment is the worst fintech failure; durability settings and the rare-but-catastrophic data-loss modes (torn pages, fsync-that-lied) directly threaten it. The strict durable settings are mandatory for money.
> - **Generics-first** — WAL, the buffer pool / page cache, the storage hierarchy, group commit, and checkpointing are universal storage-engine concepts (every serious database, filesystem journal, and log-structured store uses them); InnoDB is one realization.
> - **Tradeoff** — durability vs throughput is the central, *tunable* tradeoff (the flush matrix); buffer-pool size vs RAM; flush aggressiveness vs write amplification. Every internal is a performance/safety dial.
>
> **Prereqs:** M07 (durability as the *contract* — 7.5; atomicity/undo for rollback — 7.2; long-transaction harm — 7.15), M08 (MVCC version chains read from undo — 8.2; "undo management/purge deferred to M09"; strict 2PL), M03/3.2 (page/footprint), M05 (B+Tree pages, page splits — the structures the buffer pool caches), M01/1.3 (clustered index). **Leads into:** M10 (the binlog — a *second* durable log for replication, and its fsync `sync_binlog`), M13 (backups/PITR built on these logs; monitoring the buffer pool / history list), **M15 (Failure Modes & Data Loss — the rare catastrophic cases this module's mechanisms can fail in: torn pages, lying disks, the flush-matrix loss window, crash-recovery edge cases)**, M16 (the platform's durability posture).
