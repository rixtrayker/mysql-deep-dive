## 8.1 · Why two mechanisms: MVCC + locking ★

**Mental model.** InnoDB delivers isolation (M07/7.4) with **two cooperating mechanisms**, not one. **MVCC** (multi-version concurrency control) handles *reads*: every row keeps old versions, so a reader sees a consistent *snapshot of the past* without taking any locks — **readers never block writers, and writers never block readers**. **Locking** handles *writes* (and reads that must block): conflicting writers take row locks and serialize, while everyone else proceeds. The slogan that captures the whole engine: *"readers don't block writers, writers don't block readers" (MVCC) + "writers block conflicting writers" (locks)*. This combination is what makes REPEATABLE READ both **correct** (consistent reads, serialized conflicting writes) and **concurrent** (reads never wait) — and understanding the split is the key to reasoning about contention.

**How it actually works.** When a transaction runs a **plain `SELECT`**, InnoDB uses **MVCC**: it reads each row from the version chain (8.2) that was valid as of the transaction's snapshot (8.3) — no locks taken, no waiting, even if another transaction is concurrently writing that row (the reader sees the *old* version). When a transaction **writes** (INSERT/UPDATE/DELETE) or does a **locking read** (`SELECT … FOR UPDATE/FOR SHARE`, 8.4), InnoDB takes **locks** on the affected index records (8.7) — and conflicting writers (two transactions wanting to write the same row) *serialize* (one waits for the other to commit). So the two mechanisms partition the work: MVCC makes the common case (reads) lock-free and non-blocking; locks make the conflicting case (concurrent writes to the same data) correct by serializing only what *must* serialize. Most of a workload's concurrency comes from MVCC keeping reads out of the locking path entirely.

**Why it exists / what it solves.** A naive single-mechanism approach — lock everything, including reads (pure two-phase locking with shared read locks) — would be correct but *catastrophic for concurrency*: every reader would block writers and vice versa, so a long report would freeze all writes, and a busy write would freeze all reads. MVCC solves this by letting readers see a consistent *past* snapshot instead of blocking for the *current* state — so reads and writes proceed in parallel. Locks then handle only the genuinely-conflicting case (write-write). The two-mechanism design is what makes a database *both* strongly-isolated *and* highly-concurrent — you don't have to choose. For our domain, it's why a heavy reconciliation read (M02/2.17) and a stream of transfers can run *at the same time* without blocking each other.

**Tradeoffs & alternatives.** MVCC's cost is **keeping old versions** (storage and maintenance — the undo logs, M09, that must be retained while any transaction's snapshot might need them, and purged later, 8.10/M09) and the work of **traversing version chains** to find the visible version (longer chains = slower reads — the history-list-bloat problem when a long transaction pins old versions, M07/7.15, M09). Pure locking (no MVCC) avoids version storage but destroys read concurrency. Pure optimistic/lock-free schemes avoid locks but push conflict-handling to retries. InnoDB's MVCC-for-reads + locks-for-writes is the dominant design precisely because it gives the best *common-case* concurrency (reads dominate most workloads) while keeping writes correct — at the cost of version maintenance.

**Generics / first-principles.** "Separate the non-conflicting common case (reads) from the conflicting case (writes), and use the cheapest correct mechanism for each — versioning for reads, locking for writes." This MVCC + locking split is the dominant concurrency-control architecture across databases (Postgres, Oracle, InnoDB all use versions for reads + locks for writes), and it reflects a universal principle: *don't make the common, non-conflicting path pay the cost of the rare, conflicting one*. The transferable instinct: when most operations don't actually conflict (reads of mostly-stable data), give them a lock-free path (snapshots/versions) and reserve serialization for the genuine conflicts (writes to the same item) — the same idea as RCU, copy-on-write, and persistent data structures.

**MySQL-specific reality.** InnoDB implements exactly this: **plain `SELECT` = non-locking MVCC snapshot read** (reads from the undo version chain, 8.2, per the read view, 8.3 — no locks); **writes and `SELECT … FOR UPDATE/FOR SHARE` = locking** (row/next-key locks on index records, 8.7). This is *why* InnoDB's default REPEATABLE READ (M07/7.13b) gives strong, non-blocking reads. The version chains live in the **undo logs** (M09), the same undo that powers rollback (M07/7.2) — so MVCC and atomicity share machinery. The practical payoff for our domain: a reconciliation `SUM` over the ledger (a plain SELECT) reads a consistent snapshot *without blocking* the concurrent transfers writing new entries, and the transfers don't block the read — true read/write concurrency. The rest of this module zooms into each half: MVCC (8.2–8.4) and locking (8.5–8.13).

## 8.2 · MVCC: multi-version concurrency control ★

**Mental model.** The core trick that makes non-blocking reads possible: **InnoDB never overwrites a row in place destructively — it keeps a *chain of old versions*.** Each row has its current version in the clustered index (M01/1.3), and a backward-linked chain of *prior* versions stored in the **undo logs** (M09). When a transaction reads, it walks this chain to find the version that was valid *as of its snapshot* — so a reader sees a consistent *past* state even while writers are creating *newer* versions. "Multi-version" = many versions of each row coexist; "concurrency control" = transactions stay isolated by each reading the *right* version, without locking.

**How it actually works.** Each row carries hidden system columns: a **`DB_TRX_ID`** (the transaction that last modified it) and a **`DB_ROLL_PTR`** (a "roll pointer" to the *previous* version in the undo log). When a transaction updates a row, InnoDB: writes the new version in place (in the clustered index) with the updater's `DB_TRX_ID`, and sets its `DB_ROLL_PTR` to point at the *old* version now copied into the undo log — so the old version is still reachable. This builds a **version chain**: current row → undo version → older undo version → …, each tagged with the `DB_TRX_ID` that created it. A reader, instead of taking a lock, **walks the chain from the current version backward**, using its read view (8.3) to pick the first version that was *committed and visible* as of its snapshot. So two transactions reading the "same" row can legitimately see *different versions* — each sees the one consistent with its own snapshot. (Deletes are marked with a delete-bit and purged later; the old version remains in the chain until no snapshot needs it, M09.)

**Why it exists / what it solves.** MVCC solves the fundamental read/write concurrency conflict: without it, a reader wanting a consistent view would have to *lock* the rows it reads (blocking writers) or risk seeing inconsistent mid-write state. By keeping old versions, MVCC lets a reader reconstruct a consistent snapshot *of the past* without locking anything — so reads and writes proceed fully in parallel (8.1). It's the mechanism that delivers REPEATABLE READ's "stable snapshot" (M07/7.9 — re-reads return the same value because they read the same snapshot version) and READ COMMITTED's "only committed data" — all without read locks. It's the single most important reason InnoDB is highly concurrent under mixed read/write load.

**Tradeoffs & alternatives.** The cost is **version retention and chain traversal**. Old versions must be *kept* as long as any active transaction's snapshot might need them — so a **long-running transaction** pins old versions across the whole database, preventing their purge and causing the **history list to grow** (M07/7.15, M09 — undo bloat that slows everything, since reads must traverse longer chains and storage grows). Frequent updates to a hot row build a long version chain, making reads of it slower (each must walk further back). The alternative — lock-based reads (no versions) — avoids this storage/traversal cost but destroys read concurrency. MVCC trades *space and version-maintenance* for *non-blocking reads*, which is almost always the right trade (reads dominate, and version maintenance is manageable with short transactions and purge, M09).

**Generics / first-principles.** "Keep old versions so readers can see a consistent past without blocking writers — never destroy data the still-running readers might need." MVCC is a specific instance of the universal **immutability-for-concurrency** pattern: copy-on-write, persistent/immutable data structures, RCU (read-copy-update) in the Linux kernel, and even Git's immutable object model all let readers work on a stable snapshot while writers produce new versions, avoiding reader/writer locking. The transferable insight: *concurrency conflicts between readers and writers largely vanish if writers create new versions instead of mutating in place* — the cost being version storage and eventual cleanup. Recognizing "this is MVCC / copy-on-write" explains a huge class of high-concurrency designs.

**MySQL-specific reality.** InnoDB's MVCC specifics: the hidden `DB_TRX_ID` and `DB_ROLL_PTR` per row, version chains threaded through the **undo logs** (M09), and the read view (8.3) deciding visibility. Crucial scope note (the M08/M09 split): MVCC *reads* the undo version chain here; the *management* of undo — the **purge thread** that removes versions no snapshot needs, the **history-list length** that bloats when long transactions pin versions (M07/7.15) — is M09. Only the **clustered index** (PK) carries full MVCC version chains; **secondary indexes** don't store `DB_TRX_ID` per version, so a secondary-index read that needs visibility checking may have to consult the clustered index (a detail affecting some reads). For our domain: when a transfer updates a hot account's balance, it creates a new version and pushes the old one to undo; a concurrent reconciliation read sees the *old* (snapshot-consistent) balance without blocking — true MVCC concurrency, powered by the version chain this concept describes. The ★ SVG (Pass C) draws this chain (row → undo versions → txn ids).

## 8.3 · Read views & visibility (how a snapshot is chosen) ★

**Mental model.** A version chain (8.2) only helps if a reader can decide *which* version to read — and that's the **read view**: a snapshot of "which transactions' changes are visible to me." When a transaction takes its snapshot, InnoDB records the set of transactions that were *active (uncommitted)* at that moment; then, for each row version it encounters (walking the chain), it applies a **visibility rule**: "was this version created by a transaction that had *committed* before my snapshot?" — if yes, that version is visible (use it); if no (the creator was still active, or started after), skip to the older version. The read view is the precise definition of "my consistent snapshot," and the visibility check is how MVCC turns a version chain into a single consistent view.

**How it actually works.** A **read view** captures, at snapshot time: the set of transaction IDs that are currently *active* (in-progress, uncommitted), plus the low/high watermarks of transaction IDs. To determine if a given row version (tagged with its creator's `DB_TRX_ID`) is visible, InnoDB checks:
- If the version's `DB_TRX_ID` is **the reader's own** → visible (you see your own writes).
- If it **committed before the read view was created** (its ID is below the watermark and not in the active set) → visible.
- If it was created by a transaction that was **still active at snapshot time**, or **started after** the snapshot → *not* visible → follow `DB_ROLL_PTR` to the previous version and re-check.
The reader walks the chain backward until it finds the first visible version — that's the value it reads. **When the read view is created** determines the isolation level's behavior (8.14): at **REPEATABLE READ**, the read view is created at the transaction's *first read* and reused for all subsequent reads (→ stable, repeatable snapshot — same versions every time); at **READ COMMITTED**, a *new* read view is created for *each statement* (→ each statement sees the latest committed data, hence non-repeatable reads, M07/7.9). So the *timing* of the read view *is* the isolation level.

**Why it exists / what it solves.** The version chain is raw material; the read view is the *policy* that selects a consistent cut through it. It solves "given many versions of every row, which set of versions forms a consistent snapshot for *this* transaction?" — and it does so *per transaction*, so different transactions get different (but each internally-consistent) views. It's the precise mechanism behind the isolation guarantees: repeatable reads (one read view, reused) vs read-committed (fresh read view per statement) are *the same versioning machinery* with *different read-view timing*. Understanding read views is what makes "why does RR see a stable snapshot but RC sees fresh data?" concrete rather than magic.

**Tradeoffs & alternatives.** The read view itself is cheap (a small set of active transaction IDs), but it *defines how far back the version chains must be retained*: every active read view pins the versions it might need, so a long-lived read view (a long transaction at RR) holds old versions across the system (8.2's history-list bloat). The RR-vs-RC tradeoff is partly a read-view tradeoff: RR's single long-lived read view gives repeatable reads but pins more history; RC's per-statement read views see fresher data and pin less (so some high-throughput shops prefer RC, M07/7.13b). The "alternative" to read-view-based visibility — timestamp ordering, or lock-based reads — either needs different bookkeeping or sacrifices non-blocking reads. Read-view MVCC is the standard, efficient approach.

**Generics / first-principles.** "A consistent snapshot is defined by *which writers had committed* at snapshot time — record that set, and resolve each item to the latest version visible under it." The read view is a specific instance of **snapshot isolation via commit-visibility**: the universal idea that a consistent point-in-time view = "everything committed before time T, nothing after." This appears in distributed systems (snapshot reads at a timestamp, MVCC in distributed databases), in version-control (a commit is a consistent snapshot of everything merged before it), and in any system offering point-in-time consistency. The transferable insight: *consistency is defined relative to a commit boundary; capture that boundary (the active set / a timestamp) and resolve reads against it.* And the timing of *when* you capture it is exactly what distinguishes consistency levels.

**MySQL-specific reality.** InnoDB read views are created **at the first consistent read** in a transaction (not at `START TRANSACTION` — so the snapshot is "as of the first read," M07/7.13b) — unless you use `START TRANSACTION WITH CONSISTENT SNAPSHOT` to take it immediately. **RR** reuses one read view for the whole transaction (repeatable, stable snapshot); **READ COMMITTED** creates a fresh read view per statement (sees latest committed → non-repeatable reads, M07/7.9). The visibility rule (own writes + committed-before-snapshot are visible) is the mechanism. Crucial connection: **plain `SELECT` uses the read view (snapshot/MVCC); locking reads (`FOR UPDATE`) bypass it and read the *latest* committed version + lock it** (8.4) — so a read view governs *consistent reads*, not *locking reads*. For our domain, this is why a multi-query reconciliation report at RR sees *one* consistent point-in-time view of all balances (one read view, reused) even as transfers commit during it — and why the same report at READ COMMITTED could see balances drift between its queries. The ★ SVG (Pass C) shows the read view's active-set and the visibility check resolving a version chain.

## 8.4 · Consistent reads vs locking reads

**Mental model.** There are **two fundamentally different ways InnoDB reads a row**, and confusing them is a common source of bugs. A **consistent read** (plain `SELECT`) uses MVCC: it reads the **snapshot** version (8.2/8.3), takes **no locks**, never blocks, and may return *older-than-current* data (the version visible to your read view). A **locking read** (`SELECT … FOR UPDATE` or `… FOR SHARE`) bypasses the snapshot: it reads the **latest committed** version, takes a **lock** (exclusive for `FOR UPDATE`, shared for `FOR SHARE`), and *can block* (waiting for a conflicting lock to release). Plain reads are for "show me a consistent view"; locking reads are for "I'm about to make a decision and write based on this, so lock it so it can't change under me."

**How it actually works.**
- **Consistent (snapshot) read** — `SELECT * FROM account WHERE id = 42;` — reads via the read view (8.3) from the version chain (8.2). No locks, no blocking, fully concurrent with writers. At RR, repeated consistent reads return the *same* snapshot value (M07/7.9); at RC, each sees the latest committed. This is the *default* and the common case.
- **Locking read** — `SELECT * FROM account WHERE id = 42 FOR UPDATE;` — does *not* use the snapshot; it reads the **current** committed version and takes an **exclusive lock** on the matching index records (8.7), so no other transaction can modify (or `FOR UPDATE`-lock) those rows until you commit. `FOR SHARE` takes a **shared lock** (others can read-lock but not write). Locking reads *can block* (if another transaction holds a conflicting lock) and are held until commit (8.10).
The critical distinction for correctness: a consistent read can return *stale* data (your snapshot's version), so making a *write decision* based on a plain `SELECT` is the **lost-update / write-skew hazard** (M07/7.11, 7.13) — to safely "read, decide, write," you must use a **locking read** (`FOR UPDATE`) so the data can't change between your read and your write.

**Why it exists / what it solves.** The two read modes serve two genuinely different needs. *Consistent reads* solve "give me a coherent view without blocking anyone" — perfect for reporting, dashboards, and any read that doesn't drive a write. *Locking reads* solve "I need the *current* value and a guarantee it won't change while I act on it" — essential for read-modify-write logic where a stale snapshot would cause a lost update (M07/7.11) or write skew (M07/7.13). Having both lets the same transaction use cheap non-blocking reads where it's safe and locking reads only where it must — minimizing locking while staying correct. It's the practical resolution of M07's "isolation level alone doesn't prevent write anomalies" — the *locking read* is how you prevent them.

**Tradeoffs & alternatives.** **Consistent reads:** maximum concurrency (no locks, no blocking), but return snapshot (possibly stale) data and are *unsafe for read-decide-write*. **Locking reads:** correct for read-decide-write and see current data, but take locks (contention, blocking, deadlock risk, 8.11) and read the *latest* (not your snapshot) — so they can see data your consistent reads in the same transaction didn't. The decision: use a plain `SELECT` unless you're going to *write based on the read*, in which case use `FOR UPDATE` (or restructure to an atomic update, M07/7.11, which is cheaper than an explicit locking read for pure increments). `FOR SHARE` is the middle ground (lock against writers but allow concurrent readers) — useful when you need the value stable but others may also read it. Over-using locking reads kills concurrency; under-using them (plain SELECT before a write) causes lost updates.

**Generics / first-principles.** "A read that *informs a write* needs different guarantees than a read that *just observes* — the former must lock or use compare-and-set, the latter can use a non-blocking snapshot." This is the universal distinction between an *observational read* and a *read-for-update*: the same issue as `load` vs `load-linked` (before a `store-conditional`), or reading a value you'll CAS vs reading it for display. The transferable instinct: **whenever a read feeds a decision that writes, the plain read is a hazard — you need to lock the read or detect change at write time** (M07/7.11–7.12); a snapshot read is only safe when nothing you do depends on the value still being current. Recognizing "is this read going to drive a write?" tells you which mode you need.

**MySQL-specific reality.** InnoDB: **plain `SELECT` = consistent snapshot read** (MVCC, no locks); **`SELECT … FOR UPDATE` = exclusive locking read**; **`SELECT … FOR SHARE`** (formerly `LOCK IN SHARE MODE`) = shared locking read. Locking reads read the latest committed version (bypassing the read view) and take next-key locks (8.7) at RR (so they also prevent phantoms in the locked range). Modifiers: `FOR UPDATE SKIP LOCKED` (skip already-locked rows — queues, 8.12), `FOR UPDATE NOWAIT` (fail immediately instead of waiting). For our domain: a reconciliation report uses **plain SELECTs** (snapshot, non-blocking); a transfer that must "read the balance, check it's sufficient, then debit" uses **`SELECT … FOR UPDATE`** on the account row (so the balance can't change between check and debit — preventing lost update / write skew, M07/7.11/7.13) — *or*, better for a pure debit, the **atomic `UPDATE … SET balance = balance - :amt WHERE balance >= :amt`** (M07/7.11) which combines the read-check-write into one locked operation. Knowing which read mode a query uses — and that plain reads are unsafe before a write — is core to correct concurrent money code.
