## 8.11 · Deadlocks: detection, victim selection, avoidance ★

**Mental model.** A **deadlock** is a *cycle* of transactions each waiting for a lock another holds: T1 holds lock A and wants B; T2 holds B and wants A — neither can proceed, ever, because each is waiting for the other. Unlike a simple lock-wait (which resolves when the holder commits), a deadlock is a *permanent* standoff that can only be broken by *aborting* one of the transactions. InnoDB **detects** the cycle automatically and **kills a victim** (rolls back one transaction so the other can proceed) — and the application must be ready to **retry** the victim. The *avoidance* discipline: acquire locks in a **consistent order** across all transactions, so cycles can't form.

**How it actually works.** InnoDB maintains a **waits-for graph** (who's waiting for whom). When a transaction would have to wait for a lock, InnoDB checks whether granting the wait would create a **cycle** in the graph — if so, it's a deadlock. InnoDB then **chooses a victim** (typically the transaction that has done the *least work* — fewest rows modified/locked — so the rollback is cheapest) and **rolls it back** (releasing its locks), letting the other transaction(s) proceed. The victim's application gets a **deadlock error (1213)** and should **retry** the whole transaction (which usually succeeds, since the conflicting transaction has now moved on). Deadlocks arise from **inconsistent lock ordering**: if T1 locks account A then B, and T2 locks B then A, they can interleave into the cycle. They also arise from **gap-lock interactions** (8.7 — next-key locks create surprising deadlocks even without obvious ordering issues) and **lock upgrades** (S→X, 8.6 — two transactions both holding S on a row, both trying to upgrade to X, deadlock). The **avoidance** patterns: (1) always acquire locks in a **consistent global order** (e.g., always lock accounts in ascending `account_id` order — so two transfers can't form a cycle); (2) keep transactions **short** (less time holding locks = smaller deadlock window); (3) use a **lower isolation level** (RC, fewer gap locks → fewer deadlocks, M07/7.13b); (4) reduce lock footprint (index your predicates, 8.8).

**Why it exists / what it solves.** Deadlocks are an *inevitable consequence* of fine-grained locking with transactions that lock multiple rows in varying orders — you can't have multi-row locking without the *possibility* of a cycle. InnoDB's **detection + victim selection** solves the "neither can proceed" standoff *automatically* (vs. some systems that only use timeouts, where you wait for a timeout to even discover the deadlock). Detection makes deadlocks *survivable* (one transaction is sacrificed and retried, rather than the system hanging). But detection is a *recovery* mechanism, not prevention — the application still pays (a rolled-back transaction, a retry), so the real win is *avoidance* (consistent lock ordering) to make deadlocks *rare*. This concept turns deadlocks from a mysterious hang into a understood, designed-around, retryable event.

**Tradeoffs & alternatives.** Deadlock **detection** (InnoDB's approach) finds cycles immediately and aborts a victim — fast resolution, but the overhead of maintaining/checking the waits-for graph (can be a bottleneck under *extreme* concurrency, so InnoDB lets you disable detection and fall back to timeouts via `innodb_deadlock_detect`). The alternative — **timeout-only** (no detection): simpler, no graph overhead, but you wait the full `innodb_lock_wait_timeout` to even *discover* a deadlock (slow resolution). The avoidance tradeoffs: consistent lock ordering is the best fix but requires discipline (and isn't always possible if lock order is data-dependent); lower isolation (RC) reduces deadlocks but weakens isolation (M07); short transactions reduce the window but don't eliminate cycles. There's no way to *eliminate* deadlocks entirely with multi-row locking — so the design stance is **avoid (consistent ordering) + detect (InnoDB does it) + retry (application must)**.

**Generics / first-principles.** "Cyclic waiting on shared resources is unavoidable with multi-resource locking — break it by consistent acquisition order (prevention) or cycle detection + victim abort (recovery)." Deadlock is one of the most fundamental concurrency problems (the dining philosophers, OS resource deadlocks, distributed lock deadlocks all share it), and the solutions are universal: **lock ordering** (impose a total order on resources, acquire in that order → no cycles possible — the classic prevention), **detection + recovery** (find the cycle, abort one), or **timeouts** (give up after waiting). The transferable insight: *whenever code acquires multiple locks, a consistent global acquisition order prevents deadlock* — and where you can't guarantee ordering, you need detection/retry. "Always lock in the same order" is one of the most important and universal concurrency rules.

**MySQL-specific reality.** InnoDB **automatically detects deadlocks** (waits-for graph) and **rolls back the victim** (the transaction that modified fewest rows — cheapest to undo), returning **error 1213** (`ER_LOCK_DEADLOCK`) — the application must **catch it and retry** (the standard money-code pattern, M07/7.7). `SHOW ENGINE INNODB STATUS` shows the *last* deadlock (the two transactions, their locks, the cycle) — the primary debugging tool. `innodb_deadlock_detect` (on by default) can be disabled at extreme scale (falling back to `innodb_lock_wait_timeout`). Gap locks (8.7) at RR cause many "surprising" deadlocks; RC (fewer gap locks) reduces them. For our domain: **two transfers between the same two accounts in opposite directions** (transfer A→B and B→A concurrently) can deadlock if each locks the *from* account first (T1 locks A then wants B; T2 locks B then wants A) — the classic cycle. The fix is **consistent lock ordering**: always lock the lower `account_id` first, regardless of transfer direction — so both transfers lock A before B, no cycle. This (consistent ordering + retry on 1213) is the canonical deadlock-avoidance pattern for money movement, and a guaranteed interview topic. The ★ SVG (Pass C) draws the waits-for cycle and the consistent-ordering fix.

## 8.12 · Lock waits, timeouts & SKIP LOCKED / NOWAIT

**Mental model.** When a transaction wants a lock that's **held** by another (but *not* in a deadlock cycle), it **waits** — blocking until the holder releases (at commit, 8.10) or until it gives up after `innodb_lock_wait_timeout`. But sometimes waiting is the *wrong* behavior, and InnoDB offers two alternatives: **`SKIP LOCKED`** (skip the locked rows and process the unlocked ones — "give me whatever's available right now") and **`NOWAIT`** (fail immediately with an error instead of waiting — "I won't queue"). These transform locking from "always block and wait" into a tool for **queue-style concurrent processing**, where many workers pull *different* items without blocking each other.

**How it actually works.**
- **Default (wait)** — a locking statement that hits a held lock waits up to `innodb_lock_wait_timeout` (default 50s), then errors (1205, lock wait timeout) and rolls back the statement. Normal for contention you expect to clear quickly.
- **`NOWAIT`** — `SELECT … FOR UPDATE NOWAIT` — if any matching row is already locked, **fail instantly** (error 3572) instead of waiting. For "I need *these specific* rows now or I'll do something else" — avoids piling up waiters.
- **`SKIP LOCKED`** — `SELECT … FOR UPDATE SKIP LOCKED` — **skip** any rows currently locked by others and return/lock only the *available* ones. This is the **queue-processing primitive**: many workers run `SELECT … FROM jobs WHERE status='pending' LIMIT 10 FOR UPDATE SKIP LOCKED` — each worker grabs *different* unlocked rows (the others are skipped), so workers pull from the queue concurrently *without blocking each other or each other's rows*. No two workers get the same job, and no worker waits.
These change the *response to a held lock* from "block" to "skip" or "fail" — turning row locks into a coordination mechanism for distributing work.

**Why it exists / what it solves.** Default lock-waiting is right for *contention you expect to clear* (two transfers to one account — wait your turn). But for **work-queue distribution** — many workers pulling tasks from a shared table — waiting is *exactly wrong*: you don't want workers blocking on each other's in-progress items; you want each worker to grab *available* work and move on. `SKIP LOCKED` solves this elegantly: the table *is* the queue, the row lock *is* the "claimed" marker, and `SKIP LOCKED` lets workers concurrently claim *different* items with no blocking and no double-processing. `NOWAIT` solves "I have a fallback if I can't get this lock immediately — don't make me wait." Together they make InnoDB row locks a primitive for **concurrent task distribution**, not just isolation.

**Tradeoffs & alternatives.** `SKIP LOCKED` trades **completeness for non-blocking**: you get *available* rows, not *all* matching rows — so it's only correct when "skip what's claimed" is the desired semantics (queues, yes; "lock all matching rows for a consistent operation," no — skipping would be wrong). `NOWAIT` trades **waiting for fast failure** — good when you have a fallback, bad if you actually needed those rows (you must handle the failure). The alternative to a DB-table queue (with `SKIP LOCKED`) is a dedicated **message queue** (Kafka, SQS, RabbitMQ) — purpose-built for high-throughput work distribution, better at very large scale, but more infrastructure; a `SKIP LOCKED` table queue is simpler, transactional (the claim and the work can be in one transaction), and fine for moderate scale — a common pattern for outbox processing, settlement queues, and job tables (M16). The tradeoff is operational simplicity (DB queue) vs scale/throughput (dedicated MQ).

**Generics / first-principles.** "A lock can mean 'wait your turn' *or* 'this is claimed, find another' — and the latter turns locks into a work-distribution primitive." The `SKIP LOCKED` pattern is a specific instance of *non-blocking work-stealing / claim-based distribution*: each worker atomically claims available work and skips claimed work, so N workers process a shared queue without coordination overhead or double-processing. This is the universal "concurrent queue consumer" pattern (the same idea as `try_lock` + move-on, or atomic claim in any work queue). The transferable insight: *blocking is not the only response to contention — "skip and find other work" enables lock-free-feeling concurrent distribution over a shared resource*. Recognizing when you want "wait" (mutual exclusion on the same item) vs "skip" (distribute different items) is the key design choice.

**MySQL-specific reality.** InnoDB (8.0): `SELECT … FOR UPDATE SKIP LOCKED` and `… FOR UPDATE NOWAIT`; `innodb_lock_wait_timeout` (default 50s) caps default waiting (error 1205). The canonical use: a **job/settlement queue table** processed by multiple workers with `FOR UPDATE SKIP LOCKED` — each grabs distinct pending rows, no blocking, no double-processing, and the claim + processing + status-update can be one transaction (transactional queue semantics). For our domain: a **settlement queue** or **outbox** (M16) where many workers concurrently process pending transfers/events uses `SKIP LOCKED` so they scale horizontally without contending; a transfer that needs a *specific* account but has a retry fallback might use `NOWAIT`. This makes InnoDB a competent transactional queue (M12/M16 outbox pattern) without a separate message broker for moderate scale. `SKIP LOCKED` is a high-value, frequently-asked pattern for queue/worker designs.

## 8.13 · Metadata locks (MDL): the DDL-blocks-everything trap

**Mental model.** Beyond row and table data locks, there's a *separate* lock on a table's **structure (metadata)**: every query takes a **shared metadata lock (MDL)** on the tables it touches — held for the *whole transaction* — to guarantee the table's schema doesn't change mid-query. A **DDL** statement (`ALTER TABLE`, etc.) needs an **exclusive MDL**. The trap: a long-running query (or a forgotten open transaction, M07/7.15) holds its shared MDL, so the `ALTER` **waits** for the exclusive MDL — and while the `ALTER` is waiting, **every *new* query on that table queues *behind* the `ALTER`** (they can't get a shared MDL because an exclusive one is pending). Result: one slow query + one DDL = **the whole table becomes unqueryable** — a classic, severe, surprising outage.

**How it actually works.** MDL is a table-level lock on the *schema/definition*, separate from data locks (rows, 8.7) and the InnoDB intention locks (8.5). Rules: a normal query (SELECT/DML) takes a **shared MDL** on each table it references, held until the transaction commits (so the schema is stable for the query's lifetime). A DDL (`ALTER TABLE`) needs an **exclusive MDL** (it's changing the schema). Shared MDLs are compatible (many queries coexist); an exclusive MDL conflicts with *all* of them. The outage sequence: (1) a long query (or open transaction) holds a shared MDL on `account`; (2) someone runs `ALTER TABLE account …` — it needs the exclusive MDL, so it **waits** behind the long query's shared MDL; (3) now *new* queries on `account` arrive — they need a shared MDL, but MySQL **queues them behind the pending exclusive request** (to avoid starving the DDL); (4) so every new query on `account` is **blocked** until the long query finishes *and* the ALTER completes — even simple SELECTs that would've taken microseconds. The table is effectively frozen. The fix: don't run DDL while long transactions/queries are open; kill the blocking query/transaction; use `lock_wait_timeout` (caps how long the DDL waits) and online-DDL tools (gh-ost/pt-osc, M13) that avoid the long exclusive-lock window.

**Why it exists / what it solves.** MDL exists to prevent a query from running against a schema that changes underneath it (e.g., a SELECT referencing a column that an ALTER drops mid-query → corruption/crash). It's a *correctness* mechanism: the shared MDL guarantees the table definition is stable for the query's duration. The problem isn't MDL itself (it's necessary) — it's the *interaction* with long-held shared MDLs and the queue-behind-pending-exclusive behavior, which turns a long query + a DDL into a table-wide stall. Understanding MDL solves one of the most confusing production incidents ("why did every query on this table suddenly hang when I ran a quick ALTER?") — the answer is the MDL queue, not the data.

**Tradeoffs & alternatives.** MDL's correctness (stable schema per query) is non-negotiable, so the tradeoff is operational: DDL must be coordinated with the query workload. Mitigations: **`lock_wait_timeout`** (the DDL gives up after N seconds rather than holding the table hostage indefinitely while it waits — so a blocked ALTER fails instead of causing a cascading stall); **kill the blocking query/transaction** before/during the DDL; **run DDL in low-traffic windows**; and crucially **online schema-change tools** (gh-ost, pt-online-schema-change, M13) that build a shadow table and swap it in, minimizing the exclusive-MDL window to a brief atomic rename instead of a long lock. Native **online DDL** (M03/3.15) also reduces the window for many operations. The alternative — running blocking DDL on a busy table without these mitigations — is the outage. So the design stance: treat DDL on busy tables as a careful operation (online tools, timeouts, low-traffic windows), never a casual one.

**Generics / first-principles.** "A shared lock held long, plus a pending exclusive request, starves *all* new shared requests — because granting them would starve the writer." The MDL trap is a specific instance of the universal **reader-writer lock fairness/starvation** problem: to prevent writer starvation, a pending exclusive (writer) request typically blocks *new* shared (reader) requests — which means one slow reader + one writer freezes all new readers. This dynamic appears in every reader-writer lock (the "writer-preference" starvation), in RW mutexes, in filesystem locks. The transferable insight: *a long-held shared lock combined with a waiting exclusive request can block everything, because the system blocks new shared requests to let the writer through eventually* — so long-held shared locks are dangerous precisely when an exclusive request might arrive. Recognizing "this is the reader-writer starvation pattern" explains the MDL outage (and many others).

**MySQL-specific reality.** MySQL's **metadata locks** (table-definition locks, separate from InnoDB row/data locks) are taken by every statement (shared) and DDL (exclusive), held until transaction end. The canonical incident: a long query or **idle-in-transaction** connection (M07/7.15, M04/4.2) holds a shared MDL; an `ALTER TABLE` waits for the exclusive MDL; **all new queries on that table queue behind the ALTER** → table-wide stall. Diagnose with `performance_schema.metadata_locks` (who holds/waits) and `SHOW PROCESSLIST` (queries in "Waiting for metadata lock"). Mitigations: `lock_wait_timeout` (the DDL's MDL wait timeout — keep it short so a blocked ALTER fails fast rather than stalling the table), kill the blocking transaction, gh-ost/pt-osc for online schema changes (M13). This was flagged in M07's skeleton as a critical internal, and it's a top operational gotcha and interview topic. For our domain: an `ALTER TABLE ledger_entry …` (a huge, busy table) run while a long reconciliation query holds a shared MDL would stall *all* ledger queries — which is why ledger schema changes use online tools (M13) and are coordinated against long transactions.

## 8.14 · How MVCC + locks deliver each isolation level

**Mental model.** This concept **closes the loop with M07**: it explains, in terms of the *mechanism* (read-view timing + lock types), *how* each isolation level's guarantees (the M07 contract) are actually delivered. The pattern: the level is determined by **(a) when the read view is created** (8.3 — once per transaction vs once per statement) and **(b) which lock types are taken** (8.7 — record-only vs next-key/gap). READ COMMITTED = fresh read view per statement + record locks (no gaps) → sees latest committed, no phantom prevention. REPEATABLE READ = one read view for the transaction + next-key locks → stable snapshot, phantoms prevented. SERIALIZABLE = locking reads (even plain SELECTs lock) → full serialization. The M07 levels were the *what*; this is the *how*, in one unifying frame.

**How it actually works (mechanism per level):**
- **READ UNCOMMITTED:** reads don't even use a proper snapshot (reads latest, including uncommitted-ish data) → dirty reads. Rarely used.
- **READ COMMITTED:** a **new read view per statement** (8.3) → each statement sees the latest committed data (hence *non-repeatable reads*, M07/7.9 — re-reads see newer commits). Locking: **record locks only, gap locks mostly disabled** (8.7) → *phantoms possible* (new rows can be inserted into ranges), but **less locking → fewer deadlocks, more concurrency** (M07/7.13b).
- **REPEATABLE READ (InnoDB default):** **one read view, created at first read, reused** (8.3) → stable, repeatable snapshot (no non-repeatable reads). Locking: **next-key locks** (record + gap, 8.7) → *phantoms prevented* (no inserts into locked ranges) — stronger than the standard's RR (M07/7.13b). More locking (gap locks) → more deadlock potential.
- **SERIALIZABLE:** like RR, but **plain `SELECT`s become locking reads** (take shared next-key locks) → reads conflict with writes → full serialization (no anomalies, including write skew, M07/7.13). Most locking, least concurrency.
So: **read-view timing controls the read anomalies (dirty/non-repeatable), lock type controls phantoms, and SERIALIZABLE adds read locking for full serialization.** The M07 anomaly matrix (7.10) is *explained* by these two mechanism dials.

**Why it exists / what it solves.** This concept makes the M07 isolation levels *concrete and predictable* instead of abstract definitions. Once you see that "RR vs RC" = "one read view + next-key locks vs per-statement read views + record-only locks," you can *predict* behavior (RR prevents phantoms because of gap locks; RC has fewer deadlocks because no gap locks) rather than memorizing the matrix. It also explains the *costs* (RR's gap locks → more deadlocks; SERIALIZABLE's read locks → low concurrency) in mechanism terms, so you can reason about the concurrency tradeoff of each level (M07/7.14) from first principles. It's the synthesis that ties M08's mechanism back to M07's contract.

**Tradeoffs & alternatives.** The mechanism reveals *why* each level has its tradeoff (M07/7.14): RC's per-statement read views + no gap locks = freshest data, least locking, most concurrency, but phantoms and non-repeatable reads; RR's single read view + next-key locks = stable snapshot + no phantoms, but more locking/deadlocks; SERIALIZABLE's read locks = full correctness but serialized throughput. The choice of level (M07/7.14) is thus a choice of *mechanism cost*: gap locks (deadlocks) for phantom prevention, read locks (blocking) for serializability. Understanding the mechanism lets you make this tradeoff knowingly — e.g., choosing RC to *eliminate gap-lock deadlocks* on a high-write workload (accepting phantoms), a deliberate mechanism-driven decision (M07/7.13b).

**Generics / first-principles.** "Isolation guarantees decompose into two orthogonal mechanism dials: *snapshot timing* (which controls what you read) and *lock scope* (which controls what others can change) — and every level is a setting of these two." This decomposition is the deep insight: the seemingly-discrete isolation levels are really *combinations* of independent mechanisms (versioning/snapshot policy + locking policy). The transferable insight: *complex behavioral guarantees often decompose into a few orthogonal mechanisms — find them, and the guarantees become predictable rather than memorized*. Snapshot-timing + lock-scope is the database instance; the general skill is reducing a taxonomy of behaviors to its underlying mechanism dials.

**MySQL-specific reality.** InnoDB's per-level mechanism: **RC** = per-statement read view + record locks (no gaps) → fewer deadlocks, phantoms possible; **RR (default)** = first-read read view reused + next-key locks → phantom-free, more gap-lock deadlocks; **SERIALIZABLE** = plain SELECTs become `FOR SHARE` (locking reads) → full serialization. This is *the* explanation of M07/7.13b's "InnoDB RR prevents phantoms (next-key locks)" and of why some shops run RC (to avoid gap-lock deadlocks). You verify behavior via `performance_schema.data_locks` (see record vs gap vs next-key being taken at each level). For our domain: the default RR gives money reads a stable, phantom-free snapshot (good for consistent reconciliation reads); if gap-lock deadlocks become a problem on a high-write path, switching that workload to RC (record locks only) is a mechanism-informed mitigation (accepting phantoms, M07/7.13b). This concept is the bridge that makes M07's contract and M08's mechanism *one* coherent picture.

## 8.15 · Hot-row & hot-account contention (the fintech problem) ★

**Mental model.** *The* fintech concurrency bottleneck: when **many transactions all need to lock the *same* row** — a popular merchant account's balance, a global counter, a single "sequence" row — they **serialize** on that row's exclusive lock (held to commit, 8.10), so they process *one at a time* no matter how many cores or how much hardware you throw at it. Row-level locking gives great concurrency *across* different rows (8.5), but a **hot row** is a serialization point that caps throughput at "one transaction's lock-hold-time at a time." A merchant taking thousands of payments a second, all updating *one* balance row, hits this wall. Relieving hot-row contention is a core fintech scaling problem with well-known patterns.

**How it actually works.** Every transfer to account X takes an **X lock on X's balance row** (to update it) and **holds it until commit** (strict 2PL, 8.10). So concurrent transfers to *the same* account form a queue on that one lock — transfer 2 waits for transfer 1 to commit, transfer 3 for 2, etc. Throughput on that account = 1 / (lock-hold time per transfer), regardless of parallelism. (Different accounts don't contend — only the *same* hot account does, 8.5.) The mitigation patterns:
- **Keep the transaction short** (8.10, M07/7.15) — minimize lock-hold time so the queue drains faster (the cheapest, most important fix — no external calls inside the transaction).
- **Use atomic updates** (`balance = balance + :delta`, M07/7.11) instead of `FOR UPDATE` + read + write — the atomic update holds the lock for a shorter window (just the update, not a round-trip).
- **Shard the balance / counter** — split the hot account's balance into N sub-balance rows (e.g., per-shard), so transfers hash to *different* rows (spreading the contention across N locks); the true balance is the sum (reconciled, M02/2.17). The classic "counter sharding" pattern.
- **Batch / queue** — instead of each transfer updating the balance synchronously, append entries to the immutable ledger (no hot-row contention on appends, 8.9) and update the balance **asynchronously in batches** (one balance update per N entries) or via a **queue** (8.12, `SKIP LOCKED`) — moving the balance off the critical path (the M02/2.17 derived-balance pattern, reconciled).
- **Async settlement** — accept the entry immediately (append-only, fast), settle the balance later (eventual consistency for the balance, with the ledger as truth).

**Why it exists / what it solves.** Hot-row contention is the *inevitable* limit of row-level locking when load concentrates on one row — and load *does* concentrate (popular merchants, global counters, a single "platform" account). It's *the* reason a naive "every transfer synchronously updates the balance row" design hits a throughput ceiling at scale, no matter the hardware. The mitigation patterns solve "how do we process thousands of operations on one logical account per second when they all want to lock one row?" — by *not* making them all lock the same row (sharding the counter, async/batched balance updates, append-only ledger). This is the concurrency problem M16 (payments at scale) must solve, and it's where M08's mechanism meets fintech reality.

**Tradeoffs & alternatives.** The mitigations trade **simplicity/strong-consistency for throughput**: a synchronous single-balance-row update is simplest and always-consistent but doesn't scale on a hot account; counter sharding scales but complicates reads (must sum shards) and writes (hash to a shard); async/batched balance updates scale and keep appends fast but make the *balance* eventually consistent (the ledger is the source of truth; the balance lags — fine if you read authoritative balance from the ledger when needed, M02/2.17). Append-only ledger + derived balance (M02/2.17) is the canonical fintech resolution: **appends don't contend** (different rows, 8.9), the immutable ledger is always consistent, and the balance is a reconciled projection that can be updated async or sharded. The tradeoff is the M02/2.17 one (derived data + reconciliation) applied to dodge hot-row contention.

**Generics / first-principles.** "A single shared mutable location is a serialization point that caps throughput regardless of parallelism — relieve it by spreading the contention (sharding the counter), removing it from the hot path (append + async aggregate), or shortening the critical section." Hot-row contention is the database instance of the universal **contended shared resource / hot-spot** problem (a contended cache line, a single global mutex, a hot partition key) — and the solutions are universal: **partition/shard** the resource (spread load across N independent copies), **append instead of update** (CRDTs, event logs, LSM-trees — appends don't contend), or **batch** (amortize the contention). The transferable insight: *any single point all writers must touch is a scalability ceiling — design to avoid funneling load through one mutable location*. Recognizing "this is a hot-spot serialization point" and reaching for sharding/append/batch is a core scalability skill.

**MySQL-specific reality.** In InnoDB, hot-row contention = many transactions queuing on one row's X lock (held to commit, 8.10) — visible as lock waits in `performance_schema.data_lock_waits` and high lock-wait time on a specific row. The mitigations in MySQL terms: **short transactions** + **atomic updates** (M07/7.11/7.16) to minimize lock-hold time; **counter sharding** (the balance split into N rows, summed for the true value) for extreme hot accounts; the **append-only ledger + derived balance** (M02/2.17) where entries append without contention and the balance is updated in batch / reconciled; **queue/async settlement** (8.12, `SKIP LOCKED`) to move balance updates off the synchronous path. For our domain: a hot merchant account taking thousands of payments/second can't synchronously update one balance row at that rate — so the design appends entries (fast, no hot-row contention) and maintains the balance via batched/async aggregation or counter sharding, with the ledger as the source of truth (M02/2.17, M16). This is *the* fintech concurrency scaling problem, and it sets up M16 directly. The ★ SVG (Pass C) draws the hot-row queue and the mitigation patterns.

## 8.16 · Fintech capstone — concurrency-correct money movement at scale ★

**Mental model.** The capstone re-examines the **atomic transfer (M07/7.16) through the lens of its *lock footprint*** — what locks it actually takes, *where* it contends, *how* it can deadlock, and *how* to scale the hot path — synthesizing the whole module. The transfer that M07 made *correct* (atomic, isolated, durable, idempotent), M08 makes *concurrent and scalable*: it shows the precise row/next-key locks the transfer acquires, the deadlock risk (two transfers in opposite order, 8.11) and its consistent-ordering fix, the hot-account contention point (8.15) and its mitigations, and how MVCC lets reconciliation reads run concurrently without blocking the transfers. It's "the atomic transfer, now under real concurrency" — the mechanism that makes money movement both correct *and* fast.

**How it actually works — the transfer's concurrency profile:**
1. **Lock footprint:** the transfer takes an **X record lock** on each account's balance row it updates (PK lookup → tight single-row record lock, 8.8 — *not* a range, because PK equality), an **X lock** on each inserted `ledger_entry` (and insert-intention locks in the index gaps, 8.9), and a **record lock** via the UNIQUE idempotency key (8.7 — preventing a duplicate insert, the dedup mechanism). Minimal, precise footprint *because* the predicates are indexed (8.8) — a missing index would explode it.
2. **Deadlock risk + fix (8.11):** two concurrent transfers between the *same two* accounts in opposite directions can deadlock (each locks its *from* account first → cycle). **Fix: consistent lock ordering** — always lock accounts in ascending `account_id` order regardless of transfer direction, so no cycle can form; plus **retry on deadlock (1213)**.
3. **Hot-account contention (8.15):** transfers to one popular account serialize on its balance-row X lock (held to commit, 8.10). **Mitigations:** short transaction (M07/7.15, external call *outside*), atomic update (`balance = balance + :delta`, M07/7.11 — short lock-hold), and at extreme scale, **append-only ledger + async/sharded balance** (M02/2.17) so appends don't contend.
4. **MVCC concurrency (8.1):** a concurrent reconciliation read (plain SELECT) sees a consistent snapshot *without blocking* the transfers (and without being blocked) — true read/write concurrency.
5. **Isolation level (8.14):** default RR gives the transfer phantom-free reads (next-key locks); the read-modify-write safety comes from the atomic update / `FOR UPDATE`, not the level alone (M07/7.11).

**Why it exists / what it solves.** The capstone proves that money movement can be *both correct and scalable* — M07 gave correctness (ACID + idempotency), M08 gives the concurrency mechanism (precise locks, deadlock avoidance, hot-account mitigation, non-blocking reads). It answers the full question: "how does a payments platform process thousands of concurrent transfers correctly, without deadlocking, without one hot account capping throughput, while reconciliation reads run concurrently?" — the synthesis of everything in M07 and M08, and the direct setup for M16 (the platform at scale). It's where the abstract mechanism becomes the concrete, scalable, correct heart of the system.

**Tradeoffs & alternatives.** The capstone embodies the module's tradeoffs: **atomic update vs `FOR UPDATE`** (shorter lock-hold vs explicit read-decide-write, M07/7.11); **consistent lock ordering** (deadlock prevention, requires discipline, 8.11); **synchronous balance vs append-only + async** (simplicity/strong-consistency vs hot-account throughput, 8.15/M02·2.17); **RR vs RC** (phantom prevention vs fewer deadlocks, 8.14/M07·7.13b). The failure alternatives are the bugs: read-then-write balance (lost update, M07/7.11), inconsistent lock order (deadlocks, 8.11), unindexed locking predicate (table-wide lock, 8.8), synchronous single-balance-row at scale (hot-account ceiling, 8.15), long transaction (lock-hold + MDL/undo bloat, 8.10/8.13/M07·7.15). The capstone threads between them by *mechanism understanding*, not luck.

**Generics / first-principles.** "Correct concurrent operations on shared state require: minimal precise locking (index your predicates), consistent lock ordering (no deadlock cycles), short critical sections (held to commit), non-blocking reads (versioning), and avoiding hot-spots (spread/append/batch)." The capstone synthesizes M08's threads — generics-first (these are universal concurrency principles, not MySQL-specific), money-never-lies (the locks make the transfer correct under concurrency), tradeoff (locking granularity/strategy vs throughput). The transferable recipe for *any* high-concurrency correctness-critical system: **lock precisely and minimally, in a consistent order, for a short time, with non-blocking reads, avoiding shared hot-spots** — the same whether it's a database transaction, a concurrent data structure, or a distributed system. This is the concurrency-engineering core.

**MySQL-specific reality.** The concrete InnoDB realization: the transfer takes tight **X record locks** on the PK-looked-up balance rows (8.8), insert-intention + record locks on the new entries and the idempotency key (8.7/8.9), held to commit (8.10); **consistent `account_id` ordering** + **retry on 1213** for deadlock safety (8.11); **atomic `balance = balance + :delta`** for short lock-hold (M07/7.11); **append-only ledger + derived balance** (M02/2.17) to dodge hot-account contention (8.15); plain SELECTs for non-blocking reconciliation (MVCC, 8.1); default RR (8.14). Verified via `performance_schema.data_locks`/`data_lock_waits` (the transfer's actual locks) and `SHOW ENGINE INNODB STATUS` (deadlocks). This is the exact concurrency design **M16** scales across shards (keeping a transfer's two accounts on one shard so it stays a single local transaction with local locks, M02/2.16, M11) — and getting the locking right (precise, ordered, short, hot-spot-aware) is what makes a payments platform both correct *and* able to handle real concurrent load.
