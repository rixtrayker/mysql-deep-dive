# M08 · Locking & MVCC ★ — Pass A (Skeleton)

> **Pass A goal:** lock the ordered concept list, a one-line mental model per concept, and the planned diagram + bespoke-SVG inventory + worked example per concept. No prose yet — that's Pass B.
>
> **Module mental model:** *M07 told you* what *isolation guarantees (the contract); M08 is* how *InnoDB delivers it. Two mechanisms, working together: **MVCC** (multi-version concurrency control) lets readers see a consistent* snapshot *of the past without blocking writers — every row keeps a chain of old versions, and each transaction reads the version valid for its snapshot; and **locking** protects writes (and reads that must block) — InnoDB takes fine-grained row and gap locks so that conflicting writers serialize while everyone else proceeds. "Readers don't block writers, writers don't block readers" (MVCC) plus "writers block conflicting writers" (locks) is the engine that makes REPEATABLE READ both correct and concurrent. Understanding the* mechanism *is what lets you reason about contention, deadlocks, the lost-update fix, and why InnoDB's RR prevents phantoms — and it's the practical core of making a money system fast* and *correct under concurrency.*
>
> **Threads carried in this module:**
> - **Generics-first** — MVCC, the shared/exclusive lock taxonomy, two-phase locking, and deadlock detection are universal concurrency-control theory (decades of database research); InnoDB is one realization with specific lock types (next-key, gap, intention).
> - **Money-never-lies** — the locks *are* what make the atomic transfer (M07/7.16) correct under concurrency: row locks serialize the hot-account balance update (no lost update, 7.11), next-key locks prevent phantoms, `FOR UPDATE`/`SKIP LOCKED` enable safe queue processing. Hot-account contention is *the* fintech concurrency problem.
> - **Tradeoff** — lock *granularity* (row vs gap vs table) and *strategy* (pessimistic locks vs MVCC reads) trade concurrency against safety/overhead; the whole module is choosing the right amount of locking.
> - **Durability** — the undo logs that power MVCC version chains are the *same* undo that powers atomicity/rollback (M07/7.2) and that the purge thread / history list manage (preview M09).
>
> **Prereqs:** M07 (the isolation contract — levels 7.8, anomalies 7.9, lost update 7.11, write skew 7.13, InnoDB's RR + next-key locks 7.13b; the atomic transfer 7.16), M01/1.3 (clustered PK + secondary indexes — locks are on *index records*), M05 (indexes — *which* index a query uses determines *what* gets locked). **Leads into:** M09 (InnoDB internals — undo logs/history list/purge, the buffer pool these structures live in), M15 (deadlock/contention failure modes), M16 (hot-account contention at scale, queue processing).

| 8.2 | **MVCC: multi-version concurrency control** ★ | Every row keeps a *chain of old versions* (in undo logs); each transaction reads the version valid as of its snapshot — so readers see a consistent past without locking. ★ custom SVG | ★ MVCC version chain (row → undo versions, txn ids) | Two transactions reading the same row see different versions consistently |

| 8.4 | **Consistent reads vs locking reads** | A plain `SELECT` is a non-blocking snapshot read (MVCC); `SELECT … FOR UPDATE/FOR SHARE` is a *locking* read that sees the latest committed data and takes locks — two fundamentally different read modes. | Plain SELECT (snapshot) vs locking SELECT (current + lock) | When the transfer must use a locking read (read-decide-write) vs a plain read |
| 8.5 | **Lock granularity: row vs table (and intention locks)** | InnoDB locks *rows* (fine-grained → high concurrency), and uses *intention locks* at the table level to coordinate row locks with occasional table locks — the hierarchy that makes row locking efficient. | Lock hierarchy: table (intention) → row | Why row-level locking lets transfers to different accounts proceed in parallel (vs MyISAM table locks) |
| 8.6 | **Shared vs exclusive locks (the compatibility matrix)** | Two lock modes — shared (read, many can hold) and exclusive (write, only one) — and the rules for which can coexist; the foundation of all locking. ★ | The S/X lock-compatibility matrix | Two `FOR SHARE` reads coexist; a `FOR UPDATE` blocks them |
| 8.7 | **Record, gap & next-key locks** ★ | InnoDB locks *index records*, the *gaps* between them, or both (next-key = record + gap) — and gap locking is *how* InnoDB prevents phantoms at REPEATABLE READ. ★ | ★ record vs gap vs next-key on an index | Why a range `FOR UPDATE` locks the gap, blocking an insert that would be a phantom (7.13b) |
| 8.8 | **Locks are on index records (which index matters)** | InnoDB locks the *index records* a query touches — so the index the optimizer chooses determines *what* gets locked; a missing/wrong index can lock far more rows than expected. | Same query, different index → different locked set | A `WHERE` with no index locks every row scanned (not just matches) — a contention trap |
| 8.9 | **Insert-intention & auto-increment locks** | Special locks for inserts: insert-intention (a gap lock that lets concurrent inserts into the same gap coexist) and auto-inc (serializing auto-increment value assignment) — niche but contention-relevant. | Insert-intention in a gap; auto-inc lock modes | Why concurrent inserts to one table contend on the auto-inc lock; the lock modes that mitigate it |
| 8.10 | **Two-phase locking & lock lifetime** | InnoDB acquires locks as needed and holds them until commit/rollback (strict two-phase locking) — so lock duration = transaction duration, which is why short transactions matter (M07/7.15). | Locks acquired during txn, all released at commit | A long transaction holding a row lock blocks every transfer to that account until it commits |
| 8.11 | **Deadlocks: detection, victim selection, avoidance** ★ | Two transactions each holding a lock the other needs → a cycle that can never resolve; InnoDB *detects* the cycle and *aborts a victim* — and you design to avoid them. ★ | ★ deadlock cycle (waits-for graph) + resolution | Two transfers locking accounts in opposite orders deadlock; the consistent-ordering fix |
| 8.12 | **Lock waits, timeouts & SKIP LOCKED / NOWAIT** | When a lock is held, a waiter blocks (up to `innodb_lock_wait_timeout`); `SKIP LOCKED` (skip locked rows) and `NOWAIT` (fail fast) change the waiting behavior — essential for queue-style processing. | Wait / timeout / SKIP LOCKED / NOWAIT behaviors | Pulling jobs from a queue table with `FOR UPDATE SKIP LOCKED` so workers don't block each other |
| 8.13 | **Metadata locks (MDL): the DDL-blocks-everything trap** | Every query takes a metadata lock on its tables; a DDL needs an exclusive MDL, so a long-running query (or open transaction) can make an `ALTER TABLE` block — and that DDL then blocks *all* new queries. | MDL queue: query holds shared MDL → DDL waits exclusive → new queries pile up | A reporting query holds an MDL; an ALTER waits; every new query stalls behind it — a classic outage |
| 8.14 | **How MVCC + locks deliver each isolation level** | Tie the mechanism back to the contract: READ COMMITTED (new snapshot per statement, no gap locks), REPEATABLE READ (one snapshot, next-key locks → no phantoms), SERIALIZABLE (locking reads) — the M07 levels *explained*. ★ | Mechanism per level (snapshot timing + lock types) | Why InnoDB RR prevents phantoms (next-key locks) but RC doesn't (no gap locks) — M07/7.13b explained |
| 8.15 | **Hot-row & hot-account contention (the fintech problem)** ★ | Many transactions serializing on one popular row's lock (a hot account's balance, a hot counter) is *the* fintech concurrency bottleneck — and the patterns to relieve it. ★ | Hot-row contention + mitigations (sharding the counter, queueing, batching) | Thousands of transfers to one merchant account all serializing on its balance row — and how to scale it |
| 8.16 | **Fintech capstone: concurrency-correct money movement at scale** | The atomic transfer (M07/7.16) under *real* lock contention — what locks it takes, where it contends, how to avoid deadlocks, and how to scale the hot account — the mechanism that makes money movement both correct and fast. ★ | ★ the transfer's locks + contention + mitigations (annotated) | End-to-end: the transfer's lock footprint, deadlock-avoidance, hot-account relief (synthesizes M07, sets up M16) |
