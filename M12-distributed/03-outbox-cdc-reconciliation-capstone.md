## 12.11 · The outbox pattern ★

**Mental model.** The **outbox pattern** is the standard fix for the dual-write problem (12.10): instead of writing state to the DB *and then* publishing an event (two non-atomic writes that can diverge on a crash), you **write the event into the *same database transaction* as the state change** — into a dedicated **`outbox` table** — so the state change and the event commit **atomically** (both or neither, M07–M09). A *separate* process then reliably reads the outbox and publishes the events (to Kafka, etc.), marking them sent. Because the event is part of the atomic commit, it **can't be lost** (if the state change committed, the event row committed with it) and it **can't be a phantom** (if the transaction rolled back, neither exists). The outbox turns "two writes that can disagree" into "one atomic write + reliable async publishing" — the cornerstone of reliable event-driven systems.

**How it actually works.**
1. **In the business transaction:** when the transfer commits, the *same* `BEGIN…COMMIT` also `INSERT`s a row into the **`outbox`** table describing the event ("TransferCompleted", payload, a unique event ID) — so the transfer rows *and* the outbox row are part of *one atomic InnoDB commit* (M07). They can't diverge.
2. **The relay (message relay / publisher):** a separate process reads *unpublished* outbox rows and publishes them to the message broker (Kafka), then marks them published (or deletes them). Two ways to read the outbox: **polling** (periodically `SELECT … WHERE published = false` — simple, but polling latency/load) or — better — **CDC** (12.12 — read the *binlog* for inserts into the outbox table, so publishing is driven by the change log itself, low-latency, no polling).
3. **At-least-once + idempotency:** the relay guarantees *at-least-once* publishing (if it crashes after publishing but before marking sent, it republishes on restart → a possible duplicate) — so **consumers must be idempotent** (12.9/12.13) to make the *effect* exactly-once. The outbox guarantees the event is *never lost*; idempotent consumers guarantee it's *not double-applied*.
The outbox row is essentially a **durable, transactional event log** co-located with the state — the same "log the intent atomically, process it reliably" idea as the redo log (M09) and Sagas' durable state (12.8).

**Why it exists / what it solves.** The outbox exists to solve the dual-write problem (12.10) *correctly and simply* — guaranteeing **no event is ever lost** when state changes, without distributed transactions (no XA, 12.7). It solves the exact failure that makes naive event-driven systems unreliable: a crash between "commit state" and "publish event" silently dropping the event. By making the event atomic with the state change (both in the DB transaction) and publishing asynchronously-but-reliably (relay + at-least-once + idempotent consumers), the outbox ensures that **every committed state change reliably produces its event** — the foundation of *reliable* event-driven architecture and CDC pipelines. For money, it's what guarantees a committed transfer *always* produces its "TransferCompleted" event (so settlements, notifications, fraud checks, and reconciliation never silently miss a transfer).

**Tradeoffs & alternatives.** The outbox trades a little complexity (an outbox table + a relay process + idempotent consumers) for **guaranteed-no-lost-events without distributed transactions** — an excellent trade. ✓ no dual-write data loss, ✓ no XA (no blocking/lock-holding, 12.7), ✓ events atomic with state; ✗ **eventually consistent** (the event is published *shortly after* the commit, not instantly — a small delay), ✗ **at-least-once** (consumers must be idempotent, 12.9), ✗ the outbox table needs cleanup (purge published rows) and the relay is a component to operate. Alternatives: **CDC alone** (12.12 — skip the explicit outbox table; treat the *binlog* as the event stream, capturing the actual state changes — simpler if downstream just needs the row changes; the outbox is better when you want *semantic* events, e.g., "TransferCompleted" with a curated payload, not raw row diffs); **event sourcing** (M01/1.17 — make events the source of truth); **listen-to-yourself**. The common production design *combines* outbox + CDC: write semantic events to the outbox atomically, then CDC the outbox table to Kafka (low-latency, reliable). For money, **outbox (or outbox+CDC) is the standard** — never naive dual-write.

**Generics / first-principles.** "Make the event part of the state-change transaction (write both atomically to one database), then publish asynchronously and reliably — so the event can never be lost or phantom, without needing distributed transactions." This is *the* canonical reliable-event-publishing pattern (transactional outbox), foundational to event-driven and microservices architectures. The transferable insight: **to reliably notify other systems of a state change, don't publish separately (dual-write) — record the notification *atomically with the state change* (in the same transactional store) and relay it asynchronously; the single atomic write is the source of truth, and the event is derived from it reliably.** It transfers to every event-driven system (the outbox is a named pattern in microservices, Debezium's recommended approach, etc.). The deep principle (recurring through M01/1.17, M09, 12.10, 12.12): **one atomic write as the source of truth, everything else derived reliably from it** — never two independent writes that can disagree.

**MySQL-specific reality.** The outbox is implemented as a normal InnoDB table written *in the same transaction* as the business change: the transfer's `BEGIN; … debit; credit; INSERT INTO outbox …; COMMIT` makes the event atomic with the transfer (M07–M09). The relay is either a **poller** (`SELECT FROM outbox WHERE published=0` — simple) or, preferably, **Debezium/CDC reading the binlog** (12.12) for outbox inserts (low-latency, no polling load — Debezium even has a dedicated **outbox event router**). Because each shard (M11) has its own binlog, the outbox + CDC works per-shard. For our domain: a "TransferCompleted" (or "PaymentAuthorized", "TransferReversed") event is `INSERT`ed into the **outbox in the transfer's single-shard transaction** (M11/11.9 — atomic with the debit/credit/ledger-entry); a CDC/relay process publishes it to Kafka for fraud detection, notifications, the reporting warehouse, and reconciliation (12.14) — *guaranteed never lost* because it committed atomically with the transfer. Idempotent consumers (12.13) handle the at-least-once republish. This is the backbone of the platform's reliable event propagation (M16). The ★ SVG (Pass C) draws one transaction writing state + outbox row, then the relay/CDC publishing.

## 12.12 · Change Data Capture (CDC) ★

**Mental model.** **Change Data Capture (CDC)** turns a database's **change log** (MySQL's **binary log**, M10/10.2/10.14) into a **stream of change events** that other systems consume — so instead of dual-writing (12.10) or having systems query the DB, you *capture every committed change from the binlog* and publish it (via **Debezium → Kafka**) to whoever needs it (search indexes, caches, analytics, fraud detection, other services). The database's own replication log — already a reliable, ordered, durable record of every change (the same log replicas consume, M10) — becomes the **integration backbone**. CDC is how you propagate data across a system *reliably and without dual-writes*: the change is captured *because it committed* (it's in the binlog), so nothing is lost, and consumers get an ordered stream of exactly what changed. CDC + the outbox pattern (12.11) together are the standard reliable-integration toolkit.

**How it actually works.** A CDC connector (e.g., **Debezium**) acts as a **replica** (M10) — it connects to MySQL and **streams the binlog** (in **ROW format**, M10/10.3, so it gets exact before/after row images), parsing each committed change into a structured **change event** ("table X, row Y, INSERT/UPDATE/DELETE, before=…, after=…"). It publishes these events to **Kafka** (one topic per table, typically), in commit order. Consumers subscribe: a **search indexer** updates Elasticsearch on each change; a **cache invalidator** evicts changed keys; an **analytics pipeline** loads the warehouse; a **fraud service** inspects transactions; an **outbox relay** (12.11) publishes semantic events. Key properties: **reliable** (every committed change is in the binlog → captured; Debezium tracks its binlog position/GTID, M10/10.7, so it resumes exactly where it left off after a restart — no lost or skipped changes); **ordered** (binlog order = commit order); **low-latency** (streams as changes commit); **non-intrusive** (reads the binlog, doesn't query or load the primary). Requirements: **ROW binlog format** (M10/10.3 — for exact images) and **durable binlog** (`sync_binlog=1`, M09/9.10).

**Why it exists / what it solves.** CDC exists to solve **reliable data propagation/integration without dual-writes** (12.10) and **without coupling systems via synchronous queries**. It solves: (1) the **dual-write problem** (don't write to DB *and* publish — let the committed change *be* the event, captured from the binlog → no lost events, no phantom events); (2) **system integration** (keep a search index, cache, warehouse, or downstream service in sync with the DB *automatically* — they consume the change stream rather than being written-to directly); (3) **decoupling** (producers don't know their consumers — they just commit to the DB; CDC fans the changes out). It reuses the *existing* replication log (M10) — the database already produces a reliable change log for replicas, so CDC just adds *more* consumers of that same log. For money: CDC streams the ledger's changes to fraud detection (real-time), the search index, the reporting warehouse (M02/2.17), and the outbox relay — reliably, in order, without dual-writes.

**Tradeoffs & alternatives.** CDC trades a little infrastructure (a connector like Debezium + Kafka) for **reliable, low-latency, decoupled, non-intrusive data propagation**. ✓ no dual-writes (captures committed changes), ✓ reliable + ordered + resumable (binlog-based), ✓ low-latency, ✓ decoupled (producers unaware of consumers), ✓ non-intrusive (reads binlog, no primary load); ✗ **eventually consistent** (consumers lag the source by the CDC pipeline latency), ✗ **operational complexity** (Debezium + Kafka to run, M13), ✗ **schema-change handling** (DDL on the source must be propagated/handled), ✗ raw CDC gives *row changes* not *semantic events* (use the outbox, 12.11, when consumers need curated "TransferCompleted"-style events, not raw row diffs). Alternatives: **dual-write** (broken, 12.10 — don't); **synchronous queries** (consumers query the DB — coupling + load); **outbox-only with polling** (12.11 — simpler, but polling latency/load). The standard reliable design **combines outbox + CDC**: semantic events to the outbox (atomic with state), then CDC the binlog to Kafka (reliable, low-latency). CDC is *the* modern integration backbone.

**Generics / first-principles.** "The database's change log (the same one replicas use) is a reliable, ordered, durable stream of every committed change — capture it and publish it, and you get reliable data propagation and system integration without dual-writes." This is the universal **Change Data Capture** principle (Debezium, Kafka Connect, Maxwell, Postgres logical decoding, the "log as the integration backbone" / "turning the database inside-out" philosophy). The transferable insight: **every committed change is already durably recorded in the replication/change log — so that log is the most reliable possible event source; capturing and streaming it gives you integration, search/cache sync, analytics, and event-driven propagation *for free and without dual-writes*, because the events *are* the committed changes.** It transfers to every database with a change log (MySQL binlog, Postgres WAL/logical decoding, MongoDB change streams, DynamoDB streams). The deep principle (the module's recurring theme, M01/1.17, M09, M10, 12.11): **the log is the source of truth, and everything — replicas, backups (M13), search indexes, caches, analytics, downstream services — is a derived consumer of that one ordered log.** "The log is primary; derive the rest."

**MySQL-specific reality.** CDC on MySQL = **Debezium** (the dominant connector) reading the **binlog** (M10/10.2) as a replica, requiring **`binlog_format=ROW`** (M10/10.3 — for exact before/after images) and a durable binlog (`sync_binlog=1`, M09/9.10), tracking its position via **GTIDs** (M10/10.7 — so it resumes exactly, no loss/skip). It publishes to **Kafka** (via Kafka Connect), one topic per table, in commit order. Each **shard** (M11) has its own binlog, so CDC runs per-shard (and consumers see a per-shard stream). Debezium's **outbox event router** integrates the outbox pattern (12.11 — route outbox-table inserts to semantic event topics). For our domain: the **ledger's binlog** is streamed via Debezium → Kafka to: **fraud detection** (inspect each transaction in near-real-time), the **search index** (find transactions/accounts), the **reporting/reconciliation warehouse** (M02/2.17 — keep analytics in sync without loading the primary), and the **outbox relay** (publish semantic "TransferCompleted" events, 12.11). All reliable (binlog-sourced, GTID-tracked), ordered, low-latency, and *without dual-writes*. CDC also powers **resharding** (M11/11.14 — VReplication is binlog-based) and **PITR/backups** (M13). One binlog, many derived consumers — the integration backbone of the whole platform (M16). The ★ SVG (Pass C) draws binlog → Debezium → Kafka → consumers.

## 12.13 · Idempotent consumers & exactly-once (the truth)

**Mental model.** "**Exactly-once delivery**" is a **myth** in distributed messaging — it's *provably* unachievable in the general case (the network can always lose the acknowledgment, forcing a choice between redeliver-maybe-twice or don't-redeliver-maybe-zero). What you *can* have is **at-least-once delivery** (the message is delivered one-or-more times — never lost, but possibly duplicated) combined with **idempotent processing** (12.9 — duplicate deliveries have no extra effect), which yields **exactly-once *effect*** (the message's *outcome* happens exactly once, even though *delivery* happened one-or-more times). So the correct mental model is: don't chase exactly-once *delivery* (impossible); achieve exactly-once *effect* via at-least-once delivery + idempotent consumers. This corrects a pervasive misconception and ties the whole module's reliability story together (it's idempotency, 12.9, again — the load-bearing primitive).

**How it actually works.** Messaging systems (Kafka, queues, the CDC/outbox pipeline, 12.11/12.12) offer delivery guarantees: **at-most-once** (deliver, don't retry — fast, but *loses* messages on failure → unacceptable for money), **at-least-once** (deliver, retry until acknowledged — *never loses*, but a lost *ack* causes redelivery → *duplicates*), and the *illusion* of **exactly-once** (some systems offer "exactly-once semantics" *within their boundary* — e.g., Kafka transactions across Kafka topics — but this doesn't extend to *external effects* like charging a card or updating a different database). For real systems with external effects, you use **at-least-once delivery** (never lose the message) and make the **consumer idempotent** (12.9): the consumer records which messages it has already processed (by an idempotency key / the message's unique ID) and *skips duplicates* — so processing a message twice has the same effect as once. Result: **exactly-once effect**. The consumer's idempotency is *the* mechanism (the broker can't guarantee it for you when there are external side effects).

**Why it exists / what it solves.** This concept exists to *correct the "exactly-once delivery" misconception* and replace it with the achievable, correct model. Teams chase exactly-once delivery (configuring brokers, hoping the infrastructure guarantees no duplicates), get burned (duplicates still happen, or they sacrifice availability for at-most-once and lose messages), and don't realize the *correct* approach is at-least-once + idempotency. It solves the *reliability of message processing* honestly: accept that delivery can duplicate (at-least-once, so nothing is lost), and make processing tolerate duplicates (idempotent, so nothing is double-applied) → exactly-once effect. For money, this is critical: a CDC event ("credit account") or a payment message *will* sometimes be delivered twice (a relay restart, 12.11; a consumer rebalance) — and *only* idempotent processing prevents the duplicate from double-crediting. The lesson: **engineer for at-least-once + idempotency, not for exactly-once delivery.**

**Tradeoffs & alternatives.** The choice among delivery guarantees: **at-most-once** (no duplicates, but *loses* messages — unacceptable for money); **at-least-once** (never loses, but duplicates — *the* choice for money, paired with idempotency); **"exactly-once" (broker-scoped)** (e.g., Kafka transactions — works *within* Kafka, but doesn't cover external side effects like DB writes or charges, so you *still* need idempotency at the effect boundary). The only correct option for money is **at-least-once + idempotent consumers** (12.9) = exactly-once effect — there's no real alternative (at-most-once loses money; exactly-once delivery is a myth for external effects). The design work is in the consumer's idempotency (the same key/dedup mechanism as 12.9). The cost (an idempotency store, dedup check) is trivial vs. the bug (double-credit). This concept is really *idempotency applied to message consumers* — the same primitive (12.9), in the consumer role.

**Generics / first-principles.** "Exactly-once *delivery* is impossible; exactly-once *effect* is achievable via at-least-once delivery + idempotent processing — so never lose messages (at-least-once) and make processing tolerate duplicates (idempotency)." This is a foundational truth of distributed messaging (the impossibility of exactly-once delivery; the at-least-once + idempotency = effective-exactly-once pattern). The transferable insight: **you cannot guarantee a message is delivered exactly once (the network can always force a redelivery-or-loss choice), so you guarantee it's delivered *at least* once (never lost) and processed *idempotently* (duplicates harmless) — the combination gives exactly-once *effect*, which is what you actually wanted.** It transfers to every message-driven system (Kafka, SQS, RabbitMQ, CDC pipelines, webhooks). The deep principle: **reliability in messaging comes from at-least-once-delivery + idempotent-processing, not from chasing exactly-once-delivery** — the same "make duplicates harmless rather than prevent them" lesson as idempotency (12.9), now at the consumer.

**MySQL-specific reality.** In a MySQL-centric platform, the consumer of CDC/outbox events (12.11/12.12) — e.g., a service that applies a "credit account" event, or updates a read model — makes itself **idempotent** using MySQL: record the processed message's unique ID (the event ID / idempotency key) in a table with a **unique constraint** (12.9), *in the same transaction* as the effect (M07) — a duplicate event's ID insert fails the constraint → the consumer recognizes the duplicate and skips re-applying. So MySQL's unique-constraint-based idempotency (12.9) is the consumer-side dedup that turns the pipeline's at-least-once delivery into exactly-once effect. For our domain: a CDC consumer applying ledger-derived events (updating a denormalized balance read-model, M02/2.17, or triggering a downstream credit) carries the **event's unique ID**; if the same event is delivered twice (relay restart, 12.11), the second is a **no-op** (unique-constraint dedup) → no double-credit. The reporting warehouse, fraud service, and notification service are all idempotent consumers. This is idempotency (12.9) in the consumer role — the safety mechanism for the whole CDC/outbox propagation layer. The Pass C Mermaid draws at-least-once + idempotent-consumer = exactly-once effect.

## 12.14 · Reconciliation as the distributed backstop

**Mental model.** No distributed system is perfectly consistent — Sagas can half-complete (12.8), events can be missed or duplicated (12.10/12.13), replicas can diverge (12.5), bugs happen — so for money you need a **backstop that *detects* any inconsistency**: **reconciliation** (M02/2.17). Reconciliation **independently re-derives** the truth (from a separate source or by re-computing) and **compares** it against the system's recorded state, **alerting and repairing** on any drift. It's the "trust but verify" safety net: internal ledger vs external processor records; recorded balances vs sum-of-entries; state vs the events that produced it. Reconciliation is what makes eventual consistency (12.5) and distributed patterns (Saga, CDC, outbox) *safe for money* — they're allowed to be *temporarily* inconsistent *because* reconciliation catches and fixes any *persistent* inconsistency. It's the money-never-lies backstop for the entire distributed system.

**How it actually works.** Reconciliation runs as a periodic (e.g., daily, or continuous) **independent verification**:
1. **Re-derive the truth independently** — from a *different* source than the one being checked: the external payment processor's settlement file, the bank's records, or a re-computation from primitives (sum the immutable ledger entries to re-derive each balance, M02/2.17 — the balance is *derived*, so it's checkable).
2. **Compare** the independently-derived value against the system's recorded value (recorded balance vs Σ entries; internal ledger vs external processor; events emitted vs state changes).
3. **On a match:** confidence (the invariants hold). **On a mismatch (drift):** *alert* (a discrepancy is a money-correctness incident) and *repair* (investigate the cause — a half-completed Saga 12.8, a missed event 12.10, a bug — and correct it, often via a compensating entry, with an audit trail, M01/1.17).
Reconciliation relies on having an **independent source of truth** (the immutable ledger to re-derive from, M01/1.17/M02/2.17; the external processor's records) and on the system's design making invariants *checkable* (debit=credit, balance=Σentries — the double-entry invariant, M16). It's the *application* of the money-never-lies thread that has run through every module.

**Why it exists / what it solves.** Reconciliation exists because **distributed systems will have inconsistencies, and for money you cannot just hope they don't** — you must *detect* them, because an undetected discrepancy is lost or duplicated money. It solves the residual-risk problem: even with idempotency (12.9), outbox (12.11), Sagas with compensations (12.8), and careful consistency choices (12.15), *something* can still go wrong (a bug, an unhandled edge case, a missed event, a half-completed Saga, hardware corruption M09/M15) — reconciliation is the **independent check that catches it** before it becomes an unrecoverable loss. It's *why* eventual consistency and the distributed patterns are *acceptable* for money: they're permitted to be temporarily/occasionally inconsistent *because* reconciliation guarantees any real inconsistency is *detected and repaired*. Reconciliation is the safety net under the whole distributed money system — the final answer to "did money get lost or duplicated?" is "reconciliation would catch it."

**Tradeoffs & alternatives.** Reconciliation costs compute (re-deriving and comparing, often on replicas/warehouse to offload the primary, M02/2.17) and engineering (building the checks, the repair workflows) — but it's *non-negotiable* for money (the cost of *not* having it is undetected loss). It's a **backstop, not a primary mechanism** — you still need idempotency, outbox, Sagas, and correct consistency choices (reconciliation detects/repairs *residual* errors, it doesn't replace the patterns that prevent most of them). The "alternative" — trusting the system to be perfectly consistent — is naive (distributed systems aren't). The design considerations: *frequency* (daily batch vs continuous/streaming), *independence* (the re-derivation must use a genuinely independent source/computation, or it just re-confirms the same bug), *granularity* (per-account, per-batch, system-wide), and *repair* (automated compensating entries vs human investigation, always with an audit trail). For money: reconciliation is **mandatory**, runs against an **independent source** (the immutable ledger, the external processor), and is the **last line of defense** behind all the other patterns.

**Generics / first-principles.** "Independently re-derive the truth and compare it against the system's state — detect and repair any drift — because no distributed system is perfectly consistent, so verification is the safety net." This is the universal **reconciliation / auditing / verification** principle for correctness-critical distributed data (financial reconciliation, anti-entropy in Dynamo/Cassandra, checksums in distributed storage, "trust but verify"). The transferable insight: **distributed systems drift, so correctness-critical systems pair their *mechanisms* (idempotency, outbox, Sagas) with an *independent verification* that re-derives the truth and catches any residual inconsistency — design the system so its invariants are *checkable* (an independent source of truth to re-derive from), and run the check continuously.** It transfers to any system where silent inconsistency is unacceptable (finance, inventory, distributed storage integrity). The deep principle (the culmination of the money-never-lies thread): **the final guarantee that money is never lost or duplicated is not any single mechanism but the *independent reconciliation* that verifies it — make the invariants checkable, and verify them.**

**MySQL-specific reality.** Reconciliation in MySQL terms (building on M02/2.17): the **immutable ledger** (append-only entries, M01/1.17) is the independent source — re-derive each balance by `SUM`ing its entries and compare to the recorded/cached balance (any mismatch = drift to investigate); compare the internal ledger against the **external processor's settlement records** (imported and matched). These re-derivations run on **read-replicas or the reporting warehouse** (M10/M02/2.17 — offloading the primary, fed by CDC, 12.12) so reconciliation doesn't load the transactional shards. The checks verify the money-never-lies invariants (debit=credit, balance=Σentries, internal=external). For our domain: a **daily reconciliation job** re-derives every account's balance from the ledger and matches the platform's totals against the payment processor's records — catching, for example, a **cross-shard Saga (12.8) that silently half-completed** (debit applied, credit lost → balance ≠ Σentries → drift detected → repaired with a compensating entry + audit trail), or a **missed CDC event** (12.10), or a **bug**. This is the backstop behind every distributed pattern in the module, and the concrete realization of the money-never-lies thread that has run since M01. The Pass C Mermaid draws independent-re-derivation → compare → alert/repair.

## 12.15 · Choosing consistency per operation (the decision)

**Mental model.** The central practical skill of distributed data: **don't choose one consistency level for the whole system — choose *per operation*, by its cost-of-being-wrong.** Money-critical operations (authorize a payment, commit a transfer) need **strong consistency + atomicity** (any staleness or partial-completion loses money — these read the primary/strong, commit atomically single-shard or via a careful Saga). Propagation and integration (notify, update a search index, feed analytics) need **eventual consistency + idempotency + reconciliation** (staleness is harmless; idempotency makes duplicates safe; reconciliation catches drift). User-facing reads of their *own* data need **read-your-writes** (M10/10.6 — see your own payment). Reporting/dashboards need **eventual** (a few seconds stale is fine). The decision is a *mapping*: for each operation, pick the *weakest consistency that's still correct* (12.4), and attach the matching patterns (idempotency, outbox/CDC, reconciliation). This is where all the module's concepts come together into design.

**How it actually works.** For each operation, ask: *what's the cost of being wrong (stale/inconsistent/partial)?*
- **Irreversible money decisions** (authorize, transfer, debit): **strong + atomic** — read the primary (strong, 12.4 — never authorize against a stale balance → overdraft/double-spend), commit atomically (single-shard ACID where possible, M11/11.9; a careful Saga + idempotency for cross-shard, 12.8/12.9). CP under partition (12.2 — reject rather than authorize wrongly).
- **The user's own reads after a write** (their balance/history post-payment): **read-your-writes** (M10/10.6 — they must see their own action; others' staleness is fine).
- **Cross-system propagation** (notify, search, fraud, analytics): **eventual + idempotent + reconciled** — via outbox/CDC (12.11/12.12), idempotent consumers (12.13), reconciliation backstop (12.14). Staleness/duplicates are tolerable *because* of idempotency + reconciliation.
- **Reporting/display** (dashboards, third-party views): **eventual** — async replicas (M10/10.5), staleness harmless.
Then attach the patterns: strong operations get atomicity (single-shard/Saga); eventual operations get idempotency + reconciliation; everything money-related gets reconciliation as the backstop. The result is a *per-operation consistency + pattern map* — the architecture (12.16).

**Why it exists / what it solves.** This decision framework exists because **a single global consistency level is always wrong** — "everything strong" is needlessly slow/unavailable/expensive (most operations don't need it), and "everything eventual" loses money (the authorization path can't tolerate staleness). It solves the *granularity* problem: consistency is a per-operation resource (12.4), so you spend it where it's worth it (the money path) and save it where it isn't (reporting). It turns the module's theory (CAP/PACELC, the spectrum) and patterns (2PC/Saga/idempotency/outbox/CDC/reconciliation) into a *design method*: map each operation to its consistency need and its supporting patterns. For money, it's the framework that ensures the *authorization path is bulletproof* (strong, atomic) while the *propagation path is reliable-but-eventual* (idempotent, reconciled) — getting both correctness where it's critical and scale/availability where it's affordable.

**Tradeoffs & alternatives.** The framework *is* the per-operation tradeoff: strong/atomic (correct but slow/less-available/expensive — for the money path) vs eventual/idempotent/reconciled (fast/available/scalable but eventually-consistent — for propagation/reporting). The meta-tradeoff is *granularity vs simplicity*: per-operation choice is more complex than one global level, but it's the *only* way to get correctness and scale simultaneously. The anti-patterns it prevents: **everything strong** (slow, expensive, less available — over-engineered), **everything eventual** (money-loss on the authorization path — under-engineered), and **applying patterns blindly** (a Saga where co-location would do, M11/11.9; strong consistency where eventual is fine). The right design is *heterogeneous*: strong+atomic money path, eventual+idempotent+reconciled propagation, read-your-writes user reads, eventual reporting — each operation at its appropriate point. This is the synthesis the whole module builds toward.

**Generics / first-principles.** "Consistency is chosen per operation by cost-of-being-wrong — strong+atomic for irreversible/critical operations, eventual+idempotent+reconciled for propagation, with the weakest-correct level everywhere — never one global level." This is the universal distributed-data design discipline (per-operation consistency, the synthesis of consistency models + patterns). The transferable insight: **a well-designed distributed system is *heterogeneous* in consistency — it applies strong consistency and atomicity precisely where incorrectness is unacceptable (and pays the cost there), and eventual consistency with idempotency and reconciliation everywhere else (and gains the scale/availability there) — the skill is classifying each operation and attaching the right consistency level and supporting patterns.** It transfers to every distributed system design (which reads need strong, which writes need atomicity, where eventual + idempotency suffices). The deep principle: **match the guarantee to the operation's cost-of-failure — bulletproof the money path, make everything else reliable-but-eventual — never a one-size-fits-all consistency level.**

**MySQL-specific reality.** The per-operation map in MySQL/platform terms: **authorize/transfer** → read the **primary** (strong, 12.4), commit **single-shard ACID** (M11/11.9) or **Saga + idempotency** (12.8/12.9) for cross-shard, **semi-sync** (M10/10.4 — node-loss-durable), CP via group replication/fencing under partition (M10/10.11). **User's own post-write reads** → **read-your-writes** (M10/10.6 — primary or GTID-caught-up replica). **Propagation** (notify/fraud/search/analytics) → **outbox + CDC** (12.11/12.12), **idempotent consumers** (12.13), **reconciliation** (12.14). **Reporting** → **async replicas / warehouse** (M10/10.5, M02/2.17 — eventual). Every money operation is backed by **reconciliation** (12.14). For our domain, this *is* the platform's consistency architecture: a payment authorization is strong and atomic (never wrong), its propagation to downstream systems is eventual but reliable (outbox/CDC + idempotency) and verified (reconciliation), its display is eventual (replicas), and the whole thing is reconciled daily. This per-operation design is what M16 makes concrete. The Pass C Mermaid draws the per-operation decision (strong/atomic vs eventual/idempotent/reconciled).

## 12.16 · Fintech capstone: the consistent-enough payments platform ★

**Mental model.** The complete distributed design — "**consistent enough**," meaning *strongly consistent and atomic exactly where money correctness demands it, and eventually-consistent-but-reliable everywhere else* — so a payment is **never lost or duplicated across the entire distributed platform**. It composes *every* pattern: **single-shard ACID** for the common transfer (M11/11.9 — strong, atomic, free); **Sagas + idempotency** for the cross-shard minority (12.8/12.9 — never fragile 2PC); **strong reads from the primary** for money decisions and **read-your-writes** for users (12.4, M10/10.6); **outbox + CDC** for reliable event propagation (12.11/12.12 — never dual-write); **idempotent consumers** for exactly-once effect (12.9/12.13); **reconciliation** as the money-never-lies backstop (12.14, M02/2.17); and **consistency chosen per operation** (12.15). It's the distributed realization of money-never-lies — the synthesis of M07–M11 (single-node → durable → replicated → sharded) with M12's distributed patterns — and the architecture M16 builds the fintech capstone on.

**How it actually works.** The platform's distributed correctness, operation by operation:
- **The common transfer (single-shard):** co-located (M11/11.9) → one **single-shard ACID** transaction (debit + credit + ledger-entry + **idempotency key** + **outbox event**, all atomic, M07–M09, 12.9/12.11) → strong, atomic, never partial, never double-applied. Most money movement is *this* (cheapest + safest).
- **The cross-shard transfer (minority):** a **Saga** (12.8) of local transactions + compensations, each **idempotent** (12.9), state durable via outbox (12.11) so it resumes (never stuck) — eventually consistent (brief in-flight), never lost, **reconciled** (12.14) as backstop.
- **Reads:** money decisions → **primary/strong** (12.4 — never authorize against stale); user's own → **read-your-writes** (M10/10.6); reporting → **async replicas/warehouse** (eventual, M02/2.17).
- **Propagation:** every state change emits an **outbox event** (atomic, 12.11) → **CDC** (12.12) streams to fraud, notifications, search, the reporting warehouse → **idempotent consumers** (12.13) → exactly-once effect, never a lost or duplicated downstream event.
- **The backstop:** **reconciliation** (12.14) re-derives balances from the immutable ledger and matches against external records daily → detects/repairs *any* residual drift (a half-completed Saga, a missed event, a bug).
- **Per-operation consistency (12.15):** strong+atomic on the money path, eventual+idempotent+reconciled on propagation — *consistent enough*.

**Why it exists / what it solves.** This capstone solves the ultimate distributed-money problem: **how to keep money correct (never lost, never duplicated) across a system that is replicated (M10), sharded (M11), and integrated with other systems (CDC/outbox) — i.e., thoroughly distributed, where single-node guarantees don't hold (12.1).** It exists to prove that *distribution and correctness are compatible* if you (1) keep money operations strong + atomic where possible (single-shard ACID, M11/11.9), (2) handle the unavoidable distributed operations with the right patterns (Saga + idempotency, not 2PC; outbox + CDC, not dual-write), (3) make everything safely retryable (idempotency, 12.9 — the load-bearing primitive), (4) choose consistency per operation (12.15 — strong where critical, eventual where affordable), and (5) verify everything with reconciliation (12.14 — the backstop). It's the synthesis of the entire database journey's money-never-lies thread (M01 modeling → M07 ACID → M08 concurrency → M09 durability → M10 node-loss durability → M11 write scale → M12 distributed correctness) into one coherent distributed architecture. It caps Track D and is the foundation M16 (the fintech capstone) and M15 (the failure modes) build on.

**Tradeoffs & alternatives.** This design accepts: **heterogeneous complexity** (different consistency/patterns per operation) in exchange for **correctness + scale + availability simultaneously** (you can't get all three with a uniform approach); **eventual consistency on propagation** (idempotent + reconciled) in exchange for **never dual-writing or blocking**; **operational weight** (CDC/Kafka, reconciliation jobs, Saga orchestration, per-shard replication) in exchange for a **distributed system that never loses money**. The alternative — a *simpler* uniform design — fails one way or another: "everything strong/atomic" doesn't scale or stay available (and cross-shard atomicity is fragile, 12.6); "everything eventual" loses money on the authorization path; "naive dual-writes" silently drop events (12.10). So the honest capstone tradeoff is: **distributed money correctness requires this heterogeneous, pattern-rich design — there's no simpler approach that's also correct at scale** — but you *minimize* the distributed surface (co-locate so most operations are single-shard ACID, M11/11.9), so the expensive patterns apply only to the genuinely-distributed minority. The art (as in M11) is *minimizing* what must be distributed, then handling that minority rigorously.

**Generics / first-principles.** "A distributed system stays correct by being *strongly consistent and atomic where incorrectness is unacceptable* and *eventually-consistent-but-idempotent-and-reconciled everywhere else* — minimize the distributed surface, make every operation idempotent, derive propagation from atomic writes (outbox/CDC), and verify with reconciliation." This is the universal recipe for **correctness-critical distributed systems** (the synthesis of consistency models + distributed transaction patterns + reliable messaging + verification). The transferable insight: **distributed correctness isn't one mechanism — it's a *composition*: keep the critical path strong+atomic (and small, via co-location), handle distributed operations with compensation + idempotency (Saga, not 2PC), propagate reliably from atomic writes (outbox/CDC, not dual-write), make every consumer idempotent (exactly-once effect), choose consistency per operation, and verify everything with independent reconciliation.** It transfers to every distributed system that can't lose data (finance, commerce, any system of record). The deep principle (the journey's culmination): **money never lies in a distributed system not because any single node guarantees it, but because the *composition* — atomic where it counts, idempotent everywhere, reliably propagated, and independently reconciled — guarantees it.**

**MySQL-specific reality.** The concrete platform: transfers are **single-shard ACID** (Vitess-sharded by tenant/ledger_group, M11/11.9; debit+credit+ledger-entry+idempotency-key+outbox-row in one InnoDB commit, M07–M09, 12.9/12.11); cross-shard transfers are **Sagas** (12.8) over single-shard local transactions with idempotency + reconciliation; each shard is **replicated** (M10 — semi-sync for node-loss durability, primary reads for strong/RYW, async replicas for eventual reporting); **Debezium CDC** (12.12) streams each shard's binlog (ROW format, `sync_binlog=1`, M10/M09) → Kafka → fraud/notifications/search/warehouse, with the **outbox** (12.11) providing semantic events and **idempotent consumers** (12.13) giving exactly-once effect; **reconciliation** (12.14) re-derives balances from the immutable ledger on replicas/warehouse and matches external records daily; **consistency is chosen per operation** (12.15 — strong/atomic money path, eventual/idempotent/reconciled propagation). For our domain, the money-never-lies guarantees across the *whole* distribution: a payment is **atomic** (single-shard ACID or compensated Saga), **durable beyond node loss** (semi-sync, M10), **never double-applied** (idempotency keys, 12.9), **never lost in propagation** (outbox + CDC, 12.11/12.12), **exactly-once in effect** (idempotent consumers, 12.13), and **independently verified** (reconciliation, 12.14) — *consistent enough* that money is never lost or duplicated anywhere in the system. This is the distributed culmination of the entire journey (M01→M12) and the foundation for M15 (its catastrophic failure modes) and M16 (the full fintech platform). The ★ SVG (Pass C) draws the complete distributed payments architecture: per-operation consistency, single-shard ACID + Saga, outbox/CDC propagation, idempotent consumers, and reconciliation backstop.
