## 3.1 · What a data type really is

**Mental model.** A data type is not just "the kind of thing in the column" — it's a bundle of four promises made at once: a **set of legal values** (the domain), a **set of operations** (what you can do with them — arithmetic, comparison, concatenation), a **storage encoding** (the exact bytes on disk), and a **comparison/sort order** (how values rank). Choosing a type silently fixes all four. Most type bugs come from picking a type for one promise (it "holds the value") while ignoring another (it compares wrong, or rounds, or wastes bytes).

**How it actually works.** When you declare `amount DECIMAL(18,2)`, MySQL commits to: only base-10 numbers with ≤16 integer digits and exactly 2 fractional digits are valid (domain); `+ - * /` behave with exact decimal semantics (operations); each value is packed into a specific binary-coded-decimal byte layout (encoding); and ordering is true numeric order (sort). A different type for the "same" value — say `FLOAT` — changes *every* one of those: the domain becomes approximate binary fractions, arithmetic becomes lossy, the encoding is IEEE-754, and two "equal" amounts may not compare equal. The type is the contract the engine enforces on every read, write, comparison, and index operation involving that column.

**Why it exists / what it solves.** Types let the database **enforce meaning and reason about data physically.** Because it knows a column is DECIMAL, it can validate input, allocate exact storage, build an ordered index, and pick query plans. Without types (everything a string), the engine couldn't range-scan numerically, couldn't reject `"abc"` as an amount, and couldn't store compactly. Types are the first and cheapest constraint (M01/1.8 — the coarse domain) and the foundation the optimizer and storage engine stand on.

**Tradeoffs & alternatives.** Every type choice trades among the four promises: a wider integer is safer (range) but heavier (bytes); VARCHAR is flexible (any length) but carries length overhead and variable rows; ENUM is compact but rigid. The recurring mistake is "stringly typed" data — storing numbers, dates, or booleans as text — which trades away *all* the benefits (no numeric ordering, no range validation, bloated storage, broken comparison) for a hollow flexibility. The discipline: pick the type that honors the value's *true* semantics, not the most permissive one.

**Generics / first-principles.** "A type is a set of values + operations + an encoding." This is the universal definition from type theory and applies identically to programming languages, serialization formats, and APIs. The database adds two database-specific promises — *physical storage cost* and *sort/comparison order* — because in a database those have performance and correctness consequences that an in-memory type doesn't. Knowing all four promises is how you predict a type's behavior in *any* system.

**MySQL-specific reality.** MySQL's type system has notable quirks you must know: it historically did **implicit coercion** aggressively (comparing a string to a number silently converts, sometimes surprisingly — `'10abc' = 10` can be true in loose modes), which **strict SQL mode** tightens by rejecting bad conversions rather than truncating. MySQL also has many near-synonymous types and display-width artifacts (the famous `INT(11)` — the `11` is *just a display hint*, not a range limit, and is deprecated). The throughline for this module: in MySQL you must choose types deliberately *and* run strict mode, because the engine's defaults historically favored "accept and coerce" over "reject and protect."

## 3.2 · Storage footprint → performance (the core link) ★

**Mental model.** Here is the single idea that connects "type choice" to "query speed": **bytes per row decide rows per page, rows per page decide how much of your data fits in memory, and memory-fit decides your IO.** InnoDB reads and caches data in fixed **16 KB pages**; a row that's twice as fat means half as many rows per page, which means you need twice as many pages (twice the IO, twice the buffer-pool pressure) to scan the same number of rows. Type size isn't cosmetic — it's a direct, multiplicative performance lever on every table scan, range scan, and index.

**How it actually works.** InnoDB stores rows inside 16 KB pages in the clustered index (M01/1.3, detailed in M09). The number of rows that fit per page = roughly page size ÷ row size. Smaller types → smaller rows → more rows per page → fewer pages for the same data → (a) less disk IO to read it, and (b) more of it fits in the **buffer pool** (InnoDB's in-memory cache, M09), so more reads are served from RAM instead of disk. The effect compounds: a column you shrink is shrunk in the clustered index *and* in every secondary index that includes it (and the PK is embedded in all of them, M01/1.3). So an oversized `BIGINT` where `INT` would do, or a `CHAR(255)` where `VARCHAR(20)` would do, silently inflates every index and every page-read for the life of the table.

**Why it exists / what it solves.** This is *why* "pick the smallest type that safely fits" is a real rule and not pedantry. On large tables the difference between a 40-byte row and an 80-byte row can be the difference between the working set fitting in RAM (fast) or not (constant disk IO). It reframes type selection from "does it hold the value?" to "what does it cost me on every page read, forever?" — the foundation the indexing and InnoDB-internals modules build on.

**Tradeoffs & alternatives.** The tension is **safety/headroom vs footprint**: too-small risks overflow (a near-full `INT` PK that wraps — catastrophic, 3.3/3.15), too-large wastes bytes on every row. The right move is "smallest type that safely covers the *real* maximum with sane headroom," not "biggest type to be safe" (premature over-provisioning has a permanent cost) nor "tightest possible" (overflow risk). For variable data, VARCHAR vs CHAR is a footprint tradeoff too (3.7). Vertical partitioning (M02) — splitting rarely-used fat columns into a side table — is the structural escape when a hot row is bloated by cold large columns.

**Generics / first-principles.** "Data structure size determines cache behavior determines speed." This is mechanical sympathy — the same reason a struct-of-arrays beats an array-of-structs for scans, the same reason cache-line size matters in systems programming. The database's pages and buffer pool are one instance of the universal memory hierarchy: smaller working sets stay in faster memory. Type size is your lever on the working-set size.

**MySQL-specific reality.** Concrete InnoDB facts: 16 KB default page size; rows live in the clustered index ordered by PK; the buffer pool caches pages and its hit rate is the dominant performance metric (M09). MySQL gives you `INFORMATION_SCHEMA` and tools to see row/table sizes, and the practical heuristics are MySQL-specific: prefer `INT`/`BIGINT` sized to the real range, store UUIDs as `BINARY(16)` not `CHAR(36)` (3.12 — 16 vs 36+ bytes × every index), avoid `VARCHAR` lengths far beyond reality, and keep the PK compact because it's embedded everywhere. This concept is the *why* behind dozens of specific M03 recommendations and the bridge to M05 (indexing) and M09 (buffer pool).

## 3.3 · Integer types & signedness

**Mental model.** MySQL gives you a **ladder of integer widths** — TINYINT (1 byte) → SMALLINT (2) → MEDIUMINT (3) → INT (4) → BIGINT (8) — and a signed/unsigned switch. The whole skill is: **pick the smallest rung that safely holds your real maximum**, and choose signed vs unsigned on purpose. Going too small risks the worst database failure there is (silent overflow / wraparound); going too big taxes every row and index (3.2).

**How it actually works.** Each width stores a fixed byte count and covers a fixed range; **UNSIGNED** shifts the range to all-non-negative, doubling the positive ceiling (a signed INT tops out ~2.1 billion; UNSIGNED INT ~4.2 billion). The type is chosen against the column's *real-world* maximum plus headroom: a `status` code fits TINYINT; a row counter might need BIGINT. For an **AUTO_INCREMENT primary key**, the type sets the absolute ceiling on how many rows the table can ever have — and hitting that ceiling is a production emergency (3.15). Signedness also affects correctness: an UNSIGNED column rejects negatives at the type level (a weak constraint), and **mixing signed/unsigned in comparisons or arithmetic** triggers MySQL coercion that can surprise you.

**Why it exists / what it solves.** Integer sizing is the cleanest example of the footprint↔safety tradeoff (3.2). Right-sizing keeps rows and indexes tight (performance) while guaranteeing you never overflow (correctness). It matters most for high-cardinality columns — PKs, foreign keys, counters — where the type is repeated across millions of rows and embedded in indexes, so every saved byte multiplies, and every missed headroom calculation is a future outage.

**Tradeoffs & alternatives.** Signed vs unsigned is a real decision: UNSIGNED doubles range and documents "never negative," but introduces signed/unsigned comparison hazards and is a MySQL-ism (some other databases lack it; portability cost). For an auto-inc PK the modern default is **BIGINT** (often UNSIGNED) precisely because the 4-byte INT ceiling (~2–4 billion) is reachable for high-volume tables and *widening a PK later is a brutal migration* (3.15) — here the footprint cost of 8 vs 4 bytes is worth paying upfront for overflow safety. For small bounded sets, TINYINT (or ENUM, 3.10) is right.

**Generics / first-principles.** "Size the integer to the domain, with deliberate headroom; overflow is silent and catastrophic." Integer width selection is universal (C/Rust/protobuf all make you choose), and the wraparound failure mode is the same everywhere (the Y2038 problem, the Boeing/Ariane integer bugs). The principle: an integer type is a *bounded* promise, and exceeding the bound doesn't error gracefully — it wraps or rejects, so the bound must exceed reality with margin.

**MySQL-specific reality.** Specifics: the five widths and their byte sizes; UNSIGNED (a MySQL feature) doubling positive range; the **deprecated display-width** `INT(11)` (the number is a zero-fill display hint, *not* a storage or range setting — a perennial source of confusion); strict mode turning out-of-range inserts into errors rather than silent clamping (in loose modes MySQL would *clamp* an over-range value to the max — silent corruption). The load-bearing MySQL guidance: **BIGINT UNSIGNED for auto-increment PKs on anything that might grow large** (the AUTO_INCREMENT counter is itself typed, and exhausting it stops all inserts), TINYINT for small codes, and never rely on display width for anything.

## 3.4 · DECIMAL vs FLOAT/DOUBLE — exactness vs approximation ★

**Mental model.** FLOAT and DOUBLE store **approximations** of numbers in binary floating-point — they are *designed* to trade exactness for range and speed, and they cannot represent most decimal fractions (like 0.1) precisely. DECIMAL stores **exact base-10 digits**. For scientific magnitudes, approximation is fine; for **money, it is a bug**, because the approximation errors accumulate into real, auditable discrepancies — cents that appear and vanish. The rule is absolute: **never store money in FLOAT/DOUBLE.**

**How it actually works.** Binary floating-point (IEEE-754) represents numbers as sign × mantissa × 2^exponent. Many decimal values have no exact binary representation — 0.1 in binary is a repeating fraction, so it's stored as the *nearest representable* value, slightly off. Arithmetic then compounds the error: the canonical demonstration is `0.1 + 0.2` yielding `0.30000000000000004`, not `0.3`. DECIMAL(p, s) instead stores the actual decimal digits (p = total significant digits, s = digits after the point) in a packed exact encoding, so 0.1 is *exactly* 0.1 and sums are exact to the defined scale. The cost: DECIMAL is stored/computed more slowly and takes more space per digit than a fixed-width float, and you must size p and s correctly upfront.

**Why it exists / what it solves.** Floats exist for a real purpose — representing a huge dynamic range (subatomic to astronomical) compactly with fast hardware arithmetic — and for measurements, statistics, and graphics, the tiny relative error is irrelevant. DECIMAL exists specifically for the cases where the *exact* decimal value is the thing that matters: money, and any value where "off by a rounding error" is unacceptable. Choosing between them is choosing whether your domain tolerates approximation.

**Tradeoffs & alternatives.** FLOAT/DOUBLE: compact, fast, enormous range, **lossy** — right for scientific/analytic magnitudes, wrong for money. DECIMAL: exact, defined scale, **slower and larger**, right for money/financial. The third option, **integer minor units** (3.5), is also exact and is often *faster and smaller* than DECIMAL while avoiding fractional types entirely — the main alternative *to* DECIMAL for money. The decision is never "FLOAT vs DECIMAL for money" (FLOAT loses by default); it's "DECIMAL vs integer minor units" (3.5), both exact.

**Generics / first-principles.** "Binary floating point cannot exactly represent decimal fractions; never use it for exact-decimal domains." This is a universal hazard — the `0.1 + 0.2` problem exists in JavaScript, Python floats, C doubles, everywhere IEEE-754 lives. The first-principles rule transfers to any language and any storage: money and exact-decimal quantities use a decimal or integer representation, never a binary float. It's one of the most reliably-tested "do you actually know this" signals in engineering.

**MySQL-specific reality.** MySQL implements **DECIMAL as an exact, fixed-point type** (since 5.0+, stored in a packed binary format ~4 bytes per 9 digits) with a max precision of 65 digits — exact arithmetic within the declared scale. FLOAT/DOUBLE are IEEE-754 approximate types. A critical MySQL gotcha: in **non-strict mode**, inserting a value with more decimals than the column's scale **silently rounds**, and float comparisons (`WHERE amount = 0.3`) can fail to match stored approximations — so even *querying* float money is unreliable. The non-negotiable MySQL guidance, and the spine of the money-never-lies thread: **money is DECIMAL(p, s) or integer minor units, never FLOAT/DOUBLE**, with strict mode on so over-scale inputs error instead of silently rounding.

## 3.5 · Representing money: DECIMAL vs integer minor units ★

**Mental model.** There are exactly **two correct ways to store money** (both exact, both fine — the choice is a tradeoff, not a right/wrong): **DECIMAL(p, s)** — store `100.00` literally as a fixed-point decimal — or **integer minor units** — store the count of the smallest unit, e.g. `10000` cents as a BIGINT, dividing by 100 only for display. Both avoid float's lossiness (3.4); they differ in ergonomics, footprint, multi-currency handling, and arithmetic behavior.

**How it actually works.**
- **DECIMAL(p, s):** the column carries the scale (e.g., `DECIMAL(18,2)` = up to 16 integer digits, 2 decimals). Stored exactly; arithmetic is exact to the scale; human-readable in queries; rounding rules apply when results exceed scale. You size p for the largest amount you'll ever hold (with headroom) and s for the currency's exponent (3.6).
- **Integer minor units:** store money as a plain integer count of the smallest unit (cents, pennies, or for crypto, satoshi/wei). `$100.00` → `10000`. Arithmetic is exact *integer* arithmetic (fast, no fractional type), and there's no scale ambiguity within a single currency. You must track the currency's exponent separately (so you know `10000` means $100.00 not ¥10000) and convert for display. This is what many payment systems (and most cryptocurrency systems) use.

**Why it exists / what it solves.** Both representations solve the same core requirement — **exact money with no drift** — that FLOAT fails (3.4). Integer minor units additionally give you the simplest, fastest, least-surprising arithmetic (integers can't have scale-mismatch or rounding-mid-calculation surprises) and are the natural fit for currencies with extreme precision (crypto, 8–18 decimal places, where DECIMAL works but integers are idiomatic). DECIMAL gives readability and built-in scale handling, which is convenient for traditional fiat ledgers.

**Tradeoffs & alternatives.** **DECIMAL:** self-documenting, human-readable, scale carried by the type, easy SUM/AVG — but slightly larger/slower than integers and you must still define rounding (3.6). **Integer minor units:** compact, fastest arithmetic, no fractional type, ideal for crypto/high precision — but every value needs a known exponent (you *must* pair it with currency to interpret it), display requires conversion, and a wrong assumed exponent is a 100× money bug. *Decision guide (per the Pass-A note):* DECIMAL is a fine default for fiat ledgers (readability, mixed reporting); integer minor units shine for high-precision/crypto and ultra-high-throughput arithmetic. Either is correct; **FLOAT is not an option.**

**Generics / first-principles.** "Represent exact quantities either as scaled decimals or as integer counts of the smallest unit — never as binary floats." The integer-minor-units trick (scale up to an integer, do exact integer math, scale down for display) is a universal pattern for exact fractional arithmetic, used far beyond money (fixed-point DSP, currency libraries, blockchain). The deeper principle: choose a representation in which your domain's operations are *closed and exact*, then handle presentation separately.

**MySQL-specific reality.** In MySQL, DECIMAL(p, s) is the exact fixed-point type (3.4); integer minor units use BIGINT (8 bytes, range easily covering any realistic minor-unit amount — though for 18-decimal crypto wei you may need DECIMAL(65,0) or a string/binary representation since even BIGINT overflows). MySQL-specific care: **SUM over DECIMAL** is exact and is what you want for balances/reconciliation (M02/2.17); integer-minor-unit sums are exact integer sums. Storing the **currency code alongside the amount** is mandatory for integer minor units (and good practice for DECIMAL) so the exponent is never ambiguous (3.6). Whichever you pick, be consistent across the schema — mixing DECIMAL and minor-units representations for the same currency invites conversion bugs.

## 3.6 · Currency, scale & rounding

**Mental model.** "Money" isn't one shape — **different currencies have different numbers of decimal places** (their *exponent*): US dollars and euros have 2 (cents), Japanese yen has 0 (no sub-unit), Bahraini dinar has 3 (fils), and crypto has 8 (Bitcoin satoshi) to 18 (Ethereum wei). Your type's **scale must accommodate the most precise currency you support**, and you must define a **rounding rule** for any operation that produces more precision than a currency allows. Money math without a defined rounding policy is money that doesn't add up.

**How it actually works.** Each currency has a standard minor-unit exponent (ISO 4217 lists them). A schema supporting multiple currencies must either (a) use a DECIMAL scale (or integer-minor-unit interpretation) large enough for the finest currency (e.g., scale 8 for crypto, with fiat just using fewer places), or (b) carry the exponent per currency and interpret amounts accordingly. **Rounding** enters whenever a calculation (interest, FX conversion, fee splitting, tax) yields more decimals than the currency permits — you must round to the currency's scale using a defined mode (round-half-up, round-half-even/"banker's rounding," round-down), and the choice has real money and regulatory consequences. Crucially, **rounding must happen at defined points** (e.g., per-transaction, not silently per-intermediate-step) and the residual (the fractions of a cent) must be accounted for, not lost.

**Why it exists / what it solves.** Multi-currency support and any computed money (FX, interest, fees) force the scale-and-rounding question; ignoring it produces amounts that violate a currency's granularity (a fractional yen) or that don't reconcile (sum of rounded parts ≠ rounded sum). Defining scale and rounding policy makes computed money exact, consistent, and auditable — and prevents the classic "penny-shaving" and "rounding drift" bugs.

**Tradeoffs & alternatives.** A single wide scale (e.g., DECIMAL(_, 8) for everything) is simple but stores trailing zeros for fiat and can let sub-currency precision sneak into fiat amounts if not rounded. Per-currency exponent handling is precise but more complex (every amount interpreted with its currency). Rounding-mode choice is a genuine tradeoff: **banker's rounding (half-to-even)** minimizes cumulative bias over many operations (preferred in finance), while half-up is more intuitive but biases upward. The non-negotiable is that the policy is *explicit and consistent*; the specific mode is a domain/regulatory decision.

**Generics / first-principles.** "Precision and rounding are domain decisions that must be explicit, not emergent." Any system computing with real-world quantities of bounded granularity (money, physical measurements with significant figures) must define where precision is held and where/how it's reduced. The principle: never let rounding happen *implicitly* in intermediate steps; define the points and the mode, and conserve the residual. This generalizes to any fixed-precision arithmetic.

**MySQL-specific reality.** MySQL specifics: DECIMAL's scale is fixed per column, so a multi-currency `amount DECIMAL(18,8)` accommodates crypto while fiat uses fewer places (you round to the currency's scale in application/logic). MySQL's `ROUND()` uses **round-half-up by default** (and its behavior on DECIMAL vs float differs — another reason to use DECIMAL), and it does **not** offer banker's rounding natively, so half-to-even must be implemented in application code or carefully via SQL. Storing the **ISO 4217 currency code** (CHAR(3), 3.7) alongside every amount is the standard pattern so each amount's exponent and rounding rule are known. The money-never-lies consequence: scale and rounding are part of the *type-and-schema* design here, and getting them wrong creates discrepancies that surface in reconciliation (M02/2.17, M16).
