## 4.6 · The query optimizer: choosing among plans ★

**Mental model.** For a single SQL statement there are usually **many different physical ways to compute the same result** — which table to read first, which index to use (or none), which join algorithm, what order. These are *execution plans*, and they can differ in cost by orders of magnitude. The optimizer's job is to **enumerate candidate plans, estimate the cost of each, and pick the cheapest** — it's a search over strategies, guided by a cost model. The result is *the same rows* either way; the optimizer only chooses *how fast and cheaply* you get them.

**How it actually works.** Given the resolved statement (4.5), the optimizer considers the space of plans: for each table, which **access path** (full scan, index range, index lookup — 4.8); for multi-table queries, the **join order** (which table drives) and **join algorithm** (4.9); plus whether sorting/grouping can use an index or needs a separate step (4.13). It **estimates each candidate's cost** using a cost model fed by **statistics** (4.7) — roughly, how many rows each step touches and how much I/O/CPU that implies — and chooses the plan with the lowest estimated cost. Because the plan space can be huge (join order alone is factorial in the number of tables), the optimizer uses heuristics, pruning, and bounded search rather than exhaustively trying everything. The chosen plan is then handed to the executor (4.10), and `EXPLAIN` (4.15/M06) shows you what it picked.

**Why it exists / what it solves.** It's the engine of **data independence** (M01/1.1): because the optimizer decides *how*, you write *what* you want and let the database find a good execution — and that execution adapts as indexes, data sizes, and statistics change, *without you rewriting the query*. It frees you from hand-coding access paths and lets one declarative query run efficiently across very different data distributions. It is the single most important component for performance and the thing every tuning technique ultimately targets.

**Tradeoffs & alternatives.** The optimizer is a *heuristic estimator*, not an oracle — it trades optimality for tractable planning time (exhaustive search is infeasible), and it relies on *estimates* that can be wrong (4.7), so it sometimes picks a bad plan. The levers when it does: better **statistics** (4.7), **indexes** that give it cheaper options (M05), query **rewrites** that change the plan space, and **hints** (M06) that override its choice. The alternative philosophy — letting the developer specify the access path manually — is more predictable but loses adaptivity and burdens the developer; cost-based optimization is the dominant approach precisely because it usually wins and self-adapts.

**Generics / first-principles.** "Search a space of equivalent plans; estimate cost; pick the cheapest." This **cost-based optimization** is the core of every serious query engine (MySQL, Postgres, Spark, BigQuery) and mirrors optimization everywhere — compilers choosing instruction sequences, query planners, even route planners. The key insight to carry: *the optimizer reasons about estimated cost over equivalent alternatives* — so to improve performance you either give it cheaper alternatives (indexes) or better estimates (statistics), rather than fighting it.

**MySQL-specific reality.** MySQL's optimizer is **cost-based** with a long history of heuristics; modern versions add a more refined cost model, **histograms** (4.7), **hash joins** (4.9), and an **optimizer trace** (`OPTIMIZER_TRACE`) that exposes its reasoning. It has known tendencies and limitations (historically weaker at complex subqueries/join-order than some competitors, though much improved), which is why MySQL practitioners lean on EXPLAIN (M06), good indexes (M05), and occasionally hints. The practical mantra this module builds toward: **you don't command the plan, you shape the optimizer's decision** — via indexes, statistics, and (sparingly) hints. Everything in M05/M06 is "make the optimizer choose well."

## 4.7 · The cost model & statistics

**Mental model.** The optimizer can't measure the real cost of a plan without running it, so it **estimates** — and those estimates come from **statistics**: approximate metadata about your data (how many rows a table has, how many distinct values a column holds, how values are distributed). The cost model plugs these into formulas to guess each plan's cost. The critical consequence: **the optimizer is only as smart as its statistics.** Stale or missing stats → wrong estimates → bad plans (e.g., a full scan chosen over a perfect index), even though nothing about your query or indexes changed.

**How it actually works.** The optimizer maintains/consults statistics such as: **table cardinality** (estimated row count), **index cardinality** (number of distinct values in an index — drives **selectivity**: how many rows a lookup will match), and, in modern MySQL, **histograms** (the distribution of values in a column, capturing skew the simple cardinality number misses). It combines these with a **cost model** that weighs estimated row counts against per-row CPU and per-page I/O costs to score each candidate plan (4.6). These statistics are **sampled and cached**, not recomputed per query — InnoDB estimates index cardinality by sampling pages, and the numbers are refreshed periodically or on demand. When the data changes a lot but stats don't update, the optimizer reasons from a stale picture and can choose badly. `ANALYZE TABLE` forces a refresh (and computes histograms).

**Why it exists / what it solves.** Estimating beats measuring because actually running every candidate plan to compare them would be absurdly expensive — the whole point is to *avoid* doing work. Statistics let the optimizer predict, cheaply, which plan will touch the fewest rows and pages. **Selectivity** (from cardinality) is especially central: it's how the optimizer decides whether an index is worth using (a highly selective index that matches few rows is great; a low-selectivity index matching half the table may be worse than a scan). Without statistics the optimizer would be flying blind.

**Tradeoffs & alternatives.** Statistics trade **accuracy for cheapness**: more sampling / histograms → better estimates but more overhead to compute and store; less → cheaper but blinder. They're inherently approximate (sampled), so they can mislead on skewed data, correlated columns, or right after bulk changes. The failure mode — **stale statistics causing plan regressions** — is a real operational hazard (a query that was fast yesterday is slow today because data grew and stats didn't keep up). Mitigations: auto-stats updates, scheduled/post-bulk-load `ANALYZE TABLE`, histograms for skewed columns, and (when the optimizer still gets it wrong) hints (M06). The deeper limitation: cost is an *estimate*, so plans are *probably* good, not provably optimal.

**Generics / first-principles.** "Decisions from estimates are only as good as the estimates; keep your statistics fresh." This is universal to any cost-based or predictive system — a planner, a cache eviction policy, a JIT compiler's profiling — all reason from sampled/approximate data about the past to predict the future, and all degrade when that data goes stale or unrepresentative. The transferable instinct: *when a cost-based system makes a surprising choice, suspect its inputs (statistics) before its logic.*

**MySQL-specific reality.** MySQL specifics: InnoDB **index statistics** are sampled (controlled by settings like `innodb_stats_persistent` and sample-page counts) and persisted; **histograms** (column value distributions, created via `ANALYZE TABLE ... UPDATE HISTOGRAM`) since 8.0 help with skew and non-indexed-column selectivity; **`ANALYZE TABLE`** refreshes stats; the **optimizer trace** shows the cost numbers it used. The classic MySQL incident: after a big data load or growth, a previously-fast query suddenly does a full scan because cardinality estimates are stale — fixed by `ANALYZE TABLE`. For our domain, accurate statistics on `ledger_entry` (which grows forever) are what keep "account 42's recent entries" choosing the index instead of scanning a billion rows — a direct M05/M06 concern, and a *money-never-lies-adjacent* performance-stability issue (a reconciliation query that suddenly scans the whole ledger can stall the system).

## 4.8 · Access paths: how a single table is read ★

**Mental model.** Strip a query down to one table: the optimizer must still decide **how to physically fetch the rows** — and there's a *ladder* of access methods from worst to best for a given query. Read **every row** (full table scan)? Use an index to read a **range** of rows (index range scan)? Use an index to jump to **specific matching rows** (index lookup/ref)? Or answer the query **entirely from an index** without touching the table at all (covering index)? This choice — the **access path** — is the single most consequential per-table decision, and it's the fundamental unit that indexing (M05) and EXPLAIN (M06) are all about.

**How it actually works.** The common access paths, roughly best-to-worst for a selective query (introduced here, deep-dived in M05/M06):
- **`const`/`eq_ref`** — at most one matching row via a primary/unique key (e.g., look up account by PK) — the cheapest.
- **`ref`** — index lookup matching one value, possibly several rows (e.g., all entries for one `account_id` via a secondary index).
- **`range`** — an index scan over a contiguous range (e.g., entries `WHERE created_at BETWEEN ...`).
- **covering index** — the needed columns are all *in* an index, so the query is answered from the index alone, skipping the row lookup ("Using index").
- **full table scan** — read every row, used when no useful index exists or when the optimizer estimates the scan is cheaper than a low-selectivity index (which can be true!).
The optimizer picks among these per table using selectivity estimates (4.7): a highly selective index lookup beats a scan; a low-selectivity index (matching most rows) may lose to a scan. The PK clustering and index design from M01/M03/M05 determine *which* of these paths are even available.

**Why it exists / what it solves.** Access paths are *the* mechanism by which indexes make queries fast: an index gives the optimizer a cheaper access path (range/ref/covering) than a full scan. Naming the ladder turns "indexing" from magic into "providing and selecting better access paths." It also explains why an index sometimes *isn't* used (the optimizer estimated a scan was cheaper) and why *covering* indexes are special (they eliminate the row-lookup step entirely). This concept is the conceptual core that M05 (how to build indexes that create good access paths) and M06 (how to read which path was chosen) elaborate.

**Tradeoffs & alternatives.** The tradeoff is **index lookup cost vs scan cost**, mediated by selectivity: indexes add write overhead and storage (M05) and only pay off when they enable a sufficiently selective access path; for a query that returns most of the table, a full scan can genuinely be the cheapest path (sequential I/O, no index traversal + row lookups). Covering indexes are faster for reads but wider (they carry extra columns) and cost more on writes. There's no universally-best path — it depends on the query's selectivity and the data, which is exactly why the optimizer *chooses* per query using statistics.

**Generics / first-principles.** "How you traverse the data structure dominates cost; choose the access method that touches the least data for this query." This is the database face of algorithmic access-cost reasoning — scanning a list vs binary-searching a sorted array vs hashing — applied to on-disk structures. The universal lesson: the *access path*, not the data size alone, determines speed, and the right path is query-dependent. "Can this be answered without reading the whole thing?" is the question behind every index.

**MySQL-specific reality.** In MySQL/InnoDB these access paths map to EXPLAIN's **`type`** column (`const`, `eq_ref`, `ref`, `range`, `index`, `ALL`) and **Extra** notes (`Using index` = covering, `Using where`, `Using index condition`) — M06's bread and butter. InnoDB specifics that shape paths: the **clustered PK** (a PK lookup *is* the row, no extra fetch) and **secondary indexes** that point back to the PK (a non-covering secondary-index lookup costs an extra clustered-index access per row — the reason covering indexes matter so much, M05/M01·1.3). For "account 42's recent entries," the available paths depend entirely on whether an index like `(account_id, created_at)` exists (M01/1.14) — turning a full ledger scan into a tight range scan. This is the direct bridge to M05.

## 4.9 · Join execution strategies

**Mental model.** A multi-table query is computed by **combining tables two at a time** using a **join algorithm**, in a chosen **join order**. The optimizer picks both. The workhorse is the **nested-loop join** (for each row of the first/"driving" table, find matching rows in the second — *fast when the second table has an index on the join column*); when no good index exists, MySQL can use **block nested-loop** (buffer driving rows to reduce repeated inner scans) or, in modern versions, a **hash join** (build a hash table on one side, probe with the other — good for large unindexed joins). Join *order* — which table drives — matters enormously because it multiplies through the whole plan.

**How it actually works (introduced here; deep-dived in M06).**
- **Nested-loop join (NLJ):** iterate the driving table; for each row, look up matches in the inner table. If the inner table's join column is **indexed**, each lookup is a cheap index access (`ref`/`eq_ref`, 4.8) — this is the efficient, common case. Without an index, each driving row triggers an inner scan (quadratic — slow).
- **Block nested-loop (BNL):** buffer a block of driving rows and scan the inner table once per block instead of once per row — mitigates a missing index somewhat (largely superseded by hash join in 8.0).
- **Hash join (8.0+):** build an in-memory hash table on the smaller input keyed by the join column, then stream the other input and probe — efficient for **large, equi-join, unindexed** cases.
- **Join order:** the optimizer chooses which table is outermost (driving). A good order keeps intermediate result sizes small (drive from the most selective table). The plan space is factorial in table count, so the optimizer prunes/heuristically searches (4.6).

**Why it exists / what it solves.** Joins are how the relational model reassembles normalized data (M02) — and they can be cheap or catastrophic depending on algorithm and order. Index-backed nested-loop makes the common "look up related rows" pattern fast; hash join rescues large analytical joins that would be quadratic under naive NLJ; smart join order avoids exploding intermediate results. These strategies are *why* normalization's "pay with joins" cost (M02/2.1) is usually affordable — the optimizer executes joins efficiently when you give it indexes.

**Tradeoffs & alternatives.** Each algorithm has a sweet spot: **NLJ with index** is great for selective joins (few driving rows, indexed inner) but terrible without the index; **hash join** is great for large unindexed equi-joins but needs memory (can spill) and only does equi-joins; **BNL** is a middle ground now mostly replaced by hash join. Join *order* trades planning effort for execution cost — getting it wrong (driving from the wrong table) can multiply work by orders of magnitude. The main lever you control: **indexes on join columns** (M05) to enable cheap nested-loop, plus statistics (4.7) so the optimizer estimates selectivity and orders joins well.

**Generics / first-principles.** "Combine sets pairwise; the algorithm and order determine whether it's linear or quadratic." Join strategy is a classic algorithm-selection problem — index-nested-loop ≈ "lookup each in a map," hash join ≈ "build a hash index then probe," and the order question is "process the most-filtering step first to shrink the working set." These transfer to any data-combination task (merging streams, joining dataframes, set intersection). The instinct: *make the inner lookup cheap (index/hash) and drive from the smallest/most-selective side.*

**MySQL-specific reality.** MySQL realities: historically **nested-loop (and block nested-loop) only**, with **hash join added in 8.0.18** (a significant improvement for unindexed joins); the optimizer chooses join order and exposes it in EXPLAIN (the row order) and the optimizer trace. The dominant MySQL performance lesson: **index your join columns** so the optimizer can use efficient `ref`/`eq_ref` nested-loop access (4.8) instead of falling back to hash/BNL or, worst, an unindexed nested scan. For our domain, joining `account` → `ledger_entry` on `account_id` is fast *only* if `ledger_entry.account_id` is indexed (M01/1.14) — otherwise it degrades badly as the ledger grows. The full mechanics, EXPLAIN reading, and tuning of joins are M06; this concept establishes *what* the optimizer is choosing among.
