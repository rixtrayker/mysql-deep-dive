## 4.1 · The big picture: query in, rows out ★

**Mental model.** A SQL query is a *declarative wish* — "give me this set" (M01/1.1) — and MySQL is an assembly line that turns that wish into concrete physical reads and writes. The query enters as text and passes through a **fixed sequence of stages**: a connection accepts it, the parser turns text into structure, the preprocessor resolves names and checks meaning, the optimizer chooses *how* to get the data, the executor runs that plan, and underneath it all a storage engine (InnoDB) actually touches the bytes on disk. Each stage transforms the query one step closer to physical I/O. Understanding this pipeline is the prerequisite for all performance work, because tuning is just *influencing the decisions a specific stage makes*.

**How it actually works.** The canonical path for a typical statement: **(1) Connection/session** — a client thread is assigned, session state is established. **(2) Parsing** — SQL text is tokenized and built into a parse tree; syntax errors die here (4.4). **(3) Preprocessing/resolution** — names (tables, columns) are resolved against the data dictionary, privileges checked, semantics validated (4.5). **(4) Optimization** — the optimizer enumerates candidate execution plans, estimates each one's cost using statistics, and selects the cheapest (4.6–4.9). **(5) Execution** — the executor walks the chosen plan, requesting rows from the storage engine via the handler API (4.10–4.11). **(6) Storage engine** — InnoDB reads/writes pages (through the buffer pool, M09) and returns rows. Results stream back (possibly via sort buffers or temp tables, 4.13) to the client. (Older MySQL inserted a *query cache* lookup near the front; that's removed in 8.0, 4.14.)

**Why it exists / what it solves.** The staged pipeline is what makes a database *declarative*: you describe the result, and the separation of concerns (parse vs plan vs execute vs store) lets each stage specialize — the parser doesn't care about indexes, the optimizer doesn't care about disk format, the engine doesn't care about SQL syntax. This is the architectural expression of data independence (M01/1.1): because *how* is decided in the optimizer/engine stages, you can change indexes, statistics, or even the storage engine without rewriting the query.

**Tradeoffs & alternatives.** The pipeline adds per-query overhead (parsing, planning) that a hand-written access-path-coded program wouldn't pay — which is why prepared statements (cache the parse/plan, 4.14) and the optimizer's effort-vs-quality balance exist (spending more time planning to find a cheaper plan only pays if the query is expensive enough). The alternative architectures — a key-value store with no query language (no optimizer, no flexibility) — trade the pipeline's generality for raw speed on a fixed access pattern. Relational engines accept the pipeline cost to get declarative, optimizable, general queries.

**Generics / first-principles.** "Compile a declarative spec into an optimized physical plan, then execute it." This **parse → resolve → optimize → execute** shape is universal — it's how SQL engines (MySQL, Postgres, SQL Server), query layers (Spark, dataframe engines), and even programming-language compilers work (parse → semantic analysis → optimize → codegen). Learn the shape once and it transfers everywhere; the database is a *query compiler* with a storage runtime attached.

**MySQL-specific reality.** MySQL's specific pipeline: a connection/thread layer, the SQL layer (parser, preprocessor, optimizer, executor) shared across all storage engines, and the **pluggable storage engine** below it (4.11 — the architectural feature that most distinguishes MySQL). The 8.0-era reality: the **query cache is gone** (4.14), the optimizer has grown more sophisticated (histograms, hash joins, 4.7/4.9), and the data dictionary is now transactional (stored in InnoDB). This master pipeline diagram recurs throughout the module and underpins M06 (EXPLAIN shows you the optimizer's stage-4 decisions). Everything else in M04 is a zoom-in on one stage of this picture.

## 4.2 · The connection layer & a session's lifecycle

**Mental model.** Before any SQL runs, a client must *connect*, and each connection gets a **thread with its own private session state** — its current database, session variables (`sql_mode`, `time_zone` from M03), open transaction, temporary tables, prepared statements. Connections are a **finite, costly resource**: each consumes memory and a thread, and establishing one isn't free (TCP, auth handshake). This is why connection *pooling* exists and why "too many connections" is a classic outage.

**How it actually works.** A client opens a connection (TCP or socket); MySQL authenticates it (4.3) and assigns it a **thread** to handle its statements. That session holds state for its lifetime: the selected schema, session-scoped system variables, the transaction context (autocommit on/off, an open transaction's locks and undo, M07/M08), session temp tables, and server-side prepared statements. Statements on that connection run **serially** (one at a time per connection). When the client disconnects (or times out), the session state is torn down. Because each connection costs memory (buffers, caches) and a thread, the server enforces a `max_connections` limit; exceeding it rejects new connections. **Connection pools** (in the app or a proxy like ProxySQL, M10) keep a set of established connections alive and hand them out, avoiding per-request connect/auth cost and bounding concurrency.

**Why it exists / what it solves.** Sessions provide *isolation and context*: your transaction, your variables, your temp tables don't leak into other clients'. The thread-per-connection model is simple and gives each client a private execution context. Pooling solves the cost problem — without it, a high-throughput service that opens a connection per request would spend enormous overhead on connect/auth and could exhaust `max_connections`.

**Tradeoffs & alternatives.** Thread-per-connection is simple but doesn't scale to tens of thousands of mostly-idle connections (each thread costs memory and scheduler overhead) — the alternative is a **thread pool** (reuse a small pool of threads across many connections, decoupling connections from threads) for high-connection-count workloads. Connection pooling trades a little complexity (pool sizing, stale-connection handling, and the subtlety that **session state persists across pooled uses** — a leftover `time_zone` or open transaction can poison the next borrower) for large efficiency gains. Pool too small → requests queue; too large → the DB is overwhelmed with concurrency (M08 contention).

**Generics / first-principles.** "Connections are stateful, finite, and expensive to create — pool them, and bound concurrency." This is universal across resource-backed services (DB connections, HTTP keep-alive, thread pools, file handles): establish-once-reuse-many beats establish-per-request, and the resource limit is a capacity-planning constant you must respect. The session-state-leaks-across-pooled-uses hazard is the general "shared mutable resource" footgun.

**MySQL-specific reality.** MySQL specifics: default thread-per-connection model (with an optional thread pool in some editions); `max_connections` limit and the "Too many connections" error; per-session vs global system variables (M03's `sql_mode`/`time_zone` are session-scoped — a pooled connection can carry a previous setting); `wait_timeout`/`interactive_timeout` reaping idle connections; and the memory cost per connection (per-thread buffers). For fintech high-throughput services, a **connection pool** (app-side, and often a **ProxySQL** layer, M10) is standard, sized deliberately, with care that transaction/variable state is reset between borrows. This connects to M10 (proxies route pooled connections to replicas) and M08 (connection count drives concurrency and lock contention).

## 4.3 · Authentication, authorization & the grant check

**Mental model.** Before MySQL runs your query it answers two questions: **who are you?** (authentication — identity) and **what are you allowed to touch?** (authorization — privileges). Authentication happens once at connect; authorization is checked **per statement** against the privilege system (grants on databases, tables, columns). A query that's perfectly valid will still be rejected if your user lacks the privilege — security is enforced *before* execution, not after.

**How it actually works.** At connection time, MySQL authenticates the client using an **authentication plugin** (e.g., `caching_sha2_password` default in 8.0) against the `mysql.user` table, verifying credentials (and optionally requiring TLS). Once connected, every statement triggers an **authorization check**: MySQL consults the **grant tables** (global, database-, table-, and column-level privileges) to verify the user holds the required privilege (SELECT, INSERT, UPDATE, etc.) on the objects the statement touches. If not, the statement is rejected ("access denied") before parsing fully proceeds to execution. **Roles** (8.0) let you bundle privileges and assign them to users. The principle of **least privilege** — granting each user only what it needs — is the security best practice this system enables.

**Why it exists / what it solves.** A shared database holds many tenants', services', and users' data; the privilege system is how MySQL enforces that each principal can only do what it's authorized to. Checking per-statement (not just per-connection) means privileges can be fine-grained (this service can read `ledger_entry` but not write it) and revoked dynamically. It's the database's access-control layer — essential for security, compliance, and blast-radius limitation.

**Tradeoffs & alternatives.** Fine-grained privileges (column-level, many roles) are more secure but more complex to manage; coarse grants (one powerful app user) are simpler but violate least privilege (a compromised app user can do anything). The tradeoff is security granularity vs operational simplicity. Alternatives/complements: enforcing authorization in an application/service layer (more flexible business rules, but bypassable if someone reaches the DB directly), or a proxy. Best practice combines DB-level least-privilege grants (the un-bypassable floor) with app-level authorization (richer rules).

**Generics / first-principles.** "Authenticate identity once; authorize each action against least-privilege grants." The authentication-vs-authorization distinction and least-privilege are universal security principles — same in OSes, cloud IAM, and APIs. The database is one enforcement point; the principle (verify *who*, then check *what they may do*, granting the minimum) transfers everywhere. Defense in depth means enforcing it at multiple layers.

**MySQL-specific reality.** MySQL specifics: pluggable authentication (`caching_sha2_password` default in 8.0, replacing `mysql_native_password`), the `mysql.user`/grant tables, `GRANT`/`REVOKE`, privilege levels (global/db/table/column/routine), **roles** (8.0), and TLS for connection encryption. Security basics (auth, TLS, encryption at rest, least privilege, audit) are expanded in M13. For fintech: **least-privilege service users** (the payments service can write the ledger; the reporting service can only read replicas, M10), no shared superuser, TLS required, and audit logging — money systems are high-value targets, so DB-level access control is a core control, not an afterthought.

## 4.4 · Parsing & the parse tree

**Mental model.** The parser is the **grammar checker and structurer**: it reads the raw SQL text, verifies it's *grammatically* valid SQL, and converts it into a **parse tree** — a structured, in-memory representation the rest of the engine can reason about. Syntax errors (a missing comma, a misspelled keyword, unbalanced parens) are caught *here* and the query dies before any data is touched. Parsing is purely about *form*, not *meaning* — it doesn't know or care whether `account` is a real table yet (that's the next stage).

**How it actually works.** Two sub-steps: **lexing/tokenizing** (split the text into tokens — keywords, identifiers, literals, operators) and **parsing** (apply the SQL grammar to assemble tokens into a tree reflecting the statement's structure — a SELECT node with its select-list, FROM, WHERE, etc.). The output is a **parse tree** (abstract syntax tree) capturing *what kind of statement* this is and its syntactic components. If the tokens don't conform to the grammar, the parser raises a syntax error with a position. The parse tree is then handed to preprocessing (4.5), which resolves it against the actual schema.

**Why it exists / what it solves.** Text is unstructured; the engine can't optimize or execute a string. Parsing converts the human-written query into a machine-manipulable structure and provides the **first validation gate** — catching malformed SQL cheaply, before the expensive stages. It's the front door that turns "a string the user typed" into "a statement the engine understands the shape of."

**Tradeoffs & alternatives.** Parsing every statement from scratch costs CPU; the optimization is to **cache the parsed (and planned) form** for repeated statements via **prepared statements** (4.14) — parse once, execute many with different parameters. This trades a little memory and protocol complexity for skipping re-parse/re-plan on hot queries. (It also has a security benefit: parameterized prepared statements separate code from data, preventing SQL injection — the parser treats parameters as values, not parseable SQL.)

**Generics / first-principles.** "Turn unstructured text into a structured tree, validating form before meaning." This is exactly the front-end of any compiler or interpreter — lex, parse, build an AST — and the principle "validate syntax separately from and before semantics" is universal. The parse tree is the database's AST; everything downstream operates on structure, not text.

**MySQL-specific reality.** MySQL has its own SQL parser producing an internal parse tree; syntax errors surface as "You have an error in your SQL syntax... near '...'" with a location. Practical MySQL connections: **prepared statements** (`PREPARE`/`EXECUTE`, or the binary protocol via client libraries) cache the parsed form and prevent SQL injection by parameterizing; the parser enforces MySQL's specific grammar and reserved words (e.g., why `transaction` must be backticked or renamed `transaction_`, as in this resource's DDL). Parsing is rarely the bottleneck, but on extremely high-QPS simple-query workloads, parse/plan overhead is real and prepared statements help. Parsing only checks form — "does this table exist?" is the next stage (4.5).

## 4.5 · Preprocessing & semantic resolution

**Mental model.** Parsing checked that the SQL is *grammatically* valid; preprocessing checks that it's *meaningfully* valid. This stage **resolves names** (do `account` and `balance` actually exist? is `balance` ambiguous across joined tables?), checks types and that operations make sense, re-verifies privileges (4.3), and applies some early transformations — producing a *resolved* tree the optimizer can plan. "Unknown column 'blance'" is caught *here*, not at parse: the grammar was fine, the meaning wasn't.

**How it actually works.** The preprocessor takes the parse tree and:
- **Resolves identifiers** against the **data dictionary** — every table and column name is looked up; non-existent names → error ("Unknown table/column"); ambiguous names (a column existing in two joined tables without qualification) → error.
- **Validates semantics** — type compatibility, that aggregate/`GROUP BY` usage is legal (recall M02/2.3's `ONLY_FULL_GROUP_BY` functional-dependency check), that functions get sensible arguments.
- **Checks privileges** on the now-resolved objects (4.3).
- **Applies early rewrites/transformations** — some normalization and simplification (constant folding, trivial simplifications, certain subquery transformations) that prepare the statement for cost-based optimization.
The result is a validated, resolved representation handed to the optimizer (4.6).

**Why it exists / what it solves.** A grammatically-valid query can still be nonsense (reference a missing column, mismatch types, misuse aggregates). Preprocessing is the **semantic validation gate** that catches these *before* planning and execution, and it binds the abstract names in the query to concrete schema objects (the data dictionary) so the optimizer can reason about real tables, columns, indexes, and statistics. It's the bridge from "syntactically a query" to "a plannable operation over the actual schema."

**Tradeoffs & alternatives.** Splitting parse (form) from preprocess (meaning) is the standard compiler design — it localizes errors (syntax vs semantic) and lets each stage be simpler. The cost is two passes over the statement structure; the benefit is clear, well-located errors and a clean separation. There's little alternative — you *must* resolve names against the schema before you can plan, because the optimizer needs to know which indexes and statistics exist for the referenced objects.

**Generics / first-principles.** "Bind names to definitions and check meaning, after checking form and before optimizing." This is the **semantic analysis / name resolution** phase of any compiler — resolve identifiers against a symbol table, type-check, report semantic errors. The database's "data dictionary" is its symbol table. The universal principle: form errors and meaning errors are distinct and caught in distinct phases, and you can't optimize what you haven't resolved.

**MySQL-specific reality.** MySQL specifics: name resolution against the **data dictionary** (transactional, InnoDB-stored in 8.0, replacing the old `.frm` files); errors like "Unknown column 'x' in 'field list'" and "Column 'id' in field list is ambiguous"; the `ONLY_FULL_GROUP_BY` semantic check (M02/2.3) enforced here using functional-dependency reasoning; and early transformations the optimizer relies on. For our domain, this is where `account.balance` in "get account 42's balance" gets bound to the real typed column from M03/3.17, and where a query referencing a dropped/renamed column fails fast. After this stage, MySQL has a fully-resolved statement and can finally ask the central performance question: *what's the cheapest way to execute it?* — the optimizer (4.6).
