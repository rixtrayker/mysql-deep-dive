## 4.10 · The execution engine: running the plan

**Mental model.** The optimizer chose a plan; the **executor** *runs* it. A plan is a tree of operators (scan this table, join to that one, sort, aggregate), and the executor **walks that tree, pulling rows operator by operator**, calling down into the storage engine whenever it needs actual data. The declarative wish has now become concrete row movement: "for each driving row, probe the inner table; for each match, apply the filter; feed survivors to the sort." The executor is the runtime that turns the plan into rows.

**How it actually works.** Execution is typically an **iterator/pipeline model**: each operator in the plan tree exposes "give me the next row," and operators pull from their children. A join operator pulls a row from its driving child, then repeatedly asks its inner child (an index lookup on the storage engine) for matches; a filter operator pulls rows and passes only those satisfying `WHERE`; a sort operator pulls *all* its input then emits in order (a blocking step, 4.13). At the leaves, **table/index access operators call the storage engine via the handler API** (4.11) — "open this index, fetch rows in this range, give me the next one." Rows flow up the tree to the client. Some operators **stream** (pass rows through as they go — pipelined, low memory); others **block/materialize** (must collect all input first — sorts, some aggregations, temp tables, 4.13). The executor coordinates this, managing buffers and the engine calls.

**Why it exists / what it solves.** Separating "decide the plan" (optimizer) from "run the plan" (executor) keeps each focused, and the iterator model lets complex plans compose from simple operators that don't need to know about each other — a scan doesn't know it's feeding a join. It's the runtime that makes plans *executable* and the layer that actually drives storage-engine I/O. Pipelining (streaming where possible) keeps memory low and lets results start flowing before the whole query finishes.

**Tradeoffs & alternatives.** The classic **iterator ("Volcano") model** is simple and composable but has per-row function-call overhead; modern engines sometimes use **vectorized** (batch rows) or **compiled** execution for speed — a simplicity-vs-throughput tradeoff. Streaming operators are memory-cheap but some operations *can't* stream (you can't return sorted rows until you've seen them all), forcing blocking steps with their buffer/temp-table costs (4.13). The executor's job includes deciding when to spill to disk vs keep in memory — a latency cliff when it must spill.

**Generics / first-principles.** "Compose computation from pull-based operators; stream where you can, materialize only when you must." The iterator/pipeline model is a universal pattern — Unix pipes, Rust/Python iterators, stream-processing frameworks, dataflow engines all share it. The key distinction it teaches: **streaming (pipelined, low-memory, incremental) vs blocking (materialized, must-see-all-input-first)** operations — knowing which is which predicts memory use and where latency spikes. The database executor is a dataflow runtime.

**MySQL-specific reality.** MySQL 8.0 moved to a more uniform **iterator-based executor** (replacing older bespoke execution code), which made EXPLAIN output (especially `EXPLAIN FORMAT=TREE` and `EXPLAIN ANALYZE`, M06) cleanly reflect the operator tree. The executor drives storage engines through the **handler API** (4.11) — the same execution code works over InnoDB, Memory, etc. Blocking operations (filesort, temp tables, 4.13) are where MySQL execution gets expensive. For our trace query, the executor opens the index on `ledger_entry`, asks InnoDB for the range of account-42 rows, and streams them to the client (or to a sort if `ORDER BY` lacks a usable index). `EXPLAIN ANALYZE` (M06) actually *runs* the plan and reports per-operator time/rows — a direct window into this stage.

## 4.11 · The storage engine API & pluggable engines ★

**Mental model.** Here's MySQL's defining architectural feature: the **SQL layer** (parse, optimize, execute — everything in 4.1–4.10) is **separated from storage** by a **handler API**. The executor doesn't know or care *how* rows are physically stored; it just calls generic operations — "open this table, position on this index value, give me the next row, write this row" — and a **pluggable storage engine** (InnoDB, MyISAM, Memory, …) implements them however it stores data. MySQL is, in effect, a SQL front-end that can sit on top of different storage back-ends, swappable per table.

**How it actually works.** The **handler interface** defines a contract: methods to open/close a table, scan or seek within an index, read the next/previous row, insert/update/delete a row, begin/commit a transaction, lock, etc. Each **storage engine** implements this contract over its own on-disk (or in-memory) format and its own concurrency/durability mechanisms. The shared SQL layer parses, plans, and executes engine-agnostically, issuing handler calls; the engine translates them to physical reads/writes. Because the engine is chosen **per table** (`ENGINE=InnoDB`), a single database can mix engines (rare in practice, but possible). The optimizer even asks engines for cost/statistics info (4.7) through this interface, so plans account for engine characteristics.

**Why it exists / what it solves.** Pluggability lets MySQL offer **different storage tradeoffs under one SQL interface**: a transactional crash-safe engine (InnoDB), a fast volatile in-memory engine (Memory) for temp/scratch data, historically a simple fast-read engine (MyISAM), specialized engines (archive, federated). You pick the storage characteristics per table without changing your SQL or application. Architecturally, it's a clean separation of concerns — query processing vs storage — that let the ecosystem innovate on engines (InnoDB itself came from a third party) independently of the SQL layer.

**Tradeoffs & alternatives.** The abstraction has a cost: the handler API is a **row-at-a-time, generic interface**, which historically limited some optimizations (the SQL layer couldn't exploit engine-specific tricks as deeply as a monolithic design might) — though modern features push more down to engines. It also means some capabilities depend on the engine (FKs, transactions, full-text historically varied by engine — a footgun, 4.12). The alternative, a **monolithic engine** (like most other databases — Postgres has one integrated storage layer), can optimize across the storage/query boundary more freely but loses the per-table pluggability. MySQL chose flexibility; the practical reality today is "use InnoDB for almost everything," so the pluggability is more historical/architectural than a daily decision.

**Generics / first-principles.** "Separate the policy (what query to run) from the mechanism (how data is stored) behind a stable interface, and you can swap mechanisms." This is the **strategy pattern / pluggable backend** at architectural scale — like a VFS abstracting filesystems, a JDBC/ODBC abstracting databases, or an OS abstracting device drivers. The principle: a well-defined interface between layers enables independent evolution and substitution. MySQL's handler API is a textbook example, and recognizing it explains both MySQL's flexibility and some of its historical quirks.

**MySQL-specific reality.** The handler API and pluggable engines are MySQL's signature design. The practical truth in 2020s MySQL: **InnoDB is the default and the right choice for essentially all real workloads** (4.12); Memory is occasionally used for transient data; MyISAM is legacy; others are niche. The same `SELECT` runs identically (from the SQL layer's view) over any engine — but the *guarantees* differ wildly (transactions, durability, locking), which is the subject of 4.12. The optimizer consults engine statistics (4.7) via this API. For fintech, the engine choice made through this interface is a **correctness decision**: the ledger's table declares `ENGINE=InnoDB` precisely to get the transactional, crash-safe guarantees the API allows engines to provide — and that the *wrong* engine silently wouldn't.

## 4.12 · InnoDB vs MyISAM (and why InnoDB) ★

**Mental model.** Because the engine is pluggable (4.11), *which engine you choose* decides your table's fundamental guarantees: **transactions, crash-safety, locking granularity, and foreign keys.** **InnoDB** provides all of them (ACID transactions, row-level locking, crash recovery, FKs) and is the correct default for virtually everything. **MyISAM** — the old default — provides *none* of them (no transactions, no crash-safe durability, table-level locking only, no FKs). The choice isn't a performance tuning knob; it's a **correctness and durability decision**, and for anything holding money it's not a choice at all: InnoDB.

**How it actually works.** The two engines differ on guarantees that matter most:
- **Transactions:** InnoDB is fully **transactional** (ACID, M07) — multi-statement atomic commits, rollback, isolation. MyISAM has **no transactions** — each statement stands alone, no rollback, no atomic multi-row operations.
- **Crash safety/durability:** InnoDB is **crash-safe** via its redo log / write-ahead logging (M09) — a committed write survives a crash. MyISAM is **not crash-safe** — a crash mid-write can corrupt tables (requiring repair) and lose data.
- **Locking:** InnoDB does **row-level locking** (M08) — concurrent writers to different rows don't block each other. MyISAM does **table-level locking** — any write locks the whole table, serializing concurrency (catastrophic under write load).
- **Foreign keys / referential integrity:** InnoDB **enforces FKs** (M01/1.5); MyISAM **parses and silently ignores** them (a dangerous footgun).
- **Clustered index/MVCC:** InnoDB clusters on the PK (M01/1.3) and does MVCC (M08); MyISAM has a different, non-transactional structure.
MyISAM's historical appeal was simplicity and fast counts/reads for read-only/read-mostly data; in modern MySQL those edges are largely gone and InnoDB is faster *and* safe for almost all workloads.

**Why it exists / what it solves.** The comparison exists to make *explicit why InnoDB's guarantees matter* — each MyISAM limitation is a way to lose or corrupt data or to serialize concurrency. Choosing InnoDB *solves* the requirements every serious application has: atomic multi-row operations, surviving crashes without data loss, concurrent writes, and enforced relationships. It's the foundation for everything in M07 (transactions), M08 (locking/MVCC), and M09 (InnoDB internals/durability).

**Tradeoffs & alternatives.** Historically MyISAM was smaller on disk and faster for certain read-only/full-text/count(*) cases — but at the cost of zero transactionality, no crash safety, and table locks, which are disqualifying for almost any real workload. Modern InnoDB has closed most performance gaps (and full-text search now works in InnoDB too). So the "tradeoff" is largely historical: **there is essentially no modern reason to choose MyISAM for application data.** The only realistic alternatives to InnoDB today are niche engines for niche purposes (Memory for transient scratch data, specialized analytics engines) — not MyISAM for primary data.

**Generics / first-principles.** "The storage layer's guarantees (atomicity, durability, isolation) are not optional features — they determine whether your data can be trusted." This is the universal lesson that *durability and transactionality are correctness properties, not performance options*. Choosing a non-transactional, non-crash-safe store for data that must not be lost is a category error — the same mistake as using a cache as a system of record, or fire-and-forget messaging for critical events. The engine choice is where the *durability* and *money-never-lies* threads become concrete.

**MySQL-specific reality.** The bottom line: **use InnoDB** (the default since 5.5). InnoDB gives ACID transactions (M07), row locking and MVCC (M08), crash recovery via redo/undo logs (M09), foreign keys (M01/1.5), and a clustered-index layout (M01/1.3) — everything this resource assumes. MyISAM survives only in legacy schemas and as a cautionary contrast. For fintech the rule is absolute: **the ledger, accounts, and all money tables are InnoDB**, because only InnoDB can guarantee a committed transfer survives a crash, that debit and credit commit atomically (M07), that FKs prevent orphaned entries (M01/1.5), and that concurrent transactions don't corrupt each other (M08). A money table on MyISAM would silently violate every one of those — the engine choice *is* a money-safety control.

## 4.13 · Result handling, buffers & sorting (filesort, temp tables)

**Mental model.** Most query operators **stream** (pass rows through as they arrive), but some **can't** — sorting, grouping, `DISTINCT`, and certain joins need to **collect rows into a buffer or a temporary table** before they can produce output. When that working set fits in memory, it's fast; when it's too big, MySQL **spills it to disk** — an on-disk sort ("filesort") or on-disk temp table — which is a **performance cliff** (orders of magnitude slower). Many "why is this query slow?" mysteries are a blocking operation that spilled to disk.

**How it actually works.** Operations that need all their input before emitting:
- **Sorting (`ORDER BY` without a usable index):** MySQL performs a **"filesort"** — it reads the rows, sorts them in a **sort buffer** in memory; if they exceed the buffer, it sorts chunks and merges them **on disk**. (Despite the name, "filesort" isn't always on-disk — small sorts stay in memory; large ones spill.)
- **Temporary tables (`GROUP BY`, `DISTINCT`, some subqueries/UNION, complex joins):** MySQL materializes intermediate results into a **temp table** — in-memory (fast) if small, **on-disk** (slow) if it exceeds limits or contains types that can't go in memory.
The key lever: an **index that already provides the needed order or grouping** lets MySQL *skip* the sort/temp-table entirely (read rows in index order — streaming), turning a blocking, possibly-spilling step into a cheap pipelined one. This is a major reason indexes matter beyond lookups (M05).

**Why it exists / what it solves.** Some results fundamentally require seeing all input (you can't emit sorted output before reading everything; you can't know distinct values until you've seen them). Buffers and temp tables are the mechanism for these blocking operations. The in-memory-vs-disk split exists because memory is fast but finite — MySQL uses memory when it can and falls back to disk to *complete* the operation rather than fail. Understanding this explains a huge class of performance problems and points to the fix (provide an index, or shrink the working set).

**Tradeoffs & alternatives.** Bigger sort/temp buffers → more operations stay in memory (fast) but more memory per connection (and many connections × big buffers can exhaust RAM). Smaller buffers → less memory but more disk spills. The real win is usually **avoiding the operation**: an index that delivers rows pre-sorted/pre-grouped eliminates the filesort/temp table entirely (M05/M06). Other mitigations: reduce the rows being sorted (filter earlier, `LIMIT`), select fewer columns (smaller rows to sort), or restructure the query. The alternative to blocking — streaming — is only possible when the operation doesn't inherently need all input.

**Generics / first-principles.** "Operations that need all input before producing output must buffer; when the buffer overflows memory, they spill to disk — a cliff." This is the universal **blocking vs streaming** and **in-memory vs external** distinction (external sort/merge, external hash, spill-to-disk in any data system). The transferable instincts: *prefer streaming; when you must block, keep the working set in memory; and the cheapest sort is the one you avoided by reading data already in order.* External-algorithm spills are a performance cliff everywhere, not just MySQL.

**MySQL-specific reality.** MySQL specifics: **filesort** appears in EXPLAIN's Extra as "Using filesort"; **temp tables** as "Using temporary"; both are signals to investigate (M06). Buffer settings (`sort_buffer_size`, `tmp_table_size`/`max_heap_table_size`) control the in-memory thresholds; on-disk temp tables use the configured temp storage. Modern MySQL improved in-memory temp tables (TempTable engine) but disk spills are still a cliff. The fix is usually an index: e.g., for "account 42's recent entries `ORDER BY created_at`," a `(account_id, created_at)` index (M01/1.14) delivers rows *already sorted*, so no filesort. Seeing "Using filesort"/"Using temporary" on a hot query is a top tuning target in M06.

## 4.14 · Caching layers in the path (and the dead query cache)

**Mental model.** Several caches sit along the execution path, and the most important one is **not** what people often think. The **buffer pool** (InnoDB's cache of *data and index pages* in RAM, M09) is the big one — it's why a repeated query that hits already-cached pages is fast. There are also dictionary/table caches and a **prepared-statement cache** (skip re-parse/re-plan, 4.4). The notorious one: the old **SQL query cache** (which cached whole *result sets* keyed by query text) is **removed in MySQL 8.0** — a frequent misconception is that it still exists or should be enabled. It doesn't, and you can't.

**How it actually works.** Caches in the pipeline:
- **Buffer pool (the central one):** InnoDB caches data/index **pages** in memory (M09). When a query needs a page that's already cached, no disk read happens — this is the dominant performance cache, and its hit rate is a key health metric. It caches *pages*, not query results, so it speeds up *any* query touching hot data, not just identical repeats.
- **Prepared-statement cache:** parsed/planned statements reused across executions (4.4) — skips re-parse/re-optimize for repeated parameterized queries.
- **Data dictionary / table-definition caches:** metadata about tables/columns kept in memory to avoid re-reading definitions.
- **The (removed) query cache:** older MySQL had a cache mapping exact query text → result set. It was **deprecated in 5.7 and removed in 8.0** because it scaled poorly: every write to a table **invalidated all cached results** for that table, and a global mutex serialized access — on write-heavy or high-concurrency workloads it *hurt* more than helped.

**Why it exists / what it solves.** Caches avoid redoing expensive work — re-reading disk pages (buffer pool), re-parsing/planning (statement cache), re-reading metadata (dictionary cache). The buffer pool in particular is what makes a database with more data than RAM still fast for the *hot* subset: keep the working set in memory, serve most reads without disk. The query cache *tried* to cache final results but its invalidation and contention costs outweighed the benefit, so it was removed — a lesson in cache design.

**Tradeoffs & alternatives.** The buffer pool trades RAM for I/O avoidance — bigger pool = more cached pages = fewer disk reads (the main InnoDB tuning knob, M09), bounded by available RAM. The query cache's removal *is* the tradeoff lesson: a result cache with **coarse invalidation** (any table write nukes all its cached queries) and a **global lock** became a bottleneck under concurrency — caching at the wrong granularity with expensive invalidation can be net-negative. The modern alternative for result caching is an **application-level cache** (Redis/memcached) where the app controls invalidation granularity and it doesn't contend with the DB's write path.

**Generics / first-principles.** "Cache the expensive, reusable work — but the cache is only worth it if invalidation is cheaper than the savings." The query cache's death is a pure illustration of the universal caching tradeoff: **invalidation cost and contention can exceed hit benefit**, especially with coarse keys and write-heavy workloads (echoing M02/2.15's sync-mechanism reasoning). Caching pages (reused by many queries, evicted by LRU) works; caching exact-text results (invalidated wholesale on any write) didn't. The principle: cache at a granularity where reuse is high and invalidation is cheap.

**MySQL-specific reality.** Concrete MySQL facts to know (and a common interview "gotcha"): **the query cache is GONE in 8.0** — don't look for `query_cache_size`, don't recommend enabling it; if you see advice to tune it, that advice is pre-8.0. The **buffer pool** (`innodb_buffer_pool_size`) is *the* cache that matters and the primary memory-tuning knob (M09) — size it to hold the hot working set. Prepared statements cache parse/plan. For result caching, use an external cache (Redis) with app-controlled invalidation. For our domain: repeated "account 42 balance" reads are fast because the relevant InnoDB pages stay in the buffer pool — *not* because of any result cache — which is why keeping rows compact (M03/3.2) to fit more in the buffer pool matters so much. This sets up M09's deep dive on the buffer pool.

## 4.15 · Reading a plan: from concept to EXPLAIN (bridge to M06)

**Mental model.** Everything in this module — the optimizer's chosen access paths (4.8), join order and algorithms (4.9), whether it'll sort or use a temp table (4.13) — is **made visible by `EXPLAIN`**. EXPLAIN shows you the plan the optimizer *chose without running it* (and `EXPLAIN ANALYZE` runs it and reports actuals). Reading an EXPLAIN is literally **reading the optimizer's decisions** — and since you can't tune what you can't see, EXPLAIN is the instrument that turns the abstract pipeline of M04 into the concrete tuning practice of M06.

**How it actually works.** `EXPLAIN <query>` returns the plan as rows/tree, with key columns mapping directly to M04 concepts:
- **access type** (`type`: `const`/`eq_ref`/`ref`/`range`/`index`/`ALL`) → the **access path** chosen per table (4.8) — `ALL` = full scan (often a red flag), `ref`/`range` = index use, `const`/`eq_ref` = PK/unique lookup.
- **key** → which **index** the optimizer chose (or NULL = none).
- **rows** → the optimizer's **estimate** (from statistics, 4.7) of rows examined — a wrong estimate signals stale stats.
- **join order** → the row order reflects the chosen **join order** (4.9).
- **Extra** → flags like "Using index" (covering index, 4.8), "Using filesort"/"Using temporary" (blocking ops, 4.13), "Using where".
`EXPLAIN FORMAT=TREE` shows the operator tree (matching the executor, 4.10); `EXPLAIN ANALYZE` executes and reports **actual** rows/time per operator, exposing estimate-vs-reality gaps. (Full reading, interpretation, and tuning is M06.)

**Why it exists / what it solves.** EXPLAIN closes the loop between "MySQL has an optimizer that makes decisions" and "I can see and influence those decisions." It's the diagnostic that reveals *why* a query is slow (full scan? filesort? bad join order? wrong row estimate?) and therefore *what to fix* (add an index, refresh stats, rewrite). Without it, performance tuning is guessing; with it, the entire M04 pipeline becomes inspectable. It's the single most important tool in the next module.

**Tradeoffs & alternatives.** Plain `EXPLAIN` is free (it doesn't run the query) but shows *estimates*, which can be wrong (stale stats, 4.7); `EXPLAIN ANALYZE` shows *actuals* but **runs the query** (cost/side-effects — careful on writes/expensive queries). The optimizer trace gives even more detail (the cost numbers and rejected plans) at the price of verbosity. The instinct: read EXPLAIN first (cheap, reveals plan shape), use EXPLAIN ANALYZE to confirm estimate-vs-actual when a plan looks wrong, and the trace when you need to understand *why* the optimizer chose what it did.

**Generics / first-principles.** "To optimize a system, first make its decisions observable." EXPLAIN embodies the universal principle that *you can't improve what you can't measure/see* — profilers for code, query plans for databases, traces for distributed systems all serve this role. The transferable instinct: when something is slow, get the system to *show you what it's actually doing* (the plan, the trace) before changing anything. The plan is ground truth; intuition about "what the DB probably does" is often wrong.

**MySQL-specific reality.** MySQL specifics (deep-dived in M06): `EXPLAIN` (tabular), `EXPLAIN FORMAT=TREE` (operator tree, 8.0), `EXPLAIN ANALYZE` (actual execution stats, 8.0), and `OPTIMIZER_TRACE`. The columns (`type`, `key`, `rows`, `filtered`, `Extra`) each map to an M04 concept as above. The everyday MySQL tuning loop is exactly this: **write query → EXPLAIN → spot the bad access path / filesort / wrong estimate → add an index or refresh stats or rewrite → re-EXPLAIN** (M01/1.14's "design for your queries" made operational). For our trace query, EXPLAIN reveals whether "account 42's recent entries" uses the `(account_id, created_at)` index (good: `range`/`ref`, "Using index") or scans the whole ledger (`ALL` — fix with an index). This is the bridge into M06.

## 4.16 · Fintech capstone — the lifecycle of a money query ★

**Mental model.** Trace a *real* payments operation end-to-end through the entire M04 pipeline, and watch **where correctness is won (the engine) and where performance is won (the plan)** — the two things this whole module is about, made concrete on money. "Post a transfer, then read the new balance" touches every stage: connection, auth, parse, resolve, optimize (plan + access paths), execute, storage engine — and its correctness depends entirely on InnoDB's transactional guarantees while its speed depends entirely on the optimizer choosing good access paths over the typed, indexed schema from M01–M03.

**How it actually works — the full trace.** Take Alice→Bob $100 transfer + balance read:
1. **Connection/session (4.2):** the payments service borrows a pooled, least-privilege connection (4.3); its session has `sql_mode` strict, UTC `time_zone` (M03).
2. **Transaction begins (M07 preview):** `START TRANSACTION` — InnoDB opens a transaction; everything until commit is atomic.
3. **Parse + resolve (4.4/4.5):** the `INSERT`s into `ledger_entry` and `UPDATE`s of `balance` are parsed and bound to the real typed columns (M03/3.17).
4. **Optimize (4.6–4.9):** for the balance `UPDATE WHERE account_id = ...`, the optimizer chooses an **access path** — ideally a PK/`(account_id)` index lookup (`eq_ref`/`ref`, 4.8), not a scan; statistics (4.7) confirm it's selective.
5. **Execute (4.10):** the executor drives the inserts and updates, calling InnoDB through the handler API (4.11) to write the entry rows and update the balance — all within the open transaction.
6. **Engine/durability (4.12, M09 preview):** InnoDB writes via its redo log (WAL) so that on `COMMIT` the transfer is **durable** — survives a crash. Row-level locks (M08) let other accounts' transfers proceed concurrently.
7. **Commit:** debit + credit + balance update become atomically visible; the double-entry invariant (`SUM = 0`, M01/1.19) holds.
8. **Read the new balance (4.8/4.13):** "select balance for account 42" — a PK lookup (`const`/`eq_ref`), no sort, served from the buffer pool (4.14) if hot.
9. **Results stream back (4.10)** to the client.

**Why it exists / what it solves.** The capstone proves the module's thesis on the highest-stakes operation: **the engine choice (InnoDB) is why the transfer is correct and durable, and the optimizer's plan choice (index access paths) is why it's fast** — and both are visible/tunable (EXPLAIN, 4.15). It ties M04 to the threads: correctness (money-never-lies) lives at the engine layer (4.11/4.12, M07–M09), performance lives at the optimizer/access-path layer (4.6–4.9, M05–M06), and you can see and shape both.

**Tradeoffs & alternatives.** The trace embodies M04's tradeoffs: transactional safety (InnoDB) costs some throughput vs a non-transactional store, but it's non-negotiable for money (4.12); the optimizer's plan is an *estimate* that good indexes and fresh stats keep correct (4.6/4.7); hot-account balance updates serialize on a row lock (M08 — a real contention point addressed in M08/M16). The alternatives are all worse for money: a non-transactional engine (data loss), no indexes (scans that degrade as the ledger grows), or stale stats (plan regressions). The design that wins: InnoDB + query-shaped indexes (M01/1.14, M05) + fresh statistics, all observable via EXPLAIN.

**Generics / first-principles.** "Correctness is decided at the storage/transaction layer; performance is decided at the planning/access-path layer — and a mature engineer reasons about both for any query." The capstone synthesizes the module's threads: generics-first (this pipeline shape is universal), tradeoff (the optimizer is a cost machine; the engine is a guarantees-vs-speed choice), durability and money-never-lies (only a transactional crash-safe engine makes a money write trustworthy). The transferable instinct for *any* critical query: ask "what guarantees does the storage layer give me?" *and* "what plan/access path will the optimizer use?" — separately, because they're won at different layers.

**MySQL-specific reality.** The concrete MySQL realization: pooled connections (often via ProxySQL, M10); InnoDB engine for all money tables (4.12); `START TRANSACTION`/`COMMIT` with InnoDB's redo log providing durability (M09) and row locks providing concurrency (M08); the optimizer choosing index access paths over the M03-typed, M01/1.14-indexed schema; `EXPLAIN` to verify those paths (4.15/M06); the buffer pool serving hot balance reads (M09). This single trace is the spine that the rest of the resource fleshes out: **M05** makes the access paths fast (indexing), **M06** makes them visible (EXPLAIN), **M07** makes the transaction atomic, **M08** handles the concurrency/locking, **M09** provides the durability, and **M16** scales the whole thing into a platform. M04 is the map; the lifecycle of a money query is the territory it describes.
