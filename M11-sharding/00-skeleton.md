# M11 · Partitioning, Sharding & Scaling Out ★ — Pass A (Skeleton)

> **Pass A goal:** lock the ordered concept list, a one-line mental model per concept, and the planned diagram + bespoke-SVG inventory + worked example per concept. No prose yet — that's Pass B.
>
> **Module mental model:** *Replication (M10) made copies of the* same *data — it scales **reads** and buys HA, but every replica still holds the* whole *dataset and applies* every *write, so it does nothing for **write** throughput or for a dataset too big for one machine. **Sharding** is the answer: split* different *data across* different *servers (shards), so each shard holds a fraction of the rows and takes a fraction of the writes — horizontal scaling of writes and storage. The entire game is the **shard key**: the column you split on. Choose it well and most queries hit one shard, related rows (a transfer's debit and credit) live together, and load spreads evenly. Choose it badly and you get **hotspots** (one shard melts), **cross-shard queries** (fan out to every shard, slow), **cross-shard transactions** (no single-node ACID — you're now in distributed-transaction territory, M12), and **resharding pain** (re-splitting a live dataset is one of the hardest operations in data engineering). Before sharding there's **partitioning** (split one table across files* within *one server — same engine, no cross-node distribution) — a lighter tool for manageability, not write scale. Sharding is the step where you accept real distributed-systems complexity in exchange for scale a single node can't give — and where, for money, you fight to keep each transfer's two legs on* one *shard so it stays a local ACID transaction.*
>
> **Threads carried in this module:**
> - **Durability** — sharding doesn't change single-shard durability (each shard is a full InnoDB node with M09's posture, usually itself replicated — M10), but it changes *what a single durable commit covers*: a transfer spanning two shards is no longer one durable atomic commit. The durability question becomes "is this operation within one shard (still atomic+durable) or across shards (needs a distributed protocol)?"
> - **Money-never-lies** — *the* fintech sharding fight: **keep a transfer's debit and credit on the same shard** so the double-entry write stays a single-node ACID transaction (shard by a key that co-locates both legs). When an operation *must* cross shards, you've left single-node ACID and money can be lost/duplicated unless you use the distributed patterns (M12: 2PC/Saga/outbox). Hot-account contention (one popular account = one hot shard) is the recurring money hazard.
> - **Generics-first** — horizontal partitioning, the shard key / partition function, consistent hashing, the routing layer, distributed ID generation, scatter-gather, and resharding are universal distributed-data concepts (every sharded store: Vitess, Cassandra, DynamoDB, Citus, MongoDB); MySQL/Vitess is one realization.
> - **Tradeoff** — sharding trades single-node simplicity (ACID, joins, one place to look) for write/storage scale. Every cross-shard operation is the cost; the shard-key choice is the irreversible bet. Partition vs shard, range vs hash, app-level vs Vitess — each is a named tradeoff.
>
> **Prereqs:** M10 (replication — scales reads/HA; sharding scales *writes*; each shard is usually itself a replicated set — the two compose into the full topology), M09 (single-node durability — what a shard still guarantees locally), M08 (single-node locking/MVCC — what you *lose* across shards; hot-row contention generalizes to hot-shard), M07 (transactions — single-shard ACID vs the cross-shard problem that needs M12), M02 (normalization + the derived-balance/reconciliation pattern — what's hard to keep consistent across shards), M03/3.12 (ID storage — distributed ID generation: UUID/ULID/Snowflake as PKs), M05 (indexing — the shard key relates to the clustered key; routing needs an index-like map), M01/1.x (the access patterns that determine a good shard key). **Leads into:** M12 (Distributed Data Concerns — cross-shard transactions are the distributed-transaction problem: CAP/PACELC, 2PC/XA, Saga, outbox, the dual-write problem — the *generalization* of M11's cross-shard pain), M13 (Ops — resharding/migrations, backups across shards), M15 (Failure Modes — a shard loss, a botched resharding, a cross-shard partial write losing money), M16 (the platform's sharded topology — sharding the ledger by account, keeping transfers single-shard, the reporting offload).

## Concept list (ordered)

| # | Concept | One-line mental model | Diagram | Worked example |
|---|---------|----------------------|---------|----------------|
| 11.1 | **Scaling up vs out; read vs write scaling** | Vertical scaling (bigger box) hits a ceiling; horizontal scaling (more boxes) is the only path past it — and replicas scale *reads* while sharding scales *writes*, the two distinct problems. ★ | The scaling map: up vs out × read (replicas) vs write (shards) | Why a payments platform that out-scaled read-replicas still has a single write bottleneck — and what fixes it |
| 11.2 | **Partitioning: splitting a table within one server** | Native partitioning splits one big table into sub-tables (partitions) by a function (range/list/hash/key) — *inside one server* — for manageability and partition pruning, **not** write scaling. | Partitioning: one logical table → N partitions on one node | An append-only ledger range-partitioned by month: old months prune away from hot queries, drop-partition vs DELETE for retention |
| 11.3 | **Partitioning types & pruning (range/list/hash/key)** | The partition function decides which rows go where — range (time/sequence), list (discrete), hash/key (even spread) — and the optimizer *prunes* partitions a query can't match. | The four partition functions + how pruning skips partitions | Why range-by-time prunes a "last 7 days" query to one partition; why a query without the partition key scans all |
| 11.4 | **Partitioning limits & gotchas** | Partitioning's sharp edges: every unique key must include the partition column, no foreign keys, cross-partition queries scan all, and it's still *one server* — knowing when it helps vs misleads. | Partitioning limits (unique-key rule, no FK, one-node ceiling) | The trap: partitioning a hot table expecting write-scaling, getting none because it's still one machine |
| 11.5 | **Sharding: splitting data across servers** ★ | Sharding spreads *different rows* across *independent servers* (shards), each a full database holding a fraction of the data and absorbing a fraction of the writes — true horizontal write/storage scale. ★ | ★ sharding: rows split across N independent shard servers (vs partitioning/replication) | The ledger split across 8 shards by account: each shard holds ~⅛ of accounts and takes ~⅛ of the write load |
| 11.6 | **The shard key: the most important decision** ★ | The shard key is the column you split on — it determines co-location, query routing, load distribution, and hotspots; it's the hardest-to-change, highest-leverage choice in the whole system. ★ | ★ the shard-key decision (co-location · routing · even spread · hotspots) | Choosing the shard key for payments: account_id vs customer_id vs transaction_id — what each co-locates and breaks |
| 11.7 | **Sharding schemes: range, hash, directory** ★ | *How* the key maps to a shard: range (ordered, prunable but hotspot-prone), hash (even spread but no range scans), or directory/lookup (flexible, indirection) — each a tradeoff. ★ | ★ range vs hash vs directory mapping (spread vs range-query vs flexibility) | Range-by-account-id hotspots the newest shard; hashing spreads evenly but kills "accounts 1000–2000" range scans; a lookup table buys flexibility |
| 11.8 | **Consistent hashing & minimizing reshuffle** ★ | Naive `hash % N` remaps almost everything when N changes; consistent hashing (a hash ring with virtual nodes) moves only ~1/N of keys when you add/remove a shard — the resharding-friendly scheme. ★ | ★ the hash ring: virtual nodes, add-a-shard moves only its arc | Adding a 9th shard: `mod 8 → mod 9` reshuffles ~everything; the ring moves only ~1/9 — why it matters for live resharding |
| 11.9 | **Co-location: keeping related data together** ★ | Shard so that data accessed together lives together — *the* fintech rule: a transfer's debit and credit on the **same shard** keep it a single-node ACID transaction instead of a distributed one. ★ | ★ co-location: same shard key → both legs of a transfer on one shard (local ACID) | Sharding by account vs by a "ledger group" so both sides of every internal transfer land on one shard; what to do for cross-customer transfers |
| 11.10 | **Cross-shard queries & scatter-gather** | A query not filtered by the shard key must fan out to *every* shard and merge results (scatter-gather) — slow, and aggregations/joins/sorts/pagination get hard across shards. | Scatter-gather: fan out to all shards → merge (vs single-shard hit) | A "total balance across all of a customer's accounts on different shards" query fanning out to N shards and summing |
| 11.11 | **Cross-shard transactions (the hard problem)** ★ | An operation spanning shards can't use single-node ACID — you need a distributed protocol (2PC/Saga, M12) — so the design goal is to *avoid* cross-shard transactions, not to do them well. ★ | ★ single-shard ACID vs cross-shard (no atomic commit → distributed protocol) | A transfer between two accounts on *different* shards: why it's no longer atomic, and the options (move money via a single-shard intermediary, Saga, 2PC — M12) |
| 11.12 | **Distributed ID generation** | Auto-increment breaks across shards (every shard would reuse IDs) — you need globally-unique IDs without a single bottleneck: UUID/ULID, Snowflake (time+node+seq), or a central allocator. | ID schemes: UUID vs ULID vs Snowflake vs ticket-server (spread vs sortable vs coordination) | Why per-shard AUTO_INCREMENT collides; choosing ULID/Snowflake for the ledger's IDs so they're unique, sortable, and don't hotspot one shard |
| 11.13 | **The routing layer (Vitess & app-level)** ★ | Something must map each query to the right shard(s) — app-level routing (simple, intrusive) or a middleware like **Vitess/ProxySQL** (a proxy that shards transparently, handles routing/resharding) — the shard-aware "router." ★ | ★ the routing layer: app → router (shard map) → shards (+ Vitess vtgate/vttablet) | How Vitess routes a keyed query to one shard and a scatter query to all, transparently to the app |
| 11.14 | **Resharding: re-splitting live data** ★ | Changing the number of shards (or the key) on a *live* system is one of the hardest operations — move data without downtime or loss, keep routing consistent during the cutover; consistent hashing + tooling (Vitess reshard) make it survivable. ★ | ★ resharding flow: copy → sync (CDC/backfill) → cutover → verify | Going from 8 → 16 shards live: backfill + binlog-catch-up (M10/M14) + atomic routing cutover + reconciliation — without losing a transfer |
| 11.15 | **Choosing partition vs shard vs replica (the decision)** | Match the tool to the problem — partition for manageability, replicas for read scale + HA, shards for write/storage scale — and shard *only* when you must, because it's a one-way door of complexity. | Decision flow: symptom → partition / replica / shard (and shard-key derivation) | Diagnosing a payments platform: reads→replicas, big-but-one-box→partition, write-bound→shard by account; deriving the key |
| 11.16 | **Fintech capstone: the sharded ledger** ★ | The full sharded payments topology — shard the ledger by account so each transfer's legs co-locate (single-shard ACID), each shard itself replicated (M10) for HA/reads, distributed IDs, cross-shard transfers via a distributed pattern, reporting offloaded — write-scale without losing money. ★ | ★ the sharded+replicated payments topology (annotated: shard key, co-location, per-shard replicas, routing, cross-shard path, reporting) | End-to-end: the platform sharded by account — what's single-shard (most transfers), what's cross-shard and how it's handled, how it composes with M10 (sets up M12/M16) |
