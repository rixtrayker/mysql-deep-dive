## 11.1 · Scaling up vs out; read vs write scaling

**Mental model.** There are two axes to growing past a single machine, and conflating them is the classic mistake. **Up vs out:** *scaling up* (vertical) means a bigger box — more CPU, RAM, faster disks; simple, keeps everything in one place (one ACID database, normal joins), but hits a hard ceiling (the biggest box money can buy, and one box is still a single point of failure). *Scaling out* (horizontal) means *more boxes* — the only path past that ceiling, but it introduces distribution. **Read vs write:** **replicas (M10) scale reads** — copies of the *same* data absorb read fan-out — but do *nothing* for writes (every replica applies every write). **Sharding scales writes** — splitting *different* data across boxes so each takes a fraction of the writes and storage. The four-cell map (up/out × read/write) tells you which tool fits which bottleneck: read-bound → replicas; write-/storage-bound and past the biggest box → shards.

**How it actually works.** A single MySQL server has finite write throughput (one set of disks, one redo log, one buffer pool, M09) and finite storage. **Vertical scaling** raises those limits (more cores, more RAM for a bigger buffer pool, faster NVMe, more IOPS) — and is *always the first move* because it's free of distribution complexity; you ride one box as far as it goes. **Read scaling via replicas (M10):** add replicas, route SELECTs to them — read capacity multiplies with replica count, but write capacity is unchanged (the source still does every write, and each replica *also* applies every write, so replicas don't relieve write load — they *duplicate* it). **Write scaling via sharding:** partition the dataset across N independent servers so each holds ~1/N of the rows and absorbs ~1/N of the writes — write throughput and storage scale with shard count. The two compose: a sharded system where *each shard* is itself a replicated set (M10) gets both write scale (across shards) and read scale + HA (within each shard).

**Why it exists / what it solves.** Because the limits are different problems with different fixes, and applying the wrong fix wastes effort and money. Adding replicas to a *write*-bound system does nothing (the write bottleneck is unchanged — a common and expensive mistake). Buying a bigger box helps both until the ceiling, then stops. Sharding is the *only* thing that scales writes past one machine, but it's the most complex, so you reach for it *last* — after vertical scaling and read-replicas are exhausted. Naming the axes precisely (up/out, read/write) is what lets you diagnose the actual bottleneck and pick the tool that addresses it, rather than cargo-culting "add more servers."

**Tradeoffs & alternatives.** **Vertical (up):** simplest (one ACID box, normal SQL), but a ceiling + single point of failure. **Replicas (out, reads):** scale reads + HA, but not writes, and add lag/staleness (M10/10.5). **Sharding (out, writes):** scales writes/storage, but the heavy cost — cross-shard queries/transactions, resharding pain, a one-way complexity door (the rest of this module). The discipline: **exhaust the cheaper tools first** (vertical, then replicas) and shard *only* when genuinely write-/storage-bound past one box, because sharding's complexity is permanent. Alternatives that delay sharding: aggressive vertical scaling, caching (push reads off the DB entirely), archiving cold data, and *functional partitioning* (split *different tables*/services onto different servers — "vertical sharding" — simpler than splitting one table's rows).

**Generics / first-principles.** "Scale up until you can't, then scale out — and know whether your bottleneck is reads (replicate) or writes (partition the data), because they need opposite tools." This is universal capacity planning: *vertical scaling* (bigger node) vs *horizontal scaling* (more nodes), and within horizontal, *replication* (copies for read/availability) vs *partitioning/sharding* (splits for write/storage capacity). The transferable insight: **replication multiplies read capacity by copying; sharding multiplies write capacity by dividing** — copies share the read load but each still bears every write; splits divide both. Every scaled datastore (Cassandra, Mongo, Kafka, Citus, DynamoDB) is some composition of "replicate the data for reads/HA" and "partition the data for writes/storage" — the same two primitives.

**MySQL-specific reality.** MySQL's path: vertical scaling first (a well-tuned single InnoDB instance on modern hardware handles enormous load — huge buffer pool, fast NVMe, M09); then **read-replicas** (M10) for read scaling + HA (route reporting/reconciliation reads to replicas, M02/2.17); then, when write-bound, **sharding** (application-level or via **Vitess**, 11.13). MySQL has **native partitioning** (11.2) but that's a single-node manageability tool, *not* horizontal scaling (a common confusion this module dismantles). For our domain: a payments platform rides a big primary + read-replicas a long way (most fintech write volumes fit one well-tuned box); sharding the ledger (11.5/11.16) comes only when write throughput or dataset size genuinely exceeds one machine — and then each shard is itself replicated (M10) for HA/reads. Reach for sharding last; it's the irreversible step into distributed-systems complexity.

## 11.2 · Partitioning: splitting a table within one server

**Mental model.** Before sharding (across servers) there's **partitioning**: splitting *one big table* into multiple physical sub-tables (**partitions**) **inside a single server**, by a rule (a partition function on a column). The table still looks like one table to queries; underneath, its rows are filed into partitions (e.g., one partition per month). The wins are *manageability and query pruning* — you can drop a whole partition instantly (instant data retention), and the optimizer can **prune** (skip) partitions a query can't match — *not* write scaling (it's still one server, one set of disks, one buffer pool). Partitioning is a single-node organizational tool that people constantly mistake for sharding; the defining fact is **it does not distribute across machines**.

**How it actually works.** You define a table as partitioned by a **partition function** over a column (range/list/hash/key, 11.3) — e.g., `PARTITION BY RANGE` on a date column, one partition per month. InnoDB stores each partition as a separate internal structure (its own B+Tree/segment). A query that filters on the partition column lets the optimizer **prune**: it inspects only the partitions whose range can contain matching rows and skips the rest (so "last 7 days" touches one or two monthly partitions, not all 60). **Dropping a partition** (`ALTER TABLE … DROP PARTITION`) instantly removes all its rows (deallocate the segment) — vastly cheaper than `DELETE … WHERE date < …` (which is row-by-row, generates undo/redo, and leaves fragmentation). New partitions are added as time advances (`REORGANIZE`/`ADD PARTITION`). All of this happens on *one server* — the partitions share its CPU, RAM, and disks.

**Why it exists / what it solves.** Partitioning solves *single-table manageability at scale*: (1) **fast retention** — drop old data by dropping a partition (instant, no DELETE storm), critical for append-heavy tables (logs, ledger history) with retention policies; (2) **query pruning** — restrict scans/index ranges to relevant partitions, improving large-range queries that align with the partition key; (3) **maintenance** — operate on one partition at a time (rebuild, optimize). It does *not* solve write throughput or storage-past-one-box (that's sharding). Its value is making a *very large single-server table* manageable and keeping time-bounded queries fast — a real but bounded benefit, easy to over-apply.

**Tradeoffs & alternatives.** **Wins:** instant partition drop (retention), partition pruning (range queries), per-partition maintenance. **Costs/limits (11.4):** every **unique key (incl. the PK) must contain the partition column** (a hard, often disqualifying constraint), **no foreign keys** on partitioned tables, queries *not* filtering on the partition key scan *all* partitions (worse than a plain indexed table sometimes), and it's **still one server** (no horizontal scale, no HA benefit). Alternatives: a plain indexed table (often just as good if you don't need partition-drop retention); **archiving** to a separate table/store for cold data; **sharding** (11.5) if the real need is horizontal scale (different problem). Use partitioning when the win is specifically *retention-by-drop* or *time-range pruning* on one server — not as a scaling strategy.

**Generics / first-principles.** "Split one table into managed sub-ranges by a key, on one node, to prune scans and drop ranges wholesale — *intra-node* partitioning, distinct from *inter-node* sharding." This is the universal distinction between **partitioning** (organizing data within a node for locality/pruning/lifecycle) and **sharding** (distributing data across nodes for capacity) — terms often used loosely but mechanically different. The transferable insight: **partitioning is about *organizing* data (one machine, faster pruning, cheaper drops); sharding is about *distributing* data (many machines, more capacity)** — they answer different questions and can even combine (each shard internally partitioned). Many systems (Postgres declarative partitioning, time-series DBs, Hive/Parquet partition pruning) use the same intra-node partitioning idea for the same retention/pruning wins.

**MySQL-specific reality.** MySQL/InnoDB native partitioning: `PARTITION BY RANGE/LIST/HASH/KEY (...)` with named partitions; `ALTER TABLE … DROP/ADD/REORGANIZE PARTITION`; partition pruning is automatic when the query filters on the partition expression (visible in `EXPLAIN`'s `partitions` column). The **unique-key-must-include-partition-column** rule and **no-foreign-keys** limit are the big gotchas (11.4). For our domain: the **immutable ledger-entry/transaction history table** is the prime partitioning candidate — **range-partition by month** (on the entry timestamp) so retention is a monthly `DROP PARTITION` (instant, vs a catastrophic `DELETE` storm on a huge append-only table) and "recent activity" queries prune to recent partitions. But note: the PK must then include the partition date (e.g., `(entry_id, created_month)`), and you lose FKs — design accordingly. Partitioning the *hot accounts/balances* table for "write scaling" is the trap (11.4) — it's still one server. Partitioning = retention + pruning on one box; for write scale, you shard (11.5).

## 11.3 · Partitioning types & pruning (range/list/hash/key)

**Mental model.** The **partition function** decides which partition a row lands in, and there are four kinds, each fitting a different data shape. **RANGE:** partition by ordered ranges of a value (e.g., dates by month, IDs by block) — natural for *time-series/sequential* data, enables range pruning and drop-oldest retention, but newest data concentrates in the newest partition (a hotspot). **LIST:** partition by an explicit set of discrete values (e.g., region codes) — for categorical data. **HASH:** partition by `hash(column) mod N` — spreads rows *evenly* across N partitions (no hotspot), but loses range-pruning (a range query can't prune — any range spans all partitions). **KEY:** like HASH but using MySQL's internal hashing (works on non-integer keys, multi-column). **Pruning** is the payoff: when a query filters on the partition column, the optimizer skips partitions that can't match — but pruning only works for the partition *type*'s strengths (RANGE/LIST prune range/equality; HASH/KEY prune only exact-value equality).

**How it actually works.**
- **RANGE:** `PARTITION BY RANGE (col)` with `VALUES LESS THAN` boundaries — a row goes to the first partition whose upper bound exceeds its value. A query `WHERE col BETWEEN x AND y` prunes to the partitions covering [x,y]. Time-partitioning (RANGE on a date/`TO_DAYS()`/`UNIX_TIMESTAMP()`) is the canonical use: monthly partitions, prune by date, drop oldest.
- **LIST:** `PARTITION BY LIST (col) VALUES IN (...)` — each partition owns an explicit value set; rows route by membership. Prunes on equality/IN over those values.
- **HASH:** `PARTITION BY HASH (col) PARTITIONS n` — row → `MOD(col, n)` (or `MOD(hash(expr), n)`). **Even distribution** (good for spreading write/scan load across partitions on one box), but a *range* query can't prune (the range scatters across all partitions). Prunes only on exact equality of the column.
- **KEY:** like HASH but MySQL's own hash function (handles strings, multi-column, no integer requirement).
- **Pruning** (all types): `EXPLAIN … partitions` shows which partitions a query touches; non-matching ones are skipped entirely (not scanned).

**Why it exists / what it solves.** Different data has different access shapes, and the partition function must match the *queries* to deliver pruning. Time-series data (the ledger history) is queried *by time range* and aged out *oldest-first* → **RANGE by time** (prune by range, drop oldest). Categorical data queried by category → **LIST**. Data with no natural range that just needs *even spreading* (balance the load/size across partitions, no range queries) → **HASH/KEY**. The wrong function kills pruning: HASH-partition time-series and your time-range queries can't prune (they scan all partitions); RANGE-partition by a value your queries don't filter on and you get no benefit. The type choice is "match the partition key + function to how the data is queried and aged."

**Tradeoffs & alternatives.** **RANGE:** great pruning + drop-oldest retention for sequential data, but the newest partition is a write hotspot (all new rows land there) and uneven partition sizes. **LIST:** clear categorical routing, but you must enumerate values (new categories need DDL). **HASH/KEY:** even distribution (no hotspot, uniform sizes), but **no range pruning** (range queries scan all) and no meaningful retention-by-drop (rows scatter). The choice is essentially **range-pruning + easy retention (RANGE/LIST)** vs **even spread (HASH/KEY)** — you can't have both (ordered ranges cluster; even hashing scatters). For the ledger history: RANGE-by-time wins (queries are time-bounded, retention is age-based) despite the newest-partition hotspot. There's no universally best function — it's dictated by the query and lifecycle pattern.

**Generics / first-principles.** "The partition/shard function trades *range-locality* against *even-distribution* — ordered ranges prune and age well but cluster load; hashing spreads load but destroys range-locality. You pick which property you need." This is *the* universal partitioning tradeoff, and it reappears identically in **sharding schemes** (11.7 — range vs hash sharding is the same choice across nodes). The transferable insight: **any keyed distribution is either order-preserving (range queries work, but hotspots) or order-destroying (even spread, but no range queries)** — hashing buys uniformity by destroying order; ranging preserves order at the cost of uniformity. This single tradeoff governs partitioning *and* sharding *and* index design (a B+Tree is order-preserving, M05; a hash index isn't) — it's a deep, recurring structural choice.

**MySQL-specific reality.** `PARTITION BY RANGE | LIST | HASH | KEY`. RANGE commonly over `TO_DAYS(date)`/`YEAR(date)`/`UNIX_TIMESTAMP()` for time-partitioning; `RANGE COLUMNS`/`LIST COLUMNS` allow multi-column and non-integer range/list partitioning. `EXPLAIN` exposes the `partitions` touched (verify pruning works). Subpartitioning (HASH/KEY within RANGE/LIST) exists but is rarely worth the complexity. For our domain: the **ledger history → RANGE by month** (time-bounded queries prune; monthly `DROP PARTITION` for retention) — the dominant pattern. A **HASH/KEY** partition would only fit a table queried purely by exact key with a need to spread size evenly on one box (rare for us). The key lesson carried into sharding: **RANGE = pruning + hotspots; HASH = even spread + no range queries** — exactly the shard-scheme tradeoff (11.7), just within one node here.

## 11.4 · Partitioning limits & gotchas

**Mental model.** Native partitioning has sharp, often-disqualifying edges that you must know *before* choosing it — otherwise you adopt it expecting one thing and get bitten. The big three: (1) **every unique key (including the primary key) must contain the partition column** — because MySQL enforces uniqueness *per partition* (it can't cheaply check global uniqueness across partitions), so the partition column must be part of every unique constraint, which often forces an unnatural PK; (2) **no foreign keys** — partitioned tables can't have FKs (in or out), so referential integrity must move to the application; (3) **it's still one server** — partitioning gives *zero* write-scaling, HA, or storage-past-one-box benefit (the most common misconception). Plus: queries not filtering on the partition column scan *all* partitions (sometimes worse than an unpartitioned indexed table).

**How it actually works.** The **unique-key rule** is structural: MySQL checks uniqueness within each partition's own index; to guarantee *global* uniqueness without cross-partition checks on every insert, the partitioning column must be part of the unique key (so equal-unique-key rows always land in the *same* partition where the local check suffices). This means a ledger table partitioned by month can't have a plain `PRIMARY KEY (entry_id)` — it must be `PRIMARY KEY (entry_id, month)` (or partition by something already in the PK). **No FKs:** the partition engine doesn't support foreign-key constraints, so `account_id` referencing `account(id)` can't be enforced by the DB on a partitioned table — the app (or triggers) must enforce it. **All-partition scans:** a query like `WHERE account_id = ?` on a table partitioned by *month* can't prune (account_id isn't the partition key) → it scans the relevant index in *every* partition (N index lookups instead of one) — potentially *slower* than the same table unpartitioned.

**Why it exists / what it solves.** These aren't arbitrary — they fall out of partitioning being a *physical split with local-only enforcement*. Global uniqueness and cross-table FKs would require cross-partition coordination on every write (expensive), so MySQL forbids them rather than pay that cost silently. Understanding *why* (local enforcement) explains all three limits and tells you when partitioning fits: tables where the natural access and uniqueness already align with the partition key (append-only history keyed by time, where `(id, time)` is an acceptable PK and FKs aren't needed). The limits exist to keep partitioning *cheap* (no cross-partition coordination) — which is also why it doesn't scale writes (no distribution, just local filing).

**Tradeoffs & alternatives.** The limits make partitioning a *narrow* tool: great for **append-only, time-keyed, FK-free, range-queried** tables (ledger history, logs, events) with retention needs; poor for **relational, FK-rich, point-queried-by-non-partition-key** tables (the operational accounts/balances). When partitioning's limits bite, the alternatives: a **plain indexed table** (keep FKs and a natural PK; lose drop-partition retention — use scheduled archiving instead); **archiving** cold rows to a history table; or, if the real need is *scale*, **sharding** (11.5 — different tool, different problem). The decision: partitioning is worth its limits *only* when you specifically need partition-drop retention or strong range-pruning on a table whose shape tolerates the unique-key/FK constraints.

**Generics / first-principles.** "Physical partitioning enforces constraints *locally* per partition — so global constraints (unique, foreign keys) either must align with the partition key or be given up; and local filing never adds capacity." This generalizes to *all* partitioned/sharded systems: **distributing data breaks global constraints** — global uniqueness, cross-partition foreign keys, and cross-partition transactions all become expensive or impossible once data is split, whether across partitions (here) or across shards (11.10/11.11). The transferable insight: **the moment you split data by a key, anything that needs to be true *across* the split (global uniqueness, cross-key FKs, cross-key transactions) becomes hard** — partitioning is the gentle, single-node preview of the *same* constraints sharding imposes harshly across nodes. Learn the pain here; it's the whole story of sharding.

**MySQL-specific reality.** The hard rules: **unique keys (PK included) must contain the partitioning column**; **no foreign keys on partitioned tables**; **pruning only when filtering on the partition expression** (else all-partition scan, check `EXPLAIN partitions`). Also: some functions are restricted in partition expressions, `ALTER`s can be expensive, and a max partition count exists. For our domain: partitioning the **ledger history by month** works *because* `(entry_id, created_month)` is an acceptable PK, the history is append-only and queried by time, and FKs to it aren't needed (it references accounts, not vice-versa, and you can enforce that app-side). Partitioning the **accounts/balances** table by month would be wrong (no time dimension, point-queried by account_id, FK-related, needs a natural PK) — and would scan all partitions for every `WHERE account_id=?`. The deeper lesson: **these limits are sharding's constraints in miniature** — global uniqueness, cross-split FKs, and cross-split transactions are exactly what you'll fight at the shard level (11.10/11.11/11.12). Partitioning teaches the shape of the problem.

## 11.5 · Sharding: splitting data across servers ★

**Mental model.** **Sharding** is the real horizontal-scale step: spread *different rows* across *multiple independent servers* (**shards**), where each shard is a *complete, standalone database* holding a **fraction** of the data and absorbing a **fraction** of the writes. If replication (M10) is "many copies of the *same* data" (scales reads), sharding is "the data *divided* across nodes" (scales writes + storage). Account rows 0–999k live on shard 1, 1M–1.999M on shard 2, etc. (by some scheme, 11.7) — so writes to different accounts hit different shards in parallel, and total write throughput + storage scale with shard count. The price is steep and permanent: no single-node ACID across shards, no cross-shard joins for free, queries must know *which* shard, and re-splitting later is brutal (11.14). Sharding is the line where "a database" becomes "a distributed data system," and you cross it only when you must.

**How it actually works.** You pick a **shard key** (11.6) — the column that decides which shard a row lives on — and a **scheme** (11.7) mapping key → shard (range, hash, or a lookup directory). Each shard is an *independent MySQL server* (itself usually a replicated set, M10, for HA/reads) running the *same schema* but holding only *its slice* of the rows. A **routing layer** (11.13 — app-level or Vitess) inspects each query's shard key and sends it to the right shard. A write to account 1,234,567 is routed to whatever shard owns that key and committed there as a *normal single-node ACID transaction* (M07–M09 all still apply *within* a shard). Writes to different shards proceed *fully in parallel* (independent servers, independent disks/redo logs) → write throughput scales ~linearly with shards. The hard parts are everything that needs *more than one shard*: a query without the shard key (scatter-gather across all, 11.10), a transaction spanning shards (no atomic commit, 11.11), globally-unique IDs (11.12), and changing the shard count later (11.14).

**Why it exists / what it solves.** Sharding solves the *one limit replication can't*: **write throughput and storage past a single machine**. A single InnoDB server, however big, has one set of disks, one redo log, one buffer pool — a finite write ceiling and a finite max dataset (M09). Replicas don't help (each applies every write). The *only* way to do more writes-per-second or hold more data than one box allows is to **divide the data across boxes** so each handles a part — that's sharding. For a payments platform that genuinely out-scales the biggest single primary (extreme write volume or dataset), sharding the ledger by account lets write load and storage spread across N shards. It exists for exactly the case "we are write-/storage-bound past one machine and have exhausted vertical scaling and read-replicas" — and nothing else solves that.

**Tradeoffs & alternatives.** Sharding buys **linear write/storage scale** at the cost of: **loss of single-node ACID across shards** (cross-shard transactions need distributed protocols, 11.11, M12 — *the* fintech cost); **cross-shard queries are scatter-gather** (slow, hard to aggregate/sort/paginate, 11.10); **the shard key is a near-irreversible bet** (resharding is brutal, 11.14); **operational complexity multiplies** (N servers × their replicas to back up, monitor, migrate — M13); and **referential integrity / global uniqueness break** (like partitioning's limits, 11.4, but across nodes). Alternatives to *delay or avoid* it: maximal vertical scaling, read-replicas (M10), caching, archiving cold data, and **functional partitioning** (split *different tables/services* to different servers — simpler than row-sharding one table). The discipline: **shard last, shard reluctantly, and design hard to keep operations single-shard** (11.9) so you pay the cross-shard cost as rarely as possible.

**Generics / first-principles.** "Divide the dataset across nodes by a key so each node owns a disjoint slice — scaling write and storage capacity linearly, at the cost of any operation that spans slices." This is the universal **horizontal partitioning / sharding** principle behind *every* system that scales writes past one node: Cassandra/Dynamo (partition by hash of key), MongoDB (sharded collections), Citus/Vitess (sharded Postgres/MySQL), Kafka (topic partitions), Elasticsearch (index shards). The transferable insight: **sharding scales by *division* (disjoint slices, capacity adds up) where replication scales by *duplication* (identical copies, reads add up) — and division's universal tax is that cross-slice operations (multi-key transactions, multi-key queries/joins, global constraints) lose the guarantees a single node gave for free.** The entire discipline of sharded-system design is *minimizing cross-slice operations* by choosing the partition key so that what's accessed together lives together (11.9) — a principle identical across all these systems.

**MySQL-specific reality.** MySQL has **no built-in sharding** — you do it **application-level** (the app computes the shard from the key and connects to the right server — full control, but the app owns routing, resharding, cross-shard logic) or via **Vitess** (11.13 — a mature MySQL sharding middleware, used at YouTube/Slack scale: it presents a single logical database, routes queries by shard key via `vtgate`, manages shards/`vttablet`, and crucially *automates resharding* — the hardest part). Each shard is a normal MySQL server (usually a primary + replicas, M10) running the same schema over its key-range slice; within a shard, everything you know (InnoDB ACID M07–M09, indexing M05, locking M08) applies unchanged. For our domain: shard the **ledger by account** (11.6/11.9) so each shard owns a disjoint set of accounts and their entries; a transfer between two accounts on the *same* shard stays a single-node ACID transaction (the design goal, 11.9); each shard is replicated (M10) for HA + to offload reporting/reconciliation (M02/2.17); Vitess (or app-level routing) directs queries. The rest of the module is *how to shard money safely* — choosing the key (11.6), the scheme (11.7/11.8), co-locating transfers (11.9), handling the unavoidable cross-shard cases (11.10/11.11), IDs (11.12), routing (11.13), and resharding (11.14).
