## 5.15 · When indexes hurt: write amplification & page splits ★

**Mental model.** Every index is a promise you pay for on **every write**: insert/update/delete a row, and *each* index on that table must be updated to stay sorted (5.1). So N indexes mean a single insert does ~N+1 B+Tree maintenance operations — **write amplification**. Worse, when a key must be inserted into the *middle* of a full page (because it's not in increasing order), the page must **split** — half its contents move to a new page — causing fragmentation, wasted space, extra I/O, and more pages to cache. Too many indexes, or a random-order key, can drown your write throughput and bloat storage. Indexes are not free; this concept is the cost side of the tradeoff that opened the module.

**How it actually works.** Two distinct costs:
- **Write amplification (index count):** a table with 8 indexes means every `INSERT` writes the clustered index *plus* 8 secondary B+Trees (each must place the new key in sorted position), and every `UPDATE` to an indexed column updates the relevant indexes. Write cost scales with index count. This is *why* "more indexes = faster reads but slower writes" (5.1) is literal and why the goal is the *fewest* indexes that serve your queries (5.16).
- **Page splits (key order):** B+Tree pages have limited free space (5.3). If inserts arrive in **increasing key order** (a monotonic PK — auto-inc, time-ordered, M03/3.12), each new key appends to the *rightmost* page; when it fills, a clean new page is added — no split, dense pages, sequential I/O. But if inserts arrive in **random order** (a UUIDv4 PK), a new key often must go into the *middle* of an already-full page → the page **splits** (half its rows move to a new page), leaving both pages ~half-full (fragmentation → wasted space → fewer rows/page → worse cache, 5.3) and causing extra random I/O. Random-order inserts into a clustered index are therefore a documented performance killer (M03/3.12) — and the same split dynamic affects secondary indexes on randomly-changing columns.

**Why it exists / what it solves.** This concept is the **discipline-enforcing reality** behind all index design: it explains *why* you can't just index everything (write amplification), *why* PK and key choice matters so much (page splits, M03/3.12), and *why* the methodology (5.16) optimizes for the *minimum* effective index set. Understanding it converts "indexes make things fast" into the truer "indexes trade write cost and space for read speed — spend deliberately." It's the cost half that makes indexing an engineering decision rather than a reflex.

**Tradeoffs & alternatives.** The core tradeoff (5.1) in its sharpest form: each index helps some reads and taxes *every* write + storage + buffer pool. Mitigations: **minimize index count** (5.16 — drop unused/redundant indexes, use composite indexes that serve many queries via leftmost-prefix, 5.7); **use monotonic/time-ordered keys** to avoid clustered-index page splits (M03/3.12 — BIGINT auto-inc or ULID/UUIDv7, never random UUIDv4); accept that **write-heavy tables tolerate fewer indexes** than read-heavy ones. The alternative storage philosophy — **LSM-trees** (some engines) — is designed for low write amplification (buffer writes, merge in background) at the cost of read complexity; InnoDB's B+Tree favors reads, so you manage write amplification by index discipline and key choice. `OPTIMIZE TABLE` can defragment split-bloated pages (a maintenance operation, M13).

**Generics / first-principles.** "Every precomputed structure must be maintained on every change — so more structures mean more write cost; and inserting out of order into a packed structure forces expensive reorganization." Write amplification is universal (every cache, index, materialized view, replica multiplies write work) and the **append-vs-insert-in-middle** distinction is fundamental (appending to a log is cheap; inserting into the middle of a sorted structure is expensive — the reason monotonic keys, append-only logs, and LSM-trees exist). The transferable instincts: **maintain the fewest derived structures that serve your reads**, and **prefer append-order (monotonic keys) over random insertion** into any sorted structure. This is the same lesson as M02/2.15 (every copy is a sync cost) and M01/1.17/M03/3.12 (logs/monotonic keys are cheap).

**MySQL-specific reality.** Concrete InnoDB facts: write amplification scales with index count (every secondary index maintained per write); a **random PK (UUIDv4) causes clustered-index page splits and fragmentation** (M03/3.12) — *the* canonical MySQL anti-pattern — while a **monotonic PK (BIGINT auto-inc, or time-ordered ULID/UUIDv7 as BINARY(16)) appends cleanly**; `SHOW TABLE STATUS` / fragmentation metrics reveal split-bloat; `OPTIMIZE TABLE` rebuilds/defragments (M13). The change-buffer (M09) mitigates *secondary*-index write cost for non-unique indexes by batching, but not the clustered-index split problem. For our domain — the forever-growing `ledger_entry`, the hottest write target — this is decisive: a compact monotonic-or-time-ordered PK (M03/3.12) keeps inserts append-only and dense, and a *minimal* index set (5.16) keeps per-insert write amplification low, so the ledger sustains high write throughput for years. Over-indexing the ledger, or using a random PK, would progressively choke writes — the failure this concept exists to prevent.

## 5.16 · Index design methodology: choosing the fewest, best indexes

**Mental model.** Index design isn't guesswork or "add an index when something's slow" — it's a **repeatable methodology**: enumerate your real access patterns (M01/1.14), design the *minimum* set of (ideally composite, ideally covering) indexes that serve them, verify with EXPLAIN (M04/4.15, M06), and prune the unused/redundant. The goal is the **fewest indexes that make your important queries fast** — because every index costs writes and space (5.15). It's an optimization problem: maximize read coverage, minimize index count.

**How it actually works — the process:**
1. **Enumerate access patterns** (M01/1.14): list the queries that matter — the frequent ones and the latency-critical ones — with their `WHERE`, `ORDER BY`, and selected columns. You design for *queries*, not columns.
2. **Design composite indexes to cover families of queries:** apply leftmost-prefix (5.7) and column order (5.8 — equality → range/sort) so *one* index serves several related queries; add covering columns (5.6) for hot reads; align with `ORDER BY` to kill filesorts (5.10). A well-designed composite index often replaces several single-column ones.
3. **Check selectivity** (5.9): index discriminating columns; don't waste indexes on low-cardinality columns standalone.
4. **Verify with EXPLAIN** (M06): confirm each important query uses the intended index with a good access type (`ref`/`range`, `Using index`, no `filesort`) — and *measure*.
5. **Prune:** find and drop **unused indexes** (write cost for no benefit) and **redundant indexes** (e.g., `(a)` is redundant if `(a, b)` exists — leftmost-prefix already covers `(a)`). Use **invisible indexes** (5.13) to test drops safely.
6. **Iterate** as access patterns evolve — index design is ongoing, not one-shot.

**Why it exists / what it solves.** A methodology prevents the two failure modes: **over-indexing** (an index per query, drowning writes and bloating storage, 5.15) and **under-/mis-indexing** (missing or wrong-ordered indexes forcing scans and filesorts). It produces a lean, intentional index set where each index earns its keep by serving real, important queries — the engineering discipline that balances the read/write tradeoff (5.1) deliberately. It turns indexing from reactive ("this is slow, add an index") into designed ("these are our access patterns; this minimal set serves them").

**Tradeoffs & alternatives.** The methodology *is* the management of the central tradeoff (5.1/5.15): every step pushes toward fewer, more-effective indexes. Real tensions within it: covering indexes (5.6) make individual queries fast but widen indexes (write cost) — so cover only hot reads; one index can't lead with two different columns (5.8) — so sometimes you accept a second index for an important query with a different shape. The alternative — no methodology, ad-hoc indexing — reliably produces both redundant indexes (nobody dares drop them) and missing ones. Tooling helps: query logs / performance_schema (M13) reveal actual access patterns and unused indexes, grounding the methodology in real traffic rather than guesses.

**Generics / first-principles.** "Design the minimal set of access structures that serve your real workload; verify against reality; prune what's unused." This is the general optimization discipline — measure the workload, build the least machinery that serves it, validate, and remove dead weight — applied to indexes. It echoes M01/1.14 ("there's no schema without a workload") and M02/2.11 ("optimize to diminishing returns, then stop"): **let the access patterns drive the structure, and resist both gold-plating (over-indexing) and neglect (under-indexing).** The transferable instinct: structures (indexes, caches, denormalizations) should be *derived from measured access patterns and pruned when unused*, not accumulated by reflex.

**MySQL-specific reality.** MySQL gives you the tools for every step: `EXPLAIN`/`EXPLAIN ANALYZE` (M04/4.15, M06) to verify index usage; **`sys.schema_unused_indexes`** and performance_schema to find unused indexes; `SHOW INDEX` for cardinality (5.9); **invisible indexes** (5.13) to test drops safely; `pt-online-schema-change`/online DDL (M13) to add/drop indexes on large tables without downtime (M03/3.15). The redundancy rule is concrete: `INDEX(a)` is redundant when `INDEX(a, b)` exists (leftmost-prefix, 5.7) — drop the narrower one. For our domain, the methodology produces the ledger's index set (5.18): the clustered PK for statement locality, a covering composite for the hot balance/statement reads, a UNIQUE on the idempotency key (5.17) — and *nothing* redundant, so write amplification (5.15) on the high-volume ledger stays minimal. This methodology is the practical synthesis of the whole module and the bridge into M06's tuning loop.

## 5.17 · Indexes as integrity: UNIQUE indexes & constraints

**Mental model.** An index isn't only a performance tool — a **UNIQUE index is also an enforced invariant.** It does two jobs at once: it's a fast access path (a B+Tree like any index) *and* it **guarantees no two rows share that key value**, rejecting any insert/update that would create a duplicate. So the *same structure* that makes lookups fast also enforces a correctness rule. This is the structural home of uniqueness invariants — idempotency keys (M01/1.2, M16), natural keys (M01/1.4), "one active X per Y" rules — and it ties indexing directly to the money-never-lies thread: a UNIQUE index is how you make "this transfer can't be posted twice" *impossible*, not just *unlikely*.

**How it actually works.** A UNIQUE index is a B+Tree that the engine **checks on every write**: before inserting (or updating) a key, it looks up whether that key already exists; if so, the operation is **rejected** (duplicate-key error). Because the lookup uses the index itself, enforcement is efficient (no extra structure needed — the index *is* the check). The PRIMARY KEY is a special UNIQUE index (the clustered one, 5.4); other uniqueness rules are declared as UNIQUE secondary indexes. Subtleties: a UNIQUE index can be **composite** (uniqueness over a *combination* — e.g., `(account_id, external_ref)` unique *per account*); and **NULLs are exempt** in MySQL (multiple NULLs are allowed in a UNIQUE index, M01/1.7 — so a nullable UNIQUE column doesn't enforce single-occupancy for the NULL case, a real gotcha). Being an index, it *also* serves as a fast access path for lookups on that key — so the idempotency-key UNIQUE index both prevents duplicates *and* makes "find the transaction for this idempotency key" an instant lookup.

**Why it exists / what it solves.** It puts a hard uniqueness invariant where it *can't be bypassed* — in the storage engine, enforced on every write from every code path (M01/1.8 — the schema as the un-bypassable fence). Application-level "check then insert" uniqueness is **racy** (two concurrent requests both check "not present," both insert — a duplicate slips through, M07/M08); a UNIQUE index makes the check-and-insert **atomic** at the engine level, closing that race. For idempotency (M16) — *the* canonical fintech use — the UNIQUE index on the idempotency key is what makes "process this payment request exactly once" actually exact: a duplicate request hits the constraint and is rejected/deduplicated, so retries (inevitable in distributed systems) can't double-charge. It's integrity *and* performance in one structure.

**Tradeoffs & alternatives.** A UNIQUE index costs the same write/space as any index (5.15) *plus* the per-write uniqueness check — and on a **hot, contended key** the uniqueness enforcement can add locking/contention (M08). It also has the NULL-exemption gotcha (multiple NULLs allowed). The alternative — enforcing uniqueness in application code — is **strictly worse for correctness** (racy under concurrency, bypassable by other writers/scripts) though more flexible for complex conditional rules (e.g., "unique only among active rows," which may need a partial-index workaround or a generated column). The guidance: **enforce hard uniqueness with a UNIQUE index** (un-bypassable, atomic, also fast); reserve application logic for rules a UNIQUE index can't express, and even then back critical ones with a constraint where possible.

**Generics / first-principles.** "Enforce uniqueness in the structure that stores it, so the check is atomic and unbypassable — application-level uniqueness checks race." This is the deep principle that **invariants belong as close to the data as possible** (M01/1.8) and that **check-then-act across a concurrency boundary is a race unless made atomic** (M07/M08). The dual nature — *the access structure is also the integrity structure* — is elegant and recurs (a hash set is both a lookup and a dedup; a content-addressed store both finds and de-duplicates by hash). The transferable instinct: **when you need "no duplicates," push it into a uniqueness constraint at the storage layer**, not a racy application check — and notice you get a fast lookup for free.

**MySQL-specific reality.** MySQL: `UNIQUE KEY` (a unique secondary index) or `PRIMARY KEY` (the clustered unique index, 5.4); violation → duplicate-key error (which the application can catch to implement idempotent "already processed" handling, M16); composite UNIQUE for per-scope uniqueness; **multiple NULLs allowed** (M01/1.7 — use NOT NULL or a generated-column workaround if NULLs must be unique). The pattern `INSERT ... ON DUPLICATE KEY UPDATE` / `INSERT IGNORE` leverages the UNIQUE index for upsert/idempotent semantics. For our domain, the **idempotency-key UNIQUE index** on `transaction_` (M01/1.2, M16) is the load-bearing example: it makes double-posting a transfer *structurally impossible* (a retry with the same key is rejected/deduplicated) and simultaneously provides instant "find by idempotency key" lookups — performance and money-never-lies integrity from one index. This is the concept where indexing and correctness converge, and it's a frequent system-design interview point (idempotency).

## 5.18 · Fintech capstone — indexing the ledger ★

**Mental model.** The capstone assembles the whole module into **the complete, justified index set for the payments schema** — every index on `account`, `transaction_`, and the forever-growing `ledger_entry`, each earning its place by serving a real access pattern (M01/1.14) while keeping write amplification minimal (5.15). It demonstrates the synthesis: clustered PK for statement locality (5.4), covering composite indexes for hot reads (5.6/5.7), UNIQUE for idempotency integrity (5.17), and *nothing redundant* — so the ledger stays **fast for reads, cheap for writes, and correct** as it grows to billions of rows. This indexed schema is the direct input to M06 (verifying/tuning the plans) and M16 (the platform at scale).

**How it actually works — the ledger's index set (each justified):**
- **`ledger_entry` clustered PK `(account_id, created_at, transaction_id, line_no)`** (5.4, M01/1.14): clusters each account's history in time order, so "account 42's statement" / "entries in a date range" / "entries ordered by date" are all *sequential leaf walks* with **no filesort** (5.10). Compact, and *append-friendly* if `created_at` is monotonic-ish per account — keeping inserts dense (5.15). This single choice (the PK *is* the layout) is the biggest structural win.
- **Covering extension for hot reads** (5.6): include the selected columns (e.g., `amount`) so the dominant statement/balance reads are answered from the index alone (`Using index`, no clustered hop, 5.5/5.6).
- **`transaction_` UNIQUE index on `idempotency_key`** (5.17, M01/1.2, M16): makes double-posting a transfer *structurally impossible* and provides instant "find by idempotency key" lookups — integrity *and* performance.
- **Secondary indexes only for genuine access patterns** (5.16): e.g., a `(transaction_id)` index on `ledger_entry` if "all legs of a transaction" is a hot query; an index on a foreign key if joins/cascade checks need it (M01/1.5). Each is added *only* because an enumerated pattern needs it — and pruned if unused.
- **Compact, monotonic/time-ordered keys throughout** (M03/3.12): BIGINT auto-inc or ULID/UUIDv7 as BINARY(16) — small indexes (5.3), append-only inserts, no page splits (5.15).
- **No redundant indexes** (5.16): `(account_id)` alone is dropped because the clustered `(account_id, created_at, …)` covers it via leftmost-prefix (5.7).

**Why it exists / what it solves.** It proves the module's thesis on the highest-stakes table: a *deliberately designed, minimal* index set keeps the ledger **queryable for years** (statements/reconciliation stay fast as it grows), **writable at high throughput** (low write amplification, no page splits, 5.15), and **correct** (UNIQUE enforces idempotency, 5.17). It's the realization of M01/1.14 ("design for your queries") + M03/3.12 (compact keys) + this module's structural understanding — the difference between a ledger that scales and one that progressively chokes on its own indexes.

**Tradeoffs & alternatives.** The capstone *is* the central tradeoff (5.1/5.15) resolved well: enough indexes (and covering width) to make the hot reads fast, few enough to keep ledger writes cheap. The deliberate choices: cover the hot statement/balance reads (accept wider indexes for them) but *don't* cover everything (write cost); one workhorse clustered index serving the query family (5.7) rather than many narrow ones; UNIQUE for idempotency (accept the per-write check, 5.17, for un-bypassable correctness). The failure alternatives are instructive: a random UUID PK (page splits choke writes, 5.15/M03·3.12); over-indexing the ledger (write amplification kills throughput); under-indexing (statements/reconciliation full-scan billions of rows). The capstone threads between them by *method* (5.16), not luck.

**Generics / first-principles.** "Design the minimal index set that makes your dominant access patterns fast and your invariants enforced, on a compact, append-friendly key layout — and you get a structure that scales in reads, writes, and correctness together." This synthesizes the module's threads: tradeoff (every index costed, minimized), generics-first (B+Tree/clustering/leftmost-prefix reasoning applied), durability (append-friendly keys avoid split/fragmentation), money-never-lies (UNIQUE = enforced idempotency). The transferable instinct for any high-scale, correctness-critical store: **let access patterns drive a minimal set of access structures, choose the physical key order to match the dominant access, push invariants into uniqueness constraints, and keep keys compact and append-ordered.**

**MySQL-specific reality.** The concrete InnoDB realization: `ledger_entry` as an InnoDB table clustered on `(account_id, created_at, transaction_id, line_no)` (M01/1.14, 5.4) with a covering extension for hot reads (5.6); `transaction_.idempotency_key` UNIQUE (5.17); compact BIGINT/BINARY(16) keys (M03/3.12) for small indexes and append-only inserts (5.15); a minimal secondary-index set verified via `EXPLAIN` and pruned via `sys.schema_unused_indexes`/invisible indexes (5.13/5.16); money as DECIMAL/minor-units (M03/3.4) so covered sums are exact. This is the exact indexed schema **M06** will EXPLAIN-and-tune, **M07/M08** will transact-and-lock, **M09** will store in the buffer pool, and **M16** will scale across shards — and it's where M05's deep structural understanding becomes a ledger that's simultaneously fast, cheap to write, and correct. Indexing the ledger well is one of the highest-leverage decisions in the entire fintech data design.
