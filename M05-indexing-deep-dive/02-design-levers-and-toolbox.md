## 5.7 · Composite indexes & the leftmost-prefix rule ★

**Mental model.** A composite (multi-column) index is **sorted by its columns in order** — first by column 1, then by column 2 *within* equal column-1 values, and so on — exactly like sorting a list of names by (last name, then first name). The defining consequence is the **leftmost-prefix rule**: the index can efficiently serve a query that constrains a **leading prefix** of its columns (column 1; or columns 1+2; or 1+2+3) but **not** a query that skips the leading column(s). An index on `(account_id, created_at)` is great for "account 42" and "account 42 in June," but **useless** for "all entries in June across all accounts" — because the data isn't sorted by `created_at` first.

**How it actually works.** In the B+Tree (5.2), composite keys are ordered lexicographically by column sequence. So within the index, all rows for `account_id = 42` are contiguous, and *within* those, rows are ordered by `created_at`. This means:
- `WHERE account_id = 42` → efficient (a contiguous range in the index).
- `WHERE account_id = 42 AND created_at >= '2025-06-01'` → efficient (a sub-range within account 42's contiguous block).
- `WHERE created_at >= '2025-06-01'` (no `account_id`) → **cannot use** the index efficiently — `created_at` values are scattered across every account's block, not contiguous; the index isn't sorted by `created_at` at the top level.
- `WHERE account_id = 42 ORDER BY created_at` → the index *already provides this order* (5.10) — no sort needed.
The rule generalizes: an index on `(a, b, c)` serves prefixes `(a)`, `(a, b)`, `(a, b, c)`, and can do a *range* on the *last* used column — but a gap (constraining `a` and `c` but not `b`) means `c` can't be used as an index range (only `a` is used, then `c` is filtered).

**Why it exists / what it solves.** The leftmost-prefix rule is a *direct consequence* of how sorted multi-column data works — it's not an arbitrary MySQL rule but a property of lexicographic ordering. Understanding it is what lets you design *one* composite index that serves *several* related queries (all the prefix queries) instead of building a separate index per query. It's the single most important rule for designing efficient, minimal index sets (5.16): a well-ordered composite index covers a family of access patterns.

**Tradeoffs & alternatives.** The tradeoff is in **column order** (5.8 — the design decision that follows from this rule): a given composite index serves only the prefixes starting from its leading column, so the order determines *which* queries it can serve. You can't have one index serve both "by account then date" and "by date then account" as leading-column queries — you'd need two indexes (more write cost, 5.15). The art is choosing column order so one index serves the most/most-important queries (5.8), and accepting that some access patterns need their own index. The alternative to a composite index — multiple single-column indexes — is usually *worse* (the optimizer can sometimes merge them, but a composite index purpose-built for the query is far more efficient).

**Generics / first-principles.** "Lexicographic ordering means you can seek by any leading prefix but not by a non-leading subset." This is universal to any sorted-by-multiple-keys structure — a phone book sorted by (last, first) lets you find all "Smith"s or "Smith, John" but not all "John"s; a sorted composite key in any system has the same property. The transferable instinct: **the order of keys in a sorted structure is a contract about which queries are cheap** — leading-prefix queries are cheap, others aren't, and you design the order around your queries.

**MySQL-specific reality.** This is one of the most important and most-tested MySQL indexing rules. EXPLAIN (M04/4.15, M06) reveals it: a query using a proper prefix shows the index in `key` with a good `type` (`ref`/`range`); a query that violates leftmost-prefix shows `key: NULL` and `type: ALL` (full scan) or only partial use. The InnoDB nuance: a composite index can use an equality on leading columns plus a *range* on the next column efficiently, but once a range is used, subsequent columns can't be used for further index range access (only filtering) — which drives column order (5.8). For our domain, `(account_id, created_at)` is the workhorse ledger index serving "this account's entries," "this account's entries in a date range," and "this account's entries ordered by date" — three patterns, one index — *because* of leftmost-prefix.

## 5.8 · Column order in composite indexes (the design decision)

**Mental model.** Given that a composite index serves leftmost prefixes (5.7), **the order you list its columns is the single highest-leverage index-design choice** — it determines which queries the index can serve and how efficiently. The reliable default heuristic: **equality columns first, then the range/sort column last.** Put the columns you'll match *exactly* (`=`, `IN`) at the front, and the column you'll *range over* or *sort by* at the end — because once a range is used, no later column can be used for index seeking.

**How it actually works.** The reasoning chains from 5.7:
- **Equality columns first:** a column matched with `=` produces a single contiguous block in the index; stacking equality columns keeps narrowing to a contiguous sub-block. So `(account_id, created_at)` with `account_id = 42` lands you in account 42's contiguous block.
- **Range/sort column last:** within that block, a *range* on the next column (`created_at >= …`) is a contiguous sub-range — efficient. But a range "uses up" the index's seeking ability: columns *after* the range column can't be used for further index seeks (the data within the range isn't sorted by them). So the range column goes last (or last-among-seekable).
- **Sort alignment:** if the query has `ORDER BY`, ordering the index to match it lets the index satisfy the sort too (5.10), eliminating a filesort.
So you order by: equality predicates → the range/sort column. Selectivity (5.9) refines ties: among equality columns, leading with the more selective one can help, though for *prefix-serving* the dominant concern is matching the query's shape.

**Why it exists / what it solves.** Column order decides whether one index serves your query at all and whether it can also satisfy the sort. Getting it right means one composite index efficiently serves a whole family of queries (5.7) *and* kills the filesort (5.10); getting it wrong means the index is partially used or unused and you add more indexes (write cost) to compensate. It's the lever that turns "I have an index on those columns" into "the index actually makes my query fast."

**Tradeoffs & alternatives.** The tension: different queries want different leading columns, and one index can't lead with two different columns. You resolve it by (a) ordering for the most important/frequent query, (b) exploiting that one good order often serves several prefix queries, and (c) adding a second index only when a genuinely important query needs a different leading column — accepting the write cost (5.15). There's also an equality-column-order subtlety: among equality columns the order doesn't affect *which* queries are served (any permutation of equalities works as a prefix set) but can affect index size/compression and how well it serves partial-equality queries — so lead with the column most often queried alone. The headline rule (equality → range/sort) is the high-leverage default; selectivity and sort needs break ties.

**Generics / first-principles.** "Order keys by how they're queried: exact-match keys first, the scanned/ordered key last." This follows from sorted-structure mechanics (5.7) and is the practical design rule that falls out of it. The transferable instinct: **a multi-key sort order is a design artifact you shape around your access patterns** — put the dimensions you pin down first and the dimension you sweep last, because you can only sweep the last pinned dimension efficiently. This is the same logic behind partition+sort keys in any sorted store.

**MySQL-specific reality.** Concretely in MySQL: for `WHERE account_id = 42 AND created_at >= X ORDER BY created_at`, the index **`(account_id, created_at)`** is correct (equality `account_id` first, range/sort `created_at` last) — it seeks to account 42, range-scans the dates in order, and needs no filesort. The reverse, `(created_at, account_id)`, would be wrong for this query (the leading `created_at` range scatters account 42's rows and can't seek `account_id`). Add a covering column (5.6) at the end for the read: `(account_id, created_at, amount)`. EXPLAIN confirms: good order → `type: range`, `Extra: Using index condition`/`Using index`, no `Using filesort`. This per-query column-ordering reasoning is the heart of practical index design (5.16) and a staple interview exercise.

## 5.9 · Selectivity & cardinality: when an index helps

**Mental model.** An index only helps if using it lets the engine **touch few rows** — i.e., if the indexed condition is **selective**. **Cardinality** is the number of distinct values in a column; **selectivity** is roughly cardinality ÷ row count (how finely the column partitions the data). A high-selectivity column (`account_id`, millions of distinct values — each matches few rows) is a great index; a low-selectivity column (`status`, 3 distinct values — each matches a third of the table) is a poor one, and the optimizer may **ignore** the index in favor of a full scan, *correctly*, because fetching a third of the table via index bookmark lookups (5.5) is slower than scanning.

**How it actually works.** The optimizer estimates, using statistics (M04/4.7, cardinality + histograms), how many rows a given index access would match. If that's a small fraction, the index path (descend B+Tree, fetch the few rows) is cheap → use it. If it's a large fraction, the index path means *many* bookmark lookups (5.5) — random I/O per row — which can exceed the cost of a sequential full scan, so the optimizer chooses the scan (M04/4.8). This is *why* an index you created might show as unused in EXPLAIN: not a bug, but the optimizer correctly judging it non-selective for that query. Selectivity also explains composite-index design (5.8): leading with a selective column narrows the search fast. **Histograms** (M04/4.7) help with *skew* — a column might be selective for rare values but not common ones (e.g., a `status` that's 99% 'active' and 1% 'frozen': indexing helps find the rare 'frozen' rows but not the common 'active' ones).

**Why it exists / what it solves.** Selectivity is the principle that tells you **which columns are worth indexing** and *why the optimizer sometimes won't use an index*. It prevents two mistakes: indexing low-cardinality columns (wasted write cost for an index the optimizer ignores) and being confused when a "perfect" index isn't used. It also drives composite-index column choice and order. It's the quantitative core of "an index helps only if it narrows the search."

**Tradeoffs & alternatives.** Indexing a low-selectivity column is usually wasteful — but not always: it can help if (a) it's part of a *composite* index that's selective overall, (b) it enables a *covering* index (5.6, no bookmark lookups, so selectivity matters less because there's no per-row fetch), or (c) you query the *rare* values (with histograms guiding the optimizer). So "don't index low-cardinality columns" is a default, not a law — the real question is whether the index produces a *cheap access path for your actual query*. The alternative to a non-selective index is a scan, which for low-selectivity predicates is often genuinely the right plan.

**Generics / first-principles.** "An index pays off in proportion to how much it narrows the search; a key that doesn't discriminate isn't worth indexing." This is information-theoretic — an index is valuable to the degree it *reduces uncertainty* about where the matching rows are. A predicate matching half the data carries little 'information' for locating rows, so an index on it saves little. The transferable instinct: **index the discriminating dimensions, not the coarse ones**, and remember that "covering" can rescue a non-selective index by removing the per-row fetch cost. This selectivity reasoning governs any lookup-structure decision.

**MySQL-specific reality.** MySQL exposes cardinality via `SHOW INDEX` (the `Cardinality` column, an estimate from sampling, M04/4.7) and uses it + histograms to decide index usage. The classic observations: a `status` ENUM/lookup column (3–5 values) is rarely worth a standalone index (optimizer scans instead); `account_id` (high cardinality) is an excellent index. Stale cardinality → wrong decisions (`ANALYZE TABLE`, M04/4.7). For skewed columns, `ANALYZE TABLE ... UPDATE HISTOGRAM` helps the optimizer use an index for *rare* values while scanning for common ones. For our domain: index `account_id`, `transaction_id`, `idempotency_key` (all high-selectivity); don't bother indexing `status_code` alone (low cardinality) — but it can ride in a composite or covering index where it adds value. This explains many "why isn't my index used?" mysteries in M06.

## 5.10 · Index-only ordering & grouping (killing filesort/temp)

**Mental model.** An index stores its keys **already in sorted order** (5.2). So if a query's `ORDER BY` (or `GROUP BY`) matches an index's order, the engine can **read rows straight from the index in the required order** and *skip the sort step entirely* — eliminating the **filesort** (and `GROUP BY`'s temp table) blocking operations from M04/4.13 that otherwise spill to disk on large results. This is a major, often-overlooked reason indexes matter: not just for *finding* rows, but for *delivering them pre-sorted*. The cheapest sort is the one you never do.

**How it actually works.** Recall M04/4.13: `ORDER BY` without a usable index triggers a **filesort** (sort in a buffer, spill to disk if large — a performance cliff); `GROUP BY`/`DISTINCT` may build a **temp table**. An index whose column order matches the query's sort/group order lets MySQL avoid both: it reads the leaf pages in index order, which *is* the sorted order, and streams the results — no separate sort, no temp table. For `WHERE account_id = 42 ORDER BY created_at`, an index `(account_id, created_at)` (5.7/5.8) delivers account 42's rows already in `created_at` order → **no filesort**. The requirements: the `ORDER BY` columns must form a prefix of the index *after* the equality-constrained leading columns (so the remaining order is by the sort columns), and the sort direction must be compatible (or a descending index, 5.13, for `DESC`). `GROUP BY` similarly benefits when the grouping columns align with index order (rows for each group are contiguous).

**Why it exists / what it solves.** Sorting and grouping are *blocking* operations (M04/4.13) that can dominate a query's cost and spill to disk on large result sets — and they're extremely common (every "recent entries newest-first," every "totals by day"). An index that provides the order turns these from expensive blocking steps into cheap streaming reads. For a high-volume account's statement query, this is the difference between a fast scroll and a multi-second stall. It's a primary motivation for composite-index column order (5.8) including the sort column.

**Tradeoffs & alternatives.** The "cost" is just designing the index to match the sort (which often aligns naturally with the filter columns anyway, since you usually filter and sort the same data). The constraint: an index serves *one* sort order (and its exact reverse via backward scan); a query needing a *different* sort order than your index provides still needs a filesort, or a second index (write cost, 5.15). Mixed-direction sorts (`ORDER BY a ASC, b DESC`) need a descending index on the right column (5.13). The alternative — let MySQL filesort — is acceptable for small result sets or rare queries but a real problem for hot, large ones. The instinct: for hot queries, design the index to satisfy the `ORDER BY`, not just the `WHERE`.

**Generics / first-principles.** "If data is already stored in the order you need, reading it *is* sorting it — for free." Maintaining a useful sort order in your access structure so reads come out ordered is a universal optimization (sorted indexes, sort-keys in columnar/warehouse stores, merge-sorted runs in LSM-trees). The transferable instinct: **a sort you can satisfy from existing order is a sort you don't pay for** — so align your stored order with your dominant query order, and reserve explicit sorting for the cases you can't pre-order. Blocking operations are expensive; eliminating them via pre-ordering is high-leverage.

**MySQL-specific reality.** EXPLAIN (M04/4.15, M06) is how you see it: the *presence* of `Extra: Using filesort`/`Using temporary` means the index *isn't* satisfying the order — a tuning target; their *absence* (with the index in `key`) means the index delivers the order for free. For our ledger, `(account_id, created_at)` serves `WHERE account_id = 42 ORDER BY created_at` with no filesort; add `amount` to cover (5.6) and it's a pure index-only read, ordered, no row fetch. A `... ORDER BY created_at DESC` is served by reading the same index *backward* (or a descending index, 5.13). This is one of the most impactful and frequently-missed indexing wins, and a standard M06 tuning move ("an ORDER BY is filesorting → add/adjust an index to provide the order").

## 5.11 · Prefix indexes (indexing part of a long column)

**Mental model.** For a **long string column** (a long reference code, a URL, a description), you can index just the **first N characters** instead of the whole value — a **prefix index**. It makes the index much smaller (and so faster and more buffer-pool-friendly, 5.3), at the cost of being **less selective** (many values may share the first N chars) and unable to fully **cover** or **sort** by the column. It's a space-vs-selectivity tradeoff for long columns where indexing the full value would be wastefully large.

**How it actually works.** `INDEX (ref_code(12))` indexes only the first 12 characters of `ref_code`. The B+Tree stores 12-char keys instead of full ones → smaller index, more keys per page (5.3), shallower tree. The engine uses the prefix to *narrow* the search to rows whose first 12 chars match, then **rechecks the full value** on the candidate rows (the prefix can't distinguish values that agree in the first 12 chars). The design question is **choosing N**: long enough to be selective (few false-positive matches per prefix) but short enough to stay small. You pick N by measuring **prefix selectivity** — how close the distinct-prefix-count gets to the distinct-full-value-count — and choosing the smallest N that captures most of the selectivity. Limitations: a prefix index **can't be a covering index** for that column (it only has the prefix, not the full value), **can't fully satisfy `ORDER BY`** on the column (prefix order ≠ full-value order beyond the prefix), and is mandatory for indexing **TEXT/BLOB** columns (which can *only* be prefix-indexed, M03/3.7).

**Why it exists / what it solves.** Indexing the full text of a long column can produce a huge index (bloating storage and buffer pool, slowing everything, 5.3) — and often the first N chars are already selective enough to locate rows efficiently. Prefix indexes capture most of the lookup benefit at a fraction of the size. They're also the *only* way to index long TEXT/BLOB columns (M03/3.7), which can't be fully indexed. So they solve "I need to look up by a long string without a giant index."

**Tradeoffs & alternatives.** Prefix length N is a tunable tradeoff: too short → low selectivity (the index narrows little, many full-value rechecks, the optimizer may not use it); too long → large index (the benefit shrinks). And a prefix index *loses* covering (5.6) and full ordering (5.10) on that column. Alternatives: index the **full column** (if it's not too long and you need covering/sorting); store and index a **hash of the value** (a generated column, 5.12 — exact-match lookups via a compact hash, but no range/ordering); or, for genuinely document-shaped data, restructure (M03/3.11). Choose prefix indexes when the column is long, lookups are by prefix/equality, and you don't need covering/sorting on it.

**Generics / first-principles.** "Index a discriminating *prefix* of a large key to get most of the selectivity at a fraction of the size — accepting false positives you recheck." This is a universal space-saving lookup technique: prefix/partial keys, bloom-filter-like "narrow then verify" two-stage lookups, hashing for compact exact-match. The transferable instinct: **you rarely need the whole key to locate data — find the shortest discriminating portion, then verify** — trading a small recheck cost for a much smaller index. The recurring sub-principle: a smaller index is a faster index (5.3), so shrink keys wherever selectivity allows.

**MySQL-specific reality.** Syntax: `INDEX (col(N))`. **TEXT/BLOB can *only* be prefix-indexed** (M03/3.7) — you must specify N. Choosing N: compare `COUNT(DISTINCT LEFT(col, N))` to `COUNT(DISTINCT col)` to find the smallest N that captures the selectivity (a standard tuning recipe). Prefix indexes **don't cover** (the full value isn't in the index) and **don't fully order**. For our domain, a long external `reference` string on `transaction_` might be prefix-indexed if lookups are by a selective prefix; but a fixed-length, already-compact identifier (an `idempotency_key`, M01/1.2) is indexed in full (it needs to be a covering UNIQUE index for exact-match dedup, 5.17). Prefix indexes are a targeted tool for long, low-structure string columns — not a default.

## 5.12 · Functional & expression indexes

**Mental model.** Normally an index can only help a query whose `WHERE` references the *bare column*. But queries often filter on a **function of a column** — `DATE(created_at)`, `LOWER(email)`, a value extracted from JSON (M03/3.11) — and a plain column index *can't* serve those (the engine would have to compute the function on every row, defeating the index). A **functional (expression) index** indexes the **result of the expression**, so such queries can use an index instead of scanning. It extends indexing from "columns" to "computed values."

**How it actually works.** Two equivalent approaches in MySQL:
- **Generated-column index (works since 5.7):** create a **STORED generated column** holding the expression's result (M03/3.11 — e.g., `entry_date DATE AS (DATE(created_at)) STORED`), then index *that* column. The expression is computed once at write time and materialized; queries filtering on `entry_date` (or written to match it) use the index.
- **Functional index (8.0+):** index the expression directly — `CREATE INDEX ix ON t ((DATE(created_at)))` — MySQL maintains a hidden generated column under the hood. Cleaner syntax, same idea.
Either way, the index stores the *computed* values in sorted order; a query whose predicate matches the indexed expression (`WHERE DATE(created_at) = '2025-06-01'`) gets an index access path instead of a full-scan-with-per-row-computation. The crucial requirement: the query's expression must **match** the indexed expression (the optimizer recognizes the match) — a subtly different expression won't use the index. This is also the canonical way to **index into JSON** (M03/3.11): extract a path into a generated column and index it.

**Why it exists / what it solves.** Without functional indexes, any filter involving a transformed column forces a full scan (the engine must apply the function to every row to test the predicate) — a common, silent performance trap (`WHERE YEAR(created_at) = 2025` scanning the whole table). Functional indexes make transformed-column lookups indexable, and they're the sanctioned bridge for **JSON queryability** (M03/3.11): promote a JSON path to an indexed, typed access path while keeping the flexible JSON. They turn a class of unavoidable scans into index lookups.

**Tradeoffs & alternatives.** Functional indexes cost what their backing entails: a **STORED** generated column takes storage and write-time computation (the value is materialized); the index adds the usual write/space cost (5.1/5.15). The expression must be **deterministic** (same inputs → same output) to be indexable. And the **query must match the indexed expression** exactly — a maintenance gotcha (change the query's transform and the index silently stops being used). The often-better alternative is to **rewrite the query to avoid the function on the column** — e.g., `WHERE created_at >= '2025-06-01' AND created_at < '2025-07-01'` is *sargable* (uses a plain `(created_at)` index) and avoids needing a functional index at all. So: prefer a sargable rewrite when possible; use a functional/generated-column index when the transform is genuinely needed (JSON extraction, case-folding, computed keys).

**Generics / first-principles.** "Index the value you actually query by — even if it's computed — or rewrite the query so it queries an indexed value." The deeper principle is **sargability**: a predicate can use an index only if it's expressed in terms of the indexed value, so either *make the index match the predicate* (functional index) or *make the predicate match the index* (rewrite). This transfers everywhere: precompute-and-index a derived value, or restructure the query to hit existing structure. The instinct: **a function wrapped around an indexed column usually disables the index — recognize it and fix it (rewrite or functional index).**

**MySQL-specific reality.** MySQL: **generated columns (5.7+)** + index, or **functional indexes (8.0+)** via `CREATE INDEX ... ((expr))`; both require deterministic expressions. The headline use case is **JSON** (M03/3.11): `tier VARCHAR AS (metadata->>'$.tier') STORED` + index, or a functional index on the JSON path — the controlled 1NF-escape made queryable. The classic anti-pattern they fix (or that a rewrite fixes): `WHERE DATE(created_at) = '2025-06-01'` or `WHERE YEAR(created_at) = 2025` silently full-scanning — solved by either a functional index on the expression or (better) a sargable range rewrite on the bare `created_at` column. For our domain, indexing a JSON-extracted attribute or a computed lookup key is the typical use; the everyday lesson is *don't wrap an indexed column in a function in your WHERE clause unless you've indexed that function.*

## 5.13 · Descending & invisible indexes (and other modern options)

**Mental model.** Two modern InnoDB index features worth knowing. **Descending indexes** store keys in *descending* order, so they efficiently serve `ORDER BY … DESC` (and, crucially, *mixed*-direction sorts like `ORDER BY a ASC, b DESC` that a single ascending index can't satisfy by simple backward scanning). **Invisible indexes** are indexes the optimizer *ignores* (as if dropped) while still being *maintained* — letting you safely test "what happens if I drop this index?" *without* actually dropping it (instant rollback by making it visible again). One serves a sorting need; the other is an operational safety tool.

**How it actually works.**
- **Descending indexes (8.0+):** `INDEX (a ASC, b DESC)` stores `b` in descending order within each `a`. Before 8.0, MySQL could scan an ascending index *backward* to serve a fully-`DESC` order, but it **couldn't** efficiently serve a *mixed* order (`a ASC, b DESC`) — that forced a filesort (5.10). Real descending indexes let one index satisfy mixed-direction `ORDER BY` directly, eliminating the filesort.
- **Invisible indexes (8.0+):** `ALTER TABLE t ALTER INDEX ix INVISIBLE` keeps the index fully maintained (writes still update it) but makes the **optimizer pretend it doesn't exist**. So you can verify that no query secretly depends on it (watch for regressions) *before* dropping it — and instantly revert (`VISIBLE`) if something breaks. It de-risks index removal, which is otherwise scary on a busy system (dropping a still-needed index causes immediate scans).
Other modern options in the same family: **functional indexes** (5.12), and index features like `INVISIBLE` columns — part of MySQL 8's "safer schema evolution" toolkit (M03/3.15, M13).

**Why it exists / what it solves.** **Descending indexes** solve mixed-direction sorts (common in "latest first, then by something ascending" UIs and reports) that previously forced filesorts. **Invisible indexes** solve a real operational fear: you suspect an index is unused (write overhead for nothing, 5.15) but dropping it might break a query you didn't know about — invisible indexes let you *test the drop reversibly*. Both reduce the cost/risk of good index hygiene (5.16): keep the right indexes, drop the wrong ones, safely.

**Tradeoffs & alternatives.** Descending indexes cost nothing extra structurally (same B+Tree, reversed order on a column) but, like any index, you only add one if a query needs that specific mixed order — otherwise an ascending index + backward scan handles pure-`DESC`. Invisible indexes have a subtlety: the index is *still maintained* (still costs writes/space) while invisible, so it's a *testing* state, not a way to "soft-disable" an index permanently — you make it invisible to *verify it's safe to drop*, then actually drop it. The alternative to invisible-then-drop is just dropping and hoping (risky) or analyzing query logs (incomplete) — invisible indexes are strictly safer for the verification step.

**Generics / first-principles.** "Match the index's order to the query's order (including direction); and make destructive changes reversible before committing to them." Descending indexes are the direction-aware corollary of 5.10 (an index satisfies an order only if it *matches* that order, direction included). Invisible indexes embody a universal ops principle — **test the effect of a removal reversibly before making it permanent** (feature flags, canary deploys, "soft delete before hard delete," M01/1.6 all share this instinct). The transferable lesson: reduce the blast radius of changes by making them observable and reversible first.

**MySQL-specific reality.** Both are **MySQL 8.0** features. Descending indexes: `CREATE INDEX ix ON t (a ASC, b DESC)` — needed for mixed-direction `ORDER BY`; a single-direction sort is often served by a normal index scanned forward or backward. Invisible indexes: `ALTER TABLE t ALTER INDEX ix INVISIBLE` / `VISIBLE` — the standard safe-index-removal workflow (make invisible → monitor for regressions/slow queries → drop if safe, or revert instantly). For our domain, a "newest entries first, then by amount ascending" report might need a descending index; and when pruning the ledger's index set (5.16) on a busy production system, invisible indexes are how you *safely* confirm an index is unused before dropping it. These are part of the modern, safer index-management toolkit (with M13's online DDL).

## 5.14 · Other index types: hash, fulltext, spatial, adaptive hash

**Mental model.** B+Trees are the workhorse (5.2), but a few **specialized index types** exist for jobs B+Trees do poorly: **hash indexes** for blazing exact-match (but *no* ranges/ordering), **fulltext indexes** for natural-language text search (what `LIKE '%word%'` can't do efficiently), **spatial indexes** for geographic/geometric queries, and InnoDB's **adaptive hash index** — an *automatic*, internal optimization that builds hash indexes on hot B+Tree pages for you. Knowing they exist (and their niches) rounds out the toolbox; B+Trees remain the default for almost everything.

**How it actually works.**
- **Hash index:** maps a key to a bucket via a hash function → **O(1) exact-match** lookup, faster than a B+Tree's O(log n) for pure equality. But a hash has *no order*, so it **cannot do range scans, `ORDER BY`, or leftmost-prefix** — only `=`/`IN`. In MySQL, hash indexes are the native index type of the **MEMORY** engine; InnoDB/MyISAM use B+Trees (you can *emulate* a hash index in InnoDB by indexing a hash-of-value generated column, 5.12, for compact exact-match on long values).
- **Fulltext index:** an **inverted index** (word → list of rows containing it) enabling `MATCH ... AGAINST` natural-language/boolean text search — far more efficient and capable than `LIKE '%term%'` (which can't use a normal index at all due to the leading wildcard, and does substring matching, not word/relevance search). Supported in InnoDB (and MyISAM).
- **Spatial (R-tree) index:** indexes geometric data (points, polygons) for spatial predicates (within, intersects, nearest) — used for geo/location queries.
- **Adaptive Hash Index (AHI):** InnoDB **automatically** builds an in-memory hash index over frequently-accessed B+Tree pages, so repeated lookups of hot keys become hash-fast — *transparently*, with no DDL. You don't create it; InnoDB manages it (and it can be turned off if it causes contention on certain workloads).

**Why it exists / what it solves.** Each fills a B+Tree gap: hash for *fastest possible exact-match* (where ranges/ordering aren't needed); fulltext for *real text search* (relevance, word matching — impossible to do well with `LIKE` + B+Tree); spatial for *multi-dimensional geometric* queries (which a 1-D B+Tree can't index efficiently); AHI to *automatically* accelerate hot exact lookups without you doing anything. They let MySQL serve specialized workloads (search, geo, in-memory caches) that the general-purpose B+Tree handles poorly.

**Tradeoffs & alternatives.** Hash: fastest equality but *useless* for ranges/ordering/prefix — a deal-breaker for most real query mixes, so B+Trees win by default (they do equality *and* ranges *and* ordering). Fulltext: powerful for text search but a specialized subsystem (its own syntax, relevance tuning, language considerations) — for simple prefix matching, a plain B+Tree on the column with `LIKE 'prefix%'` (no leading wildcard, so sargable) suffices; fulltext is for genuine word/relevance search. Spatial: essential for geo, irrelevant otherwise. AHI: usually a free win, but on some high-concurrency workloads its maintenance becomes a contention point and is disabled — a tuning consideration (M09). The meta-tradeoff: specialized indexes are powerful in their niche and useless/counterproductive outside it; **B+Trees are the right default** precisely because they're good at the broadest range of operations.

**Generics / first-principles.** "Match the index structure to the query's *shape*: hashing for point lookups, sorted trees for ranges/ordering, inverted indexes for text, spatial trees for multi-dimensional." The deep principle is that **the access structure must fit the access pattern** — there's no universal best index, only the best index *for a query shape*. The transferable instinct: when a query type is slow, ask whether you're using the *wrong kind* of index for that shape (e.g., trying to do text search with `LIKE`, or range scans on a hash). This "structure follows access pattern" idea (echoing M01/1.14) is why databases offer multiple index types.

**MySQL-specific reality.** Concretely: **InnoDB's persistent indexes are B+Trees** (no user-created hash indexes — hash is the MEMORY engine's native type, or emulated via a hashed generated column); **fulltext indexes work in InnoDB** (since 5.6) via `MATCH … AGAINST` — the right tool when you'd otherwise reach for `LIKE '%term%'` (which can't use an index due to the leading wildcard and isn't real search); **spatial (R-tree) indexes** for `GEOMETRY` types; and the **adaptive hash index** (`innodb_adaptive_hash_index`, on by default) automatically accelerating hot B+Tree lookups (occasionally disabled for contention, M09). For our domain these are mostly niche — the ledger lives on B+Trees — but knowing them matters: e.g., searching transaction descriptions/notes is a *fulltext* job, not a `LIKE` scan; and the AHI is silently helping your hot account/PK lookups. The everyday rule stands: **B+Tree by default; reach for a specialized index only when the query shape demands it.**
