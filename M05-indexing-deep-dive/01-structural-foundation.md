## 5.1 · What an index is (and the core tradeoff)

**Mental model.** An index is a **separate, sorted data structure that maps key values to row locations**, so the engine can *find* rows matching a condition without examining every row. Think of a book's index: instead of reading all 900 pages to find "buffer pool," you look up the word in a sorted list and jump to the page. The catch that defines everything in this module: that index must be **kept correct on every write** — insert/update/delete a row and every index on that table must be updated too. So an index **trades write cost and storage for read speed**. Indexing is not "make everything fast"; it's *choosing where to spend*.

**How it actually works.** Without an index, finding rows matching `WHERE account_id = 42` requires a **full table scan** (M04/4.8) — read every row, test each. An index on `account_id` is a structure (almost always a B+Tree, 5.2) holding `account_id` values in *sorted order*, each paired with a way to locate the full row. To answer the query, the engine searches the sorted structure (O(log n), like a binary search) to find the matching key(s), then retrieves the row(s) — touching a tiny fraction of the data instead of all of it. The cost: when you insert a new entry, the engine must place its `account_id` into the *correct sorted position* in that B+Tree (and in every other index on the table) — extra work on every write, plus disk space for each index.

**Why it exists / what it solves.** Indexes are the mechanism that makes a database usable at scale: they turn O(n) scans into O(log n) lookups and enable efficient range queries, sorting, and joins. They're how the optimizer (M04/4.6) gets *fast access paths* (M04/4.8) to choose from — without indexes, every query is a scan and performance collapses as data grows. For a billion-row `ledger_entry`, the difference between "account 42's entries" being an index lookup vs a full scan is the difference between milliseconds and minutes.

**Tradeoffs & alternatives.** This is the module's central thread, stated up front: **more indexes → faster reads but slower writes, more storage, more buffer-pool pressure (M03/3.2).** Each index added is maintained on every write to the table and consumes space in RAM and on disk. So the alternatives aren't "index or don't" but "*which* indexes, *how many*": the discipline (5.16) is the **fewest indexes that serve your real access patterns** (M01/1.14). Over-indexing drowns writes (5.15); under-indexing forces scans. There's no free lunch — every index is a bet that its read benefit exceeds its write/storage cost.

**Generics / first-principles.** "Maintain a sorted auxiliary structure to find data without scanning — paid for at write time." This is universal: a hash map's keys, a search tree, a sorted log's index, a search engine's inverted index — all trade write-time maintenance and space for read-time speed. The deep principle is **precompute an ordering/lookup so reads touch the least data**, and its inescapable cost is **keeping that precomputation current as data changes**. Every cache, every materialized view (M02/2.14), every index is the same bargain.

**MySQL-specific reality.** In MySQL/InnoDB, indexes are almost always **B+Trees** (5.2); the **primary key is a special "clustered" index** that *is* the table (5.4), and all other ("secondary") indexes point back to it (5.5). Index size is governed by type/key size (M03/3.2, 3.12) — compact keys mean smaller indexes that fit in the buffer pool (M09). MySQL exposes indexes via `CREATE INDEX`/`ADD KEY`, and the optimizer decides whether to *use* an available index based on selectivity and statistics (5.9, M04/4.7). The recurring MySQL workflow (M04/4.15, M06): design an index for an access pattern → `EXPLAIN` to confirm it's used → measure → adjust. Everything else in M05 elaborates *what* B+Tree indexes are and *how* to design them well.

## 5.2 · The B+Tree: the structure behind almost every index ★

**Mental model.** Nearly every MySQL index is a **B+Tree**: a balanced, shallow, high-fanout tree where **all the actual keys live in the leaf level**, the leaves are **linked in sorted order**, and the upper levels are just a sparse "routing" directory to find the right leaf fast. This one structure gives you *both* superpowers an index needs: **O(log n) point lookups** (descend from root to leaf) *and* **efficient range scans** (find the start leaf, then walk the linked leaves in order). Understanding the B+Tree is the key that unlocks every indexing rule in this module — column order, covering, page splits, why ranges work — they're all consequences of this shape.

**How it actually works.** A B+Tree has a **root page**, optional **internal pages**, and **leaf pages**. Internal pages hold *separator keys + pointers to child pages* (a routing directory, not data); **leaf pages hold the indexed keys in sorted order** (plus, depending on index type, the row or a row-pointer — 5.4/5.5). Key properties:
- **Balanced:** every leaf is the same distance from the root, so every lookup costs the same (the tree's *height*), and the tree stays shallow.
- **High fanout:** each page holds *many* keys/pointers (hundreds — because a page is 16KB and keys are small, 5.3), so the tree is very wide and only a few levels deep even for billions of rows. A 3–4 level B+Tree can index billions of rows.
- **Linked leaves:** leaf pages point to the next leaf, so once you find the start of a range, you read sequentially without re-descending — making range scans (`BETWEEN`, `>`, `ORDER BY`) efficient.
A **point lookup** descends root→internal→leaf (a few page reads). A **range scan** descends to the start key, then follows the leaf links. This is *why* indexes are fast for both equality and ranges — and why the "B" relatives (plain B-Tree, with data in internal nodes too) are worse for ranges (you'd revisit upper levels).

**Why it exists / what it solves.** B+Trees solve the exact problem disk-based databases have: **minimize the number of page reads** (disk/SSD I/O is the bottleneck, not CPU). High fanout keeps the tree shallow → few page reads per lookup; sorted linked leaves make ranges sequential → cache- and disk-friendly; balance guarantees consistent performance. They simultaneously support point lookups, range scans, ordering, and prefix matching — the full set of operations an index must serve — in one structure. No simpler structure (a sorted array: expensive inserts; a hash table: no ranges/ordering; a plain binary tree: too deep, too many page reads) does all of this on disk.

**Tradeoffs & alternatives.** B+Trees are excellent generalists but not optimal for every case: a **hash index** beats them for pure exact-match (O(1) vs O(log n)) but can't do ranges or ordering (5.14); **LSM-trees** (used by some engines, not InnoDB's default) trade read speed for much faster writes (better write amplification). B+Trees also degrade with **random-order inserts** (page splits, 5.15) — they love sequential/monotonic key insertion (M03/3.12). The tradeoff: B+Trees are the best *general* index structure (lookups + ranges + ordering + decent writes), which is why they're the default everywhere, but specialized structures win in specialized regimes.

**Generics / first-principles.** "A shallow, high-fanout, sorted, balanced tree minimizes I/O for both lookups and ranges." The B+Tree is one of the most important data structures in computing precisely because it's *the* answer to "index on-disk data." The transferable insights: **fanout determines depth determines I/O count** (wider = shallower = fewer reads); **keeping data sorted in linked leaves makes ranges cheap**; **balance gives predictable performance**. These ideas recur in filesystems, key-value stores, and any disk-resident sorted structure. If you understand the B+Tree, you understand why almost every database index looks the way it does.

**MySQL-specific reality.** InnoDB indexes are B+Trees stored in **16KB pages** (5.3). The clustered (PK) index's leaves hold *full rows* (5.4); secondary indexes' leaves hold *indexed columns + the PK value* (5.5). Typical tree height is 3–4 levels even for huge tables (high fanout), so a point lookup is a handful of page reads — often served from the buffer pool (M09). The linked-leaf structure is what makes `(account_id, created_at)` serve "account 42's entries in June, in order" as a sequential leaf walk (5.7, 5.10). InnoDB's preference for *monotonic* PK insertion (M03/3.12, append to the right edge) vs random (page splits, 5.15) is a direct B+Tree consequence. (The deep page-format/buffer-pool internals are M09; here the B+Tree is the structure that *explains the index-design rules*.)

## 5.3 · Pages: the unit of index (and table) storage ★

**Mental model.** Indexes and rows don't live as loose keys — they live in **fixed-size pages** (16KB in InnoDB), and the B+Tree is really **pages pointing to pages**. The page is the unit of I/O (the engine reads/writes whole pages, never single rows) and the unit of caching (the buffer pool holds pages, M09). The single most important consequence: **how many keys fit in a page sets the tree's fanout, which sets its height, which sets how many page reads a lookup costs.** Smaller keys → more per page → wider, shallower tree → fewer I/Os. This is *why* M03/3.2's "compact types" rule translates directly into index speed.

**How it actually works.** A 16KB InnoDB page holds as many keys/rows as fit (page header + sorted keys/records + page directory + free space). An **internal (non-leaf) page** packs *separator keys + child-page pointers* — since these are small, hundreds fit, giving high fanout. A **leaf page** packs the actual keys (and rows or PK-pointers). Fanout = keys-per-page; tree height ≈ log_fanout(rows). With a fanout of, say, several hundred, a 3-level tree indexes hundreds-of-millions to billions of rows — so a lookup is ~3 page reads. Crucially, **key size drives fanout**: a 4-byte INT key packs far more per page than a 36-byte CHAR UUID (M03/3.12), so the compact key yields a shallower tree *and* smaller index *and* better buffer-pool fit. Pages also have **free space** for inserts; when a page fills and a new key must go in the middle, the page **splits** (5.15).

**Why it exists / what it solves.** Fixed-size pages are how databases bridge the gap between "rows/keys" and "disk/memory blocks": disk and SSD are read/written efficiently in blocks, and the buffer pool caches in fixed units, so organizing data into uniform pages makes I/O and caching tractable and predictable. The page-as-I/O-unit is *why* sequential/clustered access is fast (read a page, get many useful rows) and random access is slow (one useful row per page read), and it's the physical basis for the footprint→performance link (M03/3.2): fewer bytes per row/key → more useful data per page read.

**Tradeoffs & alternatives.** A fixed 16KB page is a compromise: larger pages → higher fanout (shallower trees) and better sequential throughput, but more wasted I/O for tiny random reads and coarser caching; smaller pages → finer caching but deeper trees and more overhead. InnoDB defaults to 16KB (configurable) as a good general balance. The key design lever you control isn't page size but **what you put in the page**: compact keys and rows (M03) maximize useful keys/rows per page, which is the highest-leverage, lowest-effort index-and-table optimization there is.

**Generics / first-principles.** "Storage is organized in fixed blocks; the count of useful items per block determines I/O cost." This is the universal block/page principle — filesystems, OS virtual memory (pages), CPU cache lines — all move data in fixed units, and packing more useful data per unit reduces the number of (slow) fetches. The transferable instinct: **density per access unit is a first-order performance lever.** "How many of the things I need fit in one read?" governs B+Tree fanout exactly as it governs cache-line utilization in systems code.

**MySQL-specific reality.** InnoDB's default page is **16KB** (`innodb_page_size`, set at init). The B+Tree is pages of separator-keys (internal) and records (leaf); fanout depends on key/row size, so **compact keys (M03/3.2, 3.12) directly yield shallower trees, smaller indexes, and better buffer-pool hit rates** (M09). Pages have fill factor and free space; mid-page inserts trigger splits (5.15), and `OPTIMIZE TABLE` can defragment. The deep page anatomy (record format, page directory, the FIL header, off-page overflow for large columns from M03/3.14) is detailed in M09; here the essential takeaway for *index design* is: **the page is the I/O unit, fanout (keys-per-page) sets tree height, and compact keys make everything faster.** This is the structural reason behind half the recommendations in this module.

## 5.4 · Clustered index: the table IS its primary-key B+Tree ★

**Mental model.** Here's the InnoDB fact that surprises people and explains a huge amount: **the table is not stored separately from its primary-key index — the table's rows live *inside* the leaf pages of the PK's B+Tree.** The clustered index *is* the table. The primary key isn't a pointer *to* a row; it's the row's *physical address* (its position in the sorted PK B+Tree). So a PK lookup descends the tree and finds the **full row right there in the leaf** — no second hop. This is why PK choice is a *physical layout* decision (M01/1.3), why PK lookups are the fastest access path (M04/4.8 `const`/`eq_ref`), and why range scans on the PK are sequential.

**How it actually works.** In InnoDB, every table is stored *as* a B+Tree keyed on the primary key (if you don't declare one, InnoDB invents a hidden rowid, M01/1.2). The **leaf pages contain the complete rows**, sorted by PK. Consequences that cascade through everything:
- **PK lookup = one descent, row in hand** (no extra fetch) — the cheapest possible access.
- **Rows are physically ordered by PK** — so a range scan on the PK (e.g., `ledger_entry` clustered on `(account_id, created_at)`, "account 42's entries over time") reads *physically adjacent* leaf pages sequentially (fast), exactly the M01/1.14 "design for your queries" payoff.
- **PK is embedded in every secondary index** (5.5) — because secondary indexes must point back to the row's location, which *is* its PK.
- **PK choice = clustering choice** — a monotonic PK (auto-inc, M03/3.12) appends to the right edge (no splits); a random PK (UUIDv4) scatters inserts across the tree (page splits, fragmentation, 5.15).

**Why it exists / what it solves.** Clustering rows by PK means the most common access (by primary key) is maximally efficient (no indirection) and that related rows (sharing a PK prefix) are stored together (range locality). For the ledger, clustering on `(account_id, created_at)` physically groups each account's history in time order, turning statement queries into sequential reads — the single biggest structural win for the dominant access pattern. It's the physical realization of M01/1.14.

**Tradeoffs & alternatives.** Clustering has costs: the PK is duplicated into every secondary index (5.5), so a **fat PK bloats all indexes** (M03/3.12 — keep it compact); inserts must go to the *right place in PK order*, so a **non-monotonic PK causes page splits** (5.15 — prefer monotonic/time-ordered keys). The alternative design (used by some engines, e.g., a "heap" table where rows live in insertion order and *all* indexes point to a physical rowid) avoids PK-embedding and insert-ordering concerns but loses PK-range locality and makes PK lookups a two-step (index→heap). InnoDB chose clustering; the tradeoff is huge wins for PK-aligned access at the cost of caring a lot about PK shape.

**Generics / first-principles.** "Store the data physically ordered by the key you most often access it by, and access by that key becomes free of indirection." Clustering is a universal locality technique — sort data by its dominant access dimension so related reads are sequential (columnar stores cluster by column; time-series DBs cluster by time; LSM-trees keep sorted runs). The principle: **physical order should match your dominant access pattern**, and whatever key defines that order pays no indirection cost while everything else pays one hop.

**MySQL-specific reality.** This is core InnoDB (M01/1.3): the table = clustered PK B+Tree, leaves = full rows, PK embedded in every secondary index. Practical rules that all derive from it: **choose a compact, monotonic PK** (BIGINT auto-inc or time-ordered ULID/UUIDv7 as BINARY(16), M03/3.12) for small indexes + append-only inserts; **shape the PK for your dominant range query** (`(account_id, created_at, …)` for the ledger, M01/1.14); **never use a random UUID PK** (page splits, 5.15). MyISAM (M04/4.12) is *non*-clustered (rows in a separate heap, all indexes point to a row offset) — a contrast that highlights how much InnoDB's clustering shapes design. The next concept (5.5) is the direct consequence: how *secondary* indexes work given that the row's address is its PK.

## 5.5 · Secondary indexes: pointers back to the clustered index ★

**Mental model.** Since the row lives in the clustered (PK) index (5.4), any *other* index — a **secondary index** — can't store the row itself; instead its leaves hold **the indexed columns + the primary-key value**. So a secondary-index lookup is a **two-step (double lookup)**: first descend the secondary B+Tree to find the matching key and read its stored PK, then descend the *clustered* index using that PK to fetch the actual row. This "extra hop" is invisible but consequential — it's why covering indexes (5.6) are so valuable (they eliminate the hop), why the PK's size matters (it's stored in every secondary index, 5.4), and why a secondary index isn't quite as cheap as it looks.

**How it actually works.** A secondary index on, say, `(account_id)` is its own B+Tree, sorted by `account_id`, whose **leaf entries store `account_id` + the PK value** of each matching row (not the row, and not a physical pointer — the *PK*). To answer `SELECT * FROM ledger_entry WHERE account_id = 42`:
1. **Secondary lookup:** descend the `account_id` B+Tree to find account 42's entries → get their PK values.
2. **Clustered lookup ("bookmark lookup"):** for *each* of those PKs, descend the clustered index to fetch the full row.
So a non-covering secondary lookup costs *two* B+Tree descents per result (and step 2 is repeated per row). This is why: (a) the **PK is embedded in every secondary index** (it's the "pointer"), so a fat PK bloats them all (5.4, M03/3.12); (b) a secondary index that must fetch many rows can be *slower* than expected (many bookmark lookups) — sometimes the optimizer correctly prefers a scan (5.9, M04/4.8); (c) eliminating step 2 via a **covering index** (5.6) is the biggest single read optimization.

**Why it exists / what it solves.** Secondary indexes give you fast access paths on columns *other than* the PK — without them, any query not on the PK is a full scan. Storing the PK (rather than a physical row pointer) as the "bookmark" is what makes InnoDB's clustering work: rows can move within the clustered index (page splits, etc.) without invalidating secondary indexes, because the bookmark is the logical PK, not a physical address. So secondary indexes provide flexibility (index any columns) and stability (survive row movement) — at the cost of the second hop.

**Tradeoffs & alternatives.** The double lookup is the core tradeoff: a secondary index is great for *selective* queries returning few rows (few bookmark lookups) but loses its edge when it would return many rows (many bookmark lookups can cost more than a scan — the optimizer weighs this via selectivity, 5.9). The PK-as-bookmark design trades a little lookup cost (the second descent) and index size (PK stored everywhere) for stability under row movement. The way to *escape* the second hop is a **covering index** (5.6) — include the needed columns so the query is answered at the secondary leaf. Alternatively, a heap-table engine (MyISAM) uses a fixed physical pointer (one hop) but loses clustering's other benefits (5.4).

**Generics / first-principles.** "An alternate index over clustered data stores a *logical reference* (the cluster key), so a lookup is two steps: find the reference, then resolve it." This indirection — index → logical id → resolve — is universal (a database secondary index, a search-engine posting list → doc id → document, a symbol table → address). The transferable insights: **indirection costs a resolution step** (so co-locating or "covering" the needed data avoids it), and **using a logical (not physical) reference buys stability at the cost of a lookup.** Recognizing "this is an extra hop" is the instinct behind covering indexes everywhere.

**MySQL-specific reality.** InnoDB secondary indexes store **indexed columns + PK** in their leaves; a non-covering lookup does a secondary descent then a clustered ("bookmark") descent per row. This makes three M05/M03 rules concrete: **keep the PK compact** (it's in every secondary index, 5.4/M03·3.12); **prefer covering indexes for hot reads** (5.6, skip the hop); and **expect the optimizer to skip a low-selectivity secondary index** in favor of a scan (5.9). In EXPLAIN (M04/4.15, M06), a covering secondary index shows `Extra: Using index` (no hop), while a non-covering one does the bookmark lookups silently. For "account 42's entries," whether the access is cheap depends on selectivity *and* whether the index covers the selected columns — which is exactly the bridge to 5.6.

## 5.6 · Covering indexes: answering from the index alone ★

**Mental model.** If **every column a query touches is present in a single index**, the engine can answer the query entirely from that index's leaf pages — and **skip the second hop into the clustered index** (5.5) completely. The index "covers" the query. This is the single biggest read optimization in indexing: you turn a "find-in-secondary-index *then* fetch-each-row-from-the-table" into a "find-in-index, done." On hot, high-volume reads, covering indexes routinely deliver order-of-magnitude wins because they eliminate per-row bookmark lookups and read far fewer pages.

**How it actually works.** Recall a secondary lookup is two steps (5.5): find matching keys + their PKs in the secondary index, then fetch each row from the clustered index. A **covering index** includes all the columns the query needs — in the `WHERE`, the `SELECT` list, the `ORDER BY` — so step 2 is unnecessary: the data is already *in the secondary leaf*. Example: for `SELECT amount FROM ledger_entry WHERE account_id = 42`, a secondary index on `(account_id)` requires a bookmark lookup per row to fetch `amount`; but an index on `(account_id, amount)` already *has* `amount` in its leaves, so the query is answered from the index alone — no clustered-index access at all. (In InnoDB, secondary indexes also *implicitly* include the PK columns, 5.5, so those are "free" to cover too.) Modern MySQL also lets you add purely-for-covering columns via `INCLUDE`-style design — in practice by appending them to the composite index.

**Why it exists / what it solves.** Covering eliminates the most expensive part of a secondary-index query — the repeated random row fetches — and reads only the (often smaller, denser) index pages. For a query run millions of times (a balance/summary read on a hot account), removing the per-row hop is transformative. It's also the reason you sometimes deliberately make an index *wider* (add columns it doesn't filter on) — to cover a hot query. It's the highest-leverage "make this specific query fast" tool, validated by `Using index` in EXPLAIN.

**Tradeoffs & alternatives.** Covering trades **index width/storage and write cost** for **read speed**: a covering index carries extra columns, so it's bigger (more storage, more buffer-pool space) and slower to maintain on writes (more data to keep updated). It's a per-query optimization — you can't cover every query (you'd recreate the whole table in indexes), so you cover the *hot, important* ones (5.16). There's tension with the "fewest, smallest indexes" goal (5.1/5.15): covering deliberately makes an index wider, so apply it judiciously to queries whose read frequency justifies the write/storage cost. The alternative for a non-covered query is the normal two-hop secondary lookup, which is fine when the query isn't hot or returns few rows.

**Generics / first-principles.** "Co-locate everything a read needs so it can be answered in one place — avoid the indirection/resolution step." Covering is the database instance of *denormalizing for the read path* (M02/2.12) and of locality optimization generally — keep what's read together, together. The transferable instinct: **when a lookup is followed by a fetch, ask whether the fetch can be eliminated by putting the needed data where the lookup already lands.** This is the same idea as a covering cache entry, an embedded document vs a reference, or a struct that inlines its hot fields. The cost is always the same: the co-located copy is bigger and must be maintained.

**MySQL-specific reality.** In InnoDB, covering works because the query's columns are all in one index (and the PK is implicitly included, 5.5); EXPLAIN shows **`Extra: Using index`** (covering) vs the silent bookmark lookups of a non-covering index. The design move: extend a composite index with the columns a hot query selects — e.g., `(account_id, created_at, amount)` covers "account 42's June amounts in date order" *and* serves the range *and* the sort (5.7/5.10) with zero clustered-index access. The caution: don't over-widen (write/storage cost, 5.15) — cover the genuinely hot reads identified by your access-pattern analysis (5.16). Covering indexes are one of the most reliably high-impact, frequently-tested MySQL optimizations, and a staple of the fintech ledger's read design (5.18).
