## 13.6 · Online schema migrations (the locking problem) ★

**Mental model.** Schemas evolve (add a column, change a type, add an index) — but a naive `ALTER TABLE` on a *huge* table is an operational disaster: depending on the change, MySQL may **rebuild the entire table** while **holding a lock** (or a metadata lock, M08) that **blocks all reads and writes** for the *hours* the rebuild takes — locking out every transfer on the ledger. **Online schema change** solves this: instead of altering the live table in place, it builds a **shadow copy** with the new schema in the *background*, **backfills** it from the original, keeps it in **sync** with ongoing changes, and then **cuts over** atomically (rename the shadow into place) — all with the original table *fully available* throughout, and only a tiny lock at the instant of cutover. It's the *same* copy → sync → cutover pattern as resharding (M11/11.14) and PITR's log-replay idea — applied to schema change, for near-zero downtime.

**How it actually works.** The online-DDL pattern (gh-ost / pt-osc, 13.7):
1. **Create a shadow table** with the *desired* (new) schema — empty, alongside the original.
2. **Backfill**: copy the original table's rows into the shadow, in throttled chunks (so it doesn't overload the server or starve live traffic).
3. **Sync ongoing changes**: while backfilling, the original table is *still live* (taking reads/writes) — so changes happening *during* the copy must also reach the shadow. Two mechanisms: **triggers** (pt-osc installs INSERT/UPDATE/DELETE triggers on the original that mirror changes to the shadow) or **binlog** (gh-ost reads the binlog, M10, to capture and replay changes — no triggers). Either way, the shadow converges to a complete, up-to-date copy with the new schema.
4. **Cut over atomically**: once the shadow is fully caught up, **atomically rename** the shadow into the original's place (and the original out) — a brief, near-instant metadata operation. The new schema is now live; the old table is dropped.
Throughout, the original is available; only the cutover rename takes a momentary lock. The tools **throttle** (pause if replication lag, M10, or load spikes) and are **abortable** (cancel cleanly before cutover if something's wrong).

**Why it exists / what it solves.** Online schema change exists because **a naive `ALTER` locks a large table for the duration of the rebuild**, which for a big production table means *hours of downtime* — utterly unacceptable for a system that must stay available (a payments ledger can't stop taking transfers for hours to add a column). It solves **zero-downtime schema evolution**: change the schema of an arbitrarily large table while it stays fully available, with only a momentary cutover lock. This is essential because schemas *must* evolve (new features, new columns, new indexes) but availability *can't* be sacrificed — and the bigger the table (and the more critical the system), the more this matters. For money, it's the difference between "we can't add that feature because the migration would lock the ledger for 4 hours" and "we shipped it with zero downtime overnight." It makes schema change *operationally safe* at scale.

**Tradeoffs & alternatives.** Online DDL trades **speed/simplicity for availability**: ✓ **near-zero downtime** (table stays available), throttleable, abortable; ✗ **slower and heavier** than a direct `ALTER` (it copies the whole table + syncs changes — more I/O, more time, ~2× disk during the migration), and adds **complexity** (triggers or binlog reading, the tooling). The alternatives: **a naive `ALTER`** (fast for small tables, but *locks* large ones — only acceptable on small/low-traffic tables or in a maintenance window the system can afford); **MySQL's own online DDL** (13.8 — `ALGORITHM=INPLACE/INSTANT` handles *some* changes online natively, no external tool — use it when it applies); **gh-ost/pt-osc** (13.7 — for changes MySQL can't do online, or for the extra control/throttling/abortability). The decision: for a *small* table, a direct `ALTER` is fine; for a *large, critical* table (the ledger), use **MySQL's native online DDL if the change supports it** (13.8 — simplest), else **gh-ost** (13.7 — most control). Never a naive locking `ALTER` on a huge production table.

**Generics / first-principles.** "Change a large table's schema without downtime by building a new-schema shadow copy in the background, keeping it in sync with live changes, and cutting over atomically — the same copy-sync-cutover pattern as any online data migration." This is the universal **online schema migration** pattern (gh-ost, pt-osc, and the same idea in every database's online-DDL story). The transferable insight: **to change a large live structure without downtime, don't modify it in place (locking) — build the new version alongside, backfill + sync it via the change log or triggers, and atomically swap it in** — the *identical* copy → sync (via change log) → verify → atomic-cutover pattern as resharding (M11/11.14), online data migration, and even blue-green deployment. It transfers everywhere (schema migrations, data migrations, infrastructure swaps). The deep principle (recurring through M11/M12/M13): **online change = build-new-alongside + sync-via-change-log + atomic-cutover**, never modify-in-place-while-locked — and the change log (binlog, M10) is what makes "sync while it's changing" possible.

**MySQL-specific reality.** MySQL online-DDL options: **(1) MySQL's native online DDL** (13.8 — `ALGORITHM=INSTANT` for metadata-only changes like adding a column at the end; `ALGORITHM=INPLACE` for some in-place rebuilds without a full copy/lock; but many changes still `ALGORITHM=COPY` which *locks*); **(2) gh-ost** (GitHub's tool — reads the **binlog**, M10, no triggers — lower overhead, more control, throttling/abort, the modern favorite); **(3) pt-online-schema-change** (Percona — uses **triggers** to sync the shadow — mature, but trigger overhead on the original table). All do copy → backfill → sync → atomic rename, *throttling* on replication lag (M10) and load. For **sharding** (M11), migrations run *per shard* (Vitess coordinates schema changes across shards). For our domain: **adding a column to a billion-row ledger table** — if it's an `INSTANT` change (add a nullable column at the end, 13.8), MySQL does it instantly; otherwise **gh-ost** builds a shadow ledger table, backfills it throttled (pausing if replica lag spikes, M10), syncs via the binlog, and atomically cuts over — *without locking out transfers*. Never a naive `ALTER` that locks the ledger for hours. The ★ SVG (Pass C) draws naive-ALTER (locks) vs online-DDL (shadow + backfill + cutover).

## 13.7 · gh-ost & pt-online-schema-change

**Mental model.** The two dominant external online-DDL tools, differing in *how they sync* the shadow table with live changes. **pt-online-schema-change (pt-osc):** installs **triggers** on the original table (INSERT/UPDATE/DELETE triggers that mirror every change to the shadow copy) — mature, widely used, but the triggers add overhead to *every* write on the original during the migration (and can interact badly with existing triggers / foreign keys). **gh-ost (GitHub's online schema transmogrifier):** **reads the binary log** (M10 — like a replica/CDC, M12) to capture changes and replay them to the shadow — **no triggers** (zero added write-overhead on the original), plus better **throttling, pausing, and observability** (it can be paused/resumed/aborted cleanly, throttles on replica lag). Both follow the *same* copy → backfill → sync → atomic-cutover pattern (13.6) — the choice is **triggers (pt-osc) vs binlog (gh-ost)**, with gh-ost the modern favorite for its lower overhead and control.

**How it actually works.** Both create a shadow table, backfill it in chunks, sync live changes, and atomically cut over (13.6) — differing in the *sync* mechanism:
- **pt-osc (triggers):** creates the shadow with the new schema, installs **AFTER INSERT/UPDATE/DELETE triggers** on the original that replicate each change to the shadow, then copies existing rows in chunks (the triggers handle concurrent changes). When the copy completes, it **atomically renames** (original → old, shadow → original) and drops the triggers + old table. *Cost:* the triggers fire on *every* write to the original during the migration (write amplification), and pt-osc can't be used if the table already has triggers (pre-8.0) or has tricky foreign keys.
- **gh-ost (binlog):** creates the shadow, then **connects as a replica and reads the binlog** (M10) to capture changes to the original, replaying them to the shadow — *while* copying existing rows in chunks. **No triggers** → no write-overhead on the original. It can **throttle** (slow/pause the copy if replica lag, M10, or load thresholds are exceeded), **pause/resume**, and **abort** cleanly (before cutover). At completion, it does the atomic cutover. *Benefit:* decoupled from the original's write path (binlog-based, like CDC, M12), lower impact, more control.

**Why it exists / what it solves.** These tools exist because **MySQL's native online DDL (13.8) can't do *every* change online** (some still `COPY` and lock), so you need an external mechanism for those changes — and because even for changes MySQL *can* do, the tools offer **throttling, pausing, abortability, and observability** that the native `ALTER` doesn't. They solve "change a large table's schema online, with control and safety, for *any* change type." gh-ost specifically solves pt-osc's trigger overhead (by using the binlog instead) and adds operational control (throttle on lag, pause, abort). For money, they provide *safe, controllable* schema evolution on the huge, critical ledger tables — with the ability to throttle (don't overwhelm the primary or lag the replicas, M10) and abort (back out cleanly if something looks wrong) that a naked `ALTER` lacks. They're the production tools for the migrations the native online DDL can't safely handle.

**Tradeoffs & alternatives.** **pt-osc (triggers):** ✓ mature, well-understood, works without binlog access; ✗ **trigger overhead** on every write during the migration, conflicts with existing triggers/some FKs, less throttling control. **gh-ost (binlog):** ✓ **no trigger overhead** (binlog-based), excellent **throttling/pause/abort**, better observability, decoupled from the write path; ✗ needs binlog access (ROW format, M10/10.3) and a bit more setup, can't run on the writer directly (reads a replica's or the primary's binlog). The choice: **gh-ost** is the modern favorite (lower overhead, more control) *unless* you can't give it binlog access or have a setup pt-osc suits better. Both beat a naive locking `ALTER` on large tables, and both are reached for when **MySQL's native online DDL (13.8) can't do the change online**. The decision: native online DDL first (if it supports the change — simplest), else **gh-ost** (else pt-osc).

**Generics / first-principles.** "Sync the shadow table with live changes via triggers (pt-osc) or the change log (gh-ost) — the binlog/change-log approach avoids write-path overhead and decouples the migration, mirroring CDC." This is the universal **online-migration sync mechanism** choice (triggers vs change-log capture — the same divide as CDC's "triggers vs log-based capture," M12/12.12). The transferable insight: **to keep a copy in sync with a changing source, you either intercept writes at the source (triggers — coupled, adds overhead) or consume the source's change log (binlog/CDC — decoupled, no overhead) — and the log-based approach is generally superior** (lower impact, more control), the *same* lesson as CDC (M12/12.12) and gh-ost using the binlog the way Debezium does. It transfers to any keep-in-sync problem (migrations, replication, CDC, caches). The deep principle: **the change log is the better synchronization primitive** — gh-ost reading the binlog is CDC applied to schema migration, decoupled and controllable.

**MySQL-specific reality.** **gh-ost** (binlog-based, ROW format M10/10.3, no triggers, throttle/pause/abort — the modern favorite) and **pt-online-schema-change** (trigger-based, mature) are the two standard MySQL online-DDL tools, both doing copy → backfill → sync → atomic-rename (13.6), both throttling on **replication lag** (M10). They're used when **InnoDB's native online DDL (13.8) can't do the change online** (a `COPY`-algorithm change on a huge table). For **sharded** platforms (M11), **Vitess** has its own schema-change workflow (often gh-ost or vitess-native, coordinated per shard). For our domain: migrating the **ledger schema** (a change the native online DDL can't do instantly) uses **gh-ost** — it builds a shadow ledger table, reads the binlog (M10) to stay in sync with live transfers, backfills throttled (pausing if replica lag spikes so reporting/reconciliation replicas don't fall behind, M10/M02), and atomically cuts over — *zero downtime, full control, abortable*. This is how the platform evolves the ledger schema safely at scale. The Pass C Mermaid draws pt-osc (triggers) vs gh-ost (binlog) — copy/sync/cutover.

## 13.8 · MySQL's own online DDL (and its limits)

**Mental model.** Modern InnoDB can do *some* schema changes **online and in-place** natively — no external tool needed — via the `ALGORITHM` clause: **`INSTANT`** (metadata-only — the change is near-instantaneous, no table rebuild, no significant lock — e.g., adding a nullable column *at the end*), **`INPLACE`** (rebuilds/modifies in place without a full external copy, usually allowing concurrent reads/writes — e.g., adding a secondary index), and **`COPY`** (the old behavior — rebuilds the whole table, *locking* it — what you must *avoid* on large tables). The catch: **many changes still require `COPY` (locking)**, so you must *know which algorithm a given change uses* — and reach for gh-ost/pt-osc (13.7) when the change would otherwise `COPY`-lock a large table. The skill is knowing what's `INSTANT` (free), what's `INPLACE` (online but rebuilds), and what's `COPY` (locks → use a tool).

**How it actually works.** When you run an `ALTER TABLE`, InnoDB picks (or you specify) an **algorithm**:
- **`ALGORITHM=INSTANT`** (the best): the change is a **metadata-only** operation — InnoDB just records the schema change without touching existing rows. Near-instant, no rebuild, no meaningful lock. Supported for a growing set of changes: **adding a column** (historically at the end; expanded over versions), adding/dropping a virtual column, renaming a column, changing a default, etc. *Always prefer this when it applies.*
- **`ALGORITHM=INPLACE`**: InnoDB modifies the table **in place** (rebuilding the table or building an index *within* InnoDB, without an external table copy), usually permitting **concurrent DML** (reads/writes proceed). E.g., **adding a secondary index** (builds the index in place while the table stays available, with a brief lock at start/end). Online, but does real work (time/I/O proportional to table size).
- **`ALGORITHM=COPY`** (the one to avoid on large tables): InnoDB **rebuilds the entire table** by copying it, **locking** it against writes (often reads too) for the duration — the *naive* behavior. Required for changes the other algorithms can't do (some type changes, some primary-key changes, etc.).
You can **force the algorithm** (`ALTER … ALGORITHM=INSTANT`) to *fail fast* if the change can't be done that way (so you don't accidentally trigger a `COPY`-lock) — then fall back to gh-ost (13.7).

**Why it exists / what it solves.** Native online DDL exists to handle the *common* schema changes online *without* an external tool — making the easy cases trivial. It solves "do this schema change with minimal disruption, using the database's built-in capability" — for the many changes that *are* `INSTANT` or `INPLACE`, you don't need gh-ost at all (simpler, no extra tooling). Its *limits* (the changes that still `COPY`-lock) exist because some changes (certain type conversions, primary-key changes, etc.) genuinely require rebuilding the table, which InnoDB can't do without copying. Knowing the limits is what tells you *when* to reach for gh-ost/pt-osc (13.7) — you use native online DDL for what it can do (the simple, common changes), and the external tools for what it can't (the `COPY`-locking changes on large tables). It solves the *easy* migrations natively and *delegates* the hard ones to the tools.

**Tradeoffs & alternatives.** **Native online DDL:** ✓ **simplest** (no external tool), `INSTANT` changes are essentially free, `INPLACE` is online; ✗ **limited** (many changes still `COPY`-lock), less throttling/abort control than gh-ost. **gh-ost/pt-osc (13.7):** ✓ handle *any* change online with full control (throttle/abort); ✗ external tooling, slower (full copy), more setup. The decision is a clean hierarchy: **(1)** if the change is `ALGORITHM=INSTANT` → just do it (instant, free); **(2)** if `INPLACE` and the table isn't *too* huge → native online DDL (online, but watch the rebuild time/load); **(3)** if it would `COPY`-lock a large table (or you need throttling/abort control) → **gh-ost** (13.7). Always **force `ALGORITHM=INSTANT/INPLACE` to fail fast** rather than silently `COPY`-locking a big table. The art is *knowing which bucket a given change falls in* — which requires checking the docs/`EXPLAIN`-equivalent for the specific change + version.

**Generics / first-principles.** "The database can do *some* schema changes online/in-place natively (instant metadata changes, in-place rebuilds) but not all — know which, use native for the easy cases, and a copy-sync-cutover tool for the rest." This is the universal **native-online-DDL-with-limits** reality (every database has *some* online-DDL capability with *some* gaps — Postgres, MySQL, etc.). The transferable insight: **modern databases handle common schema changes online natively (especially metadata-only changes, which are instant), but a meaningful set still require a full rebuild — so the skill is knowing which changes are cheap (instant/in-place) vs expensive (rebuild/lock), using the native capability for the cheap ones and an online-migration tool for the expensive ones.** It transfers to every database (know your engine's online-DDL matrix). The deep principle: **metadata-only changes are nearly free; structural rebuilds are expensive — design schema changes to be metadata-only where possible (e.g., add nullable columns at the end), and know when you've crossed into rebuild territory** (where you need a tool or a maintenance window).

**MySQL-specific reality.** InnoDB's `ALTER TABLE … ALGORITHM={INSTANT|INPLACE|COPY}, LOCK={NONE|SHARED|EXCLUSIVE}`: **`INSTANT`** for an expanding set of metadata-only changes (adding columns — including, in newer versions, not just at the end — renaming columns, changing defaults); **`INPLACE`** for in-place rebuilds/index builds (adding a secondary index, with concurrent DML); **`COPY`** for the rest (locks — avoid on large tables). **Force the algorithm** (`ALGORITHM=INSTANT`) to fail fast and avoid an accidental `COPY`-lock. Watch out: even `INPLACE` takes a **metadata lock** (M08) briefly at start/end (the "DDL blocks every query" MDL trap, M08 — a long-running query can make the `ALTER` wait, queuing everything behind it). For **sharding** (M11), changes apply per shard (Vitess coordinates). For our domain: **adding a nullable column at the end of the ledger table** → `INSTANT` (do it directly, instant); **adding an index to a large ledger table** → `INPLACE` (online, but watch the build time/load); **changing a column's type on the billion-row ledger** → would `COPY`-lock → use **gh-ost** (13.7). The skill is classifying each ledger schema change into instant/in-place/needs-gh-ost. The Pass C Mermaid draws the INSTANT/INPLACE/COPY decision (which locks).

## 13.9 · Observability: metrics, logs, the golden signals ★

**Mental model.** **You can't operate what you can't see** — observability is the discipline of *knowing what your database is doing* so problems surface *before* they become outages. It has three data types — **metrics** (numeric time-series: query rates, latencies, connection counts, buffer-pool hit ratio, replication lag — cheap, aggregatable, alertable), **logs** (discrete events: the slow query log, the error log, the audit log — detailed, for diagnosis), and **traces** (a request's path across services — for distributed diagnosis) — organized by the **four golden signals**: **Latency** (how long queries take), **Traffic** (how many — QPS/TPS), **Errors** (failed queries/connections), and **Saturation** (how full — CPU, connections, buffer pool, disk, the closest-to-the-limit resource). Watch the golden signals and you catch problems *early* (rising latency, climbing saturation) rather than as a 3am outage. Observability turns "the database is down, why?" into "saturation is climbing, intervene now."

**How it actually works.**
- **Metrics**: continuously collected numeric time-series — from MySQL's status variables (`SHOW GLOBAL STATUS` — QPS, connections, InnoDB stats), `performance_schema`/`sys` (13.10), and the OS (CPU, disk, memory) — scraped by a monitoring system (Prometheus + Grafana, etc.), graphed, and **alerted** on thresholds. Organized by the **golden signals**: *Latency* (query/transaction time — p50/p95/p99), *Traffic* (queries/sec, transactions/sec), *Errors* (error rate, failed connections, deadlocks), *Saturation* (CPU%, connection-pool fullness vs `max_connections`, buffer-pool pressure, disk space, replication lag — the **early-warning signals**, 13.11).
- **Logs**: the **slow query log** (queries exceeding `long_query_time` — finds the slow queries, 13.10), the **error log** (crashes, replication errors, warnings), the **audit log** (who did what, 13.14). Searched/aggregated for diagnosis.
- **Traces**: a request's path (app → DB → replica → other services) — for diagnosing *where* latency/errors arise in a distributed system (M12).
The goal: **alert on the golden signals** (especially *saturation* — it predicts the others) so you intervene *before* a saturation point becomes an outage, and have the *logs/traces* to diagnose *why*.

**Why it exists / what it solves.** Observability exists because **a database failing silently or degrading gradually becomes an outage you didn't see coming** — and you can't fix what you can't see. It solves the gap between "the system is misbehaving" and "we *know* it's misbehaving, *what's* wrong, and we caught it *early*." Metrics catch *trends* (latency creeping up, saturation climbing) before they're failures; logs explain *what* (which query is slow, what error occurred); traces explain *where* (which hop). The golden signals provide a *minimal, complete* set to watch (latency/traffic/errors/saturation cover the symptoms of almost any problem). For money, observability is critical: a payments DB that's silently degrading (lag growing, a replica failing, the disk filling, semi-sync degraded, M10/10.12) can become a money-losing outage or a durability gap — and observability (especially the early-warning signals, 13.11) is what catches it *while it's still preventable*. It's the eyes that keep the system operable.

**Tradeoffs & alternatives.** Observability trades **collection overhead and cost** for **visibility**: metrics/logs/traces cost CPU, storage, and money (the monitoring stack), and over-instrumenting (logging every query, tracing everything) adds overhead. The tradeoff is *overhead vs visibility* — you instrument *enough* to catch and diagnose problems without the instrumentation itself becoming a burden (e.g., the slow query log set to a sensible `long_query_time`, not logging *every* query; sampling traces; aggregating metrics). The "alternative" — flying blind (minimal monitoring) — means problems surface as outages (late, expensive). The discipline: **alert on a small set of high-signal metrics (the golden signals, especially saturation/early-warning, 13.11), keep diagnostic logs (slow query log, error log) for when you need them, and don't over-instrument.** For money, the visibility is worth the overhead — deep monitoring with early-warning signals is non-negotiable for a payments platform.

**Generics / first-principles.** "Observe via metrics (trends, alertable), logs (events, diagnosis), and traces (distributed paths), organized by the golden signals — latency, traffic, errors, saturation — so problems surface early, not as outages." This is the universal **observability / monitoring** discipline (the three pillars — metrics/logs/traces — and the golden signals, from Google's SRE practice). The transferable insight: **you operate what you can see — instrument the system with metrics (for trends and alerting), logs (for diagnosis), and traces (for distributed paths), and watch the golden signals (latency/traffic/errors/saturation) so degradation surfaces *before* it's an outage; saturation especially is the leading indicator.** It transfers to every production system (the same observability discipline for any database, service, or platform). The deep principle: **observability is the prerequisite for operability — you can't run, recover, or improve what you can't measure**, and the golden signals (especially saturation/early-warning, 13.11) turn reactive firefighting into proactive prevention.

**MySQL-specific reality.** MySQL observability sources: **status variables** (`SHOW GLOBAL STATUS` — QPS, connections, InnoDB stats, semi-sync status M10/10.12), **`performance_schema`/`sys`** (13.10 — detailed query/lock/IO instrumentation), the **slow query log** (13.10 — queries over `long_query_time`), the **error log**, and OS metrics — scraped by **Prometheus + Grafana** (or PMM — Percona Monitoring & Management, MySQL-specific) and alerted. The **golden signals** for MySQL: *Latency* (query time, from `performance_schema`/slow log), *Traffic* (`Questions`/`Com_*` rates, transactions/sec), *Errors* (`Aborted_connects`, deadlocks, replication errors), *Saturation* (`Threads_connected` vs `max_connections`, buffer-pool pressure, disk, **replication lag** M10, **history-list length** M08/M09, **checkpoint age** M09 — the early-warning signals, 13.11). For **sharding** (M11), monitor *per shard* + aggregate. For our domain: the payments platform watches latency/traffic/errors/saturation per shard, alerts on the **early-warning signals** (13.11 — lag, HLL, checkpoint age, connection saturation, semi-sync status M10/10.12, disk space), keeps the slow query log + error log for diagnosis, and runs **reconciliation** (M12/12.14) as the money-correctness watchdog — so a degrading ledger is caught early, not as a 3am outage. The ★ SVG (Pass C) draws the observability stack + the four golden signals.

## 13.10 · performance_schema, sys & the slow query log

**Mental model.** MySQL's *built-in* instrumentation for finding *what's actually slow or contended and why* — three layers. **`performance_schema`** (P_S): low-level instruments capturing detailed events — every statement's execution, every lock wait, every I/O, every stage — extremely detailed but *raw* (hundreds of cryptic tables). **`sys`** schema: *human-friendly views* over `performance_schema` — pre-built queries that answer real questions ("what are the slowest statements?", "what's causing lock waits?", "which tables have the most I/O?") in readable form — the *practical* interface to P_S. **Slow query log**: logs every query exceeding `long_query_time` (and optionally those not using indexes) — the simplest tool to *find the slow queries* (then analyzed with `pt-query-digest` to rank by total impact). Together: the slow log + `sys` views are how you find *the* problematic queries (the slow ones, the contended ones) and understand *why* (full scans, lock waits, bad plans — M06).

**How it actually works.**
- **`performance_schema`**: instruments the server at a low level — statement events (`events_statements_*`), wait events (locks, I/O, mutexes — `events_waits_*`), stages, memory, etc. — into in-memory tables. Powerful but raw; you rarely query it directly.
- **`sys` schema**: *views* over P_S that answer practical questions: `sys.statement_analysis` / `statements_with_full_table_scans` (slow/scanning queries), `sys.schema_table_lock_waits` / `innodb_lock_waits` (what's blocking what — M08), `sys.io_global_by_file_by_bytes` (I/O hotspots), `sys.schema_unused_indexes` (indexes to drop, M05). This is the *practical* P_S interface — readable answers to "what's slow/contended/wasteful?"
- **Slow query log**: logs queries exceeding `long_query_time` (e.g., >1s) and optionally those `log_queries_not_using_indexes` — the raw record of slow queries. Analyzed with **`pt-query-digest`** (Percona) which *aggregates* by query fingerprint and **ranks by total time** (so you fix the query consuming the most *cumulative* time, not just the single slowest — a fast query run a million times can dominate). 
Workflow: slow log + `pt-query-digest` finds the high-impact queries → `EXPLAIN` (M06) shows *why* they're slow → `sys`/P_S shows lock waits/I/O → you fix (index, M05; rewrite, M06; reduce contention, M08).

**Why it exists / what it solves.** This instrumentation exists because **"the database is slow" is useless without knowing *which queries* and *why*** — and these tools answer exactly that. The slow query log + `pt-query-digest` finds *which* queries cost the most (by total time), `EXPLAIN` (M06) and `sys`/P_S explain *why* (full scans, lock waits, I/O, bad plans), and you fix the *highest-impact* problems first. It solves the diagnostic core of database performance work: *find the queries that matter, understand why they're slow/contended, fix them.* For a payments platform, it's how you find the query causing **replica lag** (M10 — a slow query on the primary that the replica also runs slowly), the query causing **lock waits** (M08 — contention on hot accounts), or the missing index (M05) behind a slow report — turning vague "the DB is slow" into specific, fixable problems. It's the practical toolkit for keeping the database *fast* in production.

**Tradeoffs & alternatives.** **`performance_schema`** has *overhead* (instrumentation costs CPU/memory) — it's tunable (enable only the instruments you need); the defaults are reasonable but heavy instrumentation can be measurable. **The slow query log** has minimal overhead at a sensible `long_query_time` but logging *every* query (`long_query_time=0`) is heavy. The **`sys` views** are just queries over P_S (no extra overhead beyond P_S). The tradeoff is *instrumentation overhead vs diagnostic depth* — you keep enough enabled to diagnose (P_S defaults + slow log at a sensible threshold) without the overhead dominating. The alternatives — external tools (PMM, Datadog) — largely *build on* these same sources (P_S, slow log). The discipline: **use the slow log + `pt-query-digest` to find high-impact queries, `sys` views for lock/I/O/index analysis, and `EXPLAIN` (M06) to understand plans — without over-instrumenting P_S.** These are the *first* tools for any "the database is slow" investigation.

**Generics / first-principles.** "Instrument the database to find *which* queries cost the most (by total impact) and *why* (scans, lock waits, I/O, bad plans) — then fix the highest-impact problems first." This is the universal **database performance diagnosis** workflow (find-the-expensive-queries → understand-why → fix, present in every database's tooling). The transferable insight: **performance work is *measurement-driven* — instrument to find the queries that consume the most *total* resources (not the single slowest), diagnose *why* (the plan, the contention, the I/O), and fix the biggest wins first** — never optimize by guessing. It transfers to every database (Postgres pg_stat_statements ≈ slow log + digest; the same find-diagnose-fix loop). The deep principle: **optimize what the data says matters** — the high-total-impact queries (a fast query × a million calls beats a slow query × ten), found by aggregating, not by eyeballing the slowest single query. This is the practical extension of M06 (query optimization) into *production* (finding *which* queries to optimize, from real traffic).

**MySQL-specific reality.** MySQL's tools: **`performance_schema`** (low-level instruments, on by default with reasonable settings), **`sys`** schema (human-friendly views: `sys.statement_analysis`, `sys.innodb_lock_waits` M08, `sys.schema_unused_indexes` M05, `sys.io_global_by_file_by_bytes`), and the **slow query log** (`slow_query_log=ON`, `long_query_time`, `log_queries_not_using_indexes`) analyzed with **`pt-query-digest`** (rank by total time). Workflow: slow log + digest → find high-impact queries → `EXPLAIN`/`EXPLAIN ANALYZE` (M06) → `sys`/P_S for locks (M08)/I/O → fix (index M05, rewrite M06, reduce contention M08). For **sharding** (M11), analyze *per shard*. For our domain: to diagnose **replica lag** (M10), find the slow query (on the primary, via slow log) that the single-threaded-or-slow applier struggles with; to diagnose **lock waits** on hot accounts (M08), use `sys.innodb_lock_waits`; to find a slow **reconciliation/report**, slow log + `EXPLAIN` reveals a missing index (M05). These built-in tools are the platform's first-line diagnostics — finding *what's* slow/contended and *why* — feeding the early-warning signals (13.11) and the monitoring (13.9). The Pass C Mermaid draws P_S → sys views → slow log → finding the slow/contended queries.
