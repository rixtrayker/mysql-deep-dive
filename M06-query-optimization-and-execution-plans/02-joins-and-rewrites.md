## 6.8 · Join algorithms in depth: nested-loop, BNL, hash

**Mental model.** M04/4.9 introduced *that* MySQL joins tables with an algorithm; here's the deep version: **how each algorithm actually works, how to read which one EXPLAIN chose, and how to make it pick the efficient one.** The three that matter: **index nested-loop** (for each driving row, an indexed lookup into the inner table — *fast when the join column is indexed*), **block nested-loop** (buffer driving rows to reduce repeated inner scans — a fallback when there's no index), and **hash join** (build a hash table on the smaller input, probe with the larger — for big *unindexed* equi-joins, 8.0+). The single most important lever: **an index on the inner table's join column** flips a slow join into a fast index nested-loop.

**How it actually works.**
- **Index nested-loop join (NLJ):** iterate the driving (outer) table; for *each* row, do an **indexed lookup** (`ref`/`eq_ref`, 6.3) into the inner table on the join column. Cost ≈ (driving rows) × (one cheap index lookup each). Fast *if* the inner join column is indexed; EXPLAIN shows the inner table with `type: ref`/`eq_ref` and the index in `key`. **Without** the inner index, each driving row triggers a full inner scan → quadratic disaster.
- **Block nested-loop (BNL):** instead of one inner scan per driving row, buffer a *block* of driving rows and scan the inner table once per block — fewer inner scans. A mitigation for a missing index; EXPLAIN shows `Using join buffer (Block Nested Loop)`. Largely **superseded by hash join in 8.0**.
- **Hash join (8.0.18+):** for a large equi-join with **no usable index**, build an in-memory **hash table** on the smaller input (keyed by the join column), then stream the larger input and probe the hash — O(n+m) instead of O(n×m). EXPLAIN (`FORMAT=TREE`) shows `Hash join`. Needs memory (can spill to disk if huge). Equi-joins only.
The optimizer picks among these based on indexes and statistics; reading EXPLAIN tells you which it chose, and the fix for a slow one is almost always **index the join column** to enable index NLJ.

**Why it exists / what it solves.** Joins reassemble normalized data (M02), and the algorithm determines whether that's cheap or catastrophic. Index NLJ makes the ubiquitous "look up related rows" pattern fast (and is *why* normalization's join cost is usually affordable, M02/2.1). Hash join rescues large analytical joins that index NLJ can't help (no selective index) and that BNL handled poorly. Knowing the mechanics lets you read EXPLAIN and apply the right fix (index for NLJ) instead of guessing.

**Tradeoffs & alternatives.** Each has a sweet spot: **index NLJ** is best for selective joins (small driving set, indexed inner) but terrible without the inner index; **hash join** is best for large unindexed equi-joins but needs memory and only does equality; **BNL** is a mostly-obsolete middle ground. The main lever you control is **indexes on join columns** (M05) — provide them and the optimizer uses fast index NLJ; omit them and it falls back to hash/BNL (better than nothing, but it's scanning whole tables). For very large joins where no selective index exists, hash join is genuinely the right tool — not every join *should* be NLJ.

**Generics / first-principles.** "To combine two sets, either index-lookup each element (NLJ) or build a hash and probe (hash join) — and the right choice depends on selectivity and indexes." These are the universal join strategies (the same in any query engine, dataframe library, or hand-written merge). The transferable instinct: **make the inner lookup cheap (an index or a hash) and drive from the smaller/more-selective side** — a join is only as fast as its inner-row resolution. Recognizing "this is doing a full inner scan per outer row" (the quadratic trap) is the key diagnostic.

**MySQL-specific reality.** Historically **NLJ + BNL only**; **hash join added in 8.0.18** (a major improvement for unindexed joins). Read the algorithm in EXPLAIN: inner `type: ref`/`eq_ref` = index NLJ (good); `Using join buffer (Block Nested Loop)` = BNL (index the join column); `FORMAT=TREE` showing `Hash join` = hash join (fine for big unindexed equi-joins, but check if an index would be better). For our domain, joining `account` → `ledger_entry` on `account_id`: with `ledger_entry.account_id` indexed (M05), EXPLAIN shows fast index NLJ (`ref` on the inner); without it, hash join or a scan that degrades as the ledger grows. The dominant tuning move: **index the join columns** so the optimizer uses index NLJ — verified by re-EXPLAIN showing `ref`/`eq_ref` on the inner table.

## 6.9 · Join order & the driving table

**Mental model.** Beyond the algorithm, the optimizer chooses the **join order** — which table is the **driving (outermost) table** that the others are looked up from. This matters enormously because **the driving table's surviving-row count multiplies through the whole join**: drive from a small/selective table and intermediate results stay tiny; drive from a huge table and you multiply work by orders of magnitude. You can *read* the chosen order in EXPLAIN (the row order) and, when the optimizer gets it wrong, *influence* it (rewrite or hint).

**How it actually works.** In a join, the optimizer picks an order to process the tables; the first is the **driving table**, and for each of its (post-filter) rows it looks up matches in the next, and so on. The cost is roughly the product of rows flowing through each stage, so the order that **keeps the early row counts smallest wins** — typically **drive from the most selective / smallest-after-filtering table**. Example: joining `account` (filter `status='active'` → a few thousand rows) with `ledger_entry` (a billion rows): driving from `account` means a few thousand indexed lookups into `ledger_entry` — cheap. Driving from `ledger_entry` means scanning a billion rows looking up accounts — catastrophic. EXPLAIN's **top-to-bottom row order = the join order** (first row = driving table); `EXPLAIN ANALYZE`'s `loops` count on inner operators reveals how many times each was probed (high loops = large driving set, 6.6). The optimizer searches the order space (factorial, so it prunes, M04/4.6), guided by statistics (6.5) — so a *mis-estimate* can cause a *wrong order*.

**Why it exists / what it solves.** Join order is one of the highest-impact plan decisions — a wrong order can make an otherwise well-indexed join orders of magnitude slower. Understanding it lets you *read* whether the optimizer chose well (is it driving from the small selective table?) and *diagnose* when it didn't (a stale statistic made it think the big table was small, 6.5). It's the difference between a join that scales and one that explodes.

**Tradeoffs & alternatives.** The optimizer *usually* gets order right when statistics are accurate — so the first fix for a bad order is **fresh statistics** (`ANALYZE TABLE`, 6.5), not forcing the order. When the optimizer is genuinely wrong (a mis-estimate it can't fix, correlated columns), you can influence order via a **rewrite** (e.g., restructuring the query, sometimes a `STRAIGHT_JOIN` or join-order hint, 6.12) — accepting the brittleness of overriding the optimizer. The tradeoff is "trust the cost-based optimizer (adaptive, usually right)" vs "force the order (predictable, but brittle as data changes)" — default to the former, override only with evidence.

**Generics / first-principles.** "Process the most-filtering step first to shrink the working set early; order determines whether work compounds or stays bounded." This is the universal "filter early, drive from the small side" principle — apply the most selective operation first so everything downstream operates on less data (true of query joins, pipeline stages, set operations, even nested loops in code). The transferable instinct: **in any multi-stage combination, the stage order that minimizes intermediate sizes dominates cost** — start from the smallest/most-selective input.

**MySQL-specific reality.** EXPLAIN row order = join order (first = driving table); `EXPLAIN ANALYZE` `loops` reveals probe counts (6.6). The optimizer chooses order cost-based (M04/4.6) from statistics — so a bad order is often a **statistics** problem (fix with `ANALYZE TABLE` first). When you must override: `STRAIGHT_JOIN` forces left-to-right join order, or `JOIN_ORDER`/optimizer hints (8.0) influence it (6.12) — last-resort tools. For our domain, the `account`(filtered)→`ledger_entry` join *must* drive from `account`; if EXPLAIN shows it driving from `ledger_entry` (a billion-row scan), that's the bug — check stats first, then consider influencing the order. Reading and reasoning about driving-table choice is a core join-tuning and interview skill.

## 6.10 · Subqueries, derived tables & CTEs

**Mental model.** The same logical result can be expressed multiple ways — a **subquery** (a query nested in `WHERE`/`SELECT`/`FROM`), a **join**, a **derived table** (a subquery in `FROM`), or a **CTE** (`WITH … AS`, a named subquery) — and *how you write it* can affect the plan (sometimes a lot, sometimes not at all, depending on whether the optimizer can transform between forms). The key skills: recognize that these forms are often **interchangeable**, know which rewrites help (especially turning a **correlated subquery** into a **join**), and understand how MySQL handles **CTE/derived-table materialization vs merging**.

**How it actually works.**
- **Correlated subquery** (the inner query references the outer row, e.g., `WHERE EXISTS (SELECT 1 FROM ledger_entry e WHERE e.account_id = a.account_id …)`): conceptually re-runs *per outer row* — historically a performance trap. Often **rewritable as a join** (or semi-join), which the optimizer can execute as an efficient index NLJ (6.8). Modern MySQL *can* transform many correlated subqueries into semi-joins automatically, but not all — so the rewrite is still a tool.
- **Derived table** (subquery in `FROM`): MySQL either **merges** it into the outer query (so its conditions/indexes apply directly — good) or **materializes** it (runs it into a temp table first, then queries that — can be slower, and the temp table has no indexes). EXPLAIN shows `DERIVED`/`Using temporary`.
- **CTE** (`WITH … AS (…)`): in MySQL 8.0, a non-recursive CTE is treated like a derived table — **merged or materialized** by the optimizer. A CTE referenced *multiple times* may be materialized once (or re-evaluated) — worth checking. **Recursive CTEs** (`WITH RECURSIVE`) enable hierarchical/graph traversal.
The practical reality: these forms often compile to the *same* plan when the optimizer can transform them — but when it *can't* (a correlated subquery it won't flatten, a materialized derived table that should've been merged), rewriting to the form the optimizer handles well is the fix.

**Why it exists / what it solves.** Offering multiple equivalent syntaxes lets you express queries naturally (CTEs for readability, subqueries for "does X exist") — but it also means a *readable* form isn't always a *fast* form. Knowing the rewrites lets you keep readability where the optimizer handles it and switch to a join/merged form where it doesn't. The classic win is eliminating a correlated-subquery-per-row by rewriting as a join (turning O(n) re-evaluations into one index NLJ).

**Tradeoffs & alternatives.** **Readability vs plan control:** CTEs and subqueries can be clearer than a tangle of joins, and modern optimizers often plan them identically — so prefer the readable form *unless EXPLAIN shows a problem* (materialization, per-row re-evaluation), then rewrite. **Materialized vs merged:** materializing a derived table/CTE can *help* (compute once, reuse) or *hurt* (an un-indexed temp table, extra pass) — it depends. The judgment: write it clearly, EXPLAIN it, and rewrite only the forms the optimizer demonstrably handles poorly. Don't pre-emptively contort queries into joins "for performance" if the optimizer already flattens them — verify with EXPLAIN.

**Generics / first-principles.** "Logical equivalence doesn't guarantee equal execution — the same result expressed differently can plan differently, so verify, don't assume." A query optimizer *tries* to normalize equivalent forms to one plan, but it has limits, so the *surface form* sometimes leaks into performance. The transferable instinct: **prefer the clearest expression, but when performance matters, check whether the engine actually treats your forms as equivalent** — and reshape toward the form it optimizes well. (This echoes sargability, 6.13: how you *write* a condition affects whether the engine can use structure.)

**MySQL-specific reality.** MySQL 8.0 specifics: **CTEs** (`WITH`, including `WITH RECURSIVE`) since 8.0; **semi-join transformations** flatten many `IN`/`EXISTS` subqueries into joins automatically (much improved over old MySQL, which materialized `IN` subqueries notoriously badly); **derived table merging** (vs materialization) controlled by the optimizer (and `derived_merge`). EXPLAIN reveals `DERIVED`, `MATERIALIZED`, `Using temporary`. The enduring practical tactics: rewrite a stubborn **correlated subquery as a join** (verify EXPLAIN shows index NLJ, 6.8); check whether a **derived table/CTE is materialized** when merging would let an index apply; use **recursive CTEs** for hierarchy traversal. For our domain, "accounts with any entry over $10k" might be written as a correlated `EXISTS` or a join to a filtered `ledger_entry` — EXPLAIN tells you which the optimizer executes efficiently.

## 6.11 · Sorting & grouping at scale (filesort/temp deep dive)

**Mental model.** This is the **deep-dive tuning** of the blocking operations M04/4.13 introduced and 6.4's `Extra` column flags: when `ORDER BY`/`GROUP BY`/`DISTINCT` **can't use an index**, MySQL must **buffer rows and sort/aggregate them** — in memory if they fit, **spilling to disk** if not (the performance cliff). At scale, these blocking ops often dominate query time. The tuning hierarchy: **(1) eliminate the operation** with an index that provides the order/grouping (M05/5.10), **(2) shrink the working set** so it stays in memory, or **(3) accept it** if neither is possible. The cheapest sort is the one you designed away.

**How it actually works.**
- **Filesort** (`Extra: Using filesort`): MySQL reads the rows, sorts them in a `sort_buffer`; if they exceed the buffer, it sorts chunks and **merges them on disk** (external merge-sort — multi-pass, disk I/O). For large result sets this is the cliff. *Fix hierarchy:* an index providing the `ORDER BY` order eliminates it entirely (M05/5.10, the rows come out sorted); failing that, reduce the rows sorted (filter earlier, `LIMIT`), select fewer/narrower columns (smaller sort rows), or — last — size the buffer so it stays in memory.
- **Temp table** (`Extra: Using temporary`): `GROUP BY`/`DISTINCT`/some unions materialize an intermediate result; **in-memory** (fast, the TempTable engine in 8.0) if small, **on-disk** if it exceeds limits or holds large columns. *Fix:* an index aligned with the `GROUP BY` (rows for each group are contiguous → aggregate streaming, no temp table), a rewrite, or keeping the grouped set small.
The deep skill is reading *which* spill is happening (filesort vs temp), *whether it's on disk* (the cliff), and choosing the right fix from the hierarchy — almost always an index (M05/5.10) when one can provide the order/grouping.

**Why it exists / what it solves.** Sorting and grouping are *semantically required to see all input* (M04/4.13) — they're unavoidable as operations, but their *cost* (and especially the disk spill) is highly tunable. At scale they're frequently the real bottleneck (a query with a great access type but a disk filesort). This concept gives the systematic approach to find and eliminate them — usually the single biggest win on a hot reporting/statement query.

**Tradeoffs & alternatives.** The tuning hierarchy *is* the tradeoff structure: **eliminate** (best — an index, but write/storage cost, M05/5.15) > **shrink the working set** (filter/limit/narrow — sometimes changes semantics) > **enlarge buffers** (worst as a primary fix — costs RAM × connections, only delays the cliff, M04/4.13). Some sorts/groups *can't* be indexed away (a sort on a computed expression, an arbitrary ad-hoc grouping) — there you minimize the set and accept the in-memory sort. The key judgment: distinguish a *fixable* spill (an `ORDER BY` an index could serve) from an *intrinsic* one (and keep the latter in memory).

**Generics / first-principles.** "Blocking operations that exceed memory spill to disk — a cliff; avoid them by pre-ordering, or keep their working set in memory." This is the universal external-sort / spill-to-disk principle (M04/4.13 at the tuning level): the same dynamic governs big-data shuffle sorts, external merge in any system, and out-of-core algorithms. The transferable instinct: **identify blocking steps, eliminate them via pre-existing order/structure when you can, and when you can't, keep their working set bounded** — because the in-memory→disk transition is a discontinuity in cost, not a gradient.

**MySQL-specific reality.** EXPLAIN `Extra`: `Using filesort`, `Using temporary` (6.4) are the targets; buffers `sort_buffer_size`, `tmp_table_size`/`max_heap_table_size` set in-memory thresholds (per-connection — don't oversize, M04/4.13); 8.0's **TempTable engine** improved in-memory temp tables but disk spills remain a cliff. The dominant fix is an index (M05/5.10): for the ledger statement query, `(account_id, created_at)` delivers account 42's rows *pre-sorted* → no filesort; for "daily totals" a `GROUP BY` aligned with an index avoids the temp table, or you maintain a summary table (M02/2.14). The everyday loop move: see `Using filesort`/`Using temporary` on a hot query → design an index that provides the order/grouping → re-EXPLAIN confirms it's gone. This is among the highest-impact tuning wins and a constant in M06's worked examples.
