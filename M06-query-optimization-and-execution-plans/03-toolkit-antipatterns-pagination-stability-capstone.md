## 6.12 · Optimizer & index hints (and when to use them)

**Mental model.** Hints **override the optimizer's choices** — forcing a specific index, join order, or algorithm. They're a **last resort**, not a first tool: the optimizer is usually right, and when it's wrong the *correct* fix is almost always to fix its **inputs** (add an index, refresh statistics, rewrite the query). A hint is what you reach for when good indexes + fresh stats + a sensible query *still* produce a bad plan — and you accept that hints are **brittle** (they hard-code a decision that can become wrong as data and the optimizer evolve). "Fix the inputs first; hint only when you must."

**How it actually works.** MySQL offers two hint families:
- **Index hints** — `USE INDEX` / `FORCE INDEX` / `IGNORE INDEX`: tell the optimizer which index(es) to consider or avoid for a table (e.g., `FORCE INDEX (ix_acct_created)` to make it use an index it's wrongly skipping).
- **Optimizer hints** (8.0, `/*+ … */` comment syntax) — finer-grained: `INDEX()`/`NO_INDEX()`, `JOIN_ORDER()`/`STRAIGHT_JOIN`, `BNL`/`NO_BNL`, `HASH_JOIN`, `MAX_EXECUTION_TIME`, etc. — influence specific decisions (which index, join order, algorithm) per query block.
You apply a hint *after* diagnosing (via EXPLAIN/trace, 6.7) that the optimizer is making a specific wrong choice you can't fix by inputs — then **re-EXPLAIN to confirm** the hint produced the intended plan. The discipline: a hint is a documented override with a *reason* (the optimizer mis-estimates X here), not a reflexive "force the index."

**Why it exists / what it solves.** Cost-based optimization is heuristic and estimate-driven (M04/4.6–4.7), so it occasionally chooses badly for reasons you can't cleanly fix (a persistent mis-estimate from correlated columns, a known optimizer limitation). Hints give you an escape hatch to *guarantee* a plan for a critical query when the optimizer won't choose it on its own. They solve the "I've fixed the inputs and it's *still* wrong" case — rare, but real, and sometimes critical (a money query that must not regress).

**Tradeoffs & alternatives.** The core tradeoff: **adaptivity vs control.** The optimizer adapts as data/stats change (usually for the better); a hint *freezes* a decision that was right *today* but may be wrong *tomorrow* (the data grows, the right index changes, a new MySQL version optimizes differently) — so hints are **maintenance debt and brittleness**. The alternatives — and the *preferred* fixes — are almost always better: **add/adjust an index** (M05), **`ANALYZE TABLE`/histograms** (6.5), **rewrite the query** (6.10/6.13). The rule: exhaust the input fixes first; use a hint only as a deliberate, documented last resort, and revisit it periodically (it may no longer be needed, or may now be harmful).

**Generics / first-principles.** "Prefer fixing a system's inputs over overriding its decisions — overrides are brittle because they assume conditions that change." This is the universal "don't hard-code around the optimizer/scheduler/planner" principle: forcing a specific choice trades the system's adaptivity for short-term control, and the forced choice rots as conditions drift. The transferable instinct: **when an adaptive system makes a bad choice, fix what it reasons *from* (its inputs) before you fix what it *decides* (its output)** — and treat any override as debt to be justified and revisited.

**MySQL-specific reality.** `USE`/`FORCE`/`IGNORE INDEX` (classic) and `/*+ … */` optimizer hints (8.0). Common legitimate uses: `FORCE INDEX` when the optimizer skips a good index due to a stubborn mis-estimate (after confirming via trace, 6.7, that stats won't fix it); `STRAIGHT_JOIN` to fix a wrong join order (6.9) the optimizer won't correct; `MAX_EXECUTION_TIME` to cap a runaway query. The strong guidance for fintech: **a money query forced with a hint is a stability liability** — prefer a designed index (M05) and fresh stats that make the optimizer *naturally* choose the right plan; if you must hint a critical query, document why and monitor it (a hint that becomes wrong silently regresses, 6.15). Hints are the last rung of the tuning loop, used sparingly and deliberately.

## 6.13 · Query anti-patterns & their fixes ★

**Mental model.** A catalog of **query shapes that silently defeat indexes or explode work** — they look innocent but produce scans, filesorts, or N× round-trips. Recognizing them *by sight* is half of query tuning, because each has a known fix. The unifying theme is **sargability** ("Search ARGument ABLE"): a predicate can use an index only if it's written so the engine can *seek* on the indexed value — and these anti-patterns mostly break sargability or multiply work. This is the *explained* catalog (why + fix + worked examples); M14 has the scannable triage version.

**How it actually works — the catalog (each: why slow → fix):**
- **N+1 queries** — fetch a list (1 query), then loop and fetch each item's details (N queries). N+1 round-trips, each with latency. *Fix:* one query with a `JOIN` or an `IN (…)` batch (or eager-load in the ORM).
- **`SELECT *`** — fetches all columns, defeating **covering indexes** (M05/5.6 — the index can't cover columns you didn't need but asked for), bloating row size and network. *Fix:* select only the columns you use.
- **Non-sargable predicates** — wrapping an indexed column in a **function** (`WHERE DATE(created_at) = …`, `WHERE amount + 0 = …`) or doing arithmetic on it → the index is unusable → full scan (M05/5.12). *Fix:* rewrite to a sargable range (`created_at >= … AND < …`) or a functional index (M05/5.12).
- **Leading wildcard `LIKE '%term%'`** — can't use a B+Tree index (no prefix to seek) → scan (M05/5.14). *Fix:* anchored `LIKE 'term%'` (sargable), or a **fulltext** index for real search (M05/5.14).
- **Implicit type conversion** — comparing a column to a value of a *different type* (an indexed `VARCHAR` column compared to a number, or vice versa) → MySQL converts column-side → **index unusable** → silent full scan (a notorious, invisible trap). *Fix:* match the literal's type to the column (M03/3.1).
- **`OR` across different columns** — can prevent index use (the optimizer may not combine indexes). *Fix:* `UNION` of indexed sub-queries, or a composite/index-merge-friendly rewrite.
- **Deep `OFFSET` pagination** — `LIMIT 1000000, 20` scans and discards a million rows (6.14). *Fix:* keyset pagination (6.14).

**Why it exists / what it solves.** These patterns are *common* (they arise naturally from ORMs, intuitive-but-wrong SQL, and incremental feature growth) and *silent* (they don't error — they just scan, and the query works fine on small data then collapses at scale). Cataloging them with fixes turns "mysteriously slow query" into "oh, that's an implicit conversion / N+1 / non-sargable predicate" — instant diagnosis. They're the highest-frequency causes of slow queries in practice.

**Tradeoffs & alternatives.** Most fixes are pure wins (sargable rewrite, select fewer columns, batch the N+1) with no real downside — the "tradeoff" is mostly *writing the query more carefully*. A few involve judgment: `UNION` vs `OR` adds complexity; fulltext vs `LIKE` is a different subsystem (M05/5.14); eager-loading an N+1 can over-fetch if you don't always need the details. But the anti-patterns themselves have *no* upside — they're accidents of how the query was written, and the fixes are how it *should* be written. The meta-skill is recognizing them, which is what this catalog builds.

**Generics / first-principles.** "How you express a query determines whether the engine can use its structure — write predicates the optimizer can exploit (sargable), and avoid multiplying round-trips." Sargability is the deep principle (echoing M05/5.12): an index/structure is only usable if the query *asks* in terms of it. The N+1 principle is "batch round-trips, don't loop them" (universal to any latency-bound remote call). The transferable instinct: **the surface form of a request leaks into its cost — express it so the system can apply its fast paths**, and never turn one round-trip into N.

**MySQL-specific reality.** All these manifest in EXPLAIN (the loop, 6.1): non-sargable/implicit-conversion → `type: ALL`, `key: NULL`; `SELECT *` defeating covering → no `Using index`; leading wildcard → scan; N+1 → many fast queries in the slow log rather than one (M13). The **implicit-conversion trap** is especially MySQL-nasty: comparing an indexed `VARCHAR` to an integer literal silently converts and scans (and the reverse) — match types exactly. For our domain: an N+1 fetching each account's latest entry (fix: one join); `WHERE DATE(created_at)=…` on the ledger (fix: sargable range, M05/5.12); an idempotency-key lookup where the key column type mismatches the parameter (fix: matching types). This catalog is the most *immediately applicable* concept in M06 and a staple of code review and interviews.

## 6.14 · Pagination & large result sets

**Mental model.** Paginating with `LIMIT n OFFSET m` (`LIMIT 1000000, 20`) gets **slower the deeper you go**, because the database must **scan and discard all `m` skipped rows** to reach the page — O(offset) work that becomes brutal deep into a large list. **Keyset (a.k.a. "seek") pagination** — remember the last row's sort key and continue *after* it (`WHERE (sort_key) > (last_seen) ORDER BY sort_key LIMIT 20`) — does an **index seek straight to the page**, staying **constant-time** regardless of depth. For statements, feeds, and any deep-paginated list at scale, keyset is the correct pattern.

**How it actually works.**
- **`LIMIT/OFFSET`:** to return rows `m`..`m+n`, MySQL produces the ordered result and **reads through the first `m` rows, throwing them away**, then returns the next `n`. Even with an index providing the order, it still *traverses* all `m` skipped index entries (and may fetch rows). So `OFFSET 20` is cheap but `OFFSET 1000000` reads a million entries to return 20 — latency grows linearly with offset (and total cost of paging through the whole list is quadratic).
- **Keyset/seek pagination:** instead of an offset, you carry the **sort key of the last row seen** and ask for rows *after* it: `WHERE created_at < :last_created_at ORDER BY created_at DESC LIMIT 20` (for a newest-first feed), using an index on the sort key. This is an **index range seek** directly to the continuation point — O(1) per page, independent of how deep you are. The cost: you can't jump to an arbitrary page number (you page forward/backward via the last key, not "page 5000"), and the sort key should be **unique** (or use a tiebreaker like `(created_at, id)`) to avoid skipping/duplicating rows at boundaries.

**Why it exists / what it solves.** Deep pagination over large datasets (statements with millions of entries, activity feeds, exports) is a common scaling failure: `OFFSET` pagination works fine in testing (small data, shallow pages) and **collapses in production** at depth. Keyset pagination solves it by leveraging the index to *seek* rather than *scan-and-discard*, keeping every page fast no matter how far the user (or an export job) scrolls. It's a critical pattern for any feed/statement at scale.

**Tradeoffs & alternatives.** **`OFFSET`:** simple, supports arbitrary page jumps ("go to page 50"), fine for *shallow* pagination or small datasets — but O(offset), unusable deep. **Keyset:** constant-time at any depth, but **no random page access** (only next/prev via the last key) and requires a **stable, unique sort order** (add a tiebreaker). For most real feeds/statements (users scroll sequentially, exports page through in order), keyset's constraints are acceptable and its performance is essential. A hybrid: `OFFSET` for shallow UI pages + keyset for deep scrolling/exports. Choose by access pattern: random-page-jump UI on small data → `OFFSET`; deep/sequential at scale → keyset.

**Generics / first-principles.** "Seek to where you need to be using an ordered structure; don't scan-and-discard to get there." Keyset pagination is the database form of "use the index to jump, don't iterate from the start" — the same principle as resuming a cursor/stream from a checkpoint rather than replaying from the beginning, or binary-searching vs linear-scanning. The transferable instinct: **when continuing through an ordered dataset, carry a position (key/cursor) and seek from it, rather than counting from the start each time** — O(1) continuation vs O(n) re-traversal.

**MySQL-specific reality.** `LIMIT m, n` (offset) is convenient but O(m); EXPLAIN won't flag it as "bad" (it may even show `Using index`) — the cost is in the *traversal* of skipped rows, visible in `EXPLAIN ANALYZE` as rows read ≫ rows returned. Keyset uses an index range on the sort key with a unique tiebreaker (e.g., the ledger's `(account_id, created_at, id)` clustering, M05/5.4, makes `WHERE account_id=42 AND created_at < :last ORDER BY created_at DESC LIMIT 20` a fast seek). For our domain, a million-row statement export *must* use keyset — `OFFSET 1000000` would scan a million ledger entries per page. This is a high-impact, frequently-overlooked pattern and a common system-design interview topic (designing a scalable feed/statement).

## 6.15 · Plan stability: why a fast query suddenly gets slow

**Mental model.** A query that's been fast for months can **suddenly regress to slow — with no code or schema change** — because the *plan* changed underneath it. The optimizer's choice depends on **statistics and data**, which drift: the table grows, the data distribution skews, statistics go stale, or different parameter values hit different plans. **Plan stability** is the operational discipline of *detecting* these regressions and *keeping plans predictable* — and it's a *money-never-lies-adjacent* concern, because a reconciliation or balance query silently regressing to a full scan over a billion-row ledger can stall the entire system.

**How it actually works.** Causes of regression:
- **Stale statistics + data growth** (the most common, M04/4.7, 6.5): the ledger grew, but cardinality stats didn't update, so the optimizer's `rows` estimate is wrong and it now mis-chooses (e.g., scan over index). *Fix:* `ANALYZE TABLE` (refresh), schedule stats updates after bulk loads.
- **Data skew** appearing over time: a value that was rare becomes common (or vice versa), so a previously-selective index is no longer selective for it. *Fix:* histograms (M04/4.7).
- **Parameter sensitivity** ("parameter sniffing"-like effects): the *same* parameterized query is optimal with one plan for a common parameter value and a different plan for a rare one, but a single cached/chosen plan can't be optimal for both.
- **Optimizer/version changes**: an upgrade changes cost-model behavior, flipping a plan.
- **Crossing a cost threshold**: as a table grows, the estimated cost of "scan" vs "index" crosses over, and the optimizer flips — sometimes correctly, sometimes due to a mis-estimate.
*Detection:* monitor the slow query log / performance_schema (M13) for queries whose execution time jumped; re-EXPLAIN a suspect query and compare to its known-good plan; check the estimate-vs-actual gap (6.6).

**Why it exists / what it solves.** Performance isn't a one-time tuning achievement — it's a property that *degrades* as data and conditions change, often silently and suddenly. Plan stability is the discipline that keeps *already-tuned* queries fast in production over time. It addresses the "it was fine yesterday" incident class — which is operationally critical because these regressions hit *at scale* (the query only got slow *because* the data grew) and can cascade (a slow reconciliation holds resources, M16).

**Tradeoffs & alternatives.** There's a tension between the optimizer's **adaptivity** (re-choosing as data changes — usually good, lets plans improve) and **stability** (predictable plans — good for avoiding surprise regressions). You don't want to freeze every plan (you'd lose beneficial adaptation), but you do want to *prevent and detect* harmful regressions. Levers: keep statistics fresh (cheap, prevents the common cause), use histograms for skew, monitor for regressions (M13), and for a *critical, regression-prone* query, consider a hint (6.12) to pin its plan — accepting the brittleness in exchange for predictability. The judgment is per-query: let most adapt, stabilize the critical few.

**Generics / first-principles.** "Performance achieved is not performance maintained — systems whose behavior depends on changing inputs (data, stats) will drift, so you must monitor and re-verify, not set-and-forget." This is the universal "performance regressions happen silently as conditions change" lesson — true of caches losing hit rate, JITs de-optimizing, autoscalers drifting. The transferable instinct: **treat performance as a monitored, maintained property, not a one-time fix** — instrument for regressions, and understand that a fast result today rests on conditions (data size, distribution, stats freshness) that *will* change.

**MySQL-specific reality.** The dominant MySQL regression cause is **stale statistics after growth/skew** (M04/4.7) → fix/prevent with `ANALYZE TABLE` (and persistent stats settings, histograms). Detect via the **slow query log** and **performance_schema** `events_statements_summary` (queries whose latency jumped, M13); re-`EXPLAIN`/`EXPLAIN ANALYZE` (6.6) the suspect and compare to its expected plan; the **optimizer trace** (6.7) reveals *why* it flipped (usually a changed estimate). For our domain, the canonical incident: the nightly reconciliation `SUM` over `ledger_entry` was fast, then the ledger crossed a size threshold with stale stats and the optimizer flipped to a full scan → the job stalled → `ANALYZE TABLE` restored the index plan. Building the habit of **monitoring for and root-causing regressions** (not just tuning once) is what keeps the platform's money queries fast *and* stable over years — and it's the bridge to M13 (observability) and M14 (triage).

## 6.16 · Fintech capstone — tuning the ledger's hot queries ★

**Mental model.** The capstone runs the payments system's **critical queries through the full loop (6.1)** end-to-end: the **statement** query, the **balance** read, the **reconciliation** aggregate, and the **idempotency-key** lookup — the exact access patterns M05/5.18 indexed. For each: EXPLAIN it, confirm it uses the intended M05 index (right access type, 6.3), **kill every filesort/temp table** (6.4/6.11), verify the estimate matches reality (6.5/6.6), and ensure the plan is **stable** (6.15). It's the synthesis of M04 (the optimizer/plan) + M05 (the indexes) + M06 (reading and tuning) on the highest-stakes queries — proving they're fast *and* will stay fast as the ledger grows to billions of rows.

**How it actually works — the money queries through the loop:**
- **Statement** (`WHERE account_id=42 AND created_at >= … ORDER BY created_at DESC LIMIT 20`): EXPLAIN should show **`type: range`**, **`key: ix_acct_created`** (the M05 `(account_id, created_at)` index, 5.7), **no `Using filesort`** (the index provides the order, 5.10), and ideally **`Using index`** (covering `amount`, 5.6). Paginate with **keyset** (6.14), not OFFSET. *Before→after:* `type: ALL` + `Using filesort` → `type: range` + `Using index`.
- **Balance read** (`WHERE account_id=42` on the balance summary, M02/2.14): a **`const`/`eq_ref`** PK lookup, served from the buffer pool (M04/4.14). Trivial *if* the access path is the PK — verify it's not scanning.
- **Reconciliation** (`SUM(amount) … GROUP BY account_id` or per-account `SUM`): the heavy one — must use the index for the range, avoid a full-ledger scan, and run on a **replica** (M10) to spare the primary (M02/2.17). The **plan-stability** risk lives here (6.15): keep stats fresh so it doesn't flip to a full scan as the ledger grows.
- **Idempotency lookup** (`WHERE idempotency_key = …`): a **`const`** lookup on the UNIQUE index (M05/5.17) — instant, and the same index that enforces dedup. Verify the parameter type matches the column (no implicit conversion, 6.13).

**Why it exists / what it solves.** It proves the whole performance arc on the queries that matter most: a payments platform lives or dies by whether its statement/balance/reconciliation/idempotency queries are fast and *stay* fast under growth. The capstone shows the loop applied to each, the specific EXPLAIN signatures to expect, the fixes (M05 indexes, keyset pagination, fresh stats), and the stability discipline — turning M04/M05's design into *verified, monitored* performance. It's the operational realization of M01/1.14 + M05/5.18.

**Tradeoffs & alternatives.** The capstone embodies the arc's tradeoffs: covering indexes for hot reads (fast reads vs write/storage cost, M05/5.6/5.15); keyset vs OFFSET pagination (constant-time vs random-page-jump, 6.14); reconciliation on a replica (offload the primary vs eventual-consistency/lag, M10); plan stability via fresh stats vs a hint (adaptive vs pinned, 6.12/6.15). The failure alternatives are the incidents this prevents: a statement query filesorting on a hot account, a reconciliation regressing to a full scan, deep-OFFSET statement exports scanning millions of rows, an idempotency lookup silently scanning due to a type mismatch. The capstone threads these by *method* (the loop), not luck.

**Generics / first-principles.** "Take your most critical operations, make their execution observable, tune them to use the right structures, and monitor them to stay fast — performance is verified and maintained, not assumed." The capstone synthesizes the module's threads: tradeoff (every tuning choice costed via EXPLAIN), generics-first (the observe→diagnose→fix→verify loop is universal), money-never-lies (plan stability keeps money queries predictable). The transferable instinct for any high-stakes system: **identify the critical queries/operations, instrument them, tune to the fast paths, and monitor for regression** — the same discipline whether it's a ledger query, an API endpoint, or a distributed job.

**MySQL-specific reality.** The concrete realization: each money query EXPLAIN'd against the M05-indexed ledger; expected signatures — statement `range`+`Using index` (no filesort), balance `const`, idempotency `const` on the UNIQUE index, reconciliation an index-served aggregate on a replica; keyset pagination for statements (6.14); `ANALYZE TABLE` keeping stats fresh for stability (6.15); the slow query log / performance_schema (M13) catching regressions. This is the exact set of queries **M16** must keep fast across shards, and the tuning discipline here is what makes the platform performant at scale. Tuning the ledger's hot queries well — and keeping them tuned — is one of the highest-leverage operational practices in the entire fintech system, closing the performance arc (M04→M05→M06) that the rest of the resource builds on.
