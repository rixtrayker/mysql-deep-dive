## 7.1 · What a transaction is & why ★

**Mental model.** A transaction is a way to **group several operations into one indivisible unit** so that, to everyone else and to the database itself, they happen *all together or not at all, without interference, and permanently once done*. The reason this exists is the reason money lives in a database: a transfer is "debit Alice, credit Bob," and there must be **no observable moment** where the debit happened but the credit didn't, **no concurrent transfer** that corrupts a balance mid-operation, and **no crash** that loses a committed payment. Without transactions, every multi-step operation is a chance to leave the data half-finished or contradictory. A transaction turns "a sequence of risky individual writes" into "one safe, atomic step."

**How it actually works.** You mark a **boundary** — `START TRANSACTION` (or `BEGIN`) — do your work (the debit, the credit, the balance updates), and end with **`COMMIT`** (make it all permanent) or **`ROLLBACK`** (undo it all). Between begin and commit, the engine: keeps your changes *isolated* from other transactions (they see a consistent snapshot or are blocked, 7.4); records enough information (undo logs) to *roll back* if anything fails (7.2); checks that *invariants* hold (7.3); and on commit, makes the changes *durable* (writes them so a crash can't lose them, 7.5). Those four behaviors are **ACID** — Atomicity, Consistency, Isolation, Durability — the formal contract a transaction provides. The application doesn't implement any of this; it just declares the boundary, and the engine guarantees the rest.

**Why it exists / what it solves.** Two hard problems make transactions necessary: **partial failure** (a multi-step operation can fail halfway — a crash, an error, a constraint violation — leaving an inconsistent half-state) and **concurrency** (many users operate on the same data simultaneously, and naive interleaving corrupts it). Transactions solve both with one abstraction: they make a group of operations behave like a single, atomic, isolated, durable step, so neither failure nor concurrency can produce a state your application didn't intend. For money — where a half-finished or corrupted state means lost or duplicated funds — this isn't a convenience; it's the foundational requirement.

**Tradeoffs & alternatives.** Transactions cost performance and concurrency: isolating transactions requires locking or versioning (M08), durability requires syncing to disk (M09), and stronger guarantees reduce throughput (7.8/7.14). The alternatives are all worse for money: **no transactions** (a non-transactional engine like MyISAM, M04/4.12 — no atomicity, no rollback, data corruption on crash); **application-level "transactions"** (manually undoing partial work — racy, error-prone, can't handle crashes); or **eventual-consistency / compensating actions** (M12 Saga — necessary across distributed boundaries, but more complex and weaker than a local ACID transaction). The rule: use a real ACID transaction wherever a group of writes must be consistent *and* it fits within one database.

**Generics / first-principles.** "Group operations into an atomic unit so partial failure and concurrency can't produce an inconsistent state." The transaction is one of computing's most important abstractions — it appears far beyond databases (filesystem journaling, atomic config swaps, software transactional memory, the "all-or-nothing deploy"). The deep idea: **make a composite operation indivisible** so the system never has to reason about (or expose) its intermediate states. The transferable instinct: when several changes must be consistent together, find the transaction boundary — and if there isn't one, you have a correctness gap.

**MySQL-specific reality.** Transactions are an **InnoDB** feature (M04/4.12) — the default and only sane engine for transactional data; MyISAM has no transactions (a category error for money). InnoDB implements ACID via **MVCC + locks** for isolation (M08), **undo logs** for atomicity/rollback (M09), and the **redo log (WAL) + fsync** for durability (M09). You delimit a transaction with `START TRANSACTION`/`COMMIT`/`ROLLBACK` (and autocommit, 7.6). For our domain, the Alice→Bob transfer *is* a transaction: `START TRANSACTION; insert debit entry; insert credit entry; update both balances; COMMIT;` — and InnoDB guarantees it's atomic, isolated, durable, and that the double-entry invariant (M01/1.19) holds. Everything in this module elaborates that contract; M08/M09 show how InnoDB delivers it.

## 7.2 · Atomicity: all-or-nothing

**Mental model.** Atomicity is the **all-or-nothing** guarantee: either *every* operation in the transaction takes effect, or *none* does. If anything fails partway — an error, a constraint violation, a crash, an explicit rollback — the engine **undoes everything done so far**, leaving no trace, as if the transaction never started. There is no "half-done" outcome. For a transfer, this is the guarantee that you can *never* end up with the debit applied but the credit lost.

**How it actually works.** As a transaction makes changes, InnoDB records the information needed to reverse each one in **undo logs** (M08/M09 — the old row versions). If the transaction **commits**, those changes become permanent. If it **rolls back** (explicitly, or because of an error/crash before commit), the engine **applies the undo logs in reverse** to restore the pre-transaction state. On a crash, **recovery** (M09) rolls back any transaction that was in-flight (not committed) at crash time. So atomicity is delivered by "keep a record of how to undo, and undo everything if we don't reach commit." The unit of atomicity is the whole transaction — every statement between begin and commit succeeds together or is erased together.

**Why it exists / what it solves.** Multi-step operations *will* fail partway (errors, crashes, violated constraints are inevitable), and a partial application is often *worse than no application* — a transfer with the debit but not the credit has destroyed money. Atomicity removes the entire category of "inconsistent half-states from partial failure": the application only ever sees the before-state or the complete after-state, never an intermediate. It lets you write a multi-step operation as if it were a single step, trusting the engine to clean up completely on any failure.

**Tradeoffs & alternatives.** Atomicity costs the overhead of maintaining undo information (storage and write work, M09) and means a failed transaction's work is *wasted* (rolled back — you must retry). The "alternative" — no atomicity (manual cleanup) — is the bug it prevents: application code trying to undo partial work is racy, incomplete, and helpless against crashes. There's no good reason to forgo atomicity for an operation that must be consistent; the only real design choice is *what to put inside the atomic boundary* (7.6 — too much and you hold resources long, 7.15; too little and you split an operation that should be atomic).

**Generics / first-principles.** "All-or-nothing: a composite operation either fully completes or leaves no effect — keep an undo record and reverse on failure." This is the universal atomic-commit principle — the same idea as a two-phase commit, a journaled filesystem operation, a git commit (the working tree is all-or-nothing), or an atomic file rename. The transferable instinct: **for any multi-step change that must be consistent, ensure there's a mechanism to undo partial work completely** — and prefer a system that provides atomicity over hand-rolling rollback, because hand-rolled cleanup never handles every failure path (especially crashes).

**MySQL-specific reality.** InnoDB provides atomicity via **undo logs** (M09) and crash recovery (M09 rolls back in-flight transactions on restart). `ROLLBACK` undoes the whole transaction; an error mid-transaction may roll back the *statement* (and you typically then `ROLLBACK` the transaction). A gotcha: **DDL statements (CREATE/ALTER/DROP) cause an implicit commit** in MySQL — they're not part of a surrounding transaction and can't be rolled back (a real footgun in migration scripts, M13). For our transfer, atomicity means: if the credit insert fails (e.g., a constraint violation) after the debit insert, `ROLLBACK` erases the debit too — money is conserved, the double-entry invariant (M01/1.19) is never left broken. This is *the* reason money tables are InnoDB and operations are explicit transactions (7.6).

## 7.3 · Consistency: invariants preserved

**Mental model.** Consistency means a transaction takes the database from **one valid state to another valid state**, never leaving it in a state that violates the rules you've declared. The "rules" are your invariants: constraints (NOT NULL, UNIQUE, CHECK, foreign keys — M01/1.8), and application-level truths like "the books balance" (double-entry SUM=0, M01/1.19) and "a balance equals the sum of its entries" (M02/2.17). A transaction that *would* violate an invariant is rejected (or rolled back), so the invariant holds before *and* after — never broken in a committed state.

**How it actually works.** Consistency is partly the engine's job and partly yours. **The engine enforces declared constraints**: if a transaction's changes would violate a CHECK, UNIQUE, NOT NULL, or FK constraint (M01/1.8), the operation errors and (with atomicity, 7.2) the transaction can roll back — so no committed state violates a declared constraint. **The application defines the invariants** by writing correct transactions and declaring the right constraints: the double-entry invariant holds *because* you post balanced entries within one atomic transaction (so the SUM-to-zero is never half-applied), and the balance-equals-sum-of-entries invariant holds *because* you update the balance in the *same* transaction as the entry (M02/2.15/2.17). So consistency = (engine-enforced constraints) + (application-designed transactions that preserve the invariants atomicity/isolation protect). Notably, "Consistency" in ACID is the *least purely-engine* guarantee — it leans on the other three (atomicity to not half-apply, isolation to not interleave into an invalid state) plus correct application logic.

**Why it exists / what it solves.** Invariants are the truths your data must always satisfy for it to be correct (the books balance; no orphaned entries; no negative balance where forbidden). Consistency ensures *every committed state* satisfies them — you never observe or build on a state where an invariant is broken. It's the guarantee that makes the *meaning* of your data trustworthy: if "balance = sum of entries" is an invariant the transaction preserves, then a reader can trust the balance. For money, the invariants *are* the correctness (conservation, attribution, non-negativity), so consistency is the guarantee that the money rules always hold.

**Tradeoffs & alternatives.** Enforcing invariants costs: constraints add write-time checks (M01/1.8), and preserving cross-row invariants (balance = sum) requires doing the related updates in one transaction (coupling them, with the contention that implies, 7.15). The tradeoff is correctness vs the cost/contention of enforcement — but for money the invariants are non-negotiable, so the question is *where* to enforce (schema constraints for hard rules, transaction design for cross-row invariants) not *whether*. An "alternative" — enforcing invariants only in application code without transactions/constraints — is the bug: races and partial failures break the invariant, and there's no engine backstop (M01/1.8, M02/2.15).

**Generics / first-principles.** "Every committed state satisfies the system's invariants — define them explicitly and ensure transactions move only between valid states." This is the universal "preserve invariants across state transitions" principle — the same idea as a type system's invariants, a class's representation invariant maintained across method calls, or a state machine that only allows valid transitions. The transferable instinct: **identify your invariants explicitly, push the hard ones into the layer that enforces them unbypassably (constraints), and design each state transition (transaction) to preserve them** — because an invariant that's only "usually true" is a latent corruption.

**MySQL-specific reality.** InnoDB enforces **declared constraints** within transactions (CHECK since 8.0.16, FK, UNIQUE, NOT NULL — M01/1.8); a violation errors and the transaction can roll back (7.2) so no committed state violates them. **Application-level invariants** (double-entry SUM=0, balance=sum-of-entries) are preserved by *transaction design*: post all of a transfer's entries + balance updates in **one** transaction (atomicity ensures they're all-or-nothing; isolation ensures no other transaction interleaves to break the invariant mid-way). MySQL won't *know* your double-entry rule unless you encode it (constraints, or a trigger, or — most commonly — disciplined transaction logic + reconciliation, M02/2.17). For our transfer, consistency = FK/CHECK constraints (no orphaned entry, no negative where forbidden) + the *atomic, single-transaction* posting of balanced entries that keeps SUM=0 true in every committed state. Consistency is where the engine's guarantees and your application's correctness meet.

## 7.4 · Isolation: concurrent transactions don't interfere

**Mental model.** Isolation is the guarantee that **each transaction runs as if it were the only one** — even when thousands run concurrently. Without it, transactions interleave and see each other's half-finished work, corrupting results (two transfers hitting one balance, a report reading a mid-transfer state). Isolation is the **hardest and most expensive** ACID property, and uniquely it comes in **levels** — a spectrum from weak-but-fast to strong-but-slow — because perfect isolation (every transaction truly serial) is too costly, so databases let you *choose* how much isolation you need versus how much concurrency you can afford. Most real transaction bugs live in choosing this level wrong.

**How it actually works.** Conceptually, the gold standard is **serializability**: the concurrent execution produces the same result as *some* serial (one-at-a-time) order — as if there were no concurrency at all. Achieving full serializability is expensive (it requires extensive locking or aggressive conflict detection), so databases offer **weaker isolation levels** (7.8) that permit certain **anomalies** (7.9) in exchange for more concurrency. InnoDB delivers isolation primarily through **MVCC** (multi-version concurrency control — readers see a consistent *snapshot* without blocking writers, M08) plus **locks** (writers, and reads that need to block, take row/range locks, M08). The isolation *level* you pick determines *which snapshot* reads see and *which locks* are taken — and therefore which anomalies are possible. (This concept states *what* isolation is and *that* it's leveled; the levels are 7.8, the anomalies 7.9, the mechanism is M08.)

**Why it exists / what it solves.** Concurrency is unavoidable (a payments system has many simultaneous transfers), and naive concurrent access to shared data corrupts it — lost updates (7.11), reading uncommitted garbage (7.9), inconsistent reports. Isolation solves this by making concurrent transactions *behave as if* serialized (to the degree the level guarantees), so the application can reason about each transaction *in isolation*, ignoring the chaos of simultaneous others. It's the guarantee that lets you write correct single-transaction logic and trust it under concurrency — *if* you chose a sufficient level.

**Tradeoffs & alternatives.** This is *the* central transaction tradeoff: **stronger isolation = more correctness (fewer anomalies) but less concurrency (more locking, blocking, and aborts → lower throughput, more contention).** SERIALIZABLE is safest but slowest; READ UNCOMMITTED is fastest but lets through the most anomalies. The "alternatives" to a sufficient isolation level are explicit mechanisms at a weaker level: **application-level locking** (`SELECT … FOR UPDATE`, pessimistic, 7.12), **optimistic version checks** (7.12), or restructuring to avoid the conflict. The skill (7.14) is matching the level to the operation: strong where correctness demands it (money movement), weaker where you can tolerate it (reporting) — never blindly maximal (throughput death) or minimal (corruption).

**Generics / first-principles.** "Make concurrent operations behave as if they ran one at a time — to the degree you can afford, trading isolation for concurrency." Isolation is the database's answer to the universal concurrency problem (shared mutable state + simultaneous access = corruption), the same problem mutexes, atomics, actor models, and STM address. The deep insight unique to databases: **isolation is a dial, not a switch** — you choose how much of the expensive "as-if-serial" guarantee you actually need. The transferable instinct: **identify what could go wrong under concurrency, and apply exactly enough isolation to prevent it** — over-isolating kills throughput, under-isolating corrupts data.

**MySQL-specific reality.** InnoDB's default isolation level is **REPEATABLE READ** (notable — many databases default to READ COMMITTED, 7.13b), implemented with **MVCC** (consistent-snapshot reads that don't block writers) plus **next-key locks** (which prevent phantoms beyond what the SQL standard requires for RR, 7.13b/M08). You set the level per-session or per-transaction (`SET TRANSACTION ISOLATION LEVEL …`). The practical reality for our domain: InnoDB's RR + row/next-key locks is strong enough for most money operations, *but* the lost-update problem (7.11) still requires care (a read-modify-write needs an atomic update or a lock, not just RR), and write skew (7.13) can occur below SERIALIZABLE. The mechanics — MVCC version chains, the lock types — are M08; here the key facts are *InnoDB defaults to RR, uses MVCC + locks, and prevents more than the bare standard requires.* Isolation is the property; M08 is the machinery.

## 7.5 · Durability: committed survives a crash

**Mental model.** Durability is the promise that **once a transaction commits, its effects are permanent** — they survive a server crash, power loss, or restart that happens a microsecond after `COMMIT` returns. When the database tells you "committed," it's making a guarantee you can build on: that payment *will* still be there. Without durability, "committed" would mean only "committed *unless* we crash," which for money is no guarantee at all. This module states the *guarantee*; the *mechanism* (write-ahead logging, fsync, crash recovery) is M09 — but the contract is: **commit means it's saved, crash-proof.**

**How it actually works (the guarantee; mechanism in M09).** The naive way to be durable — write every change to its final location on disk before commit — is far too slow (random I/O). So databases use **write-ahead logging (WAL)**: before commit, the transaction's changes are written *sequentially* to a **redo log** and **fsync'd to disk** (forced past OS and disk caches to stable storage). Because the redo log is on durable storage, a crash can be recovered: on restart, the engine **replays the redo log** to re-apply any committed changes that hadn't yet reached their final data pages (and rolls back in-flight ones via undo, 7.2). So "durable" means "recorded in the redo log on stable storage at commit time" — the data pages can be written lazily later, because the log can always reconstruct them. The critical, subtle part (M09) is that the `fsync` must *actually* reach stable storage — and a whole class of data-loss bugs comes from durability that *looks* committed but wasn't truly synced (lying disks, mis-set flush settings — M09/M15).

**Why it exists / what it solves.** Systems crash — power fails, processes die, hardware faults. Durability ensures that the boundary between "before commit" and "after commit" is *crash-proof*: anything committed survives, anything not committed is cleanly undone (7.2). It's what makes a commit *trustworthy* — the application (and the user, and the auditor) can rely on a committed payment being permanently recorded. Without it, every commit would carry an invisible asterisk ("…unless we crash in the next instant"), and no money system could promise that a confirmed transfer actually happened.

**Tradeoffs & alternatives.** Durability's cost is the **`fsync` on commit** — forcing data to physical storage is slow (it's the single biggest per-commit cost, M09), so there's a real tension between durability and throughput. MySQL exposes this as a *tunable* (M09/M15): the strictest setting (`innodb_flush_log_at_trx_commit=1` + `sync_binlog=1`) fsyncs on every commit (fully durable, slower); relaxed settings (`=2`, `=0`) batch or skip the sync (faster, but a crash can lose the last fraction of a second of committed transactions). This is the durability/throughput tradeoff matrix (M09/M15) — and for money the strict setting is usually mandatory (a lost committed payment is unacceptable), accepting the throughput cost. The "alternative" to durable commit (in-memory only, or relaxed flush) is acceptable only for data you can afford to lose — never the ledger.

**Generics / first-principles.** "Once acknowledged, it must survive failure — record it to stable storage (via a sequential log) before acknowledging." Durability is the universal "make it survive a crash" guarantee, delivered by the universal write-ahead-logging pattern (log sequentially + sync, apply lazily, replay on recovery) — the same idea in filesystem journals, Kafka's commit log, Raft/consensus logs, and any system that must not lose acknowledged writes. The transferable instinct: **an acknowledgment is a promise of durability — and the promise is only as good as the sync to stable storage** (the recurring M09/M15 lesson that "I/O that didn't truly reach disk" is a silent data-loss source).

**MySQL-specific reality.** InnoDB durability is delivered by the **redo log (WAL)** + **fsync** at commit, with crash recovery replaying the log (full mechanism in M09). The key tunables (M09/M15): **`innodb_flush_log_at_trx_commit`** (1 = fsync every commit, durable; 2 = write to OS cache, fsync ~1×/sec; 0 = neither) and **`sync_binlog`** (binlog durability for replication, M10). For money, the durable settings (`=1`, `sync_binlog=1`) are the norm — a crash must not lose a committed transfer. The honest caveats (M09/M15): durability depends on the fsync *actually* reaching stable storage (lying disks, cloud-storage fsync semantics, the doublewrite buffer for torn pages) — which is exactly the rare-but-catastrophic data-loss territory M15 explores. Here the contract is what matters: **InnoDB COMMIT means durable** (with strict settings), and our transfer, once committed, survives a crash. M09 shows how; M15 shows how it can go wrong.

## 7.6 · Transaction boundaries & autocommit

**Mental model.** *Where* a transaction begins and ends is a **deliberate decision**, not automatic — and by default MySQL works against you here: **autocommit is ON**, meaning *each individual statement is its own transaction*, committed immediately. So a multi-step money operation written as separate statements is *not* atomic — it's several independent commits. To make the transfer atomic, you must **explicitly mark the boundary** (`START TRANSACTION … COMMIT`) so all its statements are one unit. Getting the boundary wrong — too wide, too narrow, or absent — is a common and serious bug.

**How it actually works.** With **autocommit = 1** (the default), every statement runs in its own implicit transaction and commits when it finishes — fine for single-statement operations, fatal for multi-statement ones (the debit commits, then *separately* the credit — no atomicity between them, 7.2). To group statements, you **explicitly start a transaction**: `START TRANSACTION;` (or `BEGIN;`, or `SET autocommit=0;`), run the statements, then `COMMIT;` (or `ROLLBACK;`). Within an explicit transaction, autocommit is suspended until you end it. The **boundary placement** is a design choice: it must be *wide enough* to include every operation that must be atomic together (all of the transfer's entries + balance updates), but *narrow enough* not to hold locks and resources longer than necessary (no user-interaction, no slow external calls, no unrelated work inside the transaction — 7.15). The boundary defines the atomic/isolated/durable unit.

**Why it exists / what it solves.** Autocommit-per-statement is a sensible default for simple, single-statement work (most queries), but multi-step operations *need* explicit grouping or they lose atomicity. Making the boundary explicit forces the developer to *consciously decide* what is atomic — which is exactly right, because "what must happen all-or-nothing together" is a correctness decision only the application knows. It solves the problem of accidentally splitting an atomic operation into independently-committing pieces (the classic "the debit committed but the credit failed and now money is gone" bug from forgetting to wrap the transfer in a transaction).

**Tradeoffs & alternatives.** The boundary is a tradeoff between **atomicity scope** and **resource-holding time** (7.15): a wider transaction guarantees more is atomic but holds locks/undo longer (more contention, more blocking, longer rollback); a narrower one releases resources sooner but risks splitting things that should be atomic. The art is making the boundary *exactly* the set of operations that must be consistent together — no more, no less. Autocommit-on (default) vs autocommit-off (everything in transactions until you commit) is a session-level choice; most applications keep autocommit on and use *explicit* `START TRANSACTION` for the multi-step operations, which is clearest. ORMs and frameworks often manage boundaries for you (a "unit of work") — convenient, but you must understand where they begin/end (a too-wide ORM transaction holding locks across a slow operation is a classic contention bug).

**Generics / first-principles.** "Explicitly delimit the atomic unit — group exactly the operations that must succeed or fail together, and hold the boundary no longer than necessary." This is the universal "scope your critical section / atomic block correctly" principle — too wide a lock/transaction kills concurrency, too narrow leaves a race. The transferable instinct: **the boundary of atomicity is a deliberate design decision driven by your invariants** (what must be consistent together), and a default that auto-commits each step is a trap for any multi-step operation. Always ask: "what is the atomic unit here, and have I drawn the boundary around exactly that?"

**MySQL-specific reality.** MySQL **defaults to `autocommit=1`** — each statement auto-commits unless inside an explicit transaction. Delimit with `START TRANSACTION`/`BEGIN` … `COMMIT`/`ROLLBACK` (or `SET autocommit=0`). Gotchas: **DDL implicitly commits** (7.2 — you can't include `ALTER TABLE` in a transaction); some statements (e.g., `LOCK TABLES`) have special interactions; and an open transaction left uncommitted (a forgotten `COMMIT`, an idle connection in a pool, M04/4.2) **holds locks and undo indefinitely** (7.15) — poisoning concurrency. For our transfer, the boundary is essential and explicit: `START TRANSACTION; insert debit; insert credit; update balance A; update balance B; COMMIT;` — *one* boundary around exactly the operations that must be atomic (the whole transfer), and nothing slow inside it. Drawing this boundary correctly is the single most important application-side transaction decision.

## 7.7 · COMMIT, ROLLBACK & savepoints

**Mental model.** Three controls end or partially-undo a transaction. **`COMMIT`** finalizes — makes all the transaction's changes permanent and durable (7.5) and visible to others. **`ROLLBACK`** abandons — undoes *everything* the transaction did, back to its start (7.2), as if it never happened. **Savepoints** add granularity *within* a transaction: you mark a named point, and can later **roll back to that savepoint** — undoing the work *after* it while keeping the work *before* it and the transaction still open. Savepoints let a long or multi-step transaction recover from a failed sub-step without throwing away everything.

**How it actually works.** `COMMIT` ends the transaction successfully: changes become durable (redo log fsync, 7.5/M09), locks are released (M08), and the changes are now visible to other transactions. `ROLLBACK` ends it by reverting: undo logs are applied to restore the pre-transaction state (7.2), locks released, nothing persisted. A **`SAVEPOINT name`** marks a point mid-transaction; **`ROLLBACK TO SAVEPOINT name`** undoes only the changes made *after* that savepoint (the transaction stays open, earlier work intact, and locks acquired before the savepoint are generally retained); **`RELEASE SAVEPOINT name`** discards the marker. So a transaction can attempt a risky sub-operation after a savepoint, and if it fails, roll back *just that part* and continue — instead of the all-or-nothing of a full rollback. This is useful for batch operations where one item's failure shouldn't abort the others, or for nested "try this, fall back if it fails" logic.

**Why it exists / what it solves.** `COMMIT`/`ROLLBACK` are the basic transaction outcomes — the means to *finalize* or *abandon* atomically (7.2). **Savepoints** solve a finer problem: in a transaction doing *several* logical sub-steps, a failure in one shouldn't necessarily doom the whole transaction. Savepoints provide *partial* rollback — recover the failed sub-step's state without losing the successful work — enabling patterns like "process a batch, skip the items that fail" within a single transaction, or implementing nested-transaction-like semantics (databases don't have true nested transactions; savepoints approximate them).

**Tradeoffs & alternatives.** Savepoints add flexibility but also complexity and subtle semantics (what locks are kept on partial rollback, how errors interact) — overusing them makes transaction logic hard to reason about. The common alternative to savepoints is **smaller transactions** (process each batch item in its own transaction — simpler, and each commits independently, but then they're not atomic *together*, and many small transactions have overhead). The tradeoff: savepoints keep things in one atomic boundary with partial-recovery flexibility (good when the batch *should* be one transaction); separate transactions are simpler but give up cross-item atomicity. Most application logic uses plain `COMMIT`/`ROLLBACK`; savepoints are for specific batch/nested patterns.

**Generics / first-principles.** "Provide both finalize and abandon, plus checkpoints for partial recovery within a unit." Commit/rollback is the universal "confirm or revert" pair; savepoints are the universal "checkpoint so you can partially undo" mechanism (like a sub-transaction, a try/catch with state restoration, or a nested undo stack). The transferable instinct: **a long atomic operation benefits from internal checkpoints** so a local failure doesn't force discarding all progress — but use them judiciously, because partial-undo semantics add complexity to reason about.

**MySQL-specific reality.** InnoDB supports `COMMIT`, `ROLLBACK`, `SAVEPOINT name`, `ROLLBACK TO SAVEPOINT name`, `RELEASE SAVEPOINT name`. Gotchas: a **deadlock or lock-wait-timeout** (M08) may roll back the whole transaction (or just the statement) — your application must be prepared to **retry** (a standard pattern for money operations under contention, M08/M16); **DDL implicitly commits** (7.2/7.6), ending any open transaction. For our domain, savepoints help a *batch* settlement transaction (process many transfers in one transaction, savepoint before each, roll back just the ones that fail) — though often separate transactions per transfer (with idempotency, M16) are simpler and more resilient. The everyday reality is `COMMIT` on success / `ROLLBACK` on error, *plus a retry loop* for transient conflicts (deadlocks, M08) — the practical shape of robust transactional money code.
