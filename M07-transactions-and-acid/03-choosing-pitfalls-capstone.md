## 7.14 · Choosing an isolation level (the decision)

**Mental model.** Choosing an isolation level is a **per-operation decision**: match the level to *what the operation's correctness requires* versus *what concurrency it can afford*. Don't blanket everything at SERIALIZABLE (throughput death) or leave everything at the default without thinking. The reasoning: identify the operation's invariants and the anomalies that would break them, find the *lowest* level (plus any targeted locks) that prevents those anomalies, and accept the rest. Most money *movements* want strong isolation (REPEATABLE READ + explicit locking for read-modify-write); most *reporting* tolerates weaker (READ COMMITTED or a snapshot read that doesn't block writers).

**How it actually works.** The decision procedure:
1. **What must be correct?** Enumerate the operation's invariants (balance conservation, non-negative balance, "exactly once," consistent report).
2. **Which anomalies break them?** Map to dirty/non-repeatable/phantom (the matrix, 7.10) *and* the write anomalies (lost update 7.11, write skew 7.13) the matrix omits.
3. **Find the minimum level + locks** that prevents exactly those: a read-modify-write needs an atomic update or `FOR UPDATE` (regardless of level, 7.11); a multi-row invariant under concurrency needs SERIALIZABLE or read-set locking (7.13); a consistent multi-statement read needs RR's stable snapshot; a tolerant dashboard read needs only READ COMMITTED (and benefits from not blocking writers).
4. **Weigh the concurrency cost** (7.8): higher level / more locking = less throughput, more blocking/deadlocks. Pick the cheapest option that's still correct.
In practice: keep InnoDB's default **REPEATABLE READ** for transactional work (strong, and phantom-protected, 7.13b), add **explicit locks** (`FOR UPDATE`) for read-modify-write and write-skew-prone operations, and consider **READ COMMITTED** for high-throughput or reporting paths that tolerate it.

**Why it exists / what it solves.** There's no single right isolation level for a whole application — different operations have different correctness needs and concurrency profiles, and using one level for everything either over-pays (everything SERIALIZABLE → bottleneck) or under-protects (everything READ COMMITTED → corruption in the money paths). The per-operation decision solves this: each operation gets *exactly* the isolation it needs. It also forces the crucial realization that *isolation level alone is insufficient* for write anomalies — so the "decision" includes *locking strategy*, not just the level.

**Tradeoffs & alternatives.** The whole concept is the tradeoff (correctness vs concurrency, per operation). The main alternatives within it: **raise the level** (blanket protection, more blocking) vs **keep a lower level + targeted locks** (more concurrency, but you must identify and lock the right things — easy to miss a write-skew hazard). For money, the bias is toward *correctness* — when unsure, use the stronger option (RR + locking, or SERIALIZABLE for a tricky multi-row invariant) and optimize concurrency only if measured contention demands it. The opposite bias (maximize throughput, minimize isolation) is dangerous for money and acceptable mainly for tolerant reads.

**Generics / first-principles.** "Apply exactly the consistency each operation needs — no more, no less — by reasoning from its invariants and the failures that would break them." This is the universal "right-size your guarantees per use case" principle (consistency levels in distributed databases, memory orderings per access, lock granularity per critical section). The transferable instinct: **don't pick one consistency setting globally; decide per operation from its correctness requirements and contention** — strong where it matters, weak where it's safe, and remember that the *default* may be wrong (too strong or too weak) for a given operation.

**MySQL-specific reality.** Set per session/transaction: `SET TRANSACTION ISOLATION LEVEL READ COMMITTED;` (next transaction) or `SET SESSION …`. Practical MySQL guidance: **default RR + `FOR UPDATE`** for money read-modify-writes and write-skew-prone operations (the common, correct choice); **READ COMMITTED** for high-write-throughput paths or reporting that tolerates non-repeatable reads (less gap locking → fewer deadlocks, more concurrency — a deliberate tradeoff some shops make globally); a **consistent snapshot read** (RR, or `START TRANSACTION WITH CONSISTENT SNAPSHOT`) for a multi-query report that must see one point-in-time view (often run on a **replica**, M10, to avoid loading the primary). For our domain: transfers run at RR with atomic balance updates / `FOR UPDATE`; reconciliation reads use a consistent snapshot on a replica; dashboards use READ COMMITTED. The decision is concrete and per-operation — and a frequent system-design interview question.

## 7.15 · Transactions, performance & pitfalls

**Mental model.** Transactions are not free, and the main rule is **keep them short and small**. A transaction **holds locks** (M08) and **pins undo/old row versions** (M09) for its *entire* duration — so a *long-running* transaction blocks other transactions (contention) and forces the engine to retain old data (history-list bloat, M09), degrading the *whole system*, not just itself. The classic pitfalls: a transaction left open across slow work (a network call, user think-time, a big computation), an oversized transaction touching too many rows, or a forgotten/uncommitted transaction holding resources indefinitely. "Begin late, commit early, and never wait on the outside world inside a transaction."

**How it actually works.** While a transaction is open: any **locks** it took (row locks from writes or `FOR UPDATE`, M08) are held until commit/rollback — so other transactions needing those rows **block**. Its **read snapshot** (under MVCC/RR, M08) pins the old versions of every row that's been modified since it started, so the engine **can't purge that undo history** — the **history list grows** (M09), bloating undo storage and slowing reads that must traverse long version chains. The longer the transaction, the worse both effects. Specific pitfalls:
- **Long transactions** (slow work inside the boundary — external API calls, user interaction, heavy computation): hold locks and pin undo for ages → contention + bloat.
- **Large transactions** (millions of rows in one transaction): huge undo, long rollback if it fails, big locks, replication impact (M10).
- **Idle-in-transaction** (an open transaction on a pooled connection that's not actively working, M04/4.2 — a forgotten `COMMIT`): holds resources indefinitely, a silent system poison.
- **Hot-row contention**: many transactions serializing on one popular row's lock (the hot-account balance, 7.11/M16).
The fix is discipline: make the boundary (7.6) *exactly* the atomic work, do slow things *outside* it, batch large work into chunks, and ensure transactions always commit/rollback promptly (no open transactions on idle connections).

**Why it exists / what it solves.** Because a transaction's resource-holding is proportional to its duration and size, transaction *hygiene* is a first-order performance and stability concern — a single misbehaving long transaction can stall a whole database (everyone blocked on its locks) and bloat the engine (undo growth). This concept exists to make that explicit: the *correctness* of transactions (ACID) is necessary, but their *cost* (held locks, pinned undo) demands discipline. It prevents the very real production incidents where one long/forgotten transaction degrades everything.

**Tradeoffs & alternatives.** The core tension is **atomicity scope vs resource-holding** (7.6): you must include everything that needs to be atomic, but every extra moment the transaction is open costs contention and bloat. Alternatives for large work: **chunk it** (process in many small transactions — gives up cross-chunk atomicity but releases resources frequently and limits rollback cost); for slow external steps: **do them outside the transaction** (e.g., call the payment provider, *then* record the result in a short transaction — often with the outbox/idempotency patterns, M12/M16, to handle the gap). The tradeoff: keeping things in one transaction is simpler and atomic but holds resources; splitting reduces resource-holding but requires handling partial-progress (idempotency, sagas, M12/M16). For money, prefer short transactions + idempotency over long all-encompassing ones.

**Generics / first-principles.** "Hold shared resources (locks, versions) for the shortest time possible — the duration of a critical section determines the contention it causes." This is the universal "minimize critical-section length" / "don't hold a lock across I/O" principle from all concurrent programming. The transferable instinct: **a transaction is a critical section; keep it short, do no slow or external work while holding it, and release it promptly** — because the cost it imposes on *others* scales with how long it's held, and one long holder can stall everyone. "Don't do I/O (or wait on humans) while holding a lock" is one of the most important concurrency rules, in databases and beyond.

**MySQL-specific reality.** InnoDB specifics: locks held until commit/rollback (M08); the **history list** (undo) grows while a transaction's snapshot pins old versions (M09 — visible via `SHOW ENGINE INNODB STATUS` "History list length", a key health metric); long transactions are a top cause of contention and undo bloat. Detect with `information_schema.INNODB_TRX` (long-running transactions), `innodb_lock_wait_timeout` (caps how long a transaction waits for a lock), and monitoring history-list length (M09/M13). Idle-in-transaction connections (pool leaks, M04/4.2) are a classic incident. For our domain: the transfer transaction must be **short** (just the entries + balance updates + commit — *no* external payment-provider call inside it; do that outside with idempotency, M16); reporting transactions on the primary should be brief or moved to a replica (M10) to avoid pinning undo. Transaction hygiene is operationally critical and previews M08 (locks) and M09 (undo/history list).

## 7.16 · Fintech capstone — the atomic transfer & money invariants ★

**Mental model.** The capstone is the **canonical money-movement transaction** — the transactional heart of the entire payments system — and it ties every M07 concept together on the highest-stakes operation. Moving $100 from Alice to Bob must be: **atomic** (debit + credit + both balance updates all-or-nothing — never half), **consistent** (double-entry SUM=0 and balance=sum-of-entries invariants preserved, M01/1.19, M02/2.17), **isolated** (concurrent transfers to the same accounts don't corrupt balances — no lost update, 7.11), **durable** (survives a crash the instant after commit), and **idempotent** (a retry can't double-post — the UNIQUE idempotency key, M05/5.17, M16). It's the synthesis of M01 (the ledger model), M02 (normalized ledger + derived balance), M05 (the indexes/UNIQUE), and M07 (the transaction that makes it all correct under concurrency and failure).

**How it actually works — the transfer, with every guarantee:**
1. **Boundary (7.6):** `START TRANSACTION` — one explicit transaction around exactly the transfer (nothing slow inside).
2. **Idempotency (M16, 7.7-retry):** insert/check the `transaction_` row with its UNIQUE `idempotency_key` (M05/5.17) — a retry hits the duplicate-key constraint and is recognized as already-processed, so it can't double-post (the structural exactly-once guarantee).
3. **Atomic, balanced entries (Atomicity 7.2, Consistency 7.3):** insert the two `ledger_entry` rows — debit Alice −100, credit Bob +100 — which **sum to zero** (double-entry, M01/1.19). Within the atomic transaction, they're all-or-nothing, so the invariant is never half-applied.
4. **Atomic balance updates (Isolation / lost-update fix 7.11):** `UPDATE account SET balance = balance − 100 WHERE id = Alice` and `… + 100 WHERE id = Bob` — **atomic increments** (not read-then-write), so concurrent transfers to the same account don't lose updates (7.11). InnoDB takes row locks, serializing the hot-account updates correctly.
5. **Commit (Durability 7.5):** `COMMIT` — all of it becomes atomically visible and durable (redo log fsync, M09); a crash an instant later can't lose it.
6. **Retry on conflict (7.7, M08):** if a deadlock/lock-timeout occurs under contention, the application **retries** the whole transaction (safe because of idempotency, step 2).
The result: money is conserved (SUM=0), correctly attributed, never lost or duplicated — under concurrency *and* crashes.

**Why it exists / what it solves.** This is the operation a payments platform exists to perform correctly, and it's where *every* failure mode this module described would manifest: a non-atomic transfer loses money (half-applied), a lost update mis-counts a hot account, a non-durable commit loses a confirmed payment, a non-idempotent retry double-charges. The capstone shows how the transaction — with the right boundary, atomic updates, isolation, durability, and idempotency — makes all of these *impossible*. It's the concrete realization of "money-never-lies" as a transaction, and the foundation M08 (locking under contention), M09 (durability mechanism), and M16 (the platform at scale) build on.

**Tradeoffs & alternatives.** The capstone embodies M07's tradeoffs: **atomic updates vs `FOR UPDATE`** for the balance (7.11 — atomic increment is cheapest and correct here); **short transaction + idempotency vs a long all-encompassing one** (7.15 — keep the external provider call *outside* the transaction, reconcile with idempotency/outbox, M16); **RR + locking vs SERIALIZABLE** (7.14 — RR with atomic updates suffices for a simple two-account transfer; a multi-account invariant might need more, 7.13); **hot-account contention** (many transfers serializing on one balance row — a real cost, addressed by sharding/batching in M08/M16). The failure alternatives are the bugs: read-then-write balances (lost update), forgetting the transaction boundary (half-applied), no idempotency (double-post on retry), non-durable settings (lost commit, M09/M15).

**Generics / first-principles.** "Wrap a multi-step state change that must be consistent in a single short atomic transaction, make each shared-state update atomic, ensure durability, and make the whole thing idempotent so it's safe to retry." This synthesizes the module's threads — money-never-lies (the invariants hold under concurrency and crashes), durability (committed survives), generics-first (ACID + idempotency are universal patterns), tradeoff (RR + atomic updates, not blanket SERIALIZABLE). The transferable instinct for *any* critical state transition: **atomic boundary around exactly the consistent unit + atomic updates to shared state + durability + idempotent retry** — the same recipe whether it's a bank transfer, an inventory decrement, or a distributed saga step (M12).

**MySQL-specific reality.** The concrete InnoDB realization: `START TRANSACTION` (autocommit off for the unit, 7.6); InnoDB engine (M04/4.12) for ACID; the UNIQUE idempotency-key index (M05/5.17) enforcing exactly-once; **atomic `balance = balance + :delta` updates** (7.11) under InnoDB row locks (M08); `COMMIT` with durable flush settings (`innodb_flush_log_at_trx_commit=1`, M09); a **retry loop** around the transaction for deadlocks/lock-timeouts (M08); money as **DECIMAL/minor units** (M03/3.4) so the sums are exact; the transfer kept **short** with the external payment call *outside* it (idempotency/outbox bridging the gap, M16). This is the exact transactional core **M08** analyzes under lock contention, **M09** makes durable, and **M16** scales across shards (keeping debit+credit on one shard so the transfer stays a single local transaction, M02/2.16, M11). The atomic transfer is the beating heart of the fintech system — and getting its transaction right is the single most important correctness decision in the entire platform.
