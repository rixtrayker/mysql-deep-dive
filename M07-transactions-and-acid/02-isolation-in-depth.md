## 7.8 · The isolation levels (the SQL standard four) ★

**Mental model.** Isolation isn't all-or-nothing — it's a **ladder of four standard levels**, each permitting *fewer* concurrency anomalies than the last, at *higher* cost: **READ UNCOMMITTED** (weakest — sees even uncommitted data) → **READ COMMITTED** (sees only committed data, but it can change between your reads) → **REPEATABLE READ** (your reads are stable within the transaction) → **SERIALIZABLE** (strongest — as if transactions ran one at a time). Climbing the ladder removes anomalies (7.9) but costs concurrency. The whole point: you *choose* the rung that's strong enough for your operation's correctness without paying for more isolation than you need.

**How it actually works.** Each level is *defined* by which anomalies (7.9) it permits:
- **READ UNCOMMITTED:** a transaction can read changes made by *other, not-yet-committed* transactions ("dirty reads") — so it might read a value that then gets rolled back (garbage). Maximum concurrency, minimum safety.
- **READ COMMITTED:** a transaction only reads *committed* data (no dirty reads) — but each statement sees the *latest committed* state, so re-reading the same row can return a *different* value if another transaction committed in between ("non-repeatable read").
- **REPEATABLE READ:** within a transaction, repeated reads of the same row return the *same* value (a stable snapshot) — no dirty or non-repeatable reads. The standard says phantoms (new rows appearing, 7.9) are still possible at RR; InnoDB prevents most of them (7.13b).
- **SERIALIZABLE:** the strongest — the result is equivalent to *some* serial execution; no anomalies (including phantoms and, with true serializability, write skew, 7.13). Achieved by extensive locking (reads take locks too), so the most blocking and lowest concurrency.
The level controls *which snapshot reads see* and *which locks are taken* (M08) — that's the mechanism behind the anomaly guarantees.

**Why it exists / what it solves.** Perfect isolation (true serial execution) is too slow for real concurrency, but *no* isolation corrupts data — so the levels offer a **spectrum of compromises**, letting each operation pick its point on the correctness/concurrency curve. A money transfer needs strong isolation (correctness is paramount); a fuzzy analytics dashboard can tolerate weaker (slightly stale reads are fine, and weaker isolation lets it not block writers). The levels exist precisely so you *don't* have to run everything at the slowest, safest setting.

**Tradeoffs & alternatives.** The ladder *is* the tradeoff: **higher level = fewer anomalies (more correctness) but more locking/blocking/aborts (less concurrency, lower throughput).** SERIALIZABLE is correct-by-default but can serialize your whole workload (and cause more deadlocks/aborts under contention); READ UNCOMMITTED is fast but unsafe for almost anything. Within a level, you can *strengthen specific operations* with explicit locks (`SELECT … FOR UPDATE`, 7.12) — so you don't have to raise the *whole transaction's* level to protect one read. The skill (7.14) is choosing the lowest level that's still correct for the operation, plus targeted locking where needed.

**Generics / first-principles.** "Isolation is a tunable dial from 'anything goes' to 'as-if-serial' — each step trades concurrency for correctness." This leveling is specific to databases but reflects the universal "consistency vs availability/performance" tension (and connects to CAP/PACELC, M12). The transferable insight: **strong consistency has a throughput cost, so systems expose it as a choice rather than a fixed property** — and competent use means *selecting the weakest guarantee that's still correct* for each operation, not defaulting to the strongest (slow) or weakest (broken).

**MySQL-specific reality.** InnoDB supports all four; **its default is REPEATABLE READ** (7.13b — notably *not* READ COMMITTED, the default in PostgreSQL/Oracle/SQL Server). You set it per session/transaction: `SET [SESSION|GLOBAL] TRANSACTION ISOLATION LEVEL …`, or `SET TRANSACTION ISOLATION LEVEL …` for the next transaction. Key InnoDB nuances (detailed in 7.13b, M08): RR uses a **consistent snapshot** (MVCC) taken at the transaction's first read, and **next-key locks** that prevent most phantoms the standard would allow at RR. SERIALIZABLE in InnoDB effectively turns plain reads into locking reads. For our domain, most money operations run at the default RR (strong enough with proper locking for the read-modify-write, 7.11); reporting may use READ COMMITTED or a snapshot read to avoid blocking writers. Choosing the level is 7.14.

## 7.9 · Read anomalies: dirty, non-repeatable, phantom ★

**Mental model.** The isolation levels are *defined* by which of three classic **concurrency anomalies** they prevent — so understanding the anomalies *is* understanding the levels. **Dirty read:** you read data another transaction wrote but *hasn't committed* (and might roll back) — reading a value that may never have "really" existed. **Non-repeatable read:** you read a row, another transaction commits a change to it, you read it *again in the same transaction* and get a *different* value — the ground shifted under you. **Phantom read:** you run a query returning a *set* of rows, another transaction inserts/deletes matching rows and commits, you run the *same query again* and get a *different set* — rows appeared or vanished. Each is a specific way concurrent transactions corrupt your view of the data.

**How it actually works (each anomaly, with the level that stops it):**
- **Dirty read** — T1 updates Alice's balance to $0 (uncommitted); T2 reads $0; T1 rolls back (balance was never really $0); T2 acted on garbage. *Prevented at READ COMMITTED and above* (only committed data is visible).
- **Non-repeatable read** — T1 reads Alice's balance = $100; T2 updates it to $50 and commits; T1 reads again = $50. The *same row's value changed* within T1's transaction — so T1's two reads disagree, breaking any logic that assumed stability. *Prevented at REPEATABLE READ and above* (a stable snapshot — re-reads return the same value).
- **Phantom read** — T1 queries "entries for account 42 in June" → 10 rows; T2 inserts a new June entry for account 42 and commits; T1 re-runs the query → 11 rows. A *new row appeared* in T1's result set (a "phantom"). *Prevented at SERIALIZABLE* per the standard (and at InnoDB's RR via next-key locks, 7.13b).
The progression is deliberate: each higher level prevents one more anomaly, which is why the levels form a ladder (7.8) and map cleanly to a matrix (7.10).

**Why it exists / what it solves.** These three anomalies are the *named, well-understood ways* concurrency breaks read consistency — they're the vocabulary the SQL standard uses to *define* isolation, and the concrete bugs you're choosing to prevent (or tolerate) when you pick a level. Knowing them lets you reason precisely: "at READ COMMITTED, can my logic break if a value changes between two reads?" (yes — non-repeatable read), and therefore choose RR. Without this vocabulary, isolation levels are abstract; with it, choosing a level is concrete reasoning about which bugs your operation can't tolerate.

**Tradeoffs & alternatives.** Tolerating an anomaly buys concurrency: allowing non-repeatable reads (running at READ COMMITTED) lets readers see fresh data and reduces locking, which is *fine* for many read patterns (a dashboard) and *catastrophic* for others (a read-modify-write on a balance, 7.11). So the "tradeoff" is per-operation: which anomalies can *this* operation tolerate? The alternative to raising the level (to prevent an anomaly) is a targeted lock (7.12) for the specific read that needs stability. Note these three are *read* anomalies; there are *write* anomalies too — lost update (7.11) and write skew (7.13) — that the standard's read-anomaly framework doesn't fully capture, which is why those get their own concepts.

**Generics / first-principles.** "Concurrent access creates specific, nameable inconsistencies in what you observe — enumerate them, and choose isolation to exclude the ones you can't tolerate." The dirty/non-repeatable/phantom taxonomy is the database-specific instance of the universal "concurrent readers and writers see inconsistent state" problem. The transferable instinct: **name the specific anomalies your concurrency can produce, then apply exactly enough synchronization to exclude the ones that break correctness** — the same discipline whether it's database isolation, a memory model's reordering, or cache coherence. Precision about *which* anomaly matters is what prevents both over- and under-synchronization.

**MySQL-specific reality.** InnoDB at its **default REPEATABLE READ** prevents dirty reads and non-repeatable reads (consistent MVCC snapshot, M08) *and* — unusually — prevents most **phantoms** via **next-key locks** (locking gaps between index records so new matching rows can't be inserted, 7.13b/M08), going beyond what the SQL standard requires at RR. So InnoDB RR is closer to "snapshot isolation with phantom protection" than the textbook RR. The anomalies map to InnoDB behavior thus: dirty read (prevented ≥ READ COMMITTED), non-repeatable read (prevented ≥ RR via snapshot), phantom (prevented at InnoDB RR via next-key locks; required only at SERIALIZABLE by the standard). For our domain, this means InnoDB's default already gives money reads strong stability — but the *write* anomalies (lost update 7.11, write skew 7.13) still need explicit care, which is the crucial nuance beyond the read-anomaly story.

## 7.10 · The isolation × anomaly matrix ★

**Mental model.** The whole topic crystallizes into **one table**: isolation levels down one axis, anomalies across the other, each cell saying "permitted" or "prevented." This matrix is *the* canonical reference — once you can read it, the relationship between levels and anomalies is no longer abstract: you look up your operation's intolerable anomaly, find the lowest level that prevents it, and you've chosen your isolation. It's the single most useful artifact in the entire isolation topic.

**How it actually works.** The standard matrix (✓ = anomaly *possible/permitted*, ✗ = *prevented*):

| Isolation level    | Dirty read | Non-repeatable read | Phantom read |
|--------------------|:----------:|:-------------------:|:------------:|
| READ UNCOMMITTED   | ✓ possible | ✓ possible          | ✓ possible   |
| READ COMMITTED     | ✗ prevented| ✓ possible          | ✓ possible   |
| REPEATABLE READ    | ✗ prevented| ✗ prevented         | ✓ possible*  |
| SERIALIZABLE       | ✗ prevented| ✗ prevented         | ✗ prevented  |

Reading it: each level prevents everything the levels below it prevent, *plus* one more anomaly. READ UNCOMMITTED prevents nothing; READ COMMITTED adds "no dirty reads"; REPEATABLE READ adds "no non-repeatable reads"; SERIALIZABLE adds "no phantoms" (and, properly, no write skew). The asterisk (*) is the crucial real-world footnote: **the SQL standard says RR permits phantoms, but InnoDB's RR prevents them** (next-key locks, 7.13b) — so MySQL's effective matrix has ✗ where the standard has ✓ for RR/phantom. This is exactly why the textbook and the real database can disagree, and why 7.13b exists.

**Why it exists / what it solves.** The matrix turns a set of definitions into a *lookup tool*. Instead of reasoning from scratch every time, you ask "which anomaly can't my operation tolerate?" and read off the minimum level that prevents it. It also makes the *structure* obvious (the staircase of ✗'s) and surfaces the standard-vs-implementation gap (the asterisk). For interviews and real decisions alike, the matrix is the artifact that makes the whole topic click — which is why it's flagged as a ★ custom diagram (a clean, colorful reference table).

**Tradeoffs & alternatives.** The matrix captures the read-anomaly tradeoffs but is *incomplete* on its own: it covers dirty/non-repeatable/phantom but not the *write* anomalies — **lost update** (7.11) and **write skew** (7.13) — which aren't in the classic three and which even RR doesn't prevent without explicit locking. So the matrix is necessary but not sufficient: you also need to know that read-modify-write and certain multi-row invariants need *more* than the level alone provides. The "alternative" framing (the Berenson et al. "A Critique of ANSI SQL Isolation Levels" / snapshot isolation literature) extends the matrix with these write anomalies — staff-level depth that 7.11/7.13 cover.

**Generics / first-principles.** "Reduce a complex set of guarantees to a lookup table indexed by the failure you must prevent." The matrix is a model of how to make a multi-dimensional tradeoff *usable*: enumerate the failure modes, enumerate the policy options, and tabulate which option prevents which failure — so the decision becomes "find the cheapest option that prevents my failure." The transferable instinct: **when guarantees come in levels, build (or recall) the level×failure matrix** — it converts open-ended reasoning into a bounded lookup, and the gaps in the matrix (here, the write anomalies it omits) tell you where you still need extra care.

**MySQL-specific reality.** MySQL's *effective* matrix differs from the standard at one cell: **InnoDB REPEATABLE READ prevents phantoms** (the ✗ where the standard has ✓), thanks to next-key locks (7.13b/M08) — so MySQL's RR is stronger than the textbook RR. The practical reading for our domain: at InnoDB's default RR, money reads are protected from dirty, non-repeatable, *and* phantom reads — strong. But the matrix's *omission* is the load-bearing caveat: **lost update (7.11) and write skew (7.13) are not on this matrix and are not prevented by RR alone** — a read-modify-write on a balance needs an atomic update or a lock regardless of isolation level. So: use the matrix to reason about *read* consistency (and know InnoDB's RR is strong), but handle the *write* anomalies explicitly (7.11–7.13). The ★ SVG of this matrix (Pass C) will include the InnoDB-vs-standard annotation and a note pointing to the omitted write anomalies.

## 7.11 · The lost update problem ★

**Mental model.** *The* classic concurrency money bug. Two transactions both **read** a value, **modify** it in memory, and **write** it back — concurrently — and one of them **silently overwrites the other's change**, so one update is *lost*. Two simultaneous $50 deposits to a $100 balance: both read $100, both compute $150, both write $150 — final balance $150, but it should be $200. **A deposit vanished.** This is why naive "read the balance, add to it, write it back" code is dangerous under concurrency — and crucially, **REPEATABLE READ does not prevent it by itself** (it's a *write* anomaly, not on the read-anomaly matrix, 7.10). It's the single most important transaction bug to understand for money.

**How it actually works.** The interleaving:
1. T1 reads `balance` = $100.
2. T2 reads `balance` = $100 (same value — both saw the pre-update state).
3. T1 computes $100 + $50 = $150, writes `balance` = $150, commits.
4. T2 computes $100 + $50 = $150 (using the *stale* $100 it read), writes `balance` = $150, commits — **overwriting T1's $150**.
Final: $150. T1's deposit is gone. The root cause is the **read-modify-write gap**: T2 made its decision based on a value that was stale by the time it wrote. MVCC/REPEATABLE READ *doesn't* save you here — each transaction reads a consistent snapshot, but they both read the *same old value*, and the last write wins. **The three fixes:**
1. **Atomic update** — don't read-then-write; do it in *one* statement the engine executes atomically: `UPDATE account SET balance = balance + 50 WHERE account_id = 42`. The engine reads-and-writes under a row lock, so concurrent increments serialize correctly (final $200). *The simplest and best fix when applicable.*
2. **Pessimistic lock** — `SELECT balance … FOR UPDATE` locks the row before reading, so the second transaction *blocks* until the first commits, then reads the *updated* value. (7.12)
3. **Optimistic concurrency** — read a version/the old value, then `UPDATE … WHERE balance = :old_value` (or `WHERE version = :v`); if another transaction changed it, the `WHERE` matches zero rows → you detect the conflict and retry. (7.12)

**Why it exists / what it solves.** Lost update is the canonical example of why *isolation levels alone aren't enough* for correct concurrent writes — and why money systems must handle read-modify-write explicitly. Understanding it (and its fixes) is the difference between balance code that silently loses money under load and code that's correct. It's the concept that forces the realization: "REPEATABLE READ protects my *reads*, but my *read-modify-write* still needs an atomic update or a lock."

**Tradeoffs & alternatives.** The three fixes trade differently: **atomic update** (best — no lost update, minimal locking, but only works when the new value is a pure function of the old, like `+50`; can't use it when you must read, decide in application code, then write); **pessimistic lock** (`FOR UPDATE` — always correct, but *blocks* concurrent writers → contention on a hot row, 7.15/M08); **optimistic** (no blocking, great when conflicts are *rare*, but requires retry logic and wastes work when conflicts are *common*). So: prefer the atomic update; use pessimistic locking when you must read-decide-write and conflicts are likely; use optimistic when conflicts are rare and you want to avoid locking. The *wrong* "alternative" — relying on a high isolation level to magically prevent it — fails (RR doesn't; even SERIALIZABLE prevents it but by *aborting* one transaction, which you must then retry).

**Generics / first-principles.** "Concurrent read-modify-write on shared state loses updates unless the read-decide-write is made atomic or conflict is detected." This is the *universal* concurrency bug — identical to the non-atomic `counter++` in multithreaded code (read, increment, write — lost increments), fixed the same three ways: atomic operation (`fetch_add`), lock (mutex), or compare-and-swap (optimistic). The transferable instinct: **any read-modify-write on shared mutable state is a lost-update hazard; make it atomic, lock it, or use compare-and-set** — and recognize that "consistent reads" (isolation/snapshots) do *not* fix it, because both readers can read the same stale value before either writes.

**MySQL-specific reality.** In InnoDB, the **atomic update** (`UPDATE … SET balance = balance + :amount`) is the idiomatic, correct fix for balance changes — InnoDB takes an exclusive row lock for the update, so concurrent increments serialize and none is lost (this is why the M02/2.17 / M16 balance-maintenance does `SET balance = balance + :delta` in the same transaction as the entry, *not* read-then-write). For read-decide-write logic that can't be a single expression, use **`SELECT … FOR UPDATE`** (pessimistic, M08) or an **optimistic version column** (`UPDATE … WHERE version = :v`, retry on zero rows affected). The hot-account case (many concurrent transfers to one popular account all locking its balance row) is a real contention problem (7.15/M08/M16). For our domain: **never read-then-write a balance in application code under concurrency** — use the atomic `balance = balance + delta`, or lock with `FOR UPDATE`. Lost update is the #1 practical transaction lesson, and a guaranteed interview topic.

## 7.12 · Optimistic vs pessimistic concurrency control

**Mental model.** Two opposite philosophies for handling concurrent writes safely. **Pessimistic:** assume conflicts are *likely*, so **lock first** — acquire a lock on the data *before* you touch it, blocking anyone else until you're done (`SELECT … FOR UPDATE`). **Optimistic:** assume conflicts are *rare*, so **don't lock — detect** — read freely, and at write time *check whether the data changed* since you read it (via a version number or comparing the old value); if it did, you have a conflict, so abort and **retry**. Pessimistic prevents conflicts by serializing access; optimistic allows concurrent access and catches conflicts after the fact. Each wins in a different contention regime.

**How it actually works.**
- **Pessimistic** — `SELECT balance FROM account WHERE id = 42 FOR UPDATE` takes an exclusive lock on Alice's row; any other transaction trying to lock it **waits** until you commit (releasing the lock). So concurrent writers *serialize* on that row — no lost update (7.11), no conflict, because only one proceeds at a time. The cost: **blocking** (waiters stall), reduced concurrency on hot rows, and deadlock risk (M08) if transactions lock multiple rows in different orders.
- **Optimistic** — add a `version` column (or use the old value). Read the row and its version (`v=7`). Do your work. Write with a guard: `UPDATE account SET balance = :new, version = 8 WHERE id = 42 AND version = 7`. If no one else changed it, the `WHERE` matches (1 row updated) — success. If someone else committed first (bumping version to 8), the `WHERE` matches **0 rows** — you detect the conflict, and **retry** the whole read-modify-write with the new value. No locks held during your work; conflicts cost a retry.
Both prevent lost update (7.11); they differ in *when* and *how* they handle the conflict (prevent-by-blocking vs detect-and-retry).

**Why it exists / what it solves.** Different workloads have different conflict rates, and the optimal strategy depends on that rate. **High contention** (many transactions fighting over the same hot rows): pessimistic locking avoids the wasted work of repeated optimistic retries (which would thrash). **Low contention** (conflicts rare): optimistic avoids the overhead and blocking of locking — most transactions sail through with a cheap version check, no waiting. Having both lets you match the strategy to the contention, rather than paying locking costs when conflicts are rare or retry-thrash costs when they're common.

**Tradeoffs & alternatives.** **Pessimistic:** guaranteed progress for the lock holder, no wasted work, but *blocks* others (lower concurrency, contention on hot rows, deadlock risk, M08) and holds locks for the transaction's duration (keep it short, 7.15). **Optimistic:** no blocking, high concurrency when conflicts are rare, but *wasted work + retries* when conflicts are common (and under heavy contention can livelock/thrash with constant retries). The decision rule: **pessimistic for high-contention hot rows; optimistic for low-contention data**. They also compose with the *atomic update* (7.11) — which is itself a form of pessimistic (the engine locks the row for the update) but cheapest when the new value is a pure function of the old. Many systems use atomic updates where possible, pessimistic locks for hot read-decide-write, and optimistic for low-contention entities.

**Generics / first-principles.** "Either prevent conflicts by locking, or allow them and detect-and-retry — choose by how often conflicts actually happen." This optimistic/pessimistic dichotomy is universal across concurrency control (lock-based vs lock-free/CAS data structures, pessimistic vs optimistic locking in any system, even mutexes vs transactional memory). The deep principle: **synchronization cost should match conflict probability** — lock when contention is high (avoid retry waste), use compare-and-set when contention is low (avoid blocking waste). The transferable instinct: measure (or estimate) your conflict rate, then pick the strategy whose *failure mode* (blocking vs retrying) is cheaper at that rate.

**MySQL-specific reality.** InnoDB pessimistic locking: **`SELECT … FOR UPDATE`** (exclusive) and **`SELECT … FOR SHARE`** (shared) — plus `SKIP LOCKED` / `NOWAIT` modifiers (M08) useful for queue-style processing (skip locked rows instead of waiting). Optimistic: a **`version` (or `updated_at`) column** + `UPDATE … WHERE version = :v`, checking *rows affected* = 0 to detect conflict and retry (often done at the application/ORM level — many ORMs have built-in optimistic locking). The atomic update (7.11, `balance = balance + :delta`) is the cheapest engine-pessimistic option. For our domain: a **hot account** (high contention) → pessimistic (`FOR UPDATE`) or atomic update + retry; a **rarely-edited entity** (e.g., account settings, low contention) → optimistic version check. The hot-account contention problem and queue processing with `SKIP LOCKED` are developed in M08/M16. Choosing optimistic vs pessimistic is a core concurrency-design and interview topic.

## 7.13 · Write skew & SERIALIZABLE's role

**Mental model.** A *subtler* anomaly than lost update, and the one that justifies SERIALIZABLE. **Write skew:** two transactions each **read** an overlapping set of data, each **checks an invariant** that currently holds, and each **writes to *different* rows** based on what it read — and individually each write is fine, but *together* they violate the invariant, because each acted on a snapshot that didn't include the other's write. Classic example: an account with $100 and a rule "total of two linked sub-balances must stay ≥ 0." Two withdrawals run concurrently; each reads the total ($100), each checks "after my $60 withdrawal, $40 ≥ 0 ✓," each writes to *its own* sub-balance — and together they withdraw $120 from $100, overdrawing it. No lost update (they wrote *different* rows!), no dirty/non-repeatable read — yet the invariant is broken. Only **SERIALIZABLE** (or explicit locking of the read set) prevents it.

**How it actually works.** The interleaving (the on-call/overdraft pattern):
1. Invariant: `sub_a + sub_b ≥ 0`. Currently `sub_a = 100, sub_b = 0` (total 100).
2. T1 reads both (total 100), wants to withdraw 60 from `sub_a`; checks "100 − 60 = 40 ≥ 0 ✓"; writes `sub_a = 40`.
3. T2, *concurrently*, reads both (also total 100 — it doesn't see T1's uncommitted/snapshot-isolated write), wants to withdraw 60 from `sub_b`'s side; checks "100 − 60 = 40 ≥ 0 ✓"; writes accordingly.
4. Both commit. Each *individually* preserved the invariant *against the state it read*, but they wrote to **different rows** based on a **shared stale read**, so the combined effect overdraws: total is now −20.
The defining feature: **disjoint writes, overlapping reads, a multi-row invariant** — so it's *not* a lost update (no overwrite) and *not* a read anomaly (each read was consistent). MVCC snapshot isolation (InnoDB RR) *permits* write skew, because each transaction's snapshot legitimately doesn't include the other's concurrent write. **SERIALIZABLE** prevents it (the executions can't be serialized — one must abort). The practical fix below SERIALIZABLE: **explicitly lock the read set** (`SELECT … FOR UPDATE` on *all* rows the invariant depends on, so the transactions can't proceed concurrently) or **materialize the conflict** (lock/insert a row representing the invariant).

**Why it exists / what it solves.** Write skew is the anomaly that shows **snapshot isolation (and InnoDB's RR) is *not* serializable** — it's strong, but a class of multi-row-invariant bugs slips through. Understanding it is what separates "I use REPEATABLE READ so I'm safe" (false for write skew) from correct reasoning. It matters wherever an invariant spans *multiple rows* and concurrent transactions can each preserve it locally while violating it jointly: overdraft across linked accounts, double-booking a resource, two on-call engineers both going off-call ("at least one must remain"). For money, write-skew-prone invariants are real (linked balances, shared limits) and require explicit handling.

**Tradeoffs & alternatives.** Preventing write skew has options with different costs: **SERIALIZABLE** (correct, but the most blocking/aborting level — and InnoDB implements it by locking reads, hurting concurrency); **explicit locking of the read set** (`FOR UPDATE` on all invariant-relevant rows at a lower isolation level — targeted, keeps the rest of the transaction concurrent, the common practical fix); or **materializing the conflict** (force the two transactions to contend on a *common* row — e.g., a lock row for the shared limit — turning write skew into a normal lock conflict). The tradeoff: SERIALIZABLE is blanket-correct but slow; targeted locking is faster but requires *recognizing* the write-skew hazard and locking the right rows (easy to miss). The dangerous non-alternative is assuming RR/snapshot isolation prevents it — it doesn't.

**Generics / first-principles.** "Concurrent transactions can each preserve a multi-item invariant locally yet violate it jointly when they read overlapping data and write disjoint items — only serializability (or locking the read set) prevents it." Write skew is the canonical demonstration that **snapshot isolation ≠ serializability**, a deep and frequently-misunderstood distinction in concurrency theory. The transferable instinct: **when an invariant spans multiple items and concurrent actors decide based on a shared read, check for write skew** — each actor's local correctness doesn't guarantee global correctness, and you need either true serialization or explicit locking of the entire decision's read set. This recurs in distributed systems (constraints that span partitions) and any optimistic multi-item check.

**MySQL-specific reality.** InnoDB's default **REPEATABLE READ (snapshot isolation via MVCC) permits write skew** — a critical caveat to "InnoDB RR is strong." InnoDB **SERIALIZABLE** prevents it (it converts plain `SELECT`s into locking reads — `LOCK IN SHARE MODE` semantics — so the overlapping reads conflict). The common practical fix at RR is **`SELECT … FOR UPDATE` on all rows the invariant depends on** before deciding (so concurrent transactions block rather than proceed on stale snapshots), or designing the invariant onto a single row that becomes the contention point (a lost-update-style atomic update instead of a write-skew-prone multi-row check). For our domain: a "total across linked sub-accounts ≥ 0" rule, or a shared spending limit, is write-skew-prone — handle it by locking the full read set (`FOR UPDATE`) or restructuring so the limit lives on one atomically-updated row. Recognizing write skew (and that RR doesn't stop it) is staff-level concurrency knowledge and a strong interview signal.

## 7.13b · MySQL/InnoDB's actual defaults & behavior

**Mental model.** The textbook isolation story (the SQL standard, the matrix in 7.10) and **what InnoDB actually does** differ in important ways — and using MySQL correctly means knowing the *real* behavior, not just the standard. The three load-bearing realities: **(1) InnoDB defaults to REPEATABLE READ** (not READ COMMITTED, the default of most other databases); **(2) InnoDB's RR prevents phantoms** (via next-key locks) — stronger than the standard's RR; **(3) InnoDB delivers isolation via MVCC (snapshot reads) + locks (writes and locking reads)** — so reads usually don't block and writers usually don't block readers, but the snapshot semantics and lock behavior have specific rules. Knowing these prevents both over-engineering (raising the level unnecessarily) and silent bugs (assuming standard behavior).

**How it actually works (the InnoDB realities; lock mechanics in M08):**
- **Default = REPEATABLE READ.** A transaction's consistent snapshot is established at its **first read** (not at `START TRANSACTION`), and all plain `SELECT`s thereafter see that same snapshot (so reads are repeatable; M08). This is stronger and more surprising than READ COMMITTED (where each statement sees the latest committed data).
- **Phantom prevention at RR via next-key locks.** For *locking* reads and writes, InnoDB locks not just matching index records but the **gaps** between them (next-key = record lock + gap lock, M08), so another transaction *can't insert* a new matching row — preventing phantoms that the standard says RR allows. (Plain non-locking `SELECT`s use the snapshot, which also doesn't see new rows.)
- **MVCC + locks.** Plain `SELECT` = non-locking snapshot read (doesn't block, doesn't take row locks, M08). `SELECT … FOR UPDATE/FOR SHARE`, and all writes = locking reads/writes (take row/next-key locks). So InnoDB combines *optimistic-feeling reads* (MVCC, no blocking) with *pessimistic writes* (locks).
- **The caveats that survive:** RR's snapshot + next-key locks prevent dirty/non-repeatable/phantom reads — but **lost update (7.11) and write skew (7.13) still require explicit atomic updates or locking**, because those are *write* anomalies the isolation level doesn't address by itself.

**Why it exists / what it solves.** This concept exists to *correct the textbook* with reality — because reasoning about MySQL from the bare SQL standard leads to wrong conclusions (expecting phantoms at RR that don't occur, expecting READ COMMITTED defaults, or — worse — assuming RR prevents lost update/write skew). Knowing InnoDB's actual defaults and behavior lets you choose isolation correctly, predict locking/blocking, and know exactly which anomalies you must still handle yourself. It's the bridge from isolation *theory* (7.8–7.13) to InnoDB *practice* (M08).

**Tradeoffs & alternatives.** InnoDB's RR default is a reasonable, fairly-strong default — but it has costs vs READ COMMITTED: RR holds gap/next-key locks (more locking, more deadlock potential under some write patterns, M08) and its snapshot can become "old" in a long transaction (history-list bloat, 7.15/M09). Some high-throughput MySQL shops deliberately switch to **READ COMMITTED** (giving up repeatable reads and phantom prevention) for *less locking and fewer deadlocks* — a real tradeoff (weaker isolation, more concurrency). The alternative of *not knowing* InnoDB's behavior is the bug: assuming standard semantics. The decision (7.14) accounts for these InnoDB-specific costs.

**Generics / first-principles.** "Implementations deviate from the specification — know your specific system's actual behavior, not just the standard." This is the universal "the spec is not the implementation" lesson (compilers, network protocols, floating-point, SQL dialects all deviate). The transferable instinct: **when correctness depends on a guarantee, verify what your *specific* implementation does, not what the standard says** — defaults, strengthenings, and quirks (InnoDB's RR-with-phantom-protection, MVCC+locks) materially change your reasoning. Building on the standard alone, without the implementation's reality, produces confident-but-wrong conclusions.

**MySQL-specific reality.** This concept *is* the MySQL reality, summarized: **default REPEATABLE READ; snapshot established at first read; next-key locks prevent phantoms (stronger than standard RR); MVCC for non-blocking snapshot reads + locks for writes/locking-reads; lost update and write skew still need explicit handling.** Check/set with `SELECT @@transaction_isolation;` and `SET TRANSACTION ISOLATION LEVEL …`. The practical implications for our domain: InnoDB's default RR gives money *reads* strong consistency without extra work; money *writes* (balance updates) still need atomic updates or `FOR UPDATE` (7.11); write-skew-prone multi-row invariants need explicit read-set locking (7.13); and very long transactions degrade (snapshot age, lock holding, 7.15/M09). The *mechanism* — exactly how MVCC version chains and next-key locks work — is the entire subject of M08; here the takeaway is *what* InnoDB actually guarantees so you choose isolation (7.14) and handle write anomalies correctly.
